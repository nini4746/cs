# DNS

## 한 줄 요약

도메인 이름(example.com)을 IP 주소로 바꾸는 분산 계층형 데이터베이스. 루트→TLD→권한 서버로 위임되며, 캐싱(TTL)으로 부하를 줄인다. 인터넷의 전화번호부.

## 왜 필요한가

- 사람은 이름, 컴퓨터는 IP - 그 사이 번역
- 왜 분산·계층형인가 (중앙 서버 하나면 불가능)
- TTL·캐싱이 왜 중요한가

## 문제: 이름 → IP

`example.com`에 접속하려면 IP가 필요 ([[ip-addressing]]). DNS가 이름을 IP로 해석:

실측 (`dig example.com`, 이 네트워크의 리졸버 응답):
```
example.com. → 104.20.23.154 (A 레코드)
TTL: 109초
```

(반환 IP는 리졸버·CDN·지역에 따라 다름. 실제 example.com 공식 IP는 별개 - 여기선 이 네트워크가 준 값. 요점은 이름→IP 해석이 됐다는 것.)

브라우저가 URL을 IP로 바꾸는 첫 단계 → web/[[what-happens-url]].

## 분산 계층 구조

중앙 서버 하나로는 불가능 (규모, 단일 장애점). DNS는 **계층적 위임**:

```
        루트 (.)                    ← 13개 루트 서버 클러스터
       /    |    \
     .com  .org  .kr               ← TLD (최상위 도메인) 서버
      |
  example.com                       ← 권한(authoritative) 서버
      |
  www.example.com                   ← 최종 레코드
```

- 각 레벨이 다음 레벨을 **위임** (알 필요 없이 "저기 물어봐")
- 부하 분산, 관리 분산 (각 도메인 소유자가 자기 레코드 관리)

## 조회 과정 (resolution)

`www.example.com` 조회:

```
1. 클라이언트 → 로컬 DNS 리졸버 (ISP 또는 8.8.8.8)
2. 리졸버 → 루트 서버: ".com 어디?"
3. 루트 → ".com TLD 서버 주소"
4. 리졸버 → TLD: "example.com 어디?"
5. TLD → "example.com 권한 서버 주소"
6. 리졸버 → 권한 서버: "www.example.com IP?"
7. 권한 서버 → "104.20.23.154"
8. 리졸버 → 클라이언트 (그리고 캐시)
```

- **재귀적 질의**: 클라이언트가 리졸버에 "다 알아서 답 줘"
- **반복적 질의**: 리졸버가 각 서버에 단계별로 물음 (위 2~7)

## 레코드 타입

| 타입 | 의미 |
|---|---|
| **A** | 이름 → IPv4 (실측의 104.20.23.154) |
| **AAAA** | 이름 → IPv6 |
| **NS** | 이 도메인의 네임서버 (실측: cloudflare.com) |
| **CNAME** | 별칭 (다른 이름으로) |
| **MX** | 메일 서버 → [[email-protocols]] |
| **TXT** | 임의 텍스트 (SPF/DKIM 검증 등 → [[email-protocols]]) |

## 캐싱과 TTL

DNS는 **캐싱으로 부하 관리** (매번 루트까지 가면 폭발):

- 각 레코드에 **TTL**(Time To Live, 실측 109초): 이 시간 동안 캐시 유효
- 리졸버·OS·브라우저가 여러 층으로 캐시 → os/[[page-cache]]의 계층 캐싱과 같은 발상
- **트레이드오프**: TTL 길면 부하↓ 하지만 변경 반영 느림 (IP 바꿔도 옛 캐시가 남음). 서버 이전 시 TTL 미리 낮춤

## DNS와 성능·보안

- **성능**: DNS 조회가 페이지 로드 첫 지연 (수십 ms). CDN·prefetch로 완화 → web/[[web-performance]]
- **로드 밸런싱**: 한 이름에 여러 A 레코드 → 분산. CDN이 지리적으로 가까운 IP 반환 → [[cdn]]
- **보안**: DNS는 평문 → 도청·조작 가능 (DNS 스푸핑). **DNSSEC**(서명), **DoH/DoT**(암호화된 DNS) → security/[[tls-deep-dive]]

## 전송 방식

- 주로 **UDP** (빠름, 작은 요청/응답) → [[udp]]
- 응답이 크거나(512B 초과) 영역 전송은 TCP
- DoH는 HTTPS 위 ([[http]]), DoT는 TLS 위

## 셀프 체크

> [!question]- DNS가 중앙 서버 하나가 아니라 분산 계층 구조인 이유는?
> 규모와 단일 장애점 때문이다. 전 세계 도메인 조회를 서버 하나가 감당할 수 없고, 그 서버가 죽으면 인터넷 전체가 멈춘다. 루트→TLD→권한 서버로 계층 위임하면 부하와 관리가 분산되고 각 도메인 소유자가 자기 레코드를 관리한다.

> [!question]- 재귀적 질의와 반복적 질의의 차이는?
> 재귀적 질의는 클라이언트가 리졸버에게 "다 알아서 최종 답을 달라"고 맡기는 것이다. 반복적 질의는 리졸버가 루트→TLD→권한 서버를 단계별로 직접 물어 각 서버가 다음 서버 주소를 알려주는 방식이다. 클라이언트-리졸버 구간은 재귀적, 리졸버-서버 구간은 반복적이다.

> [!question]- TTL의 트레이드오프는 무엇인가?
> TTL이 길면 캐시 유효 기간이 길어 조회 부하가 줄지만, IP를 바꿔도 옛 캐시가 남아 변경 반영이 느리다. TTL이 짧으면 반영은 빠르지만 조회가 잦아 부하가 는다. 그래서 서버 이전 전에는 TTL을 미리 낮춰둔다.

> [!question]- DNS가 주로 UDP를 쓰는 이유와 TCP를 쓰는 경우는?
> 요청/응답이 작고 빨라야 하므로 연결 설정 오버헤드가 없는 UDP가 적합하다. 다만 응답이 512바이트를 초과하거나 영역 전송(zone transfer)처럼 신뢰성이 필요한 경우에는 TCP를 쓴다. DoH는 HTTPS 위, DoT는 TLS 위에서 동작한다.

## 연습문제

> [!example]- 문제: 캐시가 완전히 빈 상태에서 www.example.com을 조회할 때 리졸버가 거치는 반복적 질의 단계를 순서대로 추적하라.
> **풀이**
> 1. 클라이언트 → 로컬 리졸버에 재귀적 질의.
> 2. 리졸버 → 루트 서버: ".com 네임서버는?" → 루트가 .com TLD 서버 주소 반환.
> 3. 리졸버 → .com TLD 서버: "example.com 네임서버는?" → TLD가 권한 서버 주소 반환.
> 4. 리졸버 → example.com 권한 서버: "www.example.com의 A 레코드는?" → 권한 서버가 IP 반환.
> 5. 리졸버가 결과를 캐시하고 클라이언트에 응답.
> 각 단계에서 상위 서버는 답 대신 "다음에 물을 곳"을 알려주는 위임을 한다.

> [!example]- 문제: 한 도메인이 www에 대해 A 레코드 3개(203.0.113.1, 203.0.113.2, 203.0.113.3)를 두고, 메일은 mail.example.com으로 받으며, google 검증용 TXT를 두려 한다. 필요한 레코드 구성을 설명하라.
> **풀이**
> - A 레코드 3개: www → 세 IP. 리졸버가 순서를 바꿔 반환(round-robin)해 요청을 분산한다. 간단한 로드 밸런싱이다.
> - MX 레코드: example.com → mail.example.com을 우선순위와 함께 지정. 발신 서버가 이 MX를 조회해 메일을 라우팅한다.
> - mail.example.com에는 다시 A(또는 AAAA) 레코드가 있어야 실제 IP로 해석된다.
> - TXT 레코드: 검증 문자열을 담는다(SPF/도메인 소유 검증 등).
