# 혼잡 제어 (Congestion Control)

## 한 줄 요약

네트워크가 과부하되지 않게 송신 속도를 조절한다. Slow start로 탐색하고, AIMD로 공정하게 나누며, 손실이나 지연을 혼잡 신호로 읽는다. 인터넷이 붕괴하지 않는 이유.

## 왜 필요한가

- 왜 인터넷이 과부하로 무너지지 않나
- 여러 연결이 대역폭을 어떻게 공정하게 나누나
- BBR 같은 최신 알고리즘이 뭘 바꾸나

## 문제: 혼잡 붕괴

모두가 최대 속도로 보내면 → 라우터 큐 폭발 → 손실 → 재전송 → 더 혼잡 → **혼잡 붕괴(congestion collapse)**. 1986년 실제로 인터넷이 이걸 겪음 (처리량이 1/1000로).

해법: 각 송신자가 **네트워크 상태를 추측해 속도 조절**. 하지만 네트워크 내부를 못 봄 → **손실/지연을 신호로** 추론.

## 혼잡 윈도우 (cwnd)

흐름 제어의 rwnd([[tcp-reliability]])와 별개로, **네트워크 한계**를 나타내는 cwnd:

```
실제 송신 윈도우 = min(rwnd, cwnd)
  rwnd = 수신자 버퍼 (흐름 제어)
  cwnd = 네트워크 여유 (혼잡 제어)   ← 이 노트
```

cwnd를 어떻게 조절하나가 혼잡 제어의 핵심.

## Slow Start: 지수적 탐색

연결 시작 시 네트워크 여유를 모름 → **작게 시작해 빠르게 늘림**:

- cwnd = 1(작은 값)에서 시작
- ACK 받을 때마다 cwnd **+1 MSS** → RTT마다 2배 (지수 증가)
- "slow"라는 이름과 달리 빠르게 증가 (시작이 작을 뿐)
- 임계값(ssthresh) 도달하면 혼잡 회피로 전환

## AIMD: 공정한 분배

혼잡 회피 단계의 핵심 - **Additive Increase, Multiplicative Decrease**:

- **가산 증가(AI)**: 혼잡 없으면 RTT마다 cwnd를 **+1** (선형, 조심스레 탐색)
- **곱셈 감소(MD)**: 손실 감지하면 cwnd를 **절반** (급격히 양보)

```
cwnd: 천천히 증가(+1) → 손실 → 반토막 → 다시 증가 → 손실 → 반토막 ...
      톱니(sawtooth) 모양
```

### 왜 AIMD가 공정한가

여러 연결이 AIMD를 쓰면 **대역폭을 공평하게 수렴**:
- 많이 쓰는 연결이 손실 시 더 많이 줄임(곱셈) → 공평 쪽으로
- 수학적으로 fair share에 수렴 (chiu-jain 논증)
- 인터넷이 협력적으로 대역폭을 나누는 근거

## 손실 신호 vs 지연 신호

혼잡을 어떻게 감지하나:

### 손실 기반 (전통)

- **패킷 손실 = 혼잡** (큐가 넘쳐 버려짐)
- Reno, CUBIC: 손실 시 cwnd 감소
- **CUBIC** (리눅스 기본): 3차 함수로 윈도우 조절, 고대역폭에 최적화
- 문제: 큐가 꽉 차야 손실 → 큐가 늘 차있음(bufferbloat) → 지연 증가

### 지연/대역폭 기반 (현대)

- **BBR** (Google): 손실이 아니라 **대역폭과 RTT를 측정**해 최적점 추정
- 큐를 채우지 않고 병목 대역폭에 맞춤 → 낮은 지연 + 높은 처리량
- 유튜브·구글 서비스가 사용. 손실 기반과 다른 철학 (큐 안 채움)

## 혼잡 vs 흐름 제어 (재확인)

| | 흐름 제어 | 혼잡 제어 |
|---|---|---|
| 보호 | 수신자 → [[tcp-reliability]] | 네트워크 |
| 윈도우 | rwnd | cwnd |
| 신호 | 수신자 광고 | 손실/지연 |

## 왜 이게 중요한가

- **인터넷의 안정성**: 수십억 연결이 협력적으로 대역폭 공유 → 붕괴 방지
- **성능**: 알고리즘 선택이 처리량·지연 좌우 (CUBIC vs BBR)
- **공정성**: TCP 연결끼리 공평, 하지만 UDP([[udp]])는 혼잡 제어 안 함 → 남용 시 불공평
