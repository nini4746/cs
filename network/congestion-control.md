# 혼잡 제어 (Congestion Control)

## 한 줄 요약

네트워크가 과부하되지 않게 송신 속도를 조절한다. Slow start로 탐색하고, AIMD로 공정하게 나누며, 손실이나 지연을 혼잡 신호로 읽는다. 인터넷이 붕괴하지 않는 이유.

## 왜 필요한가

- 왜 인터넷이 과부하로 무너지지 않나
- 여러 연결이 대역폭을 어떻게 공정하게 나누나
- BBR 같은 최신 알고리즘이 뭘 바꾸나

## 문제: 혼잡 붕괴

모두가 최대 속도로 보내면 → 라우터 큐 폭발 → 손실 → 재전송 → 더 혼잡 → **혼잡 붕괴(congestion collapse)**. 1986년 실제로 인터넷이 이걸 겪음 (처리량이 1/1000로).

해법: 각 송신자가 **네트워크 상태를 추측해 속도 조절**. 하지만 네트워크 내부를 못 봄 → **손실/지연을 신호로** 추론.

## 혼잡 윈도우 (cwnd)

흐름 제어의 rwnd([[tcp-reliability]])와 별개로, **네트워크 한계**를 나타내는 cwnd:

```
실제 송신 윈도우 = min(rwnd, cwnd)
  rwnd = 수신자 버퍼 (흐름 제어)
  cwnd = 네트워크 여유 (혼잡 제어)   ← 이 노트
```

cwnd를 어떻게 조절하나가 혼잡 제어의 핵심.

## Slow Start: 지수적 탐색

연결 시작 시 네트워크 여유를 모름 → **작게 시작해 빠르게 늘림**:

- cwnd = 1(작은 값)에서 시작
- ACK 받을 때마다 cwnd **+1 MSS** → RTT마다 2배 (지수 증가)
- "slow"라는 이름과 달리 빠르게 증가 (시작이 작을 뿐)
- 임계값(ssthresh) 도달하면 혼잡 회피로 전환

## AIMD: 공정한 분배

혼잡 회피 단계의 핵심 - **Additive Increase, Multiplicative Decrease**:

- **가산 증가(AI)**: 혼잡 없으면 RTT마다 cwnd를 **+1** (선형, 조심스레 탐색)
- **곱셈 감소(MD)**: 손실 감지하면 cwnd를 **절반** (급격히 양보)

```
cwnd: 천천히 증가(+1) → 손실 → 반토막 → 다시 증가 → 손실 → 반토막 ...
      톱니(sawtooth) 모양
```

### 왜 AIMD가 공정한가

여러 연결이 AIMD를 쓰면 **대역폭을 공평하게 수렴**:
- 많이 쓰는 연결이 손실 시 더 많이 줄임(곱셈) → 공평 쪽으로
- 수학적으로 fair share에 수렴 (chiu-jain 논증)
- 인터넷이 협력적으로 대역폭을 나누는 근거

## 손실 신호 vs 지연 신호

혼잡을 어떻게 감지하나:

### 손실 기반 (전통)

- **패킷 손실 = 혼잡** (큐가 넘쳐 버려짐)
- Reno, CUBIC: 손실 시 cwnd 감소
- **CUBIC** (리눅스 기본): 3차 함수로 윈도우 조절, 고대역폭에 최적화
- 문제: 큐가 꽉 차야 손실 → 큐가 늘 차있음(bufferbloat) → 지연 증가

### 지연/대역폭 기반 (현대)

- **BBR** (Google): 손실이 아니라 **대역폭과 RTT를 측정**해 최적점 추정
- 큐를 채우지 않고 병목 대역폭에 맞춤 → 낮은 지연 + 높은 처리량
- 유튜브·구글 서비스가 사용. 손실 기반과 다른 철학 (큐 안 채움)

## 혼잡 vs 흐름 제어 (재확인)

| | 흐름 제어 | 혼잡 제어 |
|---|---|---|
| 보호 | 수신자 → [[tcp-reliability]] | 네트워크 |
| 윈도우 | rwnd | cwnd |
| 신호 | 수신자 광고 | 손실/지연 |

## 왜 이게 중요한가

- **인터넷의 안정성**: 수십억 연결이 협력적으로 대역폭 공유 → 붕괴 방지
- **성능**: 알고리즘 선택이 처리량·지연 좌우 (CUBIC vs BBR)
- **공정성**: TCP 연결끼리 공평, 하지만 UDP([[udp]])는 혼잡 제어 안 함 → 남용 시 불공평

## 셀프 체크

> [!question]- 실제 송신 윈도우는 무엇으로 결정되는가?
> min(rwnd, cwnd)이다. rwnd는 수신자 버퍼 여유로 흐름 제어를 담당하고, cwnd는 네트워크 여유를 나타내 혼잡 제어를 담당한다. 둘 중 더 작은 값이 실제로 보낼 수 있는 양을 제한한다.

> [!question]- Slow start가 "느리지 않다"는 말은 무슨 뜻인가?
> cwnd를 1(작은 값)에서 시작하지만 ACK마다 +1 MSS 늘려 RTT마다 2배로 지수 증가한다. 증가 속도 자체는 빠르며, 이름의 "slow"는 시작점이 작다는 의미다. ssthresh에 도달하면 혼잡 회피(AIMD)로 전환한다.

> [!question]- AIMD가 대역폭을 공정하게 수렴시키는 이유는?
> 곱셈 감소 때문이다. 손실이 나면 각 연결이 cwnd를 절반으로 줄이는데, 많이 쓰던 연결이 절대량으로 더 많이 줄어든다. 반면 가산 증가는 모두 동일하게 +1씩 늘린다. 반복하면 fair share로 수렴한다(chiu-jain 논증).

> [!question]- 손실 기반(CUBIC)과 지연 기반(BBR)의 철학적 차이는?
> CUBIC은 패킷 손실을 혼잡 신호로 보고 큐가 꽉 차 손실이 날 때까지 밀어붙인다. 그 결과 큐가 늘 차있어 지연이 커진다(bufferbloat). BBR은 병목 대역폭과 RTT를 측정해 큐를 채우지 않고 최적점에 맞춘다. 낮은 지연과 높은 처리량을 동시에 노린다.

> [!question]- UDP가 혼잡 제어 공정성에 위협이 되는 이유는?
> UDP는 혼잡 제어를 하지 않으므로 손실이 나도 속도를 줄이지 않는다. TCP 연결들은 AIMD로 양보하는데 UDP는 계속 밀어붙여 대역폭을 독차지할 수 있다. 협력적 공유를 전제로 한 인터넷에서 불공정을 유발한다.

## 연습문제

> [!example]- 문제: cwnd = 32 MSS에서 손실이 감지되고 ssthresh가 16 MSS로 설정됐다. 이후 혼잡 회피 단계에서 cwnd가 다시 24 MSS가 되기까지 몇 RTT가 걸리는지 계산하라. (Reno 방식, MD로 절반)
> **풀이**
> 손실 시 곱셈 감소로 cwnd = 32 / 2 = 16 MSS가 된다(ssthresh와 일치).
> 이제 혼잡 회피 단계이므로 RTT마다 +1 MSS 가산 증가한다.
> 16 → 24까지 필요한 증가량 = 24 - 16 = 8 MSS.
> RTT마다 +1이므로 8 RTT가 걸린다.

> [!example]- 문제: 대역폭지연곱(BDP)이 10MB인 링크에서 두 TCP 연결이 AIMD로 경쟁한다. 한 연결이 8MB, 다른 하나가 2MB를 쓰던 중 동시에 손실을 겪었다. 손실 직후 각 연결의 윈도우와 공정성 방향을 논하라.
> **풀이**
> 곱셈 감소로 각각 절반이 된다: 8 → 4MB, 2 → 1MB.
> 손실 전 격차는 8 - 2 = 6MB, 손실 후 격차는 4 - 1 = 3MB로 절대 격차가 줄었다.
> 이후 가산 증가는 둘 다 동일 속도(+1 MSS/RTT)로 늘어나므로 격차는 유지되지만, 다음 손실에서 다시 큰 쪽이 더 많이 줄어 fair share(각 5MB)로 수렴한다. AI는 격차를 유지하고 MD가 격차를 좁히는 것이 핵심이다.
