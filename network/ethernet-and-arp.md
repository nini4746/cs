# 이더넷과 ARP (Ethernet and ARP)

## 한 줄 요약

링크 계층은 같은 로컬 네트워크 안의 인접 노드 간 전달을 담당한다. MAC 주소로 식별하고, ARP가 IP 주소를 MAC 주소로 변환한다. 스위치는 MAC 기반, 라우터는 IP 기반으로 동작한다.

## 왜 필요한가

- IP 주소가 있는데 왜 MAC 주소가 또 필요한가
- 로컬 네트워크에서 실제 전달이 어떻게 되나
- 스위치와 라우터의 차이

## 링크 계층의 역할

네트워크 계층(IP, [[ip-addressing]])이 "전체 경로"라면, 링크 계층은 **한 홉** - 같은 로컬 네트워크의 인접 노드 간 전달:

```
IP: 서울 → ... → 부산 (전체 경로, [[routing]])
링크: 각 홉마다 "이 라우터 → 다음 라우터" (한 구간)
```

각 홉이 링크 계층으로 실제 프레임을 전달. 이더넷이 가장 흔한 링크 기술.

## MAC 주소

**MAC(Media Access Control) 주소**: 네트워크 인터페이스의 하드웨어 식별자:

```
92:4e:2a:22:08:55   (48비트, 16진수 6쌍)   ← 실측한 내 인터페이스
```

- **하드웨어에 고정**(제조 시), 앞 3바이트 = 제조사
- IP는 **논리적**(바뀜, 위치 기반), MAC은 **물리적**(고정, 기기 기반)
- 로컬 네트워크 안에서 프레임은 MAC으로 전달

## IP vs MAC: 왜 둘 다

| | IP 주소 | MAC 주소 |
|---|---|---|
| 계층 | 네트워크 | 링크 |
| 범위 | 전 세계 (라우팅) | 로컬 네트워크 |
| 성격 | 논리적 (바뀜) | 물리적 (고정) |
| 용도 | 전체 경로 | 한 홉 전달 |

비유: IP = 최종 목적지 주소(계속 유효), MAC = 다음 갈아탈 곳(홉마다 바뀜). 라우터를 지날 때마다 목적 IP는 그대로, 목적 MAC은 다음 홉으로 바뀜.

## ARP: IP → MAC 변환

같은 로컬 네트워크에서 IP는 아는데 MAC을 모를 때 → **ARP(Address Resolution Protocol)**:

```
1. "10.18.0.1의 MAC 누구?" 를 로컬 네트워크에 브로드캐스트
2. 해당 IP를 가진 기기가 "나야, 내 MAC은 ..." 응답
3. IP→MAC 매핑을 ARP 캐시에 저장
```

실측 (`arp -a`):
```
10.18.0.1     → 0:0:5e:0:1:1
10.18.0.254   → b0:26:28:f5:3d:ac
10.18.108.210 → 9a:9b:6:1b:99:b7
```

IP→MAC 매핑 테이블 = ARP 캐시. os/[[page-cache]]처럼 캐싱으로 매번 브로드캐스트 회피. TTL로 만료.

### ARP 스푸핑

ARP도 검증 없음(신뢰 기반) → **ARP 스푸핑**: 공격자가 "게이트웨이 IP = 내 MAC"이라 거짓 응답 → 트래픽 가로챔(중간자 공격) → security/[[authn-authz-failures]].

## 스위치 vs 라우터

로컬 네트워크의 두 장비:

- **스위치(switch)**: **MAC 기반**, 링크 계층. MAC 주소 테이블로 프레임을 해당 포트로만 전달 (학습으로 테이블 구성). 같은 네트워크 내
- **라우터(router)**: **IP 기반**, 네트워크 계층. 다른 네트워크 간 전달, 경로 결정 → [[routing]]

```
같은 서브넷 안 → 스위치 (MAC)
다른 서브넷으로 → 라우터 (IP)
```

전달 과정: 목적이 같은 서브넷이면 ARP로 MAC 찾아 직접, 다른 서브넷이면 게이트웨이(라우터) MAC으로 보냄 → 라우터가 다음 홉으로.

## 이더넷 프레임

```
[목적 MAC][출발 MAC][타입][페이로드(IP 패킷)][CRC]
```

- **CRC**: 오류 검출 (손상 프레임 폐기)
- 페이로드에 IP 패킷 → 캡슐화 ([[internet-overview]])
- 타입 필드로 상위 프로토콜 (IPv4/IPv6/ARP)

## 충돌과 CSMA/CD (역사)

옛 이더넷(공유 매체)은 충돌 처리 필요 (CSMA/CD - 들어보고 보내고 충돌 감지). 현대 스위치 이더넷은 **전이중**(각 포트 전용) → 충돌 없음. 무선은 여전히 충돌 회피 필요 → [[wifi]].

## 셀프 체크

> [!question]- IP 주소가 있는데 왜 MAC 주소가 또 필요한가?
> IP는 전 세계 라우팅을 위한 논리적 주소로 위치에 따라 바뀐다. MAC은 로컬 네트워크 안에서 실제 프레임을 인접 노드로 전달하기 위한 물리적 하드웨어 식별자로 기기에 고정된다. 라우터를 지날 때 목적 IP는 그대로지만 목적 MAC은 다음 홉으로 바뀐다. 한 홉 전달에는 MAC이 필요하다.

> [!question]- ARP는 무엇을, 어떻게 해결하는가?
> 같은 로컬 네트워크에서 목적지 IP는 아는데 MAC을 모를 때 변환한다. "이 IP의 MAC은?"을 브로드캐스트하면 해당 IP를 가진 기기가 자기 MAC으로 응답하고, 이 매핑을 ARP 캐시에 저장해 이후 브로드캐스트를 피한다. 캐시는 TTL로 만료된다.

> [!question]- 스위치와 라우터는 각각 어느 계층에서 무엇을 기준으로 동작하는가?
> 스위치는 링크 계층에서 MAC 주소 기반으로 프레임을 해당 포트로만 전달하며 같은 네트워크 내에서 동작한다(학습으로 MAC 테이블 구성). 라우터는 네트워크 계층에서 IP 주소 기반으로 다른 네트워크 간 경로를 결정해 전달한다. 같은 서브넷은 스위치, 다른 서브넷은 라우터.

> [!question]- ARP 스푸핑은 어떻게 중간자 공격이 되는가?
> ARP는 응답을 검증하지 않는 신뢰 기반 프로토콜이다. 공격자가 "게이트웨이 IP = 내 MAC"이라 거짓 응답하면 피해자의 ARP 캐시가 오염되어 게이트웨이로 갈 트래픽이 공격자에게 향한다. 공격자가 트래픽을 가로채는 중간자 공격이 성립한다.

## 연습문제

> [!example]- 문제: 호스트 A(10.18.0.5)가 다른 서브넷의 B(203.0.113.9)로 패킷을 보낼 때, 그리고 같은 서브넷의 C(10.18.0.7)로 보낼 때 ARP와 목적 MAC이 어떻게 달라지는지 추적하라.
> **풀이**
> - 같은 서브넷의 C로: A가 C의 IP에 대해 ARP로 C의 MAC을 직접 알아내 프레임의 목적 MAC에 C의 MAC을 넣어 전송한다.
> - 다른 서브넷의 B로: A는 B가 로컬에 없음을 서브넷 마스크로 판단하고, 기본 게이트웨이(라우터)의 MAC을 ARP로 알아내 목적 MAC에 라우터 MAC을 넣는다. 목적 IP는 여전히 B다. 라우터가 프레임을 받아 다음 홉으로 다시 전달하며 매 홉마다 목적 MAC이 갱신된다.

> [!example]- 문제: 이더넷 프레임을 수신한 스위치와 노드가 헤더 필드를 어떻게 활용하는지, [목적 MAC][출발 MAC][타입][페이로드][CRC] 순으로 설명하라.
> **풀이**
> - 목적 MAC: 스위치가 MAC 테이블에서 이 주소에 해당하는 포트를 찾아 그 포트로만 전달한다. 테이블에 없으면 모든 포트로 플러딩한다.
> - 출발 MAC: 스위치가 이 주소와 수신 포트를 매핑해 테이블을 학습한다.
> - 타입: 페이로드의 상위 프로토콜(IPv4/IPv6/ARP)을 알려 노드가 올바른 처리로 넘긴다.
> - 페이로드: IP 패킷이 캡슐화되어 들어 있다.
> - CRC: 노드가 오류를 검출해 손상된 프레임을 폐기한다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: (1) IP(논리적, 전체 경로)와 MAC(물리적, 한 홉)이 왜 둘 다 필요한가. (2) ARP가 IP를 MAC으로 브로드캐스트 변환하고 캐싱하는 과정. (3) 같은 서브넷은 스위치(MAC), 다른 서브넷은 라우터(IP)로 전달되는 경로 판단.
