# 로드 밸런싱 (Load Balancing)

## 한 줄 요약

여러 서버에 트래픽을 분산해 처리량·가용성을 높이는 것. L4(전송 계층, 빠름)냐 L7(응용 계층, 똑똑함)이냐, 분배 알고리즘(라운드로빈/최소연결/해시)을 무엇으로 하느냐가 핵심. 상태 있는 요청엔 consistent hashing이 노드 추가·삭제 시 재배치를 최소화한다.

## 왜 필요한가

- 서버 하나로 감당 안 되는 트래픽을 어떻게
- L4 vs L7 로드밸런서 차이
- 왜 단순 해싱 말고 consistent hashing인가

## 왜 로드밸런싱

```
서버 1대: 용량 한계, 죽으면 전체 다운 (SPOF)
서버 N대 + LB: 수평 확장 + 하나 죽어도 계속 (고가용성)
```

- **확장성**: 트래픽 늘면 서버 추가 (수평 확장, distributed-systems/[[why-distributed]])
- **가용성**: 죽은 서버 제외 (헬스 체크)
- **성능**: 부하 고르게 → 지연↓
- CDN([[cdn]]), k8s Service(devops/[[kubernetes-networking]]), DB 읽기 복제(database/[[replication-db]])가 다 이 원리

## L4 vs L7

로드밸런서가 어느 계층에서 결정하나:

### L4 (전송 계층)

```
TCP/UDP 수준에서 분배 (IP·포트만 봄)
패킷 내용 안 봄 → 빠름, 저오버헤드
```

- **빠름**: 헤더만 보고 라우팅 (내용 파싱 X)
- 연결을 서버에 고정 (NAT/DSR)
- 내용 기반 라우팅 불가 (URL·쿠키 모름)

### L7 (응용 계층)

```
HTTP 수준에서 분배 (URL·헤더·쿠키 봄) [[http]]
경로별 라우팅, TLS 종료, 콘텐츠 기반
```

- **똑똑함**: `/api`는 A서버, `/img`는 B서버 (경로 라우팅)
- TLS 종료([[tls]]), 압축, 캐싱, 요청 재작성
- 느림·비쌈 (내용 파싱), 하지만 유연
- k8s Ingress가 L7 (devops/[[kubernetes-networking]])

## 분배 알고리즘

- **라운드로빈**: 순서대로 (단순, 서버 성능 같을 때)
- **가중 라운드로빈**: 성능 좋은 서버에 더 (용량 차이 반영)
- **최소 연결**: 연결 적은 서버로 (요청 처리 시간 편차 클 때)
- **해시 기반**: 키(IP·세션)로 해싱 → **같은 클라이언트를 같은 서버로** (세션 고정, 캐시 지역성)
- **랜덤 + two-choices**: 둘 뽑아 덜 바쁜 쪽 (단순한데 효과 좋음)

## Consistent Hashing (핵심)

해시 기반 분배의 문제 - **노드 수가 바뀌면 대량 재배치**:

```
단순 모듈로: server = hash(key) % N
N이 3→4로 바뀌면 → 거의 모든 키의 목적지가 바뀜 (캐시 전멸)
```

consistent hashing 해법:
```
해시 링(0~2^32)에 노드를 배치 → 키는 시계방향 다음 노드로
노드 추가/삭제 시 → 인접 구간만 재배치 (전체 아님)
가상 노드(vnode)로 고르게 분산
```

### 코드로 확인

10000개 키, 노드 3→4 추가 시 재배치 비교:

```python
# 모듈로: hash(key) % N
# consistent: 해시 링 + 가상 노드 100개
```

실행:
```
노드 3->4 추가 시 재배치된 키 (총 10000개):
  모듈로 해싱:       7428 (74.3%)
  consistent hash:  2537 (25.4%)  <- 이상적 ~1/4
```

- 모듈로는 **74%가 재배치**(캐시·세션 대참사) vs consistent는 **25%**(이상적 1/4에 근접)
- 노드 추가 시 **새 노드가 가져갈 몫만** 이동 → 캐시 미스·데이터 이동 최소
- 분산 캐시(memcached), DB 파티셔닝(database/[[partitioning-db]]), DHT의 표준 기법

## 상태 문제: 세션 고정

- **무상태(stateless) 서버가 이상적**: 아무 서버나 처리 가능 → LB 자유
- **상태 있으면**(세션 메모리): 같은 클라이언트를 같은 서버로 (sticky session) 또는 **세션을 외부화**(Redis 등) → 무상태로
- 무상태가 확장·복구에 유리 (software-design/의 상태 최소화)

## 헬스 체크·고가용성

- **헬스 체크**: 주기적으로 서버 생존 확인 → 죽으면 풀에서 제외 ([[icmp-and-tools]], devops/[[observability]])
- **LB 자신의 이중화**: LB가 SPOF 되면 안 됨 → 이중 LB (active-passive, VRRP)
- **DNS 로드밸런싱**: DNS로 여러 IP 반환 (조악하지만 지리적 분산, [[cdn]])

## 셀프 체크

> [!question]- L4 로드밸런서와 L7 로드밸런서의 결정 근거와 트레이드오프는?
> L4는 전송 계층에서 IP·포트만 보고 분배한다. 패킷 내용을 파싱하지 않아 빠르고 저오버헤드지만 URL·쿠키 기반 라우팅은 못 한다. L7은 응용 계층에서 HTTP의 URL·헤더·쿠키를 보고 경로별 라우팅, TLS 종료, 콘텐츠 기반 분배를 하지만 내용 파싱 때문에 느리고 비싸다.

> [!question]- 단순 모듈로 해싱(hash(key) % N)이 노드 수 변경에 취약한 이유와 consistent hashing이 이를 어떻게 완화하나?
> 모듈로 방식은 N이 바뀌면 거의 모든 키의 목적지가 재계산되어 대량 재배치(캐시 전멸)가 일어난다. consistent hashing은 해시 링에 노드를 배치하고 키를 시계방향 다음 노드에 할당해, 노드 추가/삭제 시 인접 구간만 재배치되도록 한다. 가상 노드로 분산을 고르게 한다.

> [!question]- 세션 고정(sticky session)과 세션 외부화(Redis 등) 중 무상태 확장에 유리한 쪽은 어디이고 왜인가?
> 세션 외부화가 유리하다. 서버를 무상태로 만들면 어느 서버가 요청을 받아도 되므로 LB가 자유롭게 분배하고, 서버 추가·장애 복구가 쉽다. sticky session은 특정 서버에 묶여 그 서버가 죽으면 세션이 날아가고 부하 불균형이 생긴다.

> [!question]- 로드밸런서 자신이 SPOF가 되는 문제를 어떻게 막나?
> LB를 이중화한다. active-passive 구성으로 예비 LB를 두고 VRRP 등으로 가상 IP를 넘겨받아, 주 LB가 죽어도 예비가 이어받게 한다. 또 DNS 로드밸런싱으로 여러 IP를 반환해 조악하게나마 분산·지리적 이중화를 할 수 있다.

## 연습문제

> [!example]- 문제: 10000개의 키를 노드 3개에서 4개로 늘릴 때, 모듈로 해싱과 consistent hashing 각각의 이상적 재배치 키 수를 추정하고, 실측(모듈로 74.3%, consistent 25.4%)이 왜 그렇게 나오는지 설명하라.
> **풀이**
> consistent hashing의 이상: 노드 4개가 되면 새 노드가 전체의 약 1/4 = 2500개를 인접 구간에서 가져간다. 나머지 키는 그대로 → 재배치 약 25%. 실측 25.4%가 이 이상에 근접한다.
> 모듈로의 경우: `hash(key) % 3`에서 `hash(key) % 4`로 바뀌면 나머지 연산 결과가 대부분의 키에서 달라진다. 키가 이전과 같은 노드에 남을 확률은 대략 1/N 수준이라, N=3→4에서 약 3/4 = 75%가 재배치된다. 실측 74.3%가 이와 일치한다.
> 결론: 모듈로는 노드 변경 시 캐시·세션 대참사(74%), consistent는 새 노드 몫만 이동(25%)해 캐시 미스·데이터 이동을 최소화한다.

> [!example]- 문제: L7 로드밸런서로 `/api/*`는 백엔드 A풀, `/static/*`는 B풀(CDN 캐싱), 그 외는 C풀로 라우팅하고 싶다. L4로는 왜 이게 불가능하며, L7 구성에서 추가로 얻는 이점 두 가지를 들라.
> **풀이**
> L4는 IP·포트만 보고 패킷 내용(URL 경로)을 파싱하지 않으므로 `/api`인지 `/static`인지 구분할 수 없다. 경로 기반 라우팅은 HTTP 요청 라인을 읽어야 하므로 L7이 필요하다.
> L7 구성: 요청의 경로를 보고 `/api/*`→A, `/static/*`→B, 나머지→C로 분배한다.
> 추가 이점: (1) TLS 종료를 LB에서 처리해 백엔드는 평문 HTTP로 단순화, (2) 응답 캐싱·압축·요청 재작성 등 콘텐츠 기반 최적화 가능. 대가는 내용 파싱으로 인한 지연·비용 증가다.

## 파인만

> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지. (1) 로드밸런싱이 확장성·가용성·성능을 어떻게 동시에 개선하는지 SPOF 관점에서 설명할 수 있나. (2) L4 vs L7의 결정 계층과 트레이드오프, 분배 알고리즘(라운드로빈/최소연결/해시)의 선택 기준을 말할 수 있나. (3) 모듈로 해싱의 재배치 문제와 consistent hashing(해시 링·가상 노드)의 해법을 수치로 대비할 수 있나.
