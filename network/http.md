# HTTP

## 한 줄 요약

웹의 요청-응답 프로토콜. 텍스트 기반 메서드/헤더/바디로 시작해, HTTP/2의 멀티플렉싱, HTTP/3의 QUIC로 진화했다. 무상태(stateless)라 쿠키·캐싱으로 상태와 성능을 보완한다.

## 왜 필요한가

- 웹의 근간 - 브라우저와 서버의 대화
- HTTP/1.1 → 2 → 3 진화의 이유
- 캐싱·쿠키가 왜 필요한가

## 요청-응답 구조

HTTP는 클라이언트가 요청, 서버가 응답:

```
요청:
  GET /index.html HTTP/1.1        ← 메서드 경로 버전
  Host: example.com               ← 헤더들
  User-Agent: ...
                                  ← 빈 줄
  (바디, POST 등)

응답:
  HTTP/1.1 200 OK                 ← 버전 상태코드
  Content-Type: text/html         ← 헤더들
  Content-Length: 1234
                                  ← 빈 줄
  <html>...                       ← 바디
```

텍스트 기반(HTTP/1.1) → 사람이 읽을 수 있음. 전송 계층 TCP 위에 얹힘 → [[tcp-basics]].

## 메서드

| 메서드 | 용도 | 멱등? | 안전? |
|---|---|---|---|
| GET | 조회 | ✓ | ✓ (변경 없음) |
| POST | 생성/제출 | ✗ | ✗ |
| PUT | 전체 교체 | ✓ | ✗ |
| PATCH | 부분 수정 | ✗ | ✗ |
| DELETE | 삭제 | ✓ | ✗ |

- **안전(safe)**: 서버 상태 안 바꿈 (GET)
- **멱등(idempotent)**: 여러 번 해도 결과 같음 (재시도 안전) → web/[[rest-design]], distributed-systems/[[idempotency]]

## 상태 코드

- **1xx**: 정보
- **2xx**: 성공 (200 OK, 201 Created, 204 No Content)
- **3xx**: 리다이렉트 (301 영구, 302 임시, 304 Not Modified - 캐시)
- **4xx**: 클라이언트 오류 (400 잘못된 요청, 401 인증 필요, 403 금지, 404 없음, 429 너무 많음)
- **5xx**: 서버 오류 (500 내부 오류, 502 게이트웨이, 503 사용 불가)

## 무상태 (stateless)

HTTP는 **각 요청이 독립** - 서버가 이전 요청을 기억 안 함:
- 장점: 서버 확장 쉬움 (어느 서버가 받아도 됨 → distributed-systems/)
- 단점: 로그인 상태 등을 유지하려면 별도 수단 → **쿠키**, 세션, 토큰 → web/[[web-auth]]

## HTTP 진화

### HTTP/1.1

- **keep-alive**: 한 TCP 연결 재사용 (연결마다 핸드셰이크 회피 → [[tcp-basics]])
- 문제: **head-of-line blocking** - 한 연결에서 요청을 순서대로 처리, 앞 응답이 느리면 뒤가 막힘
- 브라우저가 병렬로 여러 연결(보통 6개)로 완화

### HTTP/2

- **멀티플렉싱**: 한 연결에서 여러 요청/응답을 **동시에** (스트림으로 인터리빙) → HTTP 레벨 HOL blocking 해결
- **헤더 압축(HPACK)**: 반복 헤더 압축
- **서버 푸시**: 요청 전에 리소스 미리 (거의 안 씀, deprecated)
- **바이너리 프레이밍**: 텍스트 아닌 바이너리 (효율)
- 여전히 TCP 위 → TCP 레벨 HOL blocking은 남음 (패킷 손실 시)

### HTTP/3

- **QUIC 위에서 동작** (TCP 아닌 UDP 기반) → [[quic]]
- **TCP HOL blocking 완전 해결**: QUIC 스트림이 독립 → 한 스트림 패킷 손실이 다른 스트림 안 막음
- 연결 설정 빠름 (0-RTT), 연결 마이그레이션 (IP 바뀌어도 유지)

## 캐싱

같은 리소스 재요청을 피해 성능·대역폭 개선 → web/[[http-for-web]]:

- **Cache-Control**: `max-age`(유효 시간), `no-cache`, `private/public`
- **ETag**: 리소스 버전 태그 → 바뀌었나 확인 (조건부 요청)
- **304 Not Modified**: "안 바뀜, 캐시 쓰세요" (바디 없이) → 대역폭 절약
- CDN이 이걸 활용해 엣지에서 캐싱 → [[cdn]]

## 셀프 체크

> [!question]- HTTP가 무상태(stateless)라는 것이 정확히 무슨 뜻이고, 이 성질의 장점과 단점은?
> 각 요청이 독립적이라 서버가 이전 요청을 기억하지 않는다는 뜻이다. 장점은 어느 서버가 받아도 되므로 수평 확장이 쉽다는 것. 단점은 로그인 같은 상태 유지를 위해 쿠키·세션·토큰 같은 별도 수단이 필요하다는 것이다.

> [!question]- HTTP/1.1의 head-of-line blocking이 HTTP/2에서 어떻게 해결되고, 그럼에도 남는 HOL blocking은 무엇인가?
> HTTP/1.1은 한 연결에서 요청을 순서대로 처리해 앞 응답이 느리면 뒤가 막힌다. HTTP/2는 멀티플렉싱으로 한 연결에서 여러 스트림을 인터리빙해 HTTP 레벨 HOL blocking을 없앤다. 다만 여전히 TCP 위라 패킷 손실 시 TCP 레벨 HOL blocking은 남고, 이는 HTTP/3의 QUIC가 해결한다.

> [!question]- GET과 PUT은 멱등이지만 POST는 아니다. 멱등이 재시도 안전성과 어떻게 연결되나?
> 멱등은 같은 요청을 여러 번 보내도 결과가 같다는 것이다. 네트워크 실패로 응답을 못 받았을 때 멱등한 요청(GET/PUT/DELETE)은 그냥 다시 보내도 안전하지만, POST는 재시도하면 중복 생성 같은 부작용이 생길 수 있어 재시도 로직을 조심해야 한다.

> [!question]- ETag와 304 Not Modified가 대역폭을 어떻게 절약하나?
> 서버가 리소스에 버전 태그(ETag)를 붙이고, 클라이언트는 조건부 요청으로 "내 버전 이거 맞나" 물어본다. 안 바뀌었으면 서버가 바디 없이 304 Not Modified만 응답해 리소스 본문 전송을 생략한다. CDN이 엣지에서 이를 활용한다.

## 연습문제

> [!example]- 문제: HTTP/1.1 keep-alive로 3개의 순차 리소스를 받을 때와, HTTP/2 멀티플렉싱으로 받을 때의 지연 차이를 RTT 단위로 추정하라. (연결은 이미 맺어져 있고, 각 리소스 응답 처리 시간은 무시, RTT=100ms 가정)
> **풀이**
> HTTP/1.1(단일 연결, keep-alive)은 한 번에 한 요청씩 순서대로 처리한다. 요청→응답 각 1 RTT이므로 3개면 대략 3 RTT = 300ms.
> HTTP/2는 3개 요청을 동시에 스트림으로 보내 응답을 병렬로 받는다. 요청을 한꺼번에 보내고 응답을 함께 받으므로 대략 1 RTT = 100ms.
> 즉 순차 처리의 HOL blocking을 멀티플렉싱이 제거해 약 3배 개선. (실제로는 브라우저가 HTTP/1.1에서 6개 병렬 연결로 완화하지만 연결마다 핸드셰이크 비용이 든다.)

> [!example]- 문제: 다음 응답 헤더를 해석하라. `Cache-Control: max-age=3600, private` 와 `ETag: "abc123"`. 클라이언트가 3600초 이내/이후 재요청 시 각각 어떻게 동작하나?
> **풀이**
> `max-age=3600`: 이 리소스는 3600초(1시간)간 신선(fresh)하다. `private`: 공유 캐시(CDN·프록시)에 저장 금지, 브라우저 개인 캐시만 허용.
> 3600초 이내 재요청: 캐시가 신선하므로 네트워크 요청 없이 로컬 캐시를 바로 사용.
> 3600초 이후 재요청: 캐시가 stale이므로 조건부 요청(`If-None-Match: "abc123"`)을 보낸다. 서버가 안 바뀌었으면 304 Not Modified(바디 없음)로 응답 → 캐시 재사용, 대역폭 절약. 바뀌었으면 200 OK로 새 바디와 새 ETag를 보낸다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지. (1) 요청-응답 메시지 구조(메서드/헤더/바디, 상태코드)를 예시로 그릴 수 있나. (2) HTTP/1.1→2→3 진화에서 각 단계가 어떤 HOL blocking을 어떻게 풀었는지 설명할 수 있나. (3) 무상태 + 쿠키/캐싱(Cache-Control/ETag/304)이 왜 함께 필요한지 인과로 말할 수 있나.
