# TLS

## 한 줄 요약

TCP 위에 암호화·인증·무결성을 더하는 계층 (HTTPS의 S). 핸드셰이크로 대칭 키를 안전하게 교환하고, 인증서로 서버 신원을 검증한다. 비대칭 암호로 시작해 대칭 암호로 통신한다.

## 왜 필요한가

- HTTPS가 실제로 뭘 보호하나
- 처음 보는 서버와 어떻게 안전하게 키를 나누나
- 인증서가 뭘 검증하나 (상세는 security/[[tls-deep-dive]])

## 무엇을 보호하나

평문 통신(HTTP)은 경로의 누구나 도청·변조 가능 ([[wifi]]의 공유 매체, 라우터). TLS가 세 가지 보장:

- **기밀성(confidentiality)**: 암호화 → 도청해도 내용 못 봄
- **무결성(integrity)**: 변조 감지 → 중간에 못 바꿈
- **인증(authentication)**: 서버가 진짜인지 → 가짜 서버(피싱) 방지

TCP([[tcp-basics]]) 위, 응용([[http]]) 아래에 위치. HTTP + TLS = HTTPS.

## 핵심 문제: 키 교환

암호화하려면 양쪽이 **같은 비밀 키**를 알아야 함. 그런데 처음 보는 서버와 도청되는 네트워크에서 어떻게 키를 안전하게 나누나? → **비대칭 암호로 시작**:

- **비대칭 암호**(RSA, ECDH → security/[[crypto-basics]]): 공개키/개인키 쌍. 공개키로 암호화하면 개인키로만 복호화
- 이걸로 대칭 키를 안전하게 합의 (또는 Diffie-Hellman으로 공유 비밀 생성)
- **왜 비대칭만 안 쓰나**: 비대칭은 느림 → 키 교환에만 쓰고, 실제 데이터는 빠른 **대칭 암호**(AES)로

**하이브리드**: 비대칭(느림, 키 교환) + 대칭(빠름, 데이터). security/[[crypto-basics]]의 그 조합.

## 핸드셰이크 (개요)

TLS 1.3 기준 (상세는 security/[[tls-deep-dive]]):

```
1. 클라이언트 헬로: 지원 암호 목록, 키 교환 파라미터
2. 서버 헬로: 선택한 암호, 인증서, 키 교환 파라미터
3. 양쪽이 공유 비밀 계산 (Diffie-Hellman) → 대칭 키 유도
4. 이후 대칭 암호로 통신
```

- **TLS 1.3**: 1 RTT (이전 1.2는 2 RTT). QUIC([[quic]])은 이걸 전송과 통합해 더 빠름
- 핸드셰이크가 연결 지연에 추가 → keep-alive([[http]])로 재사용

## 인증서와 신뢰 체인

서버가 진짜임을 어떻게 믿나 → **인증서(certificate)**:

```
서버 인증서 (공개키 + 도메인) ← CA가 서명
CA 인증서 ← 상위 CA가 서명
...
루트 CA ← 브라우저/OS에 미리 내장 (신뢰 앵커)
```

- **CA(인증 기관)**: 서버의 신원을 확인하고 인증서에 **서명** (security/[[digital-signatures]])
- 브라우저가 루트 CA를 미리 신뢰 → 체인을 따라 서버 인증서 검증
- 도메인 일치, 만료, 폐지 확인
- 검증 실패 → 경고 (인증서 오류)

이 신뢰 체인(PKI)이 "처음 보는 서버를 믿는" 근거 → security/[[digital-signatures]].

## forward secrecy

TLS 1.3의 중요 성질: **과거 통신이 미래에 키가 유출돼도 안전**:

- 매 세션마다 임시(ephemeral) 키 교환 (ECDHE)
- 서버 개인키가 나중에 유출돼도 과거 세션은 복호화 불가 (그때의 임시 키는 이미 폐기)
- 대량 감청에 대한 방어 → security/[[tls-deep-dive]]

## 실무 관점

- **어디서나 HTTPS**: 이제 표준 (Let's Encrypt로 무료 인증서). HTTP는 리다이렉트
- **CDN이 TLS 종료**: 엣지에서 처리 → [[cdn]]
- **인증서 관리**: 만료 갱신 자동화 (만료되면 사이트 접속 불가)
- **HSTS**: "항상 HTTPS로" 강제 헤더 → security/[[web-vulnerabilities]]

## 셀프 체크

> [!question]- TLS는 왜 비대칭 암호로만 통신하지 않고 비대칭+대칭 하이브리드를 쓰는가?
> 비대칭 암호(RSA, ECDH)는 처음 보는 서버와 도청되는 네트워크에서 키를 안전하게 합의하기에 좋지만 연산이 느리다. 그래서 비대칭은 대칭 키를 교환하는 데만 쓰고, 실제 데이터 전송은 빠른 대칭 암호(AES)로 한다. 느린 부분을 최소화하는 조합이다.

> [!question]- 브라우저가 처음 보는 서버를 어떻게 믿는가? 신뢰 체인을 설명하라.
> 서버 인증서는 그 상위 CA가 서명하고, 그 CA 인증서는 다시 상위 CA가 서명하는 식으로 루트 CA까지 이어진다. 루트 CA는 브라우저/OS에 미리 내장된 신뢰 앵커다. 브라우저는 이 체인을 따라 서명을 검증하고 도메인 일치, 만료, 폐지를 확인해 서버 신원을 믿는다.

> [!question]- forward secrecy란 무엇이며 TLS 1.3은 이를 어떻게 달성하는가?
> forward secrecy는 서버 개인키가 나중에 유출돼도 과거에 캡처된 세션을 복호화할 수 없는 성질이다. TLS 1.3은 매 세션마다 임시(ephemeral) 키 교환(ECDHE)을 하고 그 임시 키를 세션 후 폐기하므로, 장기 개인키가 유출돼도 과거 세션의 대칭 키를 재현할 수 없다.

> [!question]- TLS가 보장하는 세 가지는 무엇이고 각각 어떤 공격을 막는가?
> 기밀성(암호화)은 도청을 막아 내용을 못 보게 하고, 무결성은 중간 변조를 감지하며, 인증은 서버가 진짜인지 검증해 가짜 서버(피싱)를 막는다. 이 셋이 함께 있어야 도청/변조/사칭을 모두 방어한다.

## 연습문제

> [!example]- 문제: TLS 1.2는 핸드셰이크에 2 RTT, TLS 1.3은 1 RTT가 든다. RTT 50ms일 때 두 버전의 핸드셰이크 지연을 비교하고, keep-alive가 왜 중요한지 설명하라.
> **풀이**
> - TLS 1.2: 2 RTT x 50ms = 100ms (TCP 핸드셰이크 1 RTT는 별도).
> - TLS 1.3: 1 RTT x 50ms = 50ms.
> TLS 1.3이 핸드셰이크만으로 50ms 절약한다. 여기에 TCP 3-way 핸드셰이크(1 RTT = 50ms)까지 더하면 연결 하나 세우는 데 드는 왕복이 상당하다. keep-alive로 기존 연결을 재사용하면 이후 요청은 핸드셰이크를 건너뛰어 매 요청마다 이 왕복 비용을 없앨 수 있다.

> [!example]- 문제: 클라이언트가 서버 인증서를 받았다. 브라우저가 "이 서버를 신뢰할지" 판단하기 위해 순서대로 수행하는 검증 단계를 나열하라.
> **풀이**
> 1. 체인 구성: 서버 인증서 → 중간 CA → 루트 CA로 이어지는 서명 체인을 만든다.
> 2. 서명 검증: 각 인증서가 상위 CA의 공개키로 검증되는지 확인하고, 루트 CA가 신뢰 저장소(브라우저/OS 내장)에 있는지 확인한다.
> 3. 도메인 일치: 인증서의 도메인(SAN/CN)이 접속하려는 호스트와 일치하는지 확인한다.
> 4. 유효 기간: 현재 시각이 인증서 유효 기간 안인지(만료 아님) 확인한다.
> 5. 폐지 확인: CRL/OCSP로 폐지되지 않았는지 확인한다.
> 하나라도 실패하면 인증서 경고를 띄운다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 도청되는 네트워크에서 처음 보는 서버와 어떻게 공유 대칭 키에 도달하는지(비대칭으로 시작 → 대칭으로 통신)를 설명할 수 있는가.
> 2. 인증서 신뢰 체인이 루트 CA(신뢰 앵커)에서 서버 인증서까지 어떻게 서명으로 이어지는지 그릴 수 있는가.
> 3. 기밀성/무결성/인증 세 보장과 forward secrecy를 구분해 말할 수 있는가.
