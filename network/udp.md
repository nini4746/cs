# UDP

## 한 줄 요약

최소한의 전송 계층 프로토콜. 연결도 신뢰성도 순서 보장도 없이 그냥 패킷을 던진다. 그 단순함과 낮은 지연이 DNS, 실시간 게임, 화상통화, QUIC의 기반이 된다.

## 왜 필요한가

- TCP가 다 해주는데 왜 UDP가 존재하나
- 언제 신뢰성을 포기하는 게 이득인가
- QUIC/HTTP3가 왜 UDP 위에 서나

## UDP = 거의 아무것도 안 함

UDP(User Datagram Protocol)는 IP([[ip-addressing]]) 위에 **최소한만** 얹음:

- **포트 번호**: 어느 앱으로 (다중화)
- **체크섬**: 손상 감지 (선택적)
- 끝. 그게 전부.

TCP([[tcp-basics]])가 하는 것 중 **아무것도 안 함**:
- 연결 설정 없음 (핸드셰이크 X)
- 신뢰성 없음 (손실돼도 재전송 X)
- 순서 보장 없음 (뒤바뀔 수 있음)
- 흐름/혼잡 제어 없음

## 왜 이걸 쓰나

"아무것도 안 함"이 장점인 경우:

1. **낮은 지연**: 핸드셰이크 없이 즉시 전송. 재전송 대기 없음
2. **오버헤드 적음**: 헤더 8바이트 (TCP는 20+)
3. **제어권**: 신뢰성이 필요하면 **앱이 직접** 원하는 방식으로 구현
4. **재전송이 무의미한 데이터**: 실시간에선 늦은 데이터보다 다음 데이터가 나음

## 어디에 쓰나

| 용도 | 이유 |
|---|---|
| **DNS** ([[dns]]) | 작은 요청/응답 - 핸드셰이크가 낭비 |
| **실시간 게임** | 늦은 위치 정보는 쓸모없음 - 최신이 중요 |
| **화상/음성 통화** | 손실된 프레임 재전송보다 다음 프레임 |
| **비디오 스트리밍** | 약간 손실 감내, 지연 회피 |
| **QUIC/HTTP3** ([[quic]]) | TCP를 우회해 유저공간에서 신뢰성 재구현 |
| **DHCP, NTP** | 간단한 요청/응답 |

## 실시간의 논리: 늦은 데이터 = 죽은 데이터

화상통화에서 0.5초 전 프레임이 재전송돼 도착하면? **이미 쓸모없음** - 지금 프레임이 필요. TCP는 순서 보장을 위해 옛 패킷을 기다리며 막지만(head-of-line blocking), 실시간에선 그게 독:

```
TCP: 패킷3 손실 → 패킷4,5 도착해도 3 기다림 (막힘)
UDP: 패킷3 손실 → 그냥 4,5 진행 (실시간엔 이게 맞음)
```

## UDP 위에 신뢰성 쌓기

"신뢰성이 필요하면 TCP 쓰면 되지"가 항상 답은 아님. UDP 위에 **맞춤 신뢰성**을 쌓는 이유:

- TCP의 HOL blocking 회피 (스트림별 독립 손실 처리)
- 앱이 필요한 만큼만 신뢰성 (일부만 재전송)
- 연결 마이그레이션 (IP 바뀌어도 유지)

**QUIC**([[quic]])가 대표: UDP 위에 신뢰성+암호화+멀티플렉싱을 유저공간에서 재구현 → HTTP/3의 기반. "TCP의 좋은 점 + UDP의 유연함".

## TCP vs UDP

| | TCP | UDP |
|---|---|---|
| 연결 | 있음 (핸드셰이크) | 없음 |
| 신뢰성 | 보장 (재전송) | 없음 |
| 순서 | 보장 | 없음 |
| 혼잡 제어 | 있음 | 없음 |
| 헤더 | 20+ 바이트 | 8 바이트 |
| 지연 | 높음 (설정, 재전송) | 낮음 |
| 용도 | 웹, 파일, 메일 | 실시간, DNS, QUIC |

**규칙**: 정확성이 중요하고 지연 감내 → TCP. 지연이 중요하고 손실 감내(또는 자체 처리) → UDP.

## 셀프 체크

> [!question]- UDP가 IP 위에 실제로 더해주는 것은 무엇이고, TCP가 하는 것 중 무엇을 안 하는가?
> UDP는 IP 위에 포트 번호(다중화)와 선택적 체크섬(손상 감지)만 더한다. 연결 설정(핸드셰이크), 신뢰성(재전송), 순서 보장, 흐름/혼잡 제어는 모두 하지 않는다. 그래서 "거의 아무것도 안 하는" 최소 전송 계층이다.

> [!question]- 실시간 통신에서 "늦은 데이터 = 죽은 데이터"라는 말과 head-of-line blocking은 어떻게 연결되는가?
> TCP는 순서 보장을 위해 앞 패킷이 손실되면 뒤 패킷이 도착해도 앱에 넘기지 않고 막는다(HOL blocking). 화상통화에서는 0.5초 전 프레임이 재전송돼 도착해도 이미 쓸모없으므로, 옛 패킷을 기다리며 막는 것이 오히려 독이다. UDP는 손실을 무시하고 최신 데이터를 진행시켜 실시간에 맞는다.

> [!question]- "신뢰성이 필요하면 그냥 TCP를 쓰면 되지"가 항상 답이 아닌 이유는?
> UDP 위에 맞춤 신뢰성을 쌓으면 TCP의 HOL blocking을 피해 스트림별로 독립적으로 손실을 처리하고, 앱이 필요한 부분만 재전송하며, IP가 바뀌어도 연결을 유지(마이그레이션)할 수 있다. QUIC이 이 방식의 대표로, UDP 위에 신뢰성+암호화+멀티플렉싱을 유저공간에서 재구현한다.

> [!question]- DNS가 UDP를 쓰는 이유는 무엇인가?
> DNS는 대개 작은 요청/응답 하나로 끝난다. TCP의 핸드셰이크를 세우고 닫는 비용이 실제 데이터보다 커서 낭비이므로, 핸드셰이크 없이 즉시 던지는 UDP가 지연과 오버헤드 면에서 이득이다.

## 연습문제

> [!example]- 문제: 같은 페이로드 500바이트를 한 번 요청/응답으로 주고받을 때, TCP와 UDP의 헤더 오버헤드와 왕복 횟수를 비교하라. (TCP 헤더 20바이트, UDP 헤더 8바이트 가정)
> **풀이**
> - 헤더 오버헤드: TCP는 세그먼트당 20바이트, UDP는 8바이트. 500바이트 페이로드 기준 오버헤드 비율은 TCP 20/520 ≈ 3.8%, UDP 8/508 ≈ 1.6%.
> - 왕복: TCP는 3-way 핸드셰이크(1 RTT)로 연결을 세운 뒤 요청/응답(1 RTT)이라 최소 2 RTT. UDP는 핸드셰이크 없이 요청 1회 + 응답 1회로 1 RTT.
> DNS처럼 단발성 소량 요청에서는 UDP가 헤더도 작고 왕복도 절반이라 명확히 유리하다.

> [!example]- 문제: 30fps 화상통화에서 프레임 하나가 손실됐다. TCP로 전송했을 때와 UDP로 전송했을 때 사용자가 겪는 결과를 시간 흐름으로 대조하라.
> **풀이**
> 30fps이면 한 프레임 간격은 1/30초 ≈ 33ms.
> - TCP: 손실 프레임을 재전송(최소 1 RTT 추가, RTT 50ms면 손실 감지+재전송에 수십 ms 이상)하고, 그동안 도착한 다음 프레임들은 순서 보장 때문에 버퍼에 막혀 재생이 멈춘다(버벅임/프리즈). 재전송돼 도착한 프레임은 이미 지난 시점이라 쓸모없다.
> - UDP: 손실 프레임은 그냥 건너뛰고 다음 프레임(33ms 뒤)을 바로 재생한다. 한 프레임이 잠깐 깨질 뿐 흐름은 끊기지 않는다.
> 실시간에서는 UDP의 "손실 무시, 최신 진행"이 사용자 체감상 낫다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. UDP가 IP 위에 더하는 최소 요소(포트, 선택적 체크섬)와 TCP가 하는데 UDP가 안 하는 것들을 나열할 수 있는가.
> 2. 실시간(게임/통화)에서 재전송·순서보장이 왜 독이 되는지(HOL blocking, 늦은 데이터 = 죽은 데이터)를 설명할 수 있는가.
> 3. QUIC이 왜 TCP 대신 UDP 위에 신뢰성을 재구현하는지 그 이점(HOL 회피, 부분 신뢰성, 연결 마이그레이션)을 말할 수 있는가.
