# UDP

## 한 줄 요약

최소한의 전송 계층 프로토콜. 연결도 신뢰성도 순서 보장도 없이 그냥 패킷을 던진다. 그 단순함과 낮은 지연이 DNS, 실시간 게임, 화상통화, QUIC의 기반이 된다.

## 왜 필요한가

- TCP가 다 해주는데 왜 UDP가 존재하나
- 언제 신뢰성을 포기하는 게 이득인가
- QUIC/HTTP3가 왜 UDP 위에 서나

## UDP = 거의 아무것도 안 함

UDP(User Datagram Protocol)는 IP([[ip-addressing]]) 위에 **최소한만** 얹음:

- **포트 번호**: 어느 앱으로 (다중화)
- **체크섬**: 손상 감지 (선택적)
- 끝. 그게 전부.

TCP([[tcp-basics]])가 하는 것 중 **아무것도 안 함**:
- 연결 설정 없음 (핸드셰이크 X)
- 신뢰성 없음 (손실돼도 재전송 X)
- 순서 보장 없음 (뒤바뀔 수 있음)
- 흐름/혼잡 제어 없음

## 왜 이걸 쓰나

"아무것도 안 함"이 장점인 경우:

1. **낮은 지연**: 핸드셰이크 없이 즉시 전송. 재전송 대기 없음
2. **오버헤드 적음**: 헤더 8바이트 (TCP는 20+)
3. **제어권**: 신뢰성이 필요하면 **앱이 직접** 원하는 방식으로 구현
4. **재전송이 무의미한 데이터**: 실시간에선 늦은 데이터보다 다음 데이터가 나음

## 어디에 쓰나

| 용도 | 이유 |
|---|---|
| **DNS** ([[dns]]) | 작은 요청/응답 - 핸드셰이크가 낭비 |
| **실시간 게임** | 늦은 위치 정보는 쓸모없음 - 최신이 중요 |
| **화상/음성 통화** | 손실된 프레임 재전송보다 다음 프레임 |
| **비디오 스트리밍** | 약간 손실 감내, 지연 회피 |
| **QUIC/HTTP3** ([[quic]]) | TCP를 우회해 유저공간에서 신뢰성 재구현 |
| **DHCP, NTP** | 간단한 요청/응답 |

## 실시간의 논리: 늦은 데이터 = 죽은 데이터

화상통화에서 0.5초 전 프레임이 재전송돼 도착하면? **이미 쓸모없음** - 지금 프레임이 필요. TCP는 순서 보장을 위해 옛 패킷을 기다리며 막지만(head-of-line blocking), 실시간에선 그게 독:

```
TCP: 패킷3 손실 → 패킷4,5 도착해도 3 기다림 (막힘)
UDP: 패킷3 손실 → 그냥 4,5 진행 (실시간엔 이게 맞음)
```

## UDP 위에 신뢰성 쌓기

"신뢰성이 필요하면 TCP 쓰면 되지"가 항상 답은 아님. UDP 위에 **맞춤 신뢰성**을 쌓는 이유:

- TCP의 HOL blocking 회피 (스트림별 독립 손실 처리)
- 앱이 필요한 만큼만 신뢰성 (일부만 재전송)
- 연결 마이그레이션 (IP 바뀌어도 유지)

**QUIC**([[quic]])가 대표: UDP 위에 신뢰성+암호화+멀티플렉싱을 유저공간에서 재구현 → HTTP/3의 기반. "TCP의 좋은 점 + UDP의 유연함".

## TCP vs UDP

| | TCP | UDP |
|---|---|---|
| 연결 | 있음 (핸드셰이크) | 없음 |
| 신뢰성 | 보장 (재전송) | 없음 |
| 순서 | 보장 | 없음 |
| 혼잡 제어 | 있음 | 없음 |
| 헤더 | 20+ 바이트 | 8 바이트 |
| 지연 | 높음 (설정, 재전송) | 낮음 |
| 용도 | 웹, 파일, 메일 | 실시간, DNS, QUIC |

**규칙**: 정확성이 중요하고 지연 감내 → TCP. 지연이 중요하고 손실 감내(또는 자체 처리) → UDP.
