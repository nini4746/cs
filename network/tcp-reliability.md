# TCP 신뢰성 (TCP Reliability)

## 한 줄 요약

TCP가 손실·손상·재정렬되는 IP 위에서 신뢰성을 만드는 법. 시퀀스 번호로 순서, ACK와 재전송으로 손실 복구, 슬라이딩 윈도우로 처리량, 흐름 제어로 수신자 보호를 한다.

## 왜 필요한가

- 신뢰 없는 IP 위에서 어떻게 신뢰성이 나오나
- 재전송·타임아웃이 어떻게 동작하나
- 흐름 제어와 혼잡 제어의 차이

## 문제: IP는 신뢰할 수 없다

IP([[ip-addressing]])는 **최선 노력(best-effort)** - 패킷을 잃거나, 손상시키거나, 순서를 바꿀 수 있음. TCP가 이 위에 신뢰성을 **소프트웨어로** 구축.

## 시퀀스 번호와 ACK

- **시퀀스 번호**: 각 바이트에 번호 → 수신자가 순서 재구성, 중복 제거
- **누적 ACK**: "번호 N까지 다 받았다" → 하나로 여러 확인
- 수신자가 순서 어긋난 패킷을 버퍼링했다가 재정렬 → 앱엔 순서대로 전달

## 재전송: 손실 복구

ACK가 안 오면 손실로 간주하고 재전송:

### 타임아웃 기반

- 패킷 보내고 **타이머** 시작. ACK 전에 만료되면 재전송
- **RTO(재전송 타임아웃)**: RTT를 측정해 동적 조정 (너무 짧으면 불필요 재전송, 길면 복구 느림)
- RTT 추정: 측정값의 이동 평균 + 변동성 (Jacobson 알고리즘)

### 빠른 재전송 (fast retransmit)

- 타임아웃을 기다리지 않고: **중복 ACK 3개** 받으면 손실로 판단해 즉시 재전송
- 수신자가 순서 어긋난 패킷 받을 때마다 "아직 N 기다림" ACK 반복 → 3번이면 N 손실 신호
- 타임아웃보다 빠른 복구

## 슬라이딩 윈도우: 처리량

한 번에 하나씩 보내고 ACK 기다리면(stop-and-wait) 너무 느림 (매번 1 RTT). **윈도우**로 여러 패킷을 ACK 전에 미리 전송:

```
[ACK된 것][전송했고 ACK 대기 중 ......][보낼 수 있음][아직 안 됨]
           ←──── 윈도우 크기 ────→
```

- 윈도우 = "ACK 없이 보낼 수 있는 양"
- 윈도우가 크면 파이프를 채워 처리량↑ ([[internet-overview]]의 bandwidth-delay product)
- ACK 오면 윈도우가 앞으로 미끄러짐(slide)

## 흐름 제어 (flow control)

**수신자를 보호** - 수신자 버퍼가 넘치지 않게:

- 수신자가 **rwnd(수신 윈도우)**를 광고: "내 버퍼에 이만큼 여유 있음"
- 송신자는 rwnd를 넘지 않게 전송
- 수신자가 느리면(앱이 안 읽음) rwnd 줄어듦 → 송신 속도 낮춤
- **수신자 속도에 맞추는 것** (네트워크가 아니라)

## 흐름 제어 vs 혼잡 제어 (혼동 주의)

둘 다 전송 속도를 제한하지만 **보호 대상이 다름**:

| | 흐름 제어 | 혼잡 제어 |
|---|---|---|
| 보호 대상 | **수신자** (버퍼) | **네트워크** (링크) |
| 신호 | rwnd (수신자 광고) | 패킷 손실/지연 |
| 이 노트 | ✓ | → [[congestion-control]] |

송신 윈도우 = min(rwnd, cwnd) - 수신자 한계와 네트워크 한계 중 작은 것. 혼잡 제어(cwnd)는 별도 → [[congestion-control]].

## 종합: 신뢰성의 조각들

```
시퀀스 번호  → 순서, 중복 제거
ACK          → 수신 확인
타임아웃/재전송 → 손실 복구
빠른 재전송   → 빠른 손실 감지
슬라이딩 윈도우 → 처리량
흐름 제어     → 수신자 보호
(혼잡 제어)   → 네트워크 보호 → [[congestion-control]]
```

이 조각들이 합쳐져 "신뢰 없는 IP 위의 신뢰성 있는 바이트 스트림".

## 셀프 체크

> [!question]- 누적 ACK(cumulative ACK)가 "번호 N까지 다 받았다"는 뜻이라면, 수신자가 순서 어긋난 패킷을 받았을 때 어떤 ACK를 보내고 그게 왜 빠른 재전송의 신호가 되는가?
> 수신자는 여전히 "아직 N을 기다린다"는 같은 ACK를 반복해 보낸다. 순서 어긋난 패킷이 올 때마다 이 중복 ACK가 쌓이고, 송신자가 같은 중복 ACK를 3개 받으면 N이 손실됐다고 판단해 타임아웃을 기다리지 않고 즉시 재전송한다. 이것이 fast retransmit이다.

> [!question]- stop-and-wait 대신 슬라이딩 윈도우를 쓰면 처리량이 왜 올라가는가?
> stop-and-wait은 패킷 하나를 보내고 ACK를 받아야 다음을 보내므로 매 패킷마다 1 RTT를 낭비한다. 슬라이딩 윈도우는 ACK를 기다리지 않고 윈도우 크기만큼 여러 패킷을 미리 보내 파이프를 채우므로, 대역폭-지연 곱에 가깝게 링크를 활용해 처리량이 올라간다.

> [!question]- 흐름 제어와 혼잡 제어는 둘 다 전송 속도를 제한하는데 무엇이 다른가?
> 흐름 제어는 수신자의 버퍼를 보호하며 신호는 수신자가 광고하는 rwnd다. 혼잡 제어는 네트워크(링크)를 보호하며 신호는 패킷 손실/지연이다. 송신 윈도우는 min(rwnd, cwnd)로, 수신자 한계와 네트워크 한계 중 작은 쪽을 따른다.

> [!question]- RTO(재전송 타임아웃)를 고정값이 아니라 RTT 측정으로 동적 조정하는 이유는?
> RTO가 너무 짧으면 아직 도착 중인 패킷을 손실로 오해해 불필요한 재전송을 하고, 너무 길면 실제 손실 복구가 느려진다. 그래서 측정한 RTT의 이동 평균과 변동성(Jacobson 알고리즘)으로 RTO를 상황에 맞게 조정한다.

## 연습문제

> [!example]- 문제: RTT 40ms, 링크 대역폭 100Mbps인 경로에서 파이프를 채우려면 최소 윈도우 크기(바이트)가 얼마여야 하는지 대역폭-지연 곱으로 계산하라.
> **풀이**
> 대역폭-지연 곱 = 대역폭 x RTT.
> 100Mbps = 100 x 10^6 bit/s, RTT = 0.04s.
> 곱 = 100 x 10^6 x 0.04 = 4 x 10^6 bit = 4,000,000 bit.
> 바이트로 = 4,000,000 / 8 = 500,000 byte ≈ 500KB.
> 즉 링크를 꽉 채우려면 ACK를 기다리지 않고 약 500KB를 실어 보낼 수 있어야 한다. 기본 TCP 헤더의 16비트 윈도우 필드는 최대 64KB뿐이라, 이런 고대역폭 경로에서는 윈도우 스케일링이 필요하다.

> [!example]- 문제: 송신자가 시퀀스 1000부터 각 1000바이트 세그먼트 5개(seq 1000, 2000, 3000, 4000, 5000)를 보냈고 seq 3000이 손실됐다. 수신자가 보내는 ACK 번호 흐름과 fast retransmit이 언제 트리거되는지 추적하라.
> **풀이**
> - seq 1000 도착 → ACK 2000 (다음 기대 바이트)
> - seq 2000 도착 → ACK 3000
> - seq 3000 손실
> - seq 4000 도착(순서 어긋남) → ACK 3000 (중복 1)
> - seq 5000 도착(순서 어긋남) → ACK 3000 (중복 2)
> 여기서 원래 3개의 중복 ACK가 쌓이면 fast retransmit. 이 예에서는 중복 ACK가 2개까지만 생겨 임계값(3)에 못 미치므로, 실제로는 추가 세그먼트가 더 오거나 타임아웃으로 seq 3000을 재전송하게 된다. 핵심은 ACK 번호가 손실 지점(3000)에 고정되고, 그 지점의 중복 카운트가 손실 신호가 된다는 점이다.
