# IP 주소 (IP Addressing)

## 한 줄 요약

인터넷의 주소 체계. IPv4는 32비트(고갈됨), IPv6는 128비트. 서브넷과 CIDR로 주소를 계층적으로 나누고, NAT로 사설 주소를 공유해 IPv4 고갈을 버텼다.

## 왜 필요한가

- 패킷이 어디로 가야 하나의 기준
- IPv4가 왜 부족한가, NAT가 어떻게 버티나
- 서브넷·CIDR이 라우팅과 어떻게 연결되나

## IPv4: 32비트

```
192.168.1.1  = 11000000.10101000.00000001.00000001
               4개 옥텟(8비트씩) = 32비트
```

- 총 2³² ≈ **43억 개** 주소
- 문제: 인터넷 성장으로 **고갈** (2011년 IANA 소진). 43억은 기기 수보다 적음
- 각 패킷 헤더에 출발/목적 IP → 라우터가 목적 IP로 경로 결정 → [[routing]]

## 서브넷과 CIDR

IP를 **네트워크 부분 + 호스트 부분**으로 나눔:

```
192.168.1.0/24
  /24 = 앞 24비트가 네트워크, 나머지 8비트가 호스트
  → 192.168.1.0 ~ 192.168.1.255 (256개, 호스트 254개)
```

- **CIDR**(Classless Inter-Domain Routing): `/n`으로 네트워크 크기 유연하게 (옛 클래스 A/B/C 대체)
- **서브넷 마스크**: 어디까지가 네트워크인지 (255.255.255.0 = /24)
- **집약(aggregation)**: 라우터가 `192.168.0.0/16`처럼 여러 서브넷을 하나로 → 라우팅 테이블 축소 ([[routing]])

CIDR이 라우팅을 계층적으로 → longest prefix match로 경로 결정 (data-structures/[[tries]]의 radix tree).

## NAT: IPv4 고갈 버티기

**NAT(Network Address Translation)**: 여러 기기가 **하나의 공인 IP를 공유**:

```
집 안: 192.168.1.x (사설 IP, 여러 기기)
          ↓ NAT 라우터
인터넷: 하나의 공인 IP (203.0.113.5)
```

- **사설 IP 대역**(RFC 1918): `10.x`, `172.16-31.x`, `192.168.x` - 인터넷에서 안 보임, 내부에서만
- NAT가 **포트로 구분**: 내부 여러 연결을 (공인IP:다른포트)로 매핑 → 응답을 올바른 내부 기기로
- 효과: 43억 공인 IP로 수백억 기기 수용 (각 가정/회사가 IP 하나 공유)

### NAT의 문제

- **외부에서 내부로 연결 불가** (내부 IP가 안 보임) → P2P·서버 어려움
- **홀 펀칭(hole punching)**: 양쪽이 동시에 나가 NAT 매핑을 뚫음 (WebRTC, 게임)
- 계층 위반 (전송 계층 포트를 네트워크 계층이 봄)

## IPv6: 근본 해결

**128비트** 주소:

```
2001:0db8:85a3::8a2e:0370:7334  (16비트씩 8그룹, 16진수)
```

- 2¹²⁸ ≈ **340간(澗)** 개 - 사실상 무한 (모래알마다 IP 줘도 남음)
- NAT 불필요 (모든 기기가 공인 IP) → 종단 간 연결 복원
- 자동 설정(SLAAC), 헤더 단순화, IPsec 통합
- 문제: 전환이 느림 (IPv4와 호환 안 됨, 병행 운영). 30년 걸쳐 점진 도입 중

## 특수 주소

- **127.0.0.1** (localhost): 자기 자신 (루프백)
- **0.0.0.0**: 모든 인터페이스 / 미지정
- **255.255.255.255**: 브로드캐스트
- **169.254.x** (IPv4), **fe80::** (IPv6): 링크 로컬 (자동 할당)

## 패킷 여정에서의 위치

```
앱 데이터 → TCP/UDP (포트) → IP (주소) → 링크 (MAC)
                             ↑ 여기
IP가 "어느 호스트로"를 담당, 라우터가 IP로 경로 결정 → [[routing]]
```

## 셀프 체크

> [!question]- `/24` 서브넷에서 총 주소 수와 실제 할당 가능한 호스트 수가 왜 다른가?
> `/24`는 앞 24비트가 네트워크, 나머지 8비트가 호스트이므로 총 2^8 = 256개 주소다. 하지만 첫 주소(네트워크 주소)와 마지막 주소(브로드캐스트)는 예약되어 실제 할당 가능한 호스트는 254개다.

> [!question]- NAT는 여러 내부 기기가 하나의 공인 IP를 공유하는데, 응답을 올바른 기기로 어떻게 되돌려 보내나?
> NAT 라우터가 포트로 구분한다. 내부의 여러 연결을 각각 (공인IP:서로 다른 포트)로 매핑해두고, 외부에서 응답이 오면 그 포트를 보고 어느 내부 기기의 연결인지 찾아 되돌린다. 이는 전송 계층 포트를 네트워크 계층이 보는 계층 위반이다.

> [!question]- CIDR의 집약(aggregation)과 longest prefix match가 라우팅 테이블에 주는 이점은?
> 집약은 `192.168.0.0/16`처럼 여러 서브넷을 하나의 프리픽스로 묶어 라우팅 테이블 항목 수를 줄인다. 라우터는 목적지 IP에 대해 가장 긴 프리픽스가 매칭되는 항목(longest prefix match)으로 경로를 정하므로, 큰 집약과 예외적 세부 경로가 공존할 수 있다.

> [!question]- IPv6가 128비트로 NAT를 불필요하게 만드는데도 전환이 느린 이유는?
> IPv6는 IPv4와 호환되지 않아 한쪽만 IPv6여서는 통신이 안 되고, 양쪽 인프라·기기가 모두 지원해야 한다. 그래서 듀얼 스택 등으로 IPv4와 병행 운영하며 30년에 걸쳐 점진적으로 도입 중이고, NAT가 IPv4 고갈을 버텨준 것도 전환 압력을 늦췄다.

## 연습문제

> [!example]- 문제: `192.168.1.0/24` 네트워크를 4개의 동일한 서브넷으로 나누어라. 각 서브넷의 CIDR 표기, 네트워크 주소, 사용 가능 호스트 수를 구하라.
> **풀이**
> 4개로 나누려면 2비트가 더 필요하다(2^2 = 4). 따라서 `/24`에서 `/26`으로 확장한다. `/26`은 호스트 비트가 6개 → 서브넷당 2^6 = 64개 주소, 사용 가능 호스트는 62개(네트워크·브로드캐스트 제외).
> - 서브넷 1: `192.168.1.0/26` (0 ~ 63)
> - 서브넷 2: `192.168.1.64/26` (64 ~ 127)
> - 서브넷 3: `192.168.1.128/26` (128 ~ 191)
> - 서브넷 4: `192.168.1.192/26` (192 ~ 255)
> 각 서브넷 사용 가능 호스트 62개, 총 248개.

> [!example]- 문제: 라우팅 테이블에 `10.0.0.0/8 → A`, `10.1.0.0/16 → B`, `10.1.2.0/24 → C` 세 항목이 있다. 목적지 `10.1.2.55`와 `10.1.5.7`은 각각 어느 경로로 나가나? longest prefix match로 판단하라.
> **풀이**
> longest prefix match는 매칭되는 것 중 프리픽스가 가장 긴 항목을 고른다.
> `10.1.2.55`: `10.0.0.0/8`(매칭), `10.1.0.0/16`(매칭), `10.1.2.0/24`(매칭) 모두 해당 → 가장 긴 `/24` 선택 → **C**.
> `10.1.5.7`: `10.0.0.0/8`(매칭), `10.1.0.0/16`(매칭), `10.1.2.0/24`(불일치, 세 번째 옥텟 2≠5) → 가장 긴 `/16` 선택 → **B**.
> 이렇게 세부 경로(/24)와 집약 경로(/8, /16)가 공존하며 가장 구체적인 규칙이 우선한다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지. (1) CIDR `/n` 표기로 네트워크/호스트 부분을 나누고 주소 개수·호스트 수를 계산할 수 있나. (2) NAT가 포트 매핑으로 하나의 공인 IP를 공유하는 원리와 그 한계(외부→내부 연결 불가)를 설명할 수 있나. (3) IPv4 고갈에 대한 NAT(임시)와 IPv6(근본) 해법을 대비하고 IPv6 전환이 느린 이유를 말할 수 있나.
