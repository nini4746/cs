# QUIC

## 한 줄 요약

UDP 위에 TCP의 신뢰성 + TLS 암호화 + 멀티플렉싱을 유저공간에서 재구현한 프로토콜. TCP의 head-of-line blocking과 느린 연결 설정을 해결하며, HTTP/3의 기반이다.

## 왜 필요한가

- TCP의 근본 한계를 왜 못 고치나
- HTTP/3가 왜 UDP 위에 서나
- 연결 마이그레이션이 뭔가

## TCP의 고착 문제

TCP는 개선하고 싶어도 못 바꿈:

1. **커널 구현**: TCP는 OS 커널에 있음 → 바꾸려면 전 세계 OS 업데이트 (수년~불가)
2. **미들박스 고착(ossification)**: 방화벽·NAT가 TCP를 특정 방식으로 검사 → 새 TCP 기능이 통과 못 함
3. **head-of-line blocking**: TCP의 순서 보장이 근본적 → HTTP/2 멀티플렉싱도 TCP 레벨에선 못 피함 ([[tcp-basics]], [[http]])

QUIC의 전략: **UDP 위에 유저공간에서** 새로 구현 → 커널·미들박스 우회, 앱 업데이트로 진화.

## QUIC = UDP + 재구현

QUIC은 UDP([[udp]]) 위에 TCP의 좋은 점을 얹음:

- **신뢰성**: 재전송, 순서 (스트림별) → [[tcp-reliability]]
- **혼잡 제어**: → [[congestion-control]]
- **TLS 1.3 내장**: 암호화가 프로토콜의 일부 (선택 아님) → security/[[tls-deep-dive]]
- **멀티플렉싱**: 여러 스트림 (아래)
- 전부 **유저공간 라이브러리**로 → 앱이 업데이트하면 즉시 개선

## 핵심 1: 스트림별 독립 (HOL blocking 해결)

TCP의 근본 문제 해결:

```
TCP: 한 연결 = 하나의 바이트 스트림
     패킷3 손실 → 스트림 전체가 3을 기다림 (HTTP/2도 이걸 못 피함)

QUIC: 한 연결 = 여러 독립 스트림
      스트림A 패킷 손실 → 스트림B,C는 계속 진행 (독립!)
```

- 각 스트림이 자기 순서만 관리 → 한 스트림의 손실이 다른 스트림 안 막음
- HTTP/3가 이걸로 진짜 멀티플렉싱 달성 → [[http]]

## 핵심 2: 빠른 연결 설정

TCP는 핸드셰이크(1 RTT, [[tcp-basics]]) + TLS 핸드셰이크(1~2 RTT) = 2~3 RTT:

- **QUIC**: 연결 + 암호화를 **한 번에** (TLS 1.3 통합) → **1 RTT**
- **0-RTT**: 이전에 연결했던 서버면 **0 RTT** (첫 패킷에 데이터) - 재접속 즉시
- 모바일·고지연 환경에서 큰 이득 (RTT 절약)

## 핵심 3: 연결 마이그레이션

TCP 연결은 **4-tuple(출발IP:포트, 목적IP:포트)로 식별** → IP 바뀌면 끊김:

- WiFi → LTE 전환 시 TCP 연결 재설정 필요 (IP 바뀜)
- QUIC은 **연결 ID**로 식별 (IP 아님) → **IP 바뀌어도 연결 유지**
- 모바일에서 네트워크 전환해도 다운로드/통화 안 끊김

## QUIC vs TCP

| | TCP | QUIC |
|---|---|---|
| 기반 | IP 직접 | UDP 위 |
| 위치 | 커널 | 유저공간 |
| HOL blocking | 있음 | 스트림별 해결 |
| 암호화 | 별도 TLS | 내장 |
| 연결 설정 | 2~3 RTT | 1 RTT (0-RTT 가능) |
| 마이그레이션 | ✗ | ✓ (연결 ID) |
| 진화 | 느림 (커널) | 빠름 (앱) |

## 대가와 논란

- **CPU 비용**: 유저공간 처리 + 암호화 → TCP보다 CPU 더 씀 (커널 오프로드 없음)
- **UDP 차단**: 일부 네트워크가 UDP를 제한/느리게 → QUIC 실패 시 TCP로 폴백
- **관측성**: 암호화가 많아 네트워크 디버깅 어려움
- 그래도 구글·주요 CDN([[cdn]])이 광범위 배포 → HTTP/3 채택 증가

## 셀프 체크

> [!question]- QUIC은 왜 TCP를 고치지 않고 UDP 위에 새로 구현했나?
> TCP는 OS 커널에 있어 바꾸려면 전 세계 OS 업데이트가 필요하고(수년~불가), 방화벽·NAT 같은 미들박스가 TCP를 특정 방식으로 검사해 새 기능이 통과 못 하는 고착(ossification) 문제가 있다. QUIC은 UDP 위 유저공간 라이브러리로 구현해 커널·미들박스를 우회하고 앱 업데이트만으로 진화한다.

> [!question]- QUIC의 스트림별 독립이 TCP의 head-of-line blocking을 어떻게 해결하나?
> TCP는 한 연결이 하나의 바이트 스트림이라 앞 패킷 하나가 손실되면 뒤가 도착해도 순서 때문에 전부 대기한다(HTTP/2도 못 피함). QUIC은 한 연결에 여러 독립 스트림을 두어 각 스트림이 자기 순서만 관리하므로, 스트림 A의 손실이 스트림 B·C의 진행을 막지 않는다.

> [!question]- 연결 마이그레이션이 무엇이고, TCP는 왜 못 하나?
> TCP 연결은 4-tuple(출발IP:포트, 목적IP:포트)로 식별되므로 WiFi→LTE처럼 IP가 바뀌면 연결이 끊긴다. QUIC은 IP가 아닌 연결 ID로 식별하므로 네트워크가 바뀌어도 같은 연결을 유지한다 - 이것이 연결 마이그레이션이다.

> [!question]- QUIC의 연결 설정이 TCP보다 몇 RTT 빠른가, 0-RTT는 언제 가능한가?
> TCP는 핸드셰이크 1 RTT + TLS 1~2 RTT로 총 2~3 RTT가 드는데, QUIC은 연결과 TLS 1.3 암호화를 한 번에 묶어 1 RTT에 끝낸다. 이전에 연결했던 서버라면 첫 패킷에 데이터를 실어 0-RTT로 즉시 재접속할 수 있다.

## 연습문제

> [!example]- 문제: RTT가 80ms인 모바일 경로에서 최초 페이지 로드에 TCP+TLS 1.3과 QUIC의 연결 설정 지연을 각각 계산하고, 0-RTT 재접속 시 QUIC의 이득을 구하라.
> **풀이**
> TCP+TLS 1.3: TCP 핸드셰이크 1 RTT + TLS 1.3 1 RTT = 2 RTT = 160ms 뒤에 첫 요청 전송 가능.
> QUIC(초기 연결): 연결+암호화 통합 1 RTT = 80ms.
> 절감액 = 160 - 80 = 80ms (1 RTT).
> QUIC 0-RTT 재접속: 0 RTT이므로 첫 패킷에 바로 데이터 → 설정 지연 0ms, TCP+TLS 대비 160ms 절약. 고지연일수록 이득이 커진다.

> [!example]- 문제: HTTP/3로 한 연결에서 이미지 10개를 병렬로 받는 중 3번째 이미지 스트림의 패킷 하나가 손실됐다. 나머지 이미지 전송은 어떻게 되는지 TCP 기반 HTTP/2와 비교해 설명하라.
> **풀이**
> HTTP/2(TCP): 단일 바이트 스트림이라 손실된 세그먼트가 재전송돼 도착할 때까지 그 뒤 모든 데이터가 순서 보장 때문에 앱에 전달되지 못해 10개 전부 지연된다(HOL blocking).
> HTTP/3(QUIC): 3번 이미지 스트림만 손실 세그먼트를 기다리고 나머지 9개 스트림은 독립적으로 계속 앱에 전달된다. 손실 영향이 해당 스트림에 국한되므로 체감 지연이 크게 준다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: (1) TCP 고착(커널·미들박스) 문제와 QUIC이 유저공간+UDP로 그것을 우회하는 방식, (2) 스트림별 독립이 HOL blocking을 없애는 원리, (3) 연결 ID 기반 마이그레이션과 1-RTT/0-RTT 연결 설정을 설명할 수 있어야 한다.
