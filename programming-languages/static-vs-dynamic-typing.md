# 정적 vs 동적 타이핑 (Static vs Dynamic Typing)

## 한 줄 요약

타입을 언제 검사하나. 정적(컴파일 타임)은 실행 전 오류를 잡고, 동적(런타임)은 유연하다. "강/약 타입"은 별개 축 - 타입을 얼마나 엄격히 지키나. 넷을 혼동하기 쉽다.

## 왜 필요한가

- 타입 오류가 언제 잡히나 (컴파일 vs 실행)
- Python은 왜 실행해봐야 타입 오류를 아나
- "강타입/약타입"과 "정적/동적"의 혼동 정리

## 정적 vs 동적: 검사 시점

핵심 차이 = **타입 검사를 언제** 하나:

- **정적(static)**: **컴파일 타임**에 타입 검사. 변수에 타입이 붙음. C, Java, Rust, Go, TypeScript
- **동적(dynamic)**: **런타임**에 타입 검사. 값에 타입이 붙고 변수는 아무거나. Python, JavaScript, Ruby

```
정적(Java): String s = 5;      // 컴파일 오류 - 실행 전에 잡힘
동적(Python): s = 5; s.upper()  // 런타임 오류 - 그 줄 실행해야 터짐
```

## 트레이드오프

| | 정적 | 동적 |
|---|---|---|
| 오류 시점 | 컴파일 (일찍) | 런타임 (늦게) |
| 안전성 | 실행 전 보장 | 실행해야 확인 |
| 유연성 | 제약 | 자유 (덕 타이핑) |
| 리팩토링 | 컴파일러가 도움 | 위험 (테스트 의존) |
| 성능 | 최적화 유리 (타입 앎) | 런타임 타입 조회 오버헤드 |
| 개발 속도 | 초기 타이핑 비용 | 빠른 프로토타이핑 |
| 도구 | 자동완성·검증 강력 | 제한적 |

- 정적: 대규모·장수명 코드베이스 (오류 조기 발견, 도구 지원). 큰 팀
- 동적: 스크립트·프로토타입·빠른 반복. 하지만 커지면 유지보수 어려움 → TypeScript 같은 점진적 타이핑 등장

## 강 vs 약: 별개의 축 (혼동 주의)

"강타입/약타입"은 정적/동적과 **다른 축**. **타입을 얼마나 엄격히 지키나** (암묵적 변환 허용 정도):

- **강타입(strong)**: 암묵적 타입 변환 거의 안 함. `"1" + 1`이 오류
- **약타입(weak)**: 암묵적 변환 관대. `"1" + 1`이 `"11"` 또는 `2`

```
Python(강, 동적): "1" + 1  → TypeError (변환 안 함)
JavaScript(약, 동적): "1" + 1 → "11" (문자열로 강제)
C(약, 정적): 포인터를 int로 캐스팅 등 관대
```

**네 조합 다 존재:**

| | 강타입 | 약타입 |
|---|---|---|
| 정적 | Java, Rust | C (암묵 변환 관대) |
| 동적 | Python, Ruby | JavaScript, PHP |

흔한 오해: "정적=강, 동적=약"은 **틀림**. Python은 동적인데 강타입. C는 정적인데 약타입.

## 타입 추론 (type inference)

정적 타입인데 **타입을 안 적어도** 컴파일러가 추론:

```
Rust:  let x = 5;          // x가 i32임을 추론
Haskell: 함수 타입 전체 추론 (Hindley-Milner)
```

- 정적의 안전성 + 동적의 간결함
- Rust, Haskell, ML, Swift, TypeScript, 최신 Java/C++ (`auto`, `var`)
- "타입을 명시 안 함 = 동적"이 아님. 추론된 정적 타입일 수 있음

## 점진적 타이핑 (gradual typing)

동적 언어에 정적 타입을 **선택적으로** 추가:

- **TypeScript** (JavaScript), **Python type hints + mypy**, Ruby Sorbet
- 동적의 유연함 + 정적의 안전함을 부분적으로
- 큰 동적 코드베이스가 유지보수를 위해 도입하는 추세
- 런타임엔 여전히 동적 (타입 힌트는 검사 도구용, 실행엔 무시되기도)

## 실행에 미치는 영향

- 정적: 컴파일러가 타입을 알아 **최적화** (인라인, 레이아웃 → computer-architecture/[[data-layout]], 디스패치 정적 결정)
- 동적: 런타임에 타입 조회 → 오버헤드. **JIT**가 실행 중 타입을 관찰해 최적화(V8) → [[compiled-vs-interpreted]]

## 셀프 체크

> [!question]- 정적 타이핑과 동적 타이핑을 가르는 핵심 차이는?
> 타입 검사를 언제 하느냐다. 정적은 컴파일 타임(변수에 타입이 붙음, 실행 전 오류 발견), 동적은 런타임(값에 타입이 붙고 변수는 아무거나, 그 줄을 실행해야 오류가 터짐).

> [!question]- "강/약 타입"은 "정적/동적"과 왜 별개의 축인가?
> 정적/동적은 "검사 시점", 강/약은 "타입을 얼마나 엄격히 지키나(암묵적 변환 허용 정도)"를 가리킨다. 서로 독립이라 네 조합이 모두 존재한다.

> [!question]- Python이 동적인데 강타입이라는 게 무슨 뜻인가?
> 타입 검사는 런타임에 하지만(동적) 암묵적 변환을 거의 안 한다(강). `"1" + 1`이 자동 변환 없이 TypeError를 낸다. 반대로 C는 정적이지만 약타입이다.

> [!question]- 타입을 안 적었다고 동적 타입인가?
> 아니다. Rust `let x = 5`나 Haskell처럼 컴파일러가 타입을 추론(inference)한 정적 타입일 수 있다. 명시하지 않았을 뿐 컴파일 타임에 타입이 확정된다.

## 연습문제

> [!example]- 문제: 아래 언어들을 (정적/동적)×(강/약) 네 칸에 배치하라 - Java, C, Python, JavaScript, Rust, Ruby, PHP.
> 
> **풀이**
> 
> - 정적 · 강: Java, Rust
> - 정적 · 약: C (포인터↔정수 캐스팅 등 암묵 변환 관대)
> - 동적 · 강: Python, Ruby
> - 동적 · 약: JavaScript, PHP
> 
> 흔한 오해 "정적=강, 동적=약"은 틀렸다. C(정적·약)와 Python(동적·강)이 반례다.

> [!example]- 문제: `"1" + 1`을 Python·JavaScript·C 관점에서 각각 어떻게 처리하는지 추적하라.
> 
> **풀이**
> 
> - Python(강, 동적): 런타임에 str+int 암묵 변환을 거부 → TypeError.
> - JavaScript(약, 동적): 1을 문자열로 강제 변환 → `"11"`.
> - C(약, 정적): 문자열 리터럴은 포인터라 `"1" + 1`은 포인터 산술(주소+1)로 조용히 해석 - 약타입의 관대함이 다른 방향의 함정을 만든다.
> 
> 같은 식이 강/약 축에 따라 오류/강제변환/포인터연산으로 갈린다.

## 파인만

> [!note]- 가로축 "검사 시점(정적/동적)", 세로축 "엄격함(강/약)"인 2×2 격자를 그리고 각 칸에 언어를 채운 뒤, 왜 두 축이 독립인지 남에게 설명하듯 써보라. 막히면 그 칸만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지 - (1) "검사 시점"과 "엄격함"의 구분, (2) Python이 강타입인 근거, (3) 타입 추론이 왜 동적이 아닌가.
