# 파인튜닝 (Fine-tuning)

## 한 줄 요약

사전학습된 모델의 가중치를 추가 데이터로 마저 조정해 특정 행동·형식·도메인에 맞추는 것. 핵심 원칙은 **지식은 RAG, 행동은 파인튜닝, 먼저 프롬프팅**. LoRA는 전체가 아니라 작은 저랭크 행렬만 학습해 비용을 급감시킨 대표 기법. 언제 쓸지 고르는 기준이 실무의 요점.

## 왜 필요한가

- 파인튜닝 vs RAG vs 프롬프팅 언제 무엇을
- LoRA가 뭔가 (왜 싸게 되나)
- 파인튜닝의 함정

## 파인튜닝이란

```
사전학습 모델([[llm-training]]) → 내 데이터로 추가 학습 → 가중치 조정
```

- [[llm-training]]의 SFT와 같은 메커니즘 (베이스 모델을 특정 용도로)
- **가중치를 바꿈**(프롬프팅과 근본 차이) → 행동이 모델에 각인
- 목적: 일관된 형식·스타일·톤, 도메인 특화, 반복 작업 자동화

## 세 방법 비교 (핵심)

무엇을 바꾸려는가로 선택:

```
프롬프팅: 문맥만 (가중치 X)     - 빠름·싸다·즉시 → 먼저 시도 [[prompting]]
RAG:      외부 지식 검색·주입    - 최신·비공개 지식 → [[rag]]
파인튜닝:  가중치 조정          - 행동·형식·스타일 각인
```

### 지식 vs 행동 (제일 중요한 구분)

```
지식(사실·최신·비공개) → RAG ✅  (파인튜닝으로 지식 주입은 비효율·환각 ❌)
행동(형식·톤·스타일·태스크 패턴) → 파인튜닝 ✅
```

- **흔한 실수**: "우리 문서 알게 하려고 파인튜닝" → 대개 틀림. **지식은 RAG**가 나음 (갱신 쉽고 환각↓)
- 파인튜닝은 "**어떻게** 답하나"(행동)에, RAG는 "**무엇을** 아나"(지식)에

### 선택 순서

```
1. 프롬프팅으로 되나? (few-shot, CoT) → 되면 끝 (제일 싸다)
2. 지식이 부족? → RAG
3. 형식·스타일이 프롬프트로 불안정하거나 대량 반복? → 파인튜닝
4. 조합도 가능 (파인튜닝 + RAG)
```

## LoRA (Low-Rank Adaptation)

전체 파인튜닝은 비쌈(수십억 파라미터 다 갱신·저장) → LoRA가 해결:

```
전체 파인튜닝: 모든 가중치 W 갱신 (막대한 메모리·저장)
LoRA: W는 고정, 작은 저랭크 행렬 ΔW = A·B 만 학습
      (A: d×r, B: r×d, r은 아주 작음 → 파라미터 급감)
```

- **왜 되나**: 파인튜닝의 가중치 변화가 사실 **저랭크**(낮은 차원)라는 관찰 → 작은 행렬로 근사
- **장점**:
  - 학습 파라미터 수백~수천 배↓ (메모리·비용 급감, GPU 하나로도)
  - 원본 모델 공유 + **작은 어댑터만 교체** (태스크별 어댑터 스왑)
  - 원본 안 건드려 파괴적 망각 완화
- **QLoRA**: 양자화까지 더해 더 싸게 (소비자 GPU에서도)
- PEFT(파라미터 효율 파인튜닝)의 대표

## 함정·주의

- **파괴적 망각(catastrophic forgetting)**: 새 데이터에 치우쳐 기존 능력 상실 → 데이터 균형·낮은 학습률 ([[training-dynamics]])
- **데이터 품질**: 적어도 좋은 데이터가 많은 나쁜 데이터보다 (garbage in garbage out)
- **과적합**: 작은 데이터셋에 과적합 ([[ml-fundamentals]]) → 검증셋 필수
- **비용·유지보수**: 모델 업데이트마다 재파인튜닝, 평가 파이프라인 필요 ([[llm-evaluation]])
- **먼저 프롬프팅 다 해봤나**: 파인튜닝은 마지막 수단인 경우 많음

## 왜 중요한가

- 프롬프팅으로 안 되는 **일관성·특화**를 얻는 길
- LoRA로 **접근성** 폭발 (누구나 파인튜닝)
- 하지만 **RAG·프롬프팅과 역할 구분**이 핵심 - 오용하면 비싸고 나쁨

## 연결

- 과적합·검증 → [[ml-fundamentals]]
