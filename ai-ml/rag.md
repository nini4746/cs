# 검색 증강 생성 (RAG)

## 한 줄 요약

LLM에게 답하기 전에 **관련 문서를 검색해 문맥으로 넣어주는** 구조. 최신·비공개·사실 지식을 파인튜닝 없이 주입하고 환각을 줄인다. 검색(임베딩 유사도)→증강(프롬프트에 삽입)→생성(LLM) 세 단계. 실패는 대부분 생성이 아니라 **검색 단계**에서 난다.

## 왜 필요한가

- LLM의 지식 한계(최신·비공개·환각)를 어떻게 넘나
- RAG 구조와 청킹 전략
- 왜 RAG가 자주 실패하나

## 왜 RAG인가

LLM의 한계 ([[prompting]]):
```
- 학습 시점 이후 모름 (최신 X)
- 회사 내부 문서 모름 (비공개)
- 모르는 걸 지어냄 (환각)
```

RAG의 해법:
```
질문 → 관련 문서 검색 → "이 문맥을 근거로 답해" → LLM
```

- **파인튜닝 대신**: 지식을 가중치에 넣지 않고 **검색으로 주입** (싸고 즉시 갱신)
- 출처 제시 가능 (근거 문서 인용) → 검증·신뢰

## 파이프라인

### 사전 준비 (인덱싱)

```
문서 → 청킹(조각내기) → 임베딩([[embeddings]]) → 벡터 DB 저장
```

### 질의 시

```
1. 검색(Retrieval): 질문 임베딩 → 벡터 DB 최근접 청크 ([[embeddings]]의 유사도)
2. 증강(Augmentation): 검색된 청크를 프롬프트에 삽입 ([[prompting]])
3. 생성(Generation): LLM이 그 문맥 근거로 답 ([[llm-training]])
```

## 청킹 전략 (중요)

문서를 어떻게 자르나가 검색 품질 좌우:

- **크기 절충**:
  - 너무 작음: 문맥 부족 (조각이 의미 불완전)
  - 너무 큼: 노이즈↑, 토큰 낭비([[tokenization]]), 임베딩 흐려짐
- **의미 단위로**: 문단·섹션 경계 존중 (문장 중간 자르지 말기)
- **오버랩**: 청크 간 겹침 → 경계 걸친 정보 손실 방지
- **메타데이터**: 출처·제목·날짜 붙여 필터링·인용

## 검색 개선

단순 벡터 검색만으론 부족 → 실무 기법:

- **하이브리드 검색**: 벡터(의미) + 키워드(BM25, database/[[full-text-search]]) 결합 → 정확한 용어·의미 둘 다
- **재순위(rerank)**: 후보 많이 뽑고 → 크로스인코더로 정밀 재정렬 (2단계)
- **쿼리 변환**: 질문을 검색 친화적으로 재작성·확장 (HyDE 등)
- **메타데이터 필터**: 날짜·카테고리로 좁힘

## 실패 모드 (대부분 검색에서)

RAG 문제의 상당수가 생성 아니라 **검색 단계**:

```
검색 실패 → 관련 없는/없는 문맥 → LLM이 잘 답할 수 없음
```

- **검색 못 함**: 관련 청크를 못 찾음 (청킹·임베딩·쿼리 문제) → 흔한 원인
- **관련 없는 청크**: 노이즈가 LLM 혼란
- **"lost in the middle"**: 문맥 많으면 중간 정보 무시 ([[prompting]])
- **모순 청크**: 서로 다른 정보 → 어느 걸 믿나
- **답이 문서에 없는데** LLM이 지어냄 (여전히 환각 가능) → "모르면 모른다고" 지시 필요
- 진단: 검색 결과를 먼저 확인 (좋은 청크가 왔나?) → [[llm-evaluation]]

## RAG vs 파인튜닝 vs 긴 문맥

```
RAG:      지식이 크고 자주 바뀜, 출처 필요 (지식 주입)
파인튜닝:  형식·스타일·행동 (지식 아니라 행동) → [[fine-tuning]]
긴 문맥:   문서 적으면 그냥 다 넣기 (문맥 창 크면 RAG 불필요할 때도)
```

- 흔한 오해: 파인튜닝으로 지식 주입 → 비효율·환각. **지식은 RAG가 대개 나음**
- 셋은 배타 아님 (조합 가능)

## 왜 중요한가

- **실무 LLM 앱의 표준** (챗봇, 문서 QA, 검색)
- 최신·비공개 지식 + 환각 감소 + 출처
- 에이전트의 도구 중 하나로도 ([[agents]]의 검색 도구)

## 셀프 체크

> [!question]- RAG는 LLM의 어떤 한계들을 어떻게 넘나?
> 학습 시점 이후 최신 정보, 회사 내부 비공개 문서, 환각(모르는 걸 지어냄)을 넘는다. 질문에 관련 문서를 검색해 문맥으로 넣고 "이 근거로 답하라"고 지시해, 지식을 가중치가 아닌 검색으로 주입하고 출처를 제시한다.

> [!question]- RAG 세 단계를 순서대로 설명하라.
> (1) 검색(Retrieval) - 질문을 임베딩해 벡터 DB에서 최근접 청크를 찾음 (2) 증강(Augmentation) - 검색된 청크를 프롬프트에 삽입 (3) 생성(Generation) - LLM이 그 문맥을 근거로 답. 사전에 문서를 청킹·임베딩해 벡터 DB에 인덱싱해 둔다.

> [!question]- 청크가 너무 작거나 너무 클 때 각각 무슨 문제가 생기나?
> 너무 작으면 문맥이 부족해 조각의 의미가 불완전하다. 너무 크면 노이즈가 늘고 토큰을 낭비하며 임베딩이 흐려져 검색 정확도가 떨어진다. 그래서 의미 단위(문단·섹션)로 자르고 오버랩을 준다.

> [!question]- RAG 실패의 상당수가 왜 검색 단계에서 나나?
> 검색이 관련 없는/없는 청크를 가져오면 LLM이 아무리 좋아도 잘 답할 수 없다(garbage in). 청킹·임베딩·쿼리 문제로 관련 청크를 못 찾는 게 흔한 원인이라, 진단은 검색 결과부터 확인한다.

## 연습문제

> [!example]- 문제: 사내 문서 QA 봇이 "환불 정책은?"에 엉뚱한 답을 한다. 검색 단계부터 진단하는 절차를 설계하라.
> **풀이**
> (1) 검색 결과부터 확인 - 상위 청크에 환불 정책 문단이 실제로 왔나? 안 왔으면 검색 문제다.
> (2) 원인 좁히기: 청킹(정책이 문장 중간에 잘렸나), 임베딩(질의-문서 의미 매칭 실패), 쿼리(용어 불일치).
> (3) 처방: 하이브리드 검색(키워드 BM25 결합)으로 "환불" 정확 매칭, 재순위로 정밀화, 의미 단위 청킹+오버랩.
> (4) 좋은 청크가 왔는데도 틀리면 생성 문제 → "문서에 없으면 모른다고 답하라" 지시 추가.

> [!example]- 문제: 파인튜닝으로 지식을 넣는 것과 RAG를 비교해, 자주 갱신되는 제품 매뉴얼 QA에 어느 쪽이 맞는지 논하라.
> **풀이**
> 매뉴얼은 자주 바뀌는 사실 "지식"이다. 파인튜닝으로 넣으면 갱신 때마다 재학습해야 하고 환각·비효율 위험이 크다.
> RAG는 매뉴얼을 벡터 DB에 인덱싱만 하면 되고, 문서가 바뀌면 인덱스만 갱신해 즉시 반영되며 출처 인용도 된다. 따라서 RAG가 맞다. 답변 형식·톤이 필요하면 파인튜닝을 조합할 수 있다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: (1) 검색→증강→생성 3단계와 사전 인덱싱 흐름 (2) 청킹 절충(크기·의미 단위·오버랩)이 검색 품질을 좌우하는 이유 (3) 실패가 주로 검색에서 나는 이유와 진단 순서.

## 연결

- 검색 = 임베딩 유사도 → [[embeddings]]
- 문맥 삽입 → [[prompting]]
- 청크 토큰 예산 → [[tokenization]]
- 하이브리드 키워드 검색 → database/[[full-text-search]]
- 지식 vs 행동 (파인튜닝 대비) → [[fine-tuning]]
- 검색 품질 평가 → [[llm-evaluation]]
- 에이전트 도구로 → [[agents]]
