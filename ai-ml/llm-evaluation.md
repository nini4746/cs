# LLM 평가 (LLM Evaluation)

## 한 줄 요약

LLM 평가가 어려운 이유는 **정답이 하나가 아니고**(열린 생성), 품질이 다차원(정확성·유용성·안전성·톤)이기 때문이다. 벤치마크는 오염·포화되고, 자동 지표(BLEU 등)는 의미를 못 잰다. 그래서 LLM-as-judge, 사람 평가, 태스크별 검증을 조합하되 각각의 함정을 안다.

## 왜 필요한가

- 왜 LLM 평가가 전통 ML보다 어려운가
- LLM-as-judge가 뭔가 (와 그 함정)
- 벤치마크를 왜 그대로 믿으면 안 되나

## 왜 어려운가

전통 ML 평가([[model-evaluation]])는 정답이 명확 → precision/recall. LLM은:

```
"에세이 써줘" → 정답이 무한, 좋고 나쁨이 주관적
"요약해줘" → 여러 좋은 요약 존재
```

- **정답 하나 아님**: 생성은 열린 공간
- **다차원 품질**: 사실성·유용성·일관성·안전성·형식·톤을 동시에
- **문맥 의존**: 같은 답도 상황 따라 좋고 나쁨
- 그래서 단일 숫자로 안 됨

## 자동 지표의 한계

### 전통 NLP 지표 (BLEU/ROUGE)

- 참조 텍스트와 **n-gram 겹침** 측정 (번역·요약)
- **문제**: 표면 단어만 봄 → 의미 같아도 표현 다르면 저점, 의미 달라도 단어 겹치면 고점
- 열린 생성엔 부적합 (참조가 유일 정답 아님)

### 태스크별 검증 (되는 경우)

정답이 명확한 작업은 직접 검증 가능:
- **코드**: 테스트 통과 여부 (software-design/[[testing-strategy]]) - 객관적
- **수학**: 답 일치
- **분류·추출**: precision/recall ([[model-evaluation]])
- 가능하면 이런 **검증 가능 지표**를 우선

## LLM-as-Judge

**LLM에게 LLM 출력을 평가시킴** (확장 가능한 자동 평가):

```
평가 LLM에게: "이 응답을 정확성·유용성 기준 1-10점 매겨줘"
또는: "응답 A와 B 중 어느 게 나은가?"
```

- **장점**: 사람보다 싸고 빠름, 의미 이해 (BLEU보다 나음), 대규모 가능
- **함정** (반드시 알아야):
  - **위치 편향**: 먼저 제시된 답을 선호 (A/B 순서 바꿔 검증)
  - **길이 편향**: 긴 답을 선호 (내용 무관)
  - **자기 선호**: 자기가 만든 스타일 선호
  - **아부(sycophancy)**: 자신감 있는 답에 후함
  - 완화: 기준 명시(루브릭), 순서 무작위화, 여러 판정 앙상블, 사람과 상관 검증

## 벤치마크와 그 한계

표준 벤치마크(MMLU, GSM8K, HumanEval 등)로 비교하지만:

- **데이터 오염(contamination)**: 벤치마크가 **학습 데이터에 유출** → 암기로 고점 (실력 아님) - 심각한 문제
- **포화**: 최신 모델이 만점 근접 → 변별력 상실
- **벤치마크 ≠ 실사용**: 시험 잘 봐도 실무 다름 (분포 차이)
- **굿하트 법칙**: 지표가 목표가 되면 지표를 게이밍 (벤치마크 맞춤 최적화)
- 대책: 비공개 홀드아웃, 최신 문제, 실사용 기반 평가 (LMArena 같은 사람 투표)

## 실무 평가 전략

```
1. 태스크별 검증 가능 지표 우선 (코드=테스트, 추출=정확도)
2. LLM-as-judge로 열린 생성 (편향 완화하며)
3. 사람 평가 (골드 스탠다드, 비싸니 표본)
4. 자기 평가셋 구축 (내 실사용 사례로, 오염 없는)
5. 회귀 방지: 프롬프트·모델 바꿀 때 평가셋 재실행 ([[ci-cd-principles]] 정신)
```

- **eval 구축이 LLM 앱 개발의 핵심** (프롬프트·RAG·에이전트 개선의 나침반)
- RAG는 검색·생성 나눠 평가 ([[rag]]), 에이전트는 단계·최종 결과 ([[agents]])

## 왜 중요한가

- 평가 없이는 **개선 방향을 모름** (좋아졌는지 판단 불가)
- 모델·프롬프트 선택의 근거
- [[model-evaluation]]의 "측정 못 하면 개선 못 함"이 LLM에서 더 절실 (더 어려우니)

## 셀프 체크

> [!question]- LLM 평가가 전통 ML 평가보다 어려운 근본 이유는?
> 열린 생성이라 정답이 하나가 아니고("에세이 써줘"는 무한한 좋은 답), 품질이 사실성·유용성·일관성·안전성·톤 등 다차원이며 문맥에 의존한다. 그래서 단일 숫자로 환원되지 않는다.

> [!question]- BLEU/ROUGE 같은 n-gram 지표의 한계는?
> 참조 텍스트와 표면 단어 겹침만 잰다. 의미가 같아도 표현이 다르면 저점, 의미가 달라도 단어가 겹치면 고점이 나온다. 참조가 유일 정답이 아닌 열린 생성엔 부적합하다.

> [!question]- LLM-as-judge의 대표 편향을 셋 들고 완화법을 말하라.
> 위치 편향(먼저 제시된 답 선호), 길이 편향(긴 답 선호), 자기 선호(자기 스타일 선호), 아부(자신감 있는 답에 후함) 등이 있다. 완화: 루브릭으로 기준 명시, A/B 순서 무작위화, 여러 판정 앙상블, 사람 평가와 상관 검증.

> [!question]- 벤치마크를 그대로 믿으면 안 되는 이유는?
> 데이터 오염(벤치마크가 학습 데이터에 유출 → 암기로 고점), 포화(만점 근접 → 변별력 상실), 실사용과의 분포 차이, 굿하트 법칙(지표가 목표가 되면 게이밍) 때문이다. 비공개 홀드아웃·최신 문제·실사용 평가로 보완한다.

## 연습문제

> [!example]- 문제: 두 요약 모델 A·B를 LLM-as-judge로 비교하니 항상 먼저 보여준 쪽이 이겼다. 원인과 신뢰할 실험 설계를 제시하라.
> **풀이**
> 원인: 위치 편향 - 판정 LLM이 먼저 제시된 응답을 선호한다. 순서가 승패를 결정하므로 결과가 무의미하다.
> 설계: 각 쌍을 (A먼저), (B먼저) 두 순서로 모두 평가하고, 두 번 다 이긴 경우만 승으로 집계(순서를 뒤집으면 판정이 바뀌면 무승부 처리). 추가로 루브릭 명시, 여러 판정 앙상블, 표본 일부는 사람과 상관 검증.

> [!example]- 문제: 사내 코드 생성 도우미의 평가 파이프라인을 설계하라. 어떤 부분은 자동 검증하고 어떤 부분은 judge/사람을 쓰나?
> **풀이**
> 검증 가능 부분 우선: 생성 코드에 단위 테스트를 돌려 통과 여부(객관), 컴파일·린트 통과, 정답이 명확한 태스크는 정확도.
> judge: 코드 설명·주석 품질, 가독성처럼 정답이 열린 부분은 LLM-as-judge(편향 완화하며).
> 사람: 골드 스탠다드로 표본만 검토. 그리고 실사용 사례로 오염 없는 자기 평가셋을 구축해, 프롬프트·모델을 바꿀 때마다 재실행해 회귀를 막는다.
