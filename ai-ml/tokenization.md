# 토큰화 (Tokenization)

## 한 줄 요약

LLM은 글자도 단어도 아닌 **토큰**(자주 쓰이는 하위단어 조각)을 다룬다. BPE는 빈번한 문자쌍을 반복 병합해 어휘를 만든다. 이 "단어도 글자도 아닌" 단위가 LLM의 여러 이상한 동작(글자 세기 실패, 숫자 계산 약점, 언어별 비용 차이)의 근본 원인이다.

## 왜 필요한가

- LLM이 텍스트를 어떻게 쪼개나
- BPE가 어떻게 어휘를 만드나
- 왜 LLM이 "strawberry의 r 개수"를 틀리나

## 왜 토큰인가

단어도 글자도 아닌 **중간 단위**를 쓰는 이유:

```
글자 단위: 어휘 작지만 시퀀스 너무 길다 (비효율)
단어 단위: 어휘 폭발 + 미등록 단어(OOV) 문제
하위단어(subword): 절충 - 자주 쓰는 건 통째로, 드문 건 조각으로
```

- "tokenization" → `token` + `ization` (드문 단어도 조각 조합으로 표현, OOV 없음)
- 어휘 크기(보통 3만~15만)와 시퀀스 길이의 균형

## BPE (Byte-Pair Encoding)

가장 널리 쓰이는 방식 - **빈번한 인접 쌍을 반복 병합**:

```
1. 텍스트를 문자로 시작
2. 가장 자주 나오는 인접 쌍을 찾아 병합 → 새 토큰
3. 목표 어휘 크기까지 반복
```

## 코드로 확인

"low/lower/lowest" 코퍼스에 BPE 적용:

```python
# 가장 빈번한 인접 쌍을 반복 병합
best = pairs.most_common(1)[0][0]   # 최빈 쌍
words = [w.replace(" ".join(best), "".join(best)) for w in words]
```

실행:
```
merge 1: ('l', 'o') -> 'lo'     words=['lo w', 'lo w e r', ...]
merge 2: ('lo', 'w') -> 'low'   words=['low', 'low e r', ...]
merge 3: ('low', 'e') -> 'lowe'
merge 4: ('lowe', 'r') -> 'lower'
학습된 병합 규칙: [('l','o'), ('lo','w'), ('low','e'), ('lowe','r')]
```

- 빈번한 조각(`low`)이 통째 토큰으로 자라남 (자주 나오니 병합)
- 병합 규칙 순서가 곧 토크나이저 (학습된 것, 고정)
- 실제 LLM은 **byte-level BPE**: 유니코드 아니라 바이트에서 시작 → 어떤 문자든 표현 (이모지·모든 언어)

## 토큰이 만드는 이상한 동작

토큰 단위라서 생기는 LLM의 약점들:

### 글자 세기 실패

```
"strawberry에 r이 몇 개?" → 자주 틀림
이유: 모델은 [str][aw][berry] 같은 토큰으로 봄, 개별 글자를 안 봄
```
- 글자가 토큰 안에 숨어 있어 셀 수 없음 (토큰이 원자 단위)

### 숫자·산술 약점

- 숫자가 일관되게 안 쪼개짐 (`1234` → `123`+`4` or `12`+`34`...)
- 자리값 개념이 토큰화로 깨짐 → 산술 어려움 (최근 모델은 숫자를 자리별로 쪼개 개선)

### 언어별 비용 불평등

- 토크나이저는 주로 영어에 최적화 → 영어는 단어당 토큰 적음
- 한국어·일본어·태국어 등은 **같은 뜻인데 토큰 수↑** → 비용·문맥 더 소모
- 저자원 언어일수록 불리 (공정성 문제)

### 기타

- **공백·대소문자 민감**: `" the"`와 `"the"`가 다른 토큰
- **희귀 토큰 버그**: 학습에 거의 없던 토큰이 이상 동작 유발 (SolidGoldMagikarp 현상)

## 실무 함의

- **비용·한도가 토큰 단위**: API 요금, 문맥 창(context window)이 토큰으로 측정 ([[prompting]])
- **문맥 예산**: 긴 프롬프트·RAG 청크가 토큰 소모 → [[rag]]의 청킹
- **대략 영어 1토큰 ≈ 4글자 ≈ 0.75단어** (언어마다 다름)
- 토큰 수 세보기(tiktoken 등)가 비용·한도 관리에 필수

## 왜 중요한가

- LLM의 **입출력 원자 단위** → 모든 것이 토큰 위 ([[attention-and-transformers]]의 입력)
- 여러 "왜 이걸 못하지" 의문의 답이 토큰화
- 비용·성능·다국어 공정성에 직접 영향

## 셀프 체크

> [!question]- 왜 글자도 단어도 아닌 하위단어(subword)를 단위로 쓰나?
> 글자 단위는 어휘가 작지만 시퀀스가 너무 길어 비효율적이고, 단어 단위는 어휘가 폭발하고 미등록 단어(OOV) 문제가 있다. 하위단어는 절충으로, 자주 쓰는 건 통째로 드문 건 조각으로 표현해 OOV를 없앤다.

> [!question]- BPE는 어휘를 어떻게 만드나?
> 텍스트를 문자로 시작해, 가장 자주 나오는 인접 쌍을 찾아 하나로 병합하고 이를 목표 어휘 크기까지 반복한다. 빈번한 조각이 통째 토큰으로 자라며, 병합 규칙의 순서가 곧 학습된 토크나이저다.

> [!question]- LLM이 "strawberry의 r 개수"를 왜 자주 틀리나?
> 모델은 단어를 [str][aw][berry]처럼 토큰으로 보지 개별 글자를 보지 않는다. 글자가 토큰 안에 숨어 있어 셀 수 없다. 토큰이 원자 단위라 글자 수준 조작이 어렵다.

> [!question]- 같은 뜻인데 한국어가 영어보다 토큰이 많이 드는 이유는?
> 토크나이저가 주로 영어에 최적화돼 영어는 단어당 토큰이 적다. 한국어·일본어 등은 같은 의미도 토큰 수가 늘어 비용·문맥을 더 소모한다. 저자원 언어일수록 불리한 공정성 문제다.

## 연습문제

> [!example]- 문제: 코퍼스 "ab ab ac"에 BPE를 적용해 첫 두 병합을 손으로 수행하라(공백은 단어 경계).
> **풀이**
> 초기(문자 분리): "a b", "a b", "a c". 인접 쌍 빈도: (a,b)=2, (a,c)=1.
> 병합1: 최빈 쌍 (a,b)→"ab". 결과: "ab", "ab", "a c". 규칙 [(a,b)].
> 병합2: 남은 쌍은 (a,c)=1뿐 → (a,c)→"ac". 결과: "ab", "ab", "ac". 규칙 [(a,b),(a,c)].
> 빈번한 "ab"가 먼저 통째 토큰이 되고, 병합 규칙 순서가 곧 토크나이저다.

> [!example]- 문제: 영어 1토큰 ≈ 4글자 근사를 쓸 때, 2000자 영어 문서는 대략 몇 토큰인가? 문맥 창이 8000토큰이면 이런 문서 몇 개를 넣을 수 있나(대략)?
> **풀이**
> 2000자 / 4 ≈ 500토큰.
> 8000 / 500 = 16개. 다만 프롬프트 지시·출력 여유분을 빼야 하므로 실제로는 그보다 적게 잡는다.
> 비용·한도가 토큰 단위이므로, 긴 프롬프트나 RAG 청크가 토큰 예산을 잠식한다. tiktoken으로 실제 토큰 수를 세는 게 관리에 필수다.
