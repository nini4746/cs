# 프롬프팅 (Prompting)

## 한 줄 요약

LLM은 프롬프트(문맥)만으로 학습 없이 새 작업을 수행한다(in-context learning). 명확한 지시·예시(few-shot)·추론 유도(chain-of-thought)로 성능이 크게 달라진다. 핵심은 모델이 다음 토큰을 예측한다는 사실에서 출발해 **원하는 출력이 자연스럽게 이어지도록 문맥을 설계**하는 것.

## 왜 필요한가

- 파인튜닝 없이 어떻게 새 작업을 시키나
- few-shot, chain-of-thought가 왜 되나
- 좋은 프롬프트 원리

## 왜 프롬프트가 통하나 (in-context learning)

```
LLM = 다음 토큰 예측기 ([[llm-training]])
프롬프트 = 예측의 문맥 → 문맥을 바꾸면 출력이 바뀜
```

- **가중치 변경 없이**(학습 X) 문맥만으로 작업 수행 → **in-context learning**
- 사전학습에서 본 패턴을 프롬프트가 "불러냄"
- 그래서 프롬프트 설계 = 원하는 연속이 나올 문맥 만들기

## Zero-shot vs Few-shot

### Zero-shot

```
"이 리뷰 감정 분류해: '최악이었다' →"
예시 없이 지시만
```

### Few-shot

```
"'좋아요' → 긍정
 '별로예요' → 부정
 '최악이었다' →"
예시 몇 개를 문맥에 → 패턴 파악
```

- **few-shot**: 예시로 형식·기준을 보여줌 → 더 정확·일관
- 예시가 곧 "즉석 학습 데이터" (가중치는 안 변함)
- 형식이 까다롭거나 모호한 작업에 특히 효과

## Chain-of-Thought (CoT)

**추론 과정을 소리 내어** 유도 → 복잡한 문제 성능↑:

```
그냥: "23×17은?" → (바로 답, 자주 틀림)
CoT:  "단계별로 생각해봐: 23×17 = 23×10 + 23×7 = 230 + 161 = 391"
```

- **"단계별로 생각해봐"** 한 줄이 추론 문제 정확도 크게 올림
- 왜 되나: 답을 바로 예측 대신 **중간 단계를 토큰으로 펼침** → 각 단계가 다음 단계의 문맥 (계산을 토큰 공간에 분산)
- [[tokenization]]에서 본 산술 약점도 CoT로 완화 (자리별 전개)
- 변형: **self-consistency**(여러 번 추론 후 다수결), **CoT 예시 few-shot**

## 좋은 프롬프트 원리

- **명확·구체**: 모호하면 모델이 추측 → 원하는 것·형식·제약 명시
- **역할 부여**: "너는 전문 편집자다" - 관련 문맥 활성화
- **출력 형식 지정**: "JSON으로", "3줄 이내" - 파싱·일관성
- **예시 제공**: few-shot으로 기준 전달
- **분해**: 큰 작업을 단계로 (복잡하면 나눠서)
- **문맥 배치**: 중요한 지시를 앞·뒤에 (중간은 놓치기 쉬움 - "lost in the middle")

## 문맥 창 (context window)

프롬프트가 들어갈 **토큰 예산**([[tokenization]]):

- 모델마다 한도 (수천~수백만 토큰)
- 길수록 비용·지연↑, 너무 길면 중간 정보 놓침
- RAG는 관련 부분만 넣어 예산 아낌 ([[rag]])
- 대화가 길어지면 오래된 문맥이 밀려남

## 한계·주의

- **환각(hallucination)**: 모르는 것도 그럴듯하게 지어냄 → 사실은 검증·RAG 필요 ([[rag]])
- **프롬프트 민감성**: 사소한 표현 차이로 결과 요동 (견고하지 않음)
- **프롬프트 인젝션**: 입력에 숨은 악의적 지시 (security 문제) → [[agents]]에서 심각
- **최신 지식 부재**: 학습 시점 이후 모름 → 도구·검색 필요 ([[agents]], [[rag]])
- 프롬프팅으로 안 되면 → RAG(지식)·파인튜닝(행동) 고려 ([[fine-tuning]])

## 언제 프롬프팅 vs 다른 방법

```
프롬프팅: 빠르고 싸다, 먼저 시도 (대부분 여기서 해결)
RAG:      외부·최신 지식 필요 → [[rag]]
파인튜닝:  일관된 형식·스타일·대량 반복 작업 → [[fine-tuning]]
```

## 셀프 체크

> [!question]- in-context learning은 왜 가중치 변경 없이 새 작업을 하게 하나?
> LLM은 다음 토큰 예측기라 프롬프트(문맥)가 예측을 좌우한다. 문맥을 바꾸면 출력이 바뀌고, 사전학습에서 본 패턴을 프롬프트가 "불러낸다". 그래서 학습 없이 문맥 설계만으로 작업을 시킬 수 있다.

> [!question]- few-shot이 zero-shot보다 나은 경우와 그 이유는?
> 형식이 까다롭거나 기준이 모호한 작업에서 낫다. 예시 몇 개가 형식·판정 기준을 보여주는 "즉석 학습 데이터" 역할을 해 더 정확·일관된 출력을 유도한다(가중치는 안 변함).

> [!question]- Chain-of-Thought가 왜 추론 문제 정확도를 올리나?
> 답을 바로 예측하는 대신 중간 단계를 토큰으로 펼치게 하면, 각 단계가 다음 단계의 문맥이 되어 계산을 토큰 공간에 분산한다. "단계별로 생각해봐" 한 줄로 복잡한 추론·산술 정확도가 크게 오른다.

> [!question]- "lost in the middle"이 프롬프트 설계에 주는 함의는?
> 문맥이 길면 모델이 중간에 놓인 정보를 놓치기 쉽다. 그래서 중요한 지시는 프롬프트 앞·뒤에 배치하고, RAG로 관련 부분만 넣어 문맥을 짧게 유지하는 게 좋다.

## 연습문제

> [!example]- 문제: 리뷰를 긍정/부정으로 분류하는 few-shot 프롬프트를 설계하고, 왜 zero-shot보다 안정적인지 설명하라.
> **풀이**
> 예: "'좋아요' → 긍정 / '별로예요' → 부정 / '최악이었다' →" 처럼 라벨링된 예시 2~3개를 넣고 새 입력을 이어 붙인다.
> 예시가 출력 형식(단어 하나)과 판정 기준을 명시해, 모델이 "긍정/부정" 형식으로 일관되게 답하도록 문맥을 고정한다. zero-shot은 형식을 추측해 "이 리뷰는 부정적입니다" 같은 산만한 출력이 나올 수 있다.

> [!example]- 문제: "한 반 학생 28명, 남학생이 여학생보다 4명 많다. 여학생 수는?"에 모델이 자주 틀린다. CoT로 고치는 프롬프트와 기대 전개를 써라.
> **풀이**
> 프롬프트에 "단계별로 생각해봐"를 추가한다. 기대 전개: 여학생 x, 남학생 x+4 → x+(x+4)=28 → 2x=24 → x=12. 답 12명.
> 바로 답하면 중간 계산이 없어 틀리기 쉽지만, 중간 단계를 토큰으로 펼치면 각 등식이 다음 단계의 문맥이 되어 정확도가 오른다. self-consistency(여러 번 추론 후 다수결)로 더 견고히 할 수 있다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: (1) "다음 토큰 예측"에서 in-context learning이 나오는 논리 (2) zero-shot·few-shot·CoT가 각각 언제·왜 효과적인지 (3) 좋은 프롬프트 원리와 프롬프팅으로 안 될 때 RAG·파인튜닝으로 넘어가는 기준.
