# CI/CD 원리 (CI/CD Principles)

## 한 줄 요약

CI(지속적 통합)는 작은 변경을 자주 main에 합치고 매번 자동 테스트하는 것, CD(지속적 배포/전달)는 통과한 변경을 자동으로 프로덕션까지 흘려보내는 것. 목적은 **피드백 루프를 짧게** - 문제를 빨리, 작게, 싸게 발견.

## 왜 필요한가

- 왜 매 커밋마다 테스트를 자동으로 돌리나
- CI와 CD의 차이
- 파이프라인 단계를 어떻게 배치·최적화하나

## CI: 지속적 통합

**작게 자주 통합 + 매번 자동 검증** ([[git-workflows]]의 trunk-based):

```
커밋/PR → 자동 트리거:
  빌드 → 테스트 → 린트 → (통과해야 머지)
```

- 핵심은 **자주** (하루에도 여러 번) → 충돌·회귀를 작게 (통합 지옥 회피)
- **매 변경마다 검증** → 깨진 걸 즉시 발견 (누가 깼는지 명확)
- software-design/[[testing-strategy]]의 자동 테스트가 게이트
- "main은 항상 배포 가능한 상태" 유지

## CD: 지속적 전달 vs 배포

두 CD 구분:

- **Continuous Delivery(전달)**: 언제든 배포 **가능한** 상태로 자동 준비, 프로덕션 배포는 **수동 승인**
- **Continuous Deployment(배포)**: 통과하면 **자동으로 프로덕션까지** (사람 개입 없음)

```
CI → 스테이징 배포 → (전달: 버튼 대기 / 배포: 자동) → 프로덕션
```

- 자동 배포는 강한 테스트·모니터링·롤백 전제 ([[deployment-strategies]], [[observability]])

## 파이프라인 단계 (순서가 중요)

**빠르고 싼 것을 앞에, 느리고 비싼 것을 뒤에** (fail fast):

```
1. 린트·포맷 (초, 제일 싸다)
2. 유닛 테스트 (빠름, software-design/[[testing-strategy]]의 피라미드 하단)
3. 빌드 (이미지 등)
4. 통합 테스트 (느림)
5. E2E 테스트 (제일 느리고 불안정)
6. 배포
```

- 앞 단계 실패하면 **즉시 중단** → 비싼 뒤 단계 안 돌림 (돈·시간 절약)
- software-design/[[testing-strategy]]의 **테스트 피라미드**와 정렬: 빠른 유닛 많이, 느린 E2E 적게

## 최적화

### 캐싱

- **의존성 캐시**: node_modules, ~/.m2 등 (매번 재다운로드 안 함)
- **레이어 캐시**: Docker 빌드 캐시 ([[dockerfile-best-practices]]의 순서)
- **빌드 캐시**: 증분 빌드 (programming-languages/[[compiled-vs-interpreted]]의 증분)

### 병렬화

- 독립 작업 동시 실행 (테스트 샤딩, 여러 잡 병렬)
- 전체 파이프라인 시간 단축 (임계 경로만큼만)

### 빠른 피드백

- 파이프라인이 느리면 사람들이 CI 안 기다림 → 우회 → 무의미
- 목표: 몇 분 내 결과 (길면 자주 통합 안 함)

## 핵심 원칙

- **자동화**: 손으로 하던 걸 스크립트로 (재현성·일관성 [[iac]])
- **fail fast**: 문제를 빨리·싸게 (앞 단계에서)
- **멱등·재현**: 같은 커밋 → 같은 결과 (환경 고정, 버전 핀)
- **작은 배치**: 작은 변경 자주 → 문제 격리 쉬움, 롤백 쉬움 ([[git-workflows]])
- **파이프라인이 곧 문서**: 빌드·배포 방법이 코드로 명시 (github-actions 등 [[github-actions]])

## 왜 중요한가

- **빠른 피드백**: 버그를 개발 중에 (프로덕션 아니라)
- **낮은 위험**: 작은 변경 = 작은 폭발 반경, 쉬운 롤백
- **속도**: 배포 자동화로 자주·빠르게 릴리스
- **신뢰**: 자동 게이트가 품질 보장 → 사람 실수↓

## 셀프 체크

> [!question]- Continuous Delivery와 Continuous Deployment의 차이는?
> 둘 다 통과한 변경을 자동으로 배포 가능 상태까지 준비한다. Delivery(전달)는 프로덕션 배포에 사람의 수동 승인(버튼)이 필요하고, Deployment(배포)는 통과하면 사람 개입 없이 자동으로 프로덕션까지 나간다.

> [!question]- 파이프라인에서 린트·유닛 테스트를 앞에, E2E를 뒤에 두는 이유는?
> fail fast. 빠르고 싼 검사를 앞에 두면 문제가 있을 때 즉시 중단해 느리고 비싼 뒤 단계(통합·E2E)를 돌리지 않아 돈과 시간을 아낀다. 테스트 피라미드(빠른 유닛 다수, 느린 E2E 소수)와도 정렬된다.

> [!question]- CI에서 "자주" 통합하는 것이 왜 핵심인가?
> 작게 자주 통합하면 충돌·회귀가 작게 유지되어 통합 지옥을 피한다. 매 변경마다 검증하므로 깨진 것을 즉시, 누가 깼는지 명확하게 발견한다.

> [!question]- CI 최적화의 캐싱 세 종류는?
> 의존성 캐시(node_modules, ~/.m2 재다운로드 회피), 레이어 캐시(Docker 빌드 캐시), 빌드 캐시(증분 빌드).

## 연습문제

> [!example]- 문제: 아래 잘못 배치된 CI 단계를 fail-fast 순서로 재배치하라. `E2E 테스트 → 빌드 → 린트 → 유닛 테스트 → 배포 → 통합 테스트`
> **풀이**
> 빠르고 싼 것을 앞에, 느리고 비싼 것을 뒤에 둔다.
> 1. 린트·포맷 (초 단위, 제일 쌈)
> 2. 유닛 테스트 (빠름)
> 3. 빌드 (이미지 등)
> 4. 통합 테스트 (느림)
> 5. E2E 테스트 (제일 느리고 불안정)
> 6. 배포
> 앞 단계가 실패하면 즉시 중단되어 뒤의 비싼 단계를 돌리지 않는다.

> [!example]- 문제: CI 파이프라인이 20분 걸려 개발자들이 결과를 안 기다리고 우회한다. 시간을 줄일 방안 세 가지를 제시하라.
> **풀이**
> 1. 캐싱: 의존성 캐시(actions/cache)와 Docker 레이어 캐시로 무거운 재설치·재빌드를 건너뛴다.
> 2. 병렬화: 독립 작업을 동시 실행(테스트 샤딩, 잡 병렬) → 전체 시간이 임계 경로만큼으로 단축.
> 3. fail fast 재배치 + 조건부 실행: 싼 검사를 앞에, 영향받은 것만 빌드(모노레포)해 불필요한 잡 스킵.
> 목표는 몇 분 내 결과. 느리면 자주 통합하지 않게 되어 CI 자체가 무의미해진다.

## 파인만

> [!note]- 백지에 CI/CD 파이프라인을 팀원에게 설명하듯 그려보라. 막히면 그 단계만 다시.
> **점검 포인트**: (1) CI(자주 통합+매번 검증)와 CD(전달 vs 배포)의 구분, (2) 파이프라인 단계를 fail fast로 배치하는 이유, (3) 자동화·멱등·작은 배치가 위험을 낮추는 원리.
