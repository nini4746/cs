# 관측성 (Observability)

## 한 줄 요약

시스템 내부 상태를 밖에서 나온 신호로 추론하는 능력. 3대 신호는 로그(무슨 일이), 메트릭(얼마나·집계), 트레이스(요청이 서비스들을 어떻게 거쳤나). 모니터링이 "아는 문제를 지켜보기"라면 관측성은 "모르는 문제도 파헤치기". SLO/SLI로 신뢰성을 수치화한다.

## 왜 필요한가

- 분산 시스템에서 "왜 느리지/실패하지"를 어떻게 아나
- 로그·메트릭·트레이스의 역할 차이
- SLO/SLI가 뭔가 (신뢰성 목표)

## 왜 어려워졌나

```
모놀리스: 서버 하나, 로그 하나 → 디버깅 단순
마이크로서비스: 요청이 수십 서비스 거침 → "어디서 느려졌지?"
```

- software-design/[[monolith-vs-microservices]]로 분산되며 관측이 필수가 됨
- 프로세스 하나 디버깅([[linux-debugging]])을 **분산 규모로** 확장한 것

## 모니터링 vs 관측성

- **모니터링**: **미리 정한** 지표를 지켜봄 (CPU 80% 넘으면 알림) - "아는 미지수"
- **관측성**: **예상 못 한** 질문도 사후에 파고들 수 있음 - "모르는 미지수"
- 관측성은 충분한 신호(고카디널리티 데이터)를 남겨 **새 질문**에 답하는 능력

## 3대 신호 (three pillars)

### 로그 (logs)

**개별 이벤트 기록** - "무슨 일이 일어났나":

- 타임스탬프 + 메시지 (에러, 요청, 상태변화)
- **구조화 로그**(JSON): 기계가 파싱·검색 (`{level, msg, user_id, trace_id}`)
- 상세하지만 **양이 많음** (비용·노이즈) → 샘플링·레벨 조절
- 중앙 수집 (ELK, Loki) - 분산된 로그를 한곳에 ([[linux-debugging]]의 로그 확장)

### 메트릭 (metrics)

**수치 시계열 집계** - "얼마나":

- 카운터·게이지·히스토그램 (요청 수, 지연시간 p99, 에러율)
- **집계라 저렴·효율** (로그보다 가벼움, 장기 보관)
- 대시보드·알림에 적합 (Prometheus + Grafana)
- **RED**(Rate/Errors/Duration), **USE**(Utilization/Saturation/Errors [[linux-debugging]]) 방법론
- 주의: 평균 말고 **백분위(p50/p95/p99)** - 꼬리 지연이 진짜 문제 (database/의 tail latency)

### 트레이스 (traces)

**요청의 서비스 간 여정** - "어디서 느렸나":

```
요청 → [API 20ms] → [Auth 5ms] → [DB 200ms ←범인] → [Cache 2ms]
       하나의 trace, 각 구간이 span
```

- **분산 추적**: 요청에 **trace_id** 부여 → 서비스마다 전파 → 전체 경로 재구성
- 마이크로서비스 지연 병목 찾기의 핵심 (어느 서비스·어느 호출이 느린가)
- 세 신호를 trace_id로 상관 (로그↔트레이스↔메트릭 연결)

## 3신호 조합

```
메트릭: 에러율 급증 감지 (뭔가 잘못됨 - 알림)
   ↓
트레이스: 어느 서비스·구간이 문제인지 좁힘
   ↓
로그: 그 지점의 상세 원인 (왜)
```

넓게→좁게, [[linux-debugging]]의 계층적 진단과 같은 흐름.

## OpenTelemetry

- 신호 수집의 **표준**(벤더 중립): 계측 API·SDK 통일
- 코드는 OTel로 계측, 백엔드는 자유 선택 (락인 회피, [[cloud-basics]])

## SLI / SLO / SLA

신뢰성을 **수치화** (Google SRE):

- **SLI**(Indicator): 측정값 (가용성 %, p99 지연, 에러율)
- **SLO**(Objective): 목표 (99.9% 가용성) - 내부 목표
- **SLA**(Agreement): 고객 계약 (어기면 배상) - SLO보다 느슨하게
- **에러 버짓**: 100% - SLO (99.9% 목표면 0.1%는 실패 허용) → 이 예산 안에선 **빠르게 배포·실험**, 소진되면 안정성 우선
- 100% 신뢰성은 불가능·비경제 → 적정 목표를 정하는 게 요점

## 왜 중요한가

- **빠른 감지·복구**: MTTR 단축 ([[deployment-strategies]]의 롤백 판단, [[ci-cd-principles]])
- **자동 배포 전제**: 관측 없이 자동 배포는 위험 (문제를 못 봄)
- **카나리 판단**: 지표로 카나리 확대/롤백 결정 ([[deployment-strategies]])
- **용량 계획·비용**: 추세로 예측 ([[cloud-basics]]의 FinOps)

## 셀프 체크

> [!question]- 모니터링과 관측성의 차이는?
> 모니터링은 미리 정한 지표를 지켜보는 것("아는 미지수", CPU 80% 알림). 관측성은 예상 못 한 질문도 사후에 파고드는 능력("모르는 미지수")으로, 충분한 신호(고카디널리티)를 남겨 새 질문에 답한다.

> [!question]- 3대 신호(로그·메트릭·트레이스)의 역할 차이는?
> 로그는 개별 이벤트("무슨 일이"), 메트릭은 수치 시계열 집계("얼마나", 저렴·알림용), 트레이스는 요청이 서비스들을 거친 여정("어디서 느렸나", trace_id로 재구성).

> [!question]- 지연을 평균 대신 백분위(p95/p99)로 보는 이유는?
> 평균은 꼬리 지연(소수의 아주 느린 요청)을 가린다. 실제 사용자 경험을 망치는 것은 이 꼬리(tail latency)이므로 p50/p95/p99로 봐야 진짜 문제가 드러난다.

> [!question]- 세 신호를 조합해 문제를 좁히는 흐름은?
> 메트릭으로 에러율 급증 감지(알림) → 트레이스로 어느 서비스·구간이 문제인지 좁힘 → 로그로 그 지점의 상세 원인(왜). 넓게→좁게, trace_id로 상관.

> [!question]- SLI/SLO/SLA와 에러 버짓은?
> SLI는 측정값(가용성 %, p99), SLO는 내부 목표(99.9%), SLA는 고객 계약(어기면 배상, SLO보다 느슨). 에러 버짓=100%-SLO(0.1% 실패 허용): 예산 안에선 빠르게 배포·실험, 소진되면 안정성 우선.

## 연습문제

> [!example]- 문제: "결제가 가끔 느리다"는 제보가 들어왔다. 세 신호를 어떤 순서로 활용해 원인을 찾겠는가?
> **풀이**
> 1. 메트릭: 결제 경로 지연을 평균이 아니라 p95/p99로 확인 → 꼬리 지연·에러율 급증 시점 파악(대시보드/알림).
> 2. 트레이스: 느린 요청의 trace를 열어 span별 소요를 봄 → 어느 서비스·호출이 병목인지(예: DB 200ms) 좁힘.
> 3. 로그: 그 서비스·시점의 구조화 로그(trace_id로 상관)에서 상세 원인(느린 쿼리·락 등) 확인.
> 넓게→좁게로 감지→국소화→원인.

> [!example]- 문제: 가용성 SLO가 99.9%다. 에러 버짓이 거의 소진됐을 때와 여유 있을 때 배포 정책을 어떻게 달리하겠는가?
> **풀이**
> 에러 버짓=0.1%(월 허용 실패량).
> - 여유 있을 때: 예산 안에서 빠르게 배포·실험(위험한 변경, 카나리 확대)을 허용 - 실패해도 예산으로 흡수.
> - 거의 소진됐을 때: 안정성 우선으로 전환 - 기능 배포 동결/축소, 위험한 변경 보류, 신뢰성 개선에 집중.
> 100% 신뢰성은 비경제적이므로 예산을 속도와 안정성의 교환 수단으로 쓴다.
