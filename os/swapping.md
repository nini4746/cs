# 스와핑과 페이지 교체 (Swapping)

## 한 줄 요약

물리 메모리보다 큰 주소 공간을 쓰려고 안 쓰는 페이지를 디스크로 내보낸다. 어떤 페이지를 내보낼지가 교체 정책이고, 지역성 덕에 LRU 근사가 잘 통하지만 working set이 넘치면 thrashing으로 무너진다.

## 왜 필요한가

- RAM보다 큰 프로그램이 어떻게 도나
- 메모리 부족 시 시스템이 왜 갑자기 느려지나 (thrashing)
- 페이지 폴트가 성능에 미치는 영향

## 핵심: 디스크를 메모리의 연장으로

가상 메모리([[virtual-memory]])는 물리 RAM보다 큰 주소 공간을 허용. 초과분은 **디스크(스왑 공간)**에 보관:

- 자주 쓰는 페이지 → RAM
- 안 쓰는 페이지 → 디스크로 축출(swap out)
- 다시 필요하면 → 디스크에서 로드(swap in)

RAM은 사실상 **디스크의 캐시** ([[memory-hierarchy]]의 계층 사고 그대로). 페이지 테이블([[page-tables]])의 valid 비트가 "지금 RAM에 있나"를 표시.

## 페이지 폴트 처리

RAM에 없는 페이지 접근 → **페이지 폴트**([[exceptions-and-interrupts]]):

```
1. MMU: 페이지 테이블 엔트리 valid=0 발견 → 폴트 발생
2. OS 페이지 폴트 핸들러 진입
3. 빈 프레임 확보 (없으면 교체 정책으로 하나 축출)
4. 디스크에서 페이지를 그 프레임으로 로드
5. 페이지 테이블 갱신 (valid=1, PPN 설정)
6. 폴트 낸 명령 재시도 → 이번엔 성공
```

디스크 접근(~100μs, [[memory-hierarchy]])이라 RAM 접근(~100ns)의 1000배. 페이지 폴트 하나가 엄청 비쌈 → 최소화가 관건.

## 교체 정책: 누구를 축출하나

빈 프레임이 없으면 하나를 골라 내보내야 함. 미래에 가장 늦게 쓰일 페이지를 내보내는 게 최적(OPT/Belady)이지만 미래를 모름 → 과거로 근사:

| 정책 | 규칙 | 문제 |
|---|---|---|
| **OPT** | 가장 먼 미래에 쓸 것 | 미래를 알아야 함 (이론적 상한) |
| **FIFO** | 가장 오래 들어온 것 | 자주 쓰는 페이지도 축출. Belady 이상현상 |
| **LRU** | 가장 오래 안 쓴 것 | 지역성 활용, 좋지만 완벽 구현 비쌈 |
| **Clock** | LRU 근사 (accessed 비트 순회) | 실용적, 대부분 OS가 사용 |

### Clock (LRU 근사)

완벽한 LRU는 매 접근마다 순서 갱신 → 비쌈. Clock은 근사:

- 페이지들을 원형으로, 시곗바늘이 돌며 검사
- accessed 비트가 1이면 → 0으로 바꾸고 넘어감 (한 번 봐줌)
- accessed 비트가 0이면 → 축출 (최근 안 쓰임)
- 페이지 테이블의 accessed 비트를 하드웨어가 자동 세팅 → 싸게 근사

## Thrashing: working set이 넘칠 때

물리 메모리 < working set([[cache-misses]]의 그 개념)이면:

- 페이지를 축출하자마자 곧 다시 필요 → swap in
- 그걸 위해 또 축출 → 그것도 곧 필요
- **거의 모든 접근이 페이지 폴트** → CPU는 노는데 디스크만 미친 듯 → 시스템이 기어감

메모리 부족 시 갑자기 극단적으로 느려지는 현상. 지역성이 무너진 극한. 해결:
- working set을 줄이거나 (메모리 추가, 프로세스 줄임)
- OS가 일부 프로세스를 통째 스왑아웃(admission control)해 나머지를 살림
- 리눅스 **OOM killer**: 극한이면 프로세스를 죽여 시스템을 구함

## SSD 시대의 스왑

- 스왑이 HDD면 thrashing = 재앙 (밀리초 지연). SSD면 덜하지만 여전히 RAM보다 훨씬 느림
- **zram/압축 스왑**: 페이지를 디스크 대신 RAM에서 압축 보관 → 디스크 왕복 회피
- 모바일(iOS/안드로이드)은 스왑 대신 앱을 종료/재시작하는 전략

## 셀프 체크

> [!question]- RAM을 "디스크의 캐시"로 보는 게 무슨 뜻이고, 무엇이 그걸 표시하나?
> 가상 메모리는 물리 RAM보다 큰 주소 공간을 허용하고 초과분은 디스크(스왑 공간)에 둔다. 즉 전체 주소 공간의 진짜 원본은 디스크에 있고, 자주 쓰는 페이지만 RAM에 올려두는 셈이라 RAM이 디스크의 캐시로 동작한다. 페이지 테이블 엔트리의 valid 비트가 "이 페이지가 지금 RAM에 있나"를 표시한다.

> [!question]- 페이지 폴트 하나가 왜 그렇게 비싼가?
> 폴트를 처리하려면 디스크에서 페이지를 읽어와야 하는데, 디스크 접근은 ~100μs로 RAM 접근 ~100ns의 약 1000배다. 그래서 페이지 폴트율을 조금만 낮춰도 성능이 크게 좌우되고, 교체 정책의 목표가 폴트 최소화가 된다.

> [!question]- Clock이 완벽한 LRU보다 실용적인 이유는?
> 완벽한 LRU는 매 메모리 접근마다 사용 순서를 갱신해야 해서 너무 비싸다. Clock은 페이지를 원형으로 두고 시곗바늘이 돌며 accessed 비트만 검사한다 - 1이면 0으로 바꾸고 한 번 봐주고, 0이면 축출한다. accessed 비트는 하드웨어가 자동으로 세팅하므로 싼 값에 LRU를 근사한다.

> [!question]- thrashing은 언제, 왜 발생하나?
> 물리 메모리가 working set보다 작을 때 발생한다. 페이지를 축출하자마자 곧 다시 필요해 swap in하고, 그 자리를 만들려 또 축출한 페이지도 곧 필요해진다. 거의 모든 접근이 페이지 폴트가 되어 CPU는 노는데 디스크만 미친 듯이 돌고 시스템이 기어간다.

> [!question]- FIFO의 문제는 무엇인가?
> 들어온 순서만 보고 축출하므로 자주 쓰는 페이지도 오래됐다는 이유로 내보내 곧바로 다시 폴트를 낸다. 게다가 Belady 이상현상이 있어 프레임을 늘렸는데 오히려 폴트가 늘 수 있다. 지역성을 반영하는 LRU/Clock과 대비된다.

## 연습문제

> [!example]- 문제: FIFO와 LRU의 페이지 폴트 수를 비교하라
> 프레임 3개, 참조열 `A B C A B D A B C D`일 때 FIFO와 LRU 각각의 페이지 폴트 수를 구하라.
> **풀이**
> FIFO (괄호는 축출 대상, 큐 오래된 순):
> - A miss[A] B miss[A,B] C miss[A,B,C] A hit B hit
> - D miss(축출A)[B,C,D] A miss(축출B)[C,D,A] B miss(축출C)[D,A,B] C miss(축출D)[A,B,C] D miss(축출A)[B,C,D]
> - 폴트 = A,B,C,D,A,B,C,D = **8회** (hit: A,B)
> LRU (LRU→MRU 순 관리):
> - A miss[A] B miss[A,B] C miss[A,B,C] A hit B hit → 순서 C,A,B
> - D miss(축출 LRU=C)[A,B,D] A hit B hit → 순서 D,A,B
> - C miss(축출 LRU=D)[A,B,C] D miss(축출 LRU=A)[B,C,D]
> - 폴트 = A,B,C,D,C,D = **6회** (hit: A,B,A,B)
> 결론: 같은 참조열·같은 프레임 수에서 LRU 6 < FIFO 8. LRU가 지역성(최근 쓴 A,B를 유지)을 반영해 폴트가 적다.

> [!example]- 문제: thrashing을 진단하고 해결하라
> 어떤 서버가 갑자기 극단적으로 느려졌다. 관찰: CPU 사용률은 10%대로 낮은데 디스크 I/O는 100%로 포화, major page fault가 초당 수천 건. 원인을 진단하고 해결책 3가지를 제시하라.
> **풀이**
> - 진단: 전형적 thrashing. 실행 중인 프로세스들의 working set 합이 물리 메모리를 초과해, CPU는 대부분 디스크에서 페이지가 오길 기다리며 놀고(낮은 CPU) 디스크만 swap in/out으로 포화(높은 I/O, 높은 major fault)된 상태.
> - 해결:
> - 1) working set을 물리 메모리 아래로 줄인다 - RAM 증설, 또는 동시 실행 프로세스/메모리 사용량 감축.
> - 2) OS가 일부 프로세스를 통째 스왑아웃(admission control)해 남은 프로세스들의 working set이 RAM에 들어가게 한다.
> - 3) 극한이면 리눅스 OOM killer가 프로세스를 죽여 시스템 전체를 구한다. 스왑을 SSD/zram으로 옮기면 지연을 완화할 수 있으나 근본 원인(메모리 부족)은 남는다.

> [!example]- 문제: 페이지 폴트 처리 순서를 재구성하라
> valid=0인 페이지에 접근했을 때 폴트가 처리되는 단계를 순서대로 쓰고, 빈 프레임이 없을 때 추가로 일어나는 일을 밝혀라.
> **풀이**
> 1) MMU가 페이지 테이블 엔트리에서 valid=0을 발견 → 페이지 폴트 발생
> 2) OS 페이지 폴트 핸들러 진입
> 3) 빈 프레임 확보 - **없으면 교체 정책(Clock 등)으로 희생 페이지를 골라 축출**(dirty면 디스크에 먼저 기록)
> 4) 디스크(스왑)에서 요청된 페이지를 그 프레임으로 로드
> 5) 페이지 테이블 갱신(valid=1, PPN 설정)
> 6) 폴트를 낸 명령을 재시도 → 이번엔 성공
> 핵심: 3번의 "축출"이 교체 정책이 개입하는 지점이고, 여기서 잘못 고르면 곧 다시 폴트가 나 thrashing의 씨앗이 된다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> - 페이지 폴트가 발생해서 처리되기까지의 단계를, 특히 빈 프레임이 없을 때 교체가 끼어드는 지점을 순서대로 말할 수 있는가.
> - OPT/FIFO/LRU/Clock을 "무엇을 기준으로 축출하나 + 무슨 문제/타협이 있나"로 각각 구분할 수 있는가.
> - working set과 물리 메모리 크기의 관계로 thrashing을 정의하고, CPU 낮음+디스크 포화라는 증상과 해결책을 이을 수 있는가.
