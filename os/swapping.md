# 스와핑과 페이지 교체 (Swapping)

## 한 줄 요약

물리 메모리보다 큰 주소 공간을 쓰려고 안 쓰는 페이지를 디스크로 내보낸다. 어떤 페이지를 내보낼지가 교체 정책이고, 지역성 덕에 LRU 근사가 잘 통하지만 working set이 넘치면 thrashing으로 무너진다.

## 왜 필요한가

- RAM보다 큰 프로그램이 어떻게 도나
- 메모리 부족 시 시스템이 왜 갑자기 느려지나 (thrashing)
- 페이지 폴트가 성능에 미치는 영향

## 핵심: 디스크를 메모리의 연장으로

가상 메모리([[virtual-memory]])는 물리 RAM보다 큰 주소 공간을 허용. 초과분은 **디스크(스왑 공간)**에 보관:

- 자주 쓰는 페이지 → RAM
- 안 쓰는 페이지 → 디스크로 축출(swap out)
- 다시 필요하면 → 디스크에서 로드(swap in)

RAM은 사실상 **디스크의 캐시** ([[memory-hierarchy]]의 계층 사고 그대로). 페이지 테이블([[page-tables]])의 valid 비트가 "지금 RAM에 있나"를 표시.

## 페이지 폴트 처리

RAM에 없는 페이지 접근 → **페이지 폴트**([[exceptions-and-interrupts]]):

```
1. MMU: 페이지 테이블 엔트리 valid=0 발견 → 폴트 발생
2. OS 페이지 폴트 핸들러 진입
3. 빈 프레임 확보 (없으면 교체 정책으로 하나 축출)
4. 디스크에서 페이지를 그 프레임으로 로드
5. 페이지 테이블 갱신 (valid=1, PPN 설정)
6. 폴트 낸 명령 재시도 → 이번엔 성공
```

디스크 접근(~100μs, [[memory-hierarchy]])이라 RAM 접근(~100ns)의 1000배. 페이지 폴트 하나가 엄청 비쌈 → 최소화가 관건.

## 교체 정책: 누구를 축출하나

빈 프레임이 없으면 하나를 골라 내보내야 함. 미래에 가장 늦게 쓰일 페이지를 내보내는 게 최적(OPT/Belady)이지만 미래를 모름 → 과거로 근사:

| 정책 | 규칙 | 문제 |
|---|---|---|
| **OPT** | 가장 먼 미래에 쓸 것 | 미래를 알아야 함 (이론적 상한) |
| **FIFO** | 가장 오래 들어온 것 | 자주 쓰는 페이지도 축출. Belady 이상현상 |
| **LRU** | 가장 오래 안 쓴 것 | 지역성 활용, 좋지만 완벽 구현 비쌈 |
| **Clock** | LRU 근사 (accessed 비트 순회) | 실용적, 대부분 OS가 사용 |

### Clock (LRU 근사)

완벽한 LRU는 매 접근마다 순서 갱신 → 비쌈. Clock은 근사:

- 페이지들을 원형으로, 시곗바늘이 돌며 검사
- accessed 비트가 1이면 → 0으로 바꾸고 넘어감 (한 번 봐줌)
- accessed 비트가 0이면 → 축출 (최근 안 쓰임)
- 페이지 테이블의 accessed 비트를 하드웨어가 자동 세팅 → 싸게 근사

## Thrashing: working set이 넘칠 때

물리 메모리 < working set([[cache-misses]]의 그 개념)이면:

- 페이지를 축출하자마자 곧 다시 필요 → swap in
- 그걸 위해 또 축출 → 그것도 곧 필요
- **거의 모든 접근이 페이지 폴트** → CPU는 노는데 디스크만 미친 듯 → 시스템이 기어감

메모리 부족 시 갑자기 극단적으로 느려지는 현상. 지역성이 무너진 극한. 해결:
- working set을 줄이거나 (메모리 추가, 프로세스 줄임)
- OS가 일부 프로세스를 통째 스왑아웃(admission control)해 나머지를 살림
- 리눅스 **OOM killer**: 극한이면 프로세스를 죽여 시스템을 구함

## SSD 시대의 스왑

- 스왑이 HDD면 thrashing = 재앙 (밀리초 지연). SSD면 덜하지만 여전히 RAM보다 훨씬 느림
- **zram/압축 스왑**: 페이지를 디스크 대신 RAM에서 압축 보관 → 디스크 왕복 회피
- 모바일(iOS/안드로이드)은 스왑 대신 앱을 종료/재시작하는 전략

## 연결

- 하드웨어 변환과 valid 비트 → [[virtual-memory]], [[page-tables]]
- 페이지 폴트 메커니즘 → [[exceptions-and-interrupts]]
- working set과 지역성 → [[cache-misses]], [[memory-hierarchy]]
- RAM=디스크 캐시의 반대편 → [[page-cache]]
