# 세마포어 (Semaphores)

## 한 줄 요약

정수 카운터 + 두 원자 연산(wait/post)으로 된 동기화 도구. 하나로 락도, 조건 변수도, 자원 개수 제한도 표현할 수 있는 범용 원시타입이다.

## 왜 필요한가

- 락과 조건 변수를 하나로 통합하는 추상
- "동시에 N개까지만 허용" 같은 자원 제한을 어떻게 표현하나
- reader-writer, 생산자-소비자의 간결한 해법

## 정의

세마포어 = 정수 값 + 두 원자적 연산:

```
wait(s)  (P, down): s--; s<0 이면 블록 (대기)
post(s)  (V, up)  : s++; 대기자 있으면 하나 깨움
```

- 값이 **자원의 개수**를 의미. 음수면 절댓값 = 대기 중인 스레드 수
- 두 연산은 원자적 (내부적으로 락으로 보호)

## 세 가지 용법

### 1. 이진 세마포어 = 락

초기값 1 → 상호 배제 ([[locks]]):

```c
sem_t m; sem_init(&m, 0, 1);   // 초기값 1
sem_wait(&m);                  // 획득 (1→0)
critical_section();
sem_post(&m);                  // 해제 (0→1)
```

값이 1이면 한 스레드 진입(0으로), 다음은 0에서 wait → 블록. 락과 동일.

### 2. 카운팅 세마포어 = 자원 제한

초기값 N → **동시에 N개까지 허용**:

```c
sem_t slots; sem_init(&slots, 0, 5);   // 커넥션 풀 5개
sem_wait(&slots);   // 슬롯 하나 점유 (없으면 대기)
use_connection();
sem_post(&slots);   // 반납
```

DB 커넥션 풀, 스레드 풀, 동시 다운로드 제한 등. "N개 자원"을 자연스럽게 표현 - 락으론 어색한 것.

### 3. 순서 강제 = 조건 변수 대용

초기값 0 → **한 스레드가 다른 스레드를 기다림**:

```c
sem_t done; sem_init(&done, 0, 0);   // 초기값 0
// 스레드A: sem_wait(&done);  // B가 끝날 때까지 대기 (0에서 블록)
// 스레드B: work(); sem_post(&done);  // 완료 신호 (0→1, A 깨움)
```

값 0에서 wait은 블록, post가 깨움 → 조건 변수([[condition-variables]])의 신호/대기와 같은 효과.

## 생산자-소비자 (세마포어 버전)

[[condition-variables]]의 그 문제를 세마포어로 더 간결하게:

```c
sem_t empty; sem_init(&empty, 0, SIZE);  // 빈 슬롯 수
sem_t full;  sem_init(&full, 0, 0);      // 찬 슬롯 수
sem_t mutex; sem_init(&mutex, 0, 1);     // 버퍼 보호

// 생산자
sem_wait(&empty);   // 빈자리 하나 확보 (없으면 대기)
sem_wait(&mutex); buffer_put(item); sem_post(&mutex);
sem_post(&full);    // 항목 하나 알림

// 소비자
sem_wait(&full);    // 항목 하나 확보 (없으면 대기)
sem_wait(&mutex); item=buffer_get(); sem_post(&mutex);
sem_post(&empty);   // 빈자리 하나 알림
```

`empty`/`full`이 슬롯 개수를 카운팅 → while 루프 없이 대기 자동. 조건 변수 버전보다 간결. **wait 순서 주의**: 자원 세마포어(empty/full)를 mutex보다 먼저 - 뒤집으면 데드락 ([[deadlock]]).

## reader-writer 락

읽기는 동시 허용, 쓰기는 배타적. 세마포어로 구현:

- 읽는 중엔 여러 reader 허용 (읽기끼리 충돌 없음)
- writer는 혼자만 (읽기/쓰기 다 배제)
- reader 카운트를 세마포어로 보호, writer용 세마포어로 배타 제어
- 읽기 많은 워크로드에 유리 (락보다 병렬성↑). 단 writer 기아 주의

## 세마포어 vs 다른 도구

| | 세마포어 | 락 | 조건 변수 |
|---|---|---|---|
| 표현력 | 범용 (셋 다 가능) | 상호 배제만 | 조건 대기 |
| 상태 | 정수 카운터 | 이진 | 없음(락에 의존) |
| 함정 | 값 의미 헷갈림, wait/post 짝 실수 | 단순 | while 필수 |

세마포어는 강력하지만 값의 의미를 추적해야 해서 실수하기 쉬움. 현대 코드는 명확성을 위해 락+CV를 선호하기도. 하지만 자원 카운팅엔 세마포어가 자연스러움.

## 셀프 체크

> [!question]- 세마포어 값이 음수일 때 그 값의 의미는 무엇인가?
> 세마포어 값은 자원의 개수를 뜻하고, 음수가 되면 그 절댓값이 현재 대기(블록) 중인 스레드 수를 의미한다. 예를 들어 값이 -3이면 자원이 없어 wait에서 블록된 스레드가 3개라는 뜻이다.

> [!question]- 초기값 1, N, 0인 세마포어는 각각 어떤 용법에 대응하는가?
> 초기값 1은 이진 세마포어 = 락(상호 배제, 한 스레드만 진입). 초기값 N은 카운팅 세마포어 = 자원 제한(동시에 N개까지 허용, 예: 커넥션 풀). 초기값 0은 순서 강제 = 조건 변수 대용(한 스레드가 다른 스레드의 완료 신호를 기다림).

> [!question]- 왜 자원 카운팅에는 락보다 세마포어가 자연스러운가?
> 락은 이진 상태(잠김/풀림)만 표현해 "동시에 N개까지 허용" 같은 개수 제한을 어색하게 만든다. 세마포어는 정수 카운터라 초기값 N으로 N개 자원을 자연스럽게 표현하고, wait/post가 자동으로 개수를 세며 남는 슬롯이 없으면 대기시킨다. DB 커넥션 풀, 동시 다운로드 제한 등에 딱 맞는다.

> [!question]- 생산자-소비자에서 세마포어 버전이 조건 변수 버전보다 간결한 이유는?
> `empty`/`full` 세마포어가 빈 슬롯·찬 슬롯 개수를 카운팅하므로, wait 자체가 "자원이 생길 때까지" 자동으로 대기한다. 조건 변수처럼 상태를 다시 확인하는 while 루프(가짜 깨어남/재검사)가 필요 없다. 그래서 코드가 더 간결하다.

## 연습문제

> [!example]- 문제: 생산자-소비자 코드에서 생산자의 wait 순서를 `sem_wait(&mutex); sem_wait(&empty);`로 뒤집으면 어떤 문제가 생기는지 시나리오로 보여라.
> **풀이**
> 데드락이 생긴다. 버퍼가 꽉 찬 상태를 가정하자. 생산자가 먼저 `mutex`를 잡고(버퍼 잠금) 그 다음 `empty`를 wait하는데, 빈 슬롯이 없어 `empty`에서 블록된다. 문제는 mutex를 쥔 채 블록된다는 것.
> 소비자는 항목을 꺼내 `empty`를 post해줘야 생산자가 깨는데, 소비자가 버퍼에 접근하려면 먼저 `mutex`를 잡아야 한다. 그 mutex는 생산자가 쥐고 안 놓는다. 서로 상대를 기다리며 영원히 멈춘다.
> 해결: 자원 세마포어(empty/full)를 항상 mutex보다 먼저 wait한다. 그러면 자원을 확보한 뒤에야 잠그므로 mutex를 쥔 채 블록되지 않는다.

> [!example]- 문제: 스레드 A는 초기화(config 로드)를 끝낸 뒤에만 스레드 B가 요청 처리를 시작하게 하고 싶다. 세마포어로 설계하라.
> **풀이**
> 순서 강제 용법(초기값 0)을 쓴다.
> ```c
> sem_t ready; sem_init(&ready, 0, 0);   // 초기값 0
> // 스레드 A: load_config(); sem_post(&ready);   // 완료 신호 (0→1)
> // 스레드 B: sem_wait(&ready); handle_requests();  // A 끝날 때까지 블록
> ```
> B가 먼저 도달하면 값 0에서 wait으로 블록되고, A가 초기화 후 post하면 깨어난다. A가 먼저 post해도 값이 1로 남아 B의 wait이 즉시 통과하므로, 실행 순서와 무관하게 "A 완료 후 B 시작"이 보장된다.

> [!example]- 문제: 최대 3개까지만 동시 접속을 허용하는 서버를 세마포어로 설계하고, 4번째 요청이 어떻게 처리되는지 설명하라.
> **풀이**
> 카운팅 세마포어를 초기값 3으로 만든다.
> ```c
> sem_t slots; sem_init(&slots, 0, 3);
> // 요청 처리: sem_wait(&slots); serve(); sem_post(&slots);
> ```
> 요청 1~3이 각각 `slots`를 wait해 값이 3→2→1→0이 되고 모두 진행한다. 4번째 요청은 값 0에서 wait하며 블록된다(값은 -1, 대기자 1명). 앞선 요청 중 하나가 끝나 `sem_post`를 하면 대기 중이던 4번째가 깨어나 슬롯을 점유한다. 이렇게 동시 처리 수가 항상 3 이하로 유지된다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지를 짧게.
> 1. 세마포어 = 정수 카운터 + 원자적 wait(P, s--, s<0이면 블록)/post(V, s++, 대기자 깨움)의 정의.
> 2. 초기값 1/N/0이 각각 락·자원 제한·순서 강제(조건 변수 대용)에 대응한다는 것.
> 3. 생산자-소비자에서 empty/full/mutex의 역할과, 왜 자원 세마포어를 mutex보다 먼저 wait해야 데드락을 피하는지.
