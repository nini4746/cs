# 주소 공간 (Address Spaces)

## 한 줄 요약

OS는 각 프로세스에게 "0번지부터 시작하는 연속된 메모리를 통째로 가진 것 같은" 환상을 준다. 실제 물리 메모리는 파편화돼 있고 공유되지만, 가상 주소 공간이 이를 감춰 격리와 단순함을 동시에 준다.

## 왜 필요한가

- 포인터를 찍으면 나오는 주소가 뭘 의미하나
- 코드/데이터/힙/스택이 메모리 어디에 있나
- 왜 모든 프로세스가 비슷한 주소를 쓰는데 안 충돌하나
- 스택 오버플로우/힙 관리의 무대 ([[memory-allocation]], [[buffer-overflow]])

## 환상: 각자 통째로 가진 메모리

물리 메모리는 하나인데 여러 프로세스가 돎. OS는 각 프로세스에게 **자기만의 가상 주소 공간**을 줌:

- 프로세스 입장: "0번지(사실은 상위)부터 큰 연속 메모리가 다 내 것"
- 실제: 조각조각 흩어진 물리 페이지에 매핑, 안 쓰는 건 디스크에 ([[virtual-memory]])
- 프로세스끼리 같은 가상 주소라도 다른 물리 주소 → 격리

이 추상이 프로그래밍을 단순하게(연속 메모리처럼) + 안전하게(격리) 만듦.

## 주소 공간의 구성

전형적 레이아웃 (낮은 주소 → 높은 주소):

```
높은 주소  ┌──────────────┐
          │ 스택          │ ↓ 아래로 자람 (함수 호출)
          │   ⋮          │
          │              │
          │   ⋮          │
          │ 힙           │ ↑ 위로 자람 (malloc)
          ├──────────────┤
          │ BSS (0초기화) │
          │ 데이터 (초기화)│
          │ rodata (상수) │
          │ 코드 (text)   │
낮은 주소  └──────────────┘
```

이 머신 실측 (주소값):

```
code (func)   : 0x100610548   ← 코드 (가장 낮음)
rodata (str)  : 0x10061065c   ← 읽기전용 상수
data (init)   : 0x100618000   ← 초기화된 전역
bss  (uninit) : 0x100618010   ← 0으로 초기화될 전역
heap (malloc) : 0x100cd18b0   ← 힙 (데이터 위)
stack (local) : 0x16f7eec48   ← 스택 (훨씬 높음)
```

순서가 그대로 확인됨. 힙과 스택 사이 거대한 빈 공간 → 둘이 서로를 향해 자라며 확장.

## 각 영역

| 영역 | 내용 | 특성 |
|---|---|---|
| **코드(text)** | 기계어 명령 | 읽기+실행, 쓰기 금지 (NX 반대) |
| **rodata** | 문자열 리터럴, const | 읽기 전용 → `str[0]='x'` 크래시 |
| **데이터(data)** | 초기화된 전역/정적 | 읽기+쓰기 |
| **BSS** | 0으로 초기화되는 전역 | 파일에 0을 저장 안 함, 로드 시 0으로 채움 (실행 파일 크기 절약) |
| **힙(heap)** | malloc/free | 동적, 위로 성장 → [[memory-allocation]] |
| **스택(stack)** | 지역변수, 호출 프레임 | 아래로 성장 → [[procedures-and-stack]] |

## 세그폴트의 정체

각 영역엔 권한(읽기/쓰기/실행)이 있음. 위반하면 **세그멘테이션 폴트**:

- NULL 역참조 (0번지는 매핑 안 됨)
- rodata에 쓰기 (`char *s="x"; s[0]='y'`)
- 코드 영역 실행 외 접근
- 매핑 안 된 주소 접근

하드웨어(MMU)가 감지 → 페이지 폴트 → OS가 "불법 접근"이면 SIGSEGV ([[virtual-memory]], [[exceptions-and-interrupts]]).

## copy-on-write: fork를 싸게

[[process]]의 fork가 주소 공간을 통째 복사하면 비쌈. 실제로는 **copy-on-write(COW)**:

1. fork 직후 부모와 자식이 **같은 물리 페이지를 공유** (읽기 전용으로 표시)
2. 둘 중 하나가 쓰려 하면 → 페이지 폴트 → 그 페이지만 복사
3. 안 바뀌는 페이지는 계속 공유

대부분 페이지는 안 바뀌므로(특히 fork 직후 exec하면 전부 버려짐) 복사가 거의 없음 → fork가 빠른 이유. 가상 메모리 간접이 이걸 가능케 함.

## 셀프 체크

> [!question]- 프로세스가 "0번지부터 연속된 메모리를 통째로 가진 것 같은" 환상을 OS가 주는 이유는?
> 물리 메모리는 하나인데 여러 프로세스가 나눠 쓴다. 각 프로세스에 자기만의 가상 주소 공간을 주면, 프로그래밍이 단순해지고(연속 메모리처럼 다룸) 프로세스끼리 격리된다(같은 가상 주소라도 다른 물리 주소로 매핑). 단순함과 안전을 동시에 얻는다.

> [!question]- 힙과 스택이 주소 공간에서 서로 반대 방향으로 자라도록 배치한 이유는?
> 힙은 낮은 주소(데이터 위)에서 위로, 스택은 높은 주소에서 아래로 자란다. 둘 사이에 거대한 빈 공간을 두고 서로를 향해 자라게 하면, 한쪽이 많이 필요하면 그쪽으로 확장할 수 있어 고정 분할보다 공간을 유연하게 나눠 쓴다.

> [!question]- BSS 영역은 왜 실행 파일에 실제 0 데이터를 저장하지 않나?
> BSS는 0으로 초기화되는 전역/정적 변수 영역이다. 어차피 전부 0이므로 파일에 0을 실제로 담을 필요가 없고, 로드 시 크기만 알고 0으로 채우면 된다. 덕분에 실행 파일 크기가 줄어든다.

> [!question]- fork가 주소 공간을 통째로 복사하지 않고도 빠른 이유는?
> copy-on-write(COW) 때문이다. fork 직후 부모와 자식이 같은 물리 페이지를 읽기 전용으로 공유하다가, 누군가 쓰려 할 때만 그 페이지를 복사한다. 대부분 페이지는 안 바뀌고, 특히 fork 후 곧바로 exec하면 전부 버려지므로 복사가 거의 없다.

## 연습문제

> [!example]- 문제: 이 머신 실측 주소 6개를 낮은 주소부터 정렬하고, 각각 어느 영역인지 판정하라
> **풀이**
> 낮은 주소 → 높은 주소 순:
> 1. `0x100610548` code(text) - 기계어, 가장 낮음
> 2. `0x10061065c` rodata - 읽기 전용 상수/문자열
> 3. `0x100618000` data - 초기화된 전역
> 4. `0x100618010` bss - 0 초기화 전역 (data 바로 위)
> 5. `0x100cd18b0` heap - malloc, 데이터 위로 성장
> 6. `0x16f7eec48` stack - 지역변수, 훨씬 높은 주소에서 아래로 성장
> heap과 stack 사이 주소 격차가 거대한 빈 공간이며, 둘이 서로를 향해 자란다.

> [!example]- 문제: `char *s = "hello"; s[0] = 'H';` 가 세그폴트를 내는 이유를 권한 관점에서 설명하라
> **풀이**
> `"hello"` 리터럴은 rodata(읽기 전용) 영역에 저장된다. 각 영역에는 읽기/쓰기/실행 권한이 있는데 rodata는 쓰기 금지다. `s[0]='H'`는 이 영역에 쓰기를 시도하는 권한 위반이다. MMU가 감지 → 페이지 폴트 → OS가 불법 접근으로 판단해 SIGSEGV를 보낸다. `char s[] = "hello";`로 스택에 복사본을 두면 쓰기 가능해진다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 가상 주소 공간이 주는 두 가지 이득(단순함, 격리)을 물리 메모리 현실과 연결해 설명할 수 있는가
> 2. 코드/rodata/data/bss/힙/스택의 순서와 각 권한, 성장 방향을 그림 없이 그려낼 수 있는가
> 3. 세그폴트가 나는 경로(권한 위반 → MMU → 페이지 폴트 → SIGSEGV)와 COW가 fork를 싸게 만드는 원리를 말할 수 있는가
