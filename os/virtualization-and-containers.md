# 가상화와 컨테이너 (Virtualization and Containers)

## 한 줄 요약

VM은 하드웨어를 통째 가상화해 게스트 OS를 통째 돌리고, 컨테이너는 하나의 커널을 공유하며 프로세스를 격리한다. 컨테이너는 가볍지만 격리가 약하고, VM은 무겁지만 강하게 격리한다.

## 왜 필요한가

- Docker 컨테이너가 실제로 뭔가 (가벼운 VM? 아님)
- VM과 컨테이너의 진짜 차이
- namespace/cgroup이 뭘 하나 → devops/[[docker-internals]]의 기반

## 가상 머신 (VM)

**하이퍼바이저**가 하드웨어를 가상화 → 그 위에 게스트 OS를 통째 실행:

```
[게스트 OS + 앱]  [게스트 OS + 앱]   ← 각자 완전한 OS (커널 포함)
        하이퍼바이저 (Type 1: 베어메탈 / Type 2: 호스트 OS 위)
              물리 하드웨어
```

- 각 VM이 자기 커널·주소 공간·가상 하드웨어를 가짐
- 하이퍼바이저가 특권 명령을 가로채 가상화 (하드웨어 지원: Intel VT-x, ARM 가상화 확장)
- **강한 격리**: VM 하나가 커널째 죽어도 다른 VM 무사. 다른 OS도 가능(리눅스 위에 윈도우)
- **무거움**: OS 전체 부팅, GB 단위 메모리/디스크, 시작 수십 초

## 컨테이너

**하나의 호스트 커널을 공유**하면서 프로세스 그룹을 격리:

```
[앱]  [앱]  [앱]      ← 그냥 프로세스들, 각자 격리된 뷰
   공유 호스트 커널
     물리 하드웨어
```

컨테이너 = "격리된 뷰를 가진 프로세스". 게스트 OS 없음. 커널은 하나. 리눅스 두 기능으로 구현:

### namespace: 뭘 보나 격리

프로세스가 보는 시스템 자원을 분리 ([[process]]):

- **PID namespace**: 컨테이너 안에서 자기 프로세스만 보임. 컨테이너의 첫 프로세스가 PID 1
- **network namespace**: 자기 네트워크 스택(인터페이스, 포트)
- **mount namespace**: 자기 파일시스템 뷰 (자기 루트)
- **user, UTS, IPC namespace**: 사용자 ID, 호스트명, IPC 격리

호스트에선 그냥 프로세스인데, 컨테이너 안에선 "혼자만의 시스템"처럼 보임.

### cgroup: 얼마나 쓰나 제한

자원 사용량 제한/측정:

- CPU: 이 컨테이너는 CPU 20%까지
- 메모리: 512MB 초과하면 OOM
- I/O 대역폭, 프로세스 수 등
- 한 컨테이너가 자원을 독점해 이웃을 굶기는 것 방지

**namespace(격리된 뷰) + cgroup(자원 제한) = 컨테이너.** Docker는 이 위의 편의 도구(이미지, 배포).

## VM vs 컨테이너

| | VM | 컨테이너 |
|---|---|---|
| 격리 | 강함 (하드웨어 레벨) | 약함 (커널 공유) |
| 커널 | 각자 | 공유 (호스트) |
| 크기 | GB, 무거움 | MB, 가벼움 |
| 시작 | 수십 초 | 밀리초~초 |
| 밀도 | 낮음 (호스트당 수십) | 높음 (수백~수천) |
| 다른 OS | 가능 | 불가 (커널 공유) |
| 보안 경계 | 강함 | 약함 (커널 취약점 = 탈출) |

## 격리의 대가와 위험

컨테이너는 **커널을 공유**하므로:

- 커널 취약점 하나로 컨테이너 탈출 가능 → 호스트/이웃 침해
- 진짜 신뢰 경계가 필요하면(멀티테넌트) VM 또는 경량 VM(gVisor, Firecracker, Kata) - "VM의 격리 + 컨테이너의 속도"를 노림
- 그래서 클라우드는 종종 **VM 안에 컨테이너**를 돌림 (2중 방어)

## macOS/Windows의 Docker

Docker는 리눅스 namespace/cgroup에 의존 → macOS/Windows엔 그게 없음. 그래서 이 OS들의 Docker는 **가벼운 리눅스 VM을 띄우고 그 안에서** 컨테이너 실행 (macOS는 Virtualization.framework). "컨테이너는 리눅스 기술"인 이유.

## 셀프 체크

> [!question]- 컨테이너와 VM의 격리는 근본적으로 무엇이 다른가?
> VM은 하이퍼바이저가 하드웨어를 가상화해 각 게스트가 자기 커널을 통째로 돌린다 - 하드웨어 레벨 격리라 VM 하나가 커널째 죽어도 다른 VM은 무사하고 강하다. 컨테이너는 하나의 호스트 커널을 공유하며 프로세스를 격리할 뿐이라, 격리가 약하고 커널 취약점 하나로 경계가 뚫릴 수 있다.

> [!question]- namespace와 cgroup의 역할 차이는?
> namespace는 "뭘 보나"를 격리한다 - 프로세스가 보는 PID, 네트워크, 마운트, 사용자 등의 뷰를 분리해 컨테이너 안에서 혼자만의 시스템처럼 보이게 한다. cgroup은 "얼마나 쓰나"를 제한한다 - CPU, 메모리, I/O 대역폭 등 자원 사용량을 제한/측정해 한 컨테이너가 이웃을 굶기지 못하게 한다. 둘을 합쳐야 컨테이너가 된다.

> [!question]- 컨테이너 탈출이 왜 가능한가?
> 컨테이너는 호스트 커널을 공유하기 때문이다. 커널 취약점 하나가 있으면 격리된 뷰를 뚫고 나와 호스트나 이웃 컨테이너를 침해할 수 있다. VM은 하드웨어 경계로 나뉘어 있어 이런 탈출이 훨씬 어렵다.

> [!question]- macOS/Windows의 Docker는 왜 리눅스 VM을 띄우나?
> 컨테이너는 리눅스의 namespace/cgroup 기능에 의존하는데 macOS/Windows 커널엔 그게 없기 때문이다. 그래서 이 OS들의 Docker는 가벼운 리눅스 VM(macOS는 Virtualization.framework)을 띄우고 그 안에서 컨테이너를 실행한다. "컨테이너는 리눅스 기술"인 이유다.

> [!question]- 클라우드가 왜 VM 안에서 컨테이너를 돌리나?
> 컨테이너만으로는 커널 공유 때문에 신뢰 경계가 약해서다. 서로 신뢰할 수 없는 여러 테넌트를 한 호스트에 태울 때, 바깥은 VM으로 강하게 격리하고 그 안에서 컨테이너의 가벼움과 밀도를 누리는 2중 방어를 쓴다.

## 연습문제

> [!example]- 문제: 워크로드에 맞는 격리 방식을 골라라
> 멀티테넌트 SaaS에서 신뢰할 수 없는 고객이 올린 임의 코드를 실행해야 한다. (a) 일반 컨테이너, (b) VM, (c) 경량 VM(gVisor/Firecracker/Kata) 중 무엇을 택하고 근거를 대라.
> **풀이**
> 신뢰 경계가 핵심이다. (a) 일반 컨테이너는 호스트 커널을 공유하므로 커널 취약점 하나로 컨테이너 탈출 → 다른 고객/호스트 침해 위험이 있어 부적합. (b) VM은 하드웨어 레벨로 강하게 격리하지만 OS 전체 부팅, GB 메모리, 수십 초 시작으로 밀도가 낮아 다수 고객 코드를 잦게 띄우기엔 무겁다. (c) 경량 VM이 최적 - "VM의 격리 + 컨테이너의 속도"를 노려, 각 워크로드를 얇은 VM 경계로 감싸면서도 빠르게 뜬다. 실무에선 VM(또는 경량 VM) 안에 컨테이너를 돌리는 2중 방어가 흔하다.

> [!example]- 문제: 호스트 밀도를 계산하라
> 물리 호스트에 메모리 64GB가 있다. VM 하나는 평균 2GB, 컨테이너 하나는 평균 256MB를 쓴다(오버헤드 무시). 각각 몇 개를 태울 수 있는지 계산하고, 이 차이가 어디서 오는지 설명하라.
> **풀이**
> - VM: 64GB / 2GB = 최대 32개
> - 컨테이너: 64GB / 0.25GB = 최대 256개
> 약 8배 차이. 원인: VM은 각자 완전한 게스트 OS(커널 포함)를 메모리에 올려야 해 기본 비용이 크다. 컨테이너는 호스트 커널 하나를 공유하고 자기 프로세스 자원만 쓰므로 훨씬 가볍다. 실제로는 페이지 공유·오버헤드로 수치가 달라지지만, "커널을 공유하느냐"가 밀도 차이의 근본이다.

> [!example]- 문제: namespace 동작을 진단하라
> 컨테이너 안에서 `ps`를 실행하니 자기 프로세스 몇 개만 보이고 그 첫 프로세스가 PID 1이다. 호스트에서 같은 프로세스를 보면 PID가 전혀 다르고 수백 개 프로세스가 함께 보인다. 왜 이런지 설명하라.
> **풀이**
> PID namespace 때문이다. 컨테이너는 자기만의 PID namespace를 가져, 그 안에서 시작한 첫 프로세스가 PID 1(격리된 뷰의 init)이 되고 오직 같은 namespace의 프로세스만 보인다. 반면 호스트는 전역 PID namespace에서 그 프로세스에 다른(전역) PID를 부여하고 모든 프로세스를 본다. 즉 같은 하나의 프로세스가 namespace에 따라 다른 PID로 보이는 것이며, 컨테이너 = "격리된 뷰를 가진 프로세스"라는 정의를 그대로 보여준다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> - VM(하드웨어 가상화, 각자 커널)과 컨테이너(커널 공유, 프로세스 격리)의 구조 차이를, 격리 강도·크기·시작 속도·밀도의 트레이드오프로 이어 설명할 수 있는가.
> - namespace("뭘 보나" 격리)와 cgroup("얼마나 쓰나" 제한)이 각각 무엇을 하고 왜 둘 다 있어야 컨테이너가 되는지 말할 수 있는가.
> - 커널 공유가 왜 격리 약화(컨테이너 탈출)로 이어지고, 그래서 경량 VM이나 "VM 안 컨테이너" 2중 방어가 왜 나오는지 설명할 수 있는가.
