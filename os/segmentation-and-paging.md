# 세그멘테이션에서 페이징으로 (Segmentation and Paging)

## 한 줄 요약

가상 주소 공간을 물리 메모리에 어떻게 놓을까. 통짜(base+bound)→가변 조각(세그멘테이션)→고정 조각(페이징)으로 진화했고, 페이징이 이긴 이유는 외부 단편화를 없앴기 때문이다.

## 왜 필요한가

- [[address-spaces]]의 환상을 실제로 어떻게 구현하나
- 왜 현대 OS는 페이징을 쓰나 (세그멘테이션 아니라)
- 단편화가 뭐고 왜 문제인가
- [[virtual-memory]] 하드웨어의 소프트웨어 짝

## 1단계: base + bound (통짜 재배치)

가장 단순: 각 프로세스에 물리 메모리의 **연속 구간** 하나를 통째 줌.

- **base 레지스터**: 이 프로세스가 시작하는 물리 주소
- **bound 레지스터**: 크기 (경계 검사용)
- 변환: `물리 = base + 가상`, `가상 < bound` 검사

문제: 주소 공간 전체(코드+힙+스택)를 연속으로 놔야 함. 힙과 스택 사이 거대한 빈 공간까지 물리 메모리를 차지 → 낭비.

## 2단계: 세그멘테이션 (가변 조각)

주소 공간을 논리 단위(코드/힙/스택)로 쪼개 **각 세그먼트를 따로 배치**:

- 세그먼트마다 base+bound
- 힙-스택 사이 빈 공간은 물리 메모리에 안 놓음 → 낭비 감소
- 세그먼트별 권한(코드=실행, 데이터=쓰기) 가능

문제: **외부 단편화(external fragmentation)**. 세그먼트 크기가 제각각 → 프로세스가 뜨고 지며 물리 메모리에 여기저기 빈 구멍. 총 여유는 충분한데 연속된 큰 구멍이 없어 할당 실패:

```
[프로세스A][빈 20KB][프로세스B][빈 30KB][프로세스C]
→ 총 50KB 비었지만 40KB 세그먼트 못 넣음 (연속 아님)
```

압축(compaction)으로 모을 수 있지만 비쌈 (전부 복사).

## 3단계: 페이징 (고정 조각) - 승자

주소 공간을 **고정 크기 페이지**(4KB/16KB)로, 물리 메모리를 같은 크기 **프레임**으로 나눔. 각 가상 페이지를 아무 물리 프레임에나 매핑:

- **외부 단편화 없음**: 모든 조각이 같은 크기 → 어떤 빈 프레임에나 들어감. 구멍 문제 소멸
- 매핑은 **페이지 테이블**이 저장 (VPN → PPN) → [[page-tables]], [[virtual-memory]]
- 연속된 가상 주소가 물리적으론 흩어져도 됨 → 자유로운 배치

### 페이징의 대가

- **내부 단편화(internal fragmentation)**: 페이지 마지막이 안 차면 낭비. 하지만 페이지가 작아(4KB) 손실 적음. 외부 단편화보다 훨씬 관리 쉬움
- **페이지 테이블 크기**: 매핑 정보 저장 공간 → 멀티레벨로 해결 ([[page-tables]])
- **변환 오버헤드**: 매 접근마다 테이블 조회 → TLB로 해결 ([[virtual-memory]])

세 문제 다 감당 가능해서 페이징이 표준이 됨.

## 왜 페이징이 이겼나 정리

| | 세그멘테이션 | 페이징 |
|---|---|---|
| 조각 크기 | 가변 | 고정 |
| 외부 단편화 | 있음 (심각) | **없음** |
| 내부 단편화 | 적음 | 있음 (작음) |
| 배치 유연성 | 제한 | 자유 |
| 관리 복잡도 | 압축 필요 | 단순 |

고정 크기의 단순함이 가변 크기의 유연함을 이겼다. 현대 시스템은 순수 페이징 (x86의 세그먼트는 사실상 무력화).

## 연결

- 페이지 테이블 구조와 관리 → [[page-tables]]
- 하드웨어 변환과 TLB → [[virtual-memory]]
- 안 쓰는 페이지를 디스크로 → [[swapping]]
- 주소 공간 추상 → [[address-spaces]]
