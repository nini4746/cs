# 세그멘테이션에서 페이징으로 (Segmentation and Paging)

## 한 줄 요약

가상 주소 공간을 물리 메모리에 어떻게 놓을까. 통짜(base+bound)→가변 조각(세그멘테이션)→고정 조각(페이징)으로 진화했고, 페이징이 이긴 이유는 외부 단편화를 없앴기 때문이다.

## 왜 필요한가

- [[address-spaces]]의 환상을 실제로 어떻게 구현하나
- 왜 현대 OS는 페이징을 쓰나 (세그멘테이션 아니라)
- 단편화가 뭐고 왜 문제인가
- [[virtual-memory]] 하드웨어의 소프트웨어 짝

## 1단계: base + bound (통짜 재배치)

가장 단순: 각 프로세스에 물리 메모리의 **연속 구간** 하나를 통째 줌.

- **base 레지스터**: 이 프로세스가 시작하는 물리 주소
- **bound 레지스터**: 크기 (경계 검사용)
- 변환: `물리 = base + 가상`, `가상 < bound` 검사

문제: 주소 공간 전체(코드+힙+스택)를 연속으로 놔야 함. 힙과 스택 사이 거대한 빈 공간까지 물리 메모리를 차지 → 낭비.

## 2단계: 세그멘테이션 (가변 조각)

주소 공간을 논리 단위(코드/힙/스택)로 쪼개 **각 세그먼트를 따로 배치**:

- 세그먼트마다 base+bound
- 힙-스택 사이 빈 공간은 물리 메모리에 안 놓음 → 낭비 감소
- 세그먼트별 권한(코드=실행, 데이터=쓰기) 가능

문제: **외부 단편화(external fragmentation)**. 세그먼트 크기가 제각각 → 프로세스가 뜨고 지며 물리 메모리에 여기저기 빈 구멍. 총 여유는 충분한데 연속된 큰 구멍이 없어 할당 실패:

```
[프로세스A][빈 20KB][프로세스B][빈 30KB][프로세스C]
→ 총 50KB 비었지만 40KB 세그먼트 못 넣음 (연속 아님)
```

압축(compaction)으로 모을 수 있지만 비쌈 (전부 복사).

## 3단계: 페이징 (고정 조각) - 승자

주소 공간을 **고정 크기 페이지**(4KB/16KB)로, 물리 메모리를 같은 크기 **프레임**으로 나눔. 각 가상 페이지를 아무 물리 프레임에나 매핑:

- **외부 단편화 없음**: 모든 조각이 같은 크기 → 어떤 빈 프레임에나 들어감. 구멍 문제 소멸
- 매핑은 **페이지 테이블**이 저장 (VPN → PPN) → [[page-tables]], [[virtual-memory]]
- 연속된 가상 주소가 물리적으론 흩어져도 됨 → 자유로운 배치

### 페이징의 대가

- **내부 단편화(internal fragmentation)**: 페이지 마지막이 안 차면 낭비. 하지만 페이지가 작아(4KB) 손실 적음. 외부 단편화보다 훨씬 관리 쉬움
- **페이지 테이블 크기**: 매핑 정보 저장 공간 → 멀티레벨로 해결 ([[page-tables]])
- **변환 오버헤드**: 매 접근마다 테이블 조회 → TLB로 해결 ([[virtual-memory]])

세 문제 다 감당 가능해서 페이징이 표준이 됨.

## 왜 페이징이 이겼나 정리

| | 세그멘테이션 | 페이징 |
|---|---|---|
| 조각 크기 | 가변 | 고정 |
| 외부 단편화 | 있음 (심각) | **없음** |
| 내부 단편화 | 적음 | 있음 (작음) |
| 배치 유연성 | 제한 | 자유 |
| 관리 복잡도 | 압축 필요 | 단순 |

고정 크기의 단순함이 가변 크기의 유연함을 이겼다. 현대 시스템은 순수 페이징 (x86의 세그먼트는 사실상 무력화).

## 셀프 체크

> [!question]- 외부 단편화와 내부 단편화의 차이는? 페이징은 어느 쪽을 없애고 어느 쪽을 남기나?
> 외부 단편화는 가변 크기 조각들이 뜨고 지며 물리 메모리에 흩어진 빈 구멍이 생겨, 총 여유는 충분해도 연속된 큰 공간이 없어 할당이 실패하는 것이다. 내부 단편화는 고정 크기 조각의 마지막이 다 안 차서 생기는 낭비다. 페이징은 모든 조각을 같은 크기로 만들어 외부 단편화를 없애지만, 페이지 끝이 안 차는 내부 단편화는 남긴다(다만 페이지가 작아 손실이 적다).

> [!question]- base+bound 방식의 근본 문제는 무엇이고 세그멘테이션이 그걸 어떻게 완화하나?
> base+bound는 주소 공간 전체(코드+힙+스택)를 물리 메모리에 하나의 연속 구간으로 놔야 해서, 힙과 스택 사이의 거대한 빈 공간까지 물리 메모리를 차지해 낭비가 크다. 세그멘테이션은 주소 공간을 코드/힙/스택 논리 단위로 쪼개 각 세그먼트를 따로 배치하므로, 세그먼트 사이 빈 공간은 물리 메모리에 안 놓아 낭비를 줄인다.

> [!question]- 페이징의 세 가지 대가는 무엇이고 각각 어떻게 해결하나?
> (1) 내부 단편화 - 페이지를 작게(4KB) 해서 손실을 줄인다. (2) 페이지 테이블 크기 - 멀티레벨 페이지 테이블로 해결한다. (3) 변환 오버헤드(매 접근마다 테이블 조회) - TLB(변환 캐시)로 해결한다. 세 문제 다 감당 가능해서 페이징이 표준이 됐다.

> [!question]- 왜 고정 크기(페이징)가 가변 크기(세그멘테이션)를 이겼나?
> 고정 크기는 어떤 빈 프레임에나 어떤 페이지든 들어가므로 외부 단편화 구멍 문제가 소멸하고, 압축(compaction) 같은 비싼 관리가 필요 없다. 세그멘테이션은 유연하지만 외부 단편화가 심각하고 압축 비용이 크다. 고정 크기의 단순함이 가변 크기의 유연함을 이겼다.

## 연습문제

> [!example]- 문제: 페이지 크기 4KB(=2^12), 가상 주소 16비트. 가상 주소 `0x1AF3`이 있고 페이지 테이블에서 VPN→PPN 매핑이 `VPN 1 → PPN 6`이라면 물리 주소를 구하라.
> **풀이**
> 페이지 크기 4KB = 2^12이므로 하위 12비트가 오프셋, 상위 비트가 VPN이다.
> `0x1AF3` = 0001 1010 1111 0011(2). 오프셋(하위 12비트) = `0xAF3`, VPN(상위 4비트) = `0001` = 1.
> VPN 1 → PPN 6이므로 물리 주소 = (PPN 6 << 12) | 오프셋 = `0x6000 + 0xAF3` = `0x6AF3`.
> 핵심: 변환은 VPN만 바꾸고 페이지 내 오프셋은 그대로 유지된다.

> [!example]- 문제: 다음 물리 메모리에서 40KB짜리 세그먼트를 배치할 수 있는가? `[프로세스A][빈 20KB][프로세스B][빈 30KB][프로세스C]`. 페이징이라면 어떻게 달라지나?
> **풀이**
> 세그멘테이션: 총 빈 공간은 50KB지만 연속된 최대 구멍은 30KB뿐이라 40KB 세그먼트는 못 넣는다(외부 단편화). 압축으로 두 구멍을 합치면 50KB 연속 공간이 생겨 넣을 수 있으나 전체 복사 비용이 크다.
> 페이징: 40KB를 4KB 페이지 10개로 쪼개 아무 빈 프레임에나 흩어 배치하면 된다. 빈 20KB(5프레임) + 빈 30KB(약 7프레임)에 나눠 담을 수 있어 연속 공간이 필요 없다. 외부 단편화 문제가 사라진다.

> [!example]- 문제: base+bound 시스템에서 base=0x8000, bound=0x2000일 때 가상 주소 0x1500과 0x2500 접근은 각각 어떻게 되나?
> **풀이**
> 변환 규칙: `가상 < bound` 검사 후 `물리 = base + 가상`.
> 0x1500: 0x1500 < 0x2000이므로 유효. 물리 주소 = 0x8000 + 0x1500 = 0x9500.
> 0x2500: 0x2500 >= 0x2000이므로 경계 위반 → 하드웨어가 예외(protection fault)를 발생시켜 OS가 프로세스를 처리(보통 종료)한다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지를 짧게.
> 1. base+bound → 세그멘테이션 → 페이징의 진화와 각 단계가 해결한 문제.
> 2. 외부 단편화 vs 내부 단편화의 정의와, 페이징이 어느 쪽을 없애는지.
> 3. VPN/오프셋으로 나눠 페이지 테이블(VPN→PPN)로 변환하는 방식과 페이징의 세 가지 대가(내부 단편화·테이블 크기·변환 오버헤드) 해결책.
