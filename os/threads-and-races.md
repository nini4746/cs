# 스레드와 경쟁 조건 (Threads and Races)

## 한 줄 요약

두 스레드가 공유 데이터를 동시에 수정하면, `counter++` 같은 단순 연산도 유실된다. 원인은 read-modify-write가 원자적이지 않아 중간에 끼어들 수 있기 때문. 임계 구역을 상호 배제로 보호해야 한다.

## 왜 필요한가

- 멀티스레드 버그의 근본 (동시성 전체의 출발점)
- `counter++`가 왜 안전하지 않나
- 락([[locks]])이 왜 필요한지의 동기

## 실증: 갱신 유실

스레드 4개가 각각 공유 카운터를 100만 번 증가. 기대값 400만:

```c
long counter = 0;
void* inc(void* a){ for(long i=0;i<1000000;i++) counter++; }
```

이 머신 `-O0`, 3회 실행:

```
expected: 4000000
actual  : 1056714  (LOST UPDATES!)
actual  : 1024376  (LOST UPDATES!)
actual  : 1125304  (LOST UPDATES!)
```

400만이 아니라 100만대, 게다가 **실행마다 값이 다름**. 300만 번의 증가가 사라졌다. 이것이 경쟁 조건.

(주의: `-O1`이면 컴파일러가 `counter += 1000000` 한 방으로 접어 경쟁 창이 거의 사라짐 → 우연히 맞을 수 있음. 버그가 최적화 수준·타이밍에 따라 숨는 게 경쟁 조건의 악질적 특성. `-O0`로 실제 read-modify-write를 유지해야 재현.)

## 원인: ++는 원자적이 아니다

`counter++`는 한 명령처럼 보이지만 실제로 세 단계 ([[assembly-basics]]):

```
1. load  counter → 레지스터   (읽기)
2. add   레지스터, 1          (수정)
3. store 레지스터 → counter   (쓰기)
```

두 스레드가 겹치면:

```
스레드A: load counter (=5)
스레드B: load counter (=5)     ← A가 아직 store 전
스레드A: add → 6, store (=6)
스레드B: add → 6, store (=6)   ← 5+1을 또 함. 하나 유실!
```

두 번 증가했는데 결과는 +1. 이 read-modify-write 사이 끼어듦(interleaving)이 유실의 원인.

## 용어

- **경쟁 조건(race condition)**: 결과가 스레드 실행 순서(스케줄링)에 의존하는 상황
- **임계 구역(critical section)**: 공유 자원에 접근하는 코드 (여기선 `counter++`)
- **상호 배제(mutual exclusion)**: 한 번에 하나의 스레드만 임계 구역에 들어가게 보장
- **원자성(atomicity)**: 연산이 나눌 수 없게(전부 또는 전무) 실행됨

## 왜 어려운가

1. **비결정적**: 스케줄링에 의존 → 재현이 들쭉날쭉. 테스트에서 안 나다 운영에서 터짐
2. **최적화/타이밍에 숨음**: 위처럼 `-O1`에선 안 보일 수 있음
3. **조합 폭발**: 인터리빙 경우의 수가 방대 → 전수 테스트 불가
4. 그래서 **설계로** 막아야 함 (테스트로 못 잡음)

## 해결의 방향

임계 구역을 **상호 배제로 보호**:

```c
lock(&m);
counter++;      // 한 번에 한 스레드만
unlock(&m);
```

- **락/뮤텍스** → [[locks]]: 임계 구역을 감쌈
- **원자적 연산** → CAS, atomic add: 하드웨어가 read-modify-write를 원자적으로 ([[lock-free-basics]])
- **조건 변수** → [[condition-variables]]: 조건 대기
- **세마포어** → [[semaphores]]: 카운팅 기반

위 예제는 `counter++`를 락으로 감싸거나 `atomic_long`으로 바꾸면 정확히 400만.

## 원자적 연산 미리보기

`counter++`를 하드웨어 원자 명령으로:

```c
#include <stdatomic.h>
atomic_long counter = 0;
counter++;   // 원자적 - 유실 없음, 락 불필요
```

하드웨어가 read-modify-write를 하나의 나눌 수 없는 연산으로 실행 ([[cache-coherence]]의 라인 독점 활용). 단순 카운터엔 락보다 이게 나음.

## 셀프 체크

> [!question]- `counter++`는 왜 원자적이지 않은가?
> 한 명령처럼 보이지만 실제로는 load(counter를 레지스터로 읽기) → add(레지스터에 1 더하기) → store(레지스터를 counter에 쓰기) 세 단계다. 이 세 단계 사이에 다른 스레드가 끼어들어(interleaving) 같은 옛 값을 읽으면, 두 번 증가시켰는데 결과는 +1만 되는 갱신 유실이 생긴다.

> [!question]- 경쟁 조건이 왜 테스트로 잡기 어려운가?
> 세 가지 이유다. (1) 비결정적 - 결과가 스케줄링에 의존해 재현이 들쭉날쭉하고 테스트에선 안 나다 운영에서 터진다. (2) 최적화·타이밍에 숨는다 - `-O1`처럼 조건이 바뀌면 안 보일 수 있다. (3) 인터리빙 경우의 수가 방대해 전수 테스트가 불가능하다. 그래서 테스트가 아니라 설계(상호 배제)로 막아야 한다.

> [!question]- `-O1`로 컴파일하면 왜 버그가 사라지는가?
> 컴파일러가 100만 번 반복 증가를 `counter += 1000000` 한 방으로 접어버려서, read-modify-write가 반복되는 경쟁 창이 거의 사라지기 때문이다. 버그가 최적화 수준에 따라 우연히 숨는 것이지 고쳐진 게 아니다. 재현하려면 `-O0`로 실제 세 단계 연산을 유지해야 한다.

> [!question]- 상호 배제(mutual exclusion)와 원자성(atomicity)의 차이는?
> 원자성은 한 연산이 나눌 수 없게(전부 또는 전무) 실행되는 성질이고, 상호 배제는 한 번에 하나의 스레드만 임계 구역에 들어가게 보장하는 것이다. 락으로 임계 구역을 감싸 상호 배제를 만들거나, 하드웨어 원자 명령(CAS 등)으로 연산 자체를 원자적으로 만들어 둘 다 갱신 유실을 막는다.

> [!question]- 단순 카운터에는 왜 락보다 atomic이 나은가?
> 하드웨어가 read-modify-write를 하나의 나눌 수 없는 연산으로 실행(cache line 독점을 활용)하므로, 락의 획득/해제 오버헤드와 컨텍스트 스위치 없이 유실을 막는다. `counter++` 같은 단순 연산이면 `atomic_long`이 락보다 가볍고 빠르다.

## 연습문제

> [!example]- 문제: 인터리빙으로 갱신 유실 시나리오를 그려라
> 두 스레드 A, B가 공유 변수 `counter`(현재 값 5)를 각각 한 번씩 `counter++` 한다. 최종적으로 6이 나오는(하나 유실되는) 명령 인터리빙을 단계별로 쓰고, 왜 7이 아니라 6인지 설명하라.
> **풀이**
> ```
> A: load counter (레지스터A = 5)
> B: load counter (레지스터B = 5)   ← A가 아직 store 전
> A: add → 레지스터A = 6
> A: store → counter = 6
> B: add → 레지스터B = 6            ← 낡은 5를 기반으로 계산
> B: store → counter = 6           ← A의 갱신을 덮어씀
> ```
> 두 스레드가 모두 옛 값 5를 읽었기 때문에 각자 6을 만들어 쓴다. 두 번 증가했으니 7이어야 하지만, B의 store가 A의 결과를 덮어써 하나가 유실되고 6이 된다.

> [!example]- 문제: check-then-act 경쟁을 찾아라
> 아래 출금 코드를 두 스레드가 동시에 실행한다. 잔액 `balance = 100`, 두 스레드 모두 `amount = 80`을 출금하려 한다. 버그를 찾고 최악의 결과를 보여라.
> ```c
> if (balance >= amount) {   // check
>     balance -= amount;     // act
> }
> ```
> **풀이**
> check와 act 사이가 원자적이지 않아 경쟁 조건이 있다. 인터리빙:
> ```
> A: if (100 >= 80) → 참
> B: if (100 >= 80) → 참        ← A가 아직 차감 전
> A: balance = 100 - 80 = 20
> B: balance = 20 - 80 = -60    ← 잔액이 음수로, 이중 인출
> ```
> 총 160을 출금해 잔액이 -60이 된다(초과 인출). 수정: check와 act 전체를 하나의 임계 구역으로 락으로 감싸(`lock; if(...) balance-=amount; unlock;`) 원자적으로 만든다. 단순 감소가 아니라 조건 검사까지 함께 보호해야 한다는 점이 핵심이다.

> [!example]- 문제: 경쟁 조건을 두 가지 방법으로 고쳐라
> 노트의 `counter++` 예제(스레드 4개 × 100만 증가, 기대값 400만)가 항상 정확히 400만이 나오도록 (1) 락 방식과 (2) atomic 방식으로 각각 고치고, 언제 어느 쪽을 택할지 말하라.
> **풀이**
> (1) 락:
> ```c
> lock(&m); counter++; unlock(&m);
> ```
> 임계 구역을 상호 배제로 감싸 한 번에 한 스레드만 증가. 확장성 좋고 여러 문장을 하나로 묶어야 할 때 필수.
> (2) atomic:
> ```c
> #include <stdatomic.h>
> atomic_long counter = 0;
> counter++;   // 하드웨어 원자 명령
> ```
> 하드웨어가 read-modify-write를 원자적으로 처리해 락 없이 유실 없음.
> 선택 기준: 보호할 게 단일 카운터 하나면 atomic이 가볍고 빠르다. 여러 변수/여러 문장을 하나의 불변식으로 묶어야 하면(위 check-then-act처럼) 락이 맞다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> - `counter++`의 세 단계(load-add-store)를 그리고, 두 스레드 인터리빙으로 갱신이 유실되는 과정을 직접 재현할 수 있는가.
> - 경쟁 조건이 왜 테스트가 아니라 설계로 막아야 하는지(비결정성·최적화 은닉·조합 폭발)를 설명할 수 있는가.
> - 상호 배제(락)와 원자성(CAS/atomic)의 차이, 그리고 언제 어느 쪽을 쓰는지 구분할 수 있는가.
