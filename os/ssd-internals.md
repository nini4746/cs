# SSD 내부 (SSD Internals)

## 한 줄 요약

SSD는 탐색 시간이 없지만 "제자리 덮어쓰기"가 불가능하다는 이상한 제약이 있다. 쓰기 전에 큰 블록을 통째로 지워야 하고, 이 때문에 FTL이 논리↔물리 매핑, wear leveling, garbage collection을 수행한다.

## 왜 필요한가

- SSD가 HDD와 왜 근본적으로 다른가
- 쓰기가 읽기보다 왜 복잡하고 느린가
- write amplification, TRIM이 뭔지
- DB/파일시스템이 SSD에 맞춰 설계를 바꾼 이유 → [[lsm-tree]]

## HDD와의 근본 차이

| | HDD | SSD (플래시) |
|---|---|---|
| 접근 방식 | 헤드 이동(탐색) | 전자적, 탐색 없음 |
| 랜덤 접근 | 느림 (탐색 지배) | 빠름 (균일) |
| 지연 | ~ms | ~100μs |
| 제자리 덮어쓰기 | 가능 | **불가능** (핵심 제약) |
| 마모 | 거의 없음 | 셀당 쓰기 횟수 제한 |

SSD는 탐색 시간이 없어 랜덤 접근이 빠름 → [[fast-file-system]]의 지역성 전제가 약해짐. 하지만 새 제약이 생김.

## 플래시의 이상한 규칙

플래시 메모리의 물리적 제약:

- **읽기/쓰기 단위 = 페이지**(4~16KB)
- **지우기 단위 = 블록**(수백 페이지, 훨씬 큼)
- **핵심: 이미 쓴 페이지를 덮어쓸 수 없음.** 다시 쓰려면 그 페이지가 속한 **블록 전체를 먼저 지워야** 함

```
페이지에 쓰기: 빈(지워진) 페이지에만 가능
페이지 수정: 불가. 블록 전체 지우고 다시 써야 함
```

이 "쓰기 전 지우기 + 지우기는 큰 단위" 비대칭이 SSD 복잡도의 근원.

## FTL: 이 복잡도를 감추는 계층

**FTL(Flash Translation Layer)**이 SSD 안(컨트롤러)에서 논리 블록 주소(LBA)를 물리 위치로 매핑하며 제약을 숨김. OS는 그냥 "블록 장치"로 봄 ([[io-devices]]).

### 로그 구조 쓰기 (out-of-place)

페이지를 수정하면 제자리에 안 쓰고:

1. **빈 페이지에 새 버전을 쓰고**
2. FTL 매핑을 새 위치로 갱신
3. 옛 페이지는 "무효(stale)"로 표시

덮어쓰기를 회피 → [[crash-consistency]]의 copy-on-write, [[lsm-tree]]와 같은 로그 구조 아이디어. SSD 자체가 로그 구조.

### Garbage Collection

무효 페이지가 쌓이면 회수 필요:

1. 유효 페이지가 섞인 블록에서 유효한 것만 다른 블록으로 복사
2. 원래 블록을 통째 지움 → 빈 블록 확보

**write amplification**: 사용자가 1을 쓰려는데 GC의 복사 때문에 실제 플래시엔 더 많이 씀. SSD 쓰기 성능/수명을 갉아먹는 주범.

### Wear Leveling

각 블록은 지우기 횟수 제한(수천~수만). 특정 블록만 계속 쓰면 거기만 빨리 죽음 → FTL이 **쓰기를 모든 블록에 고루 분산**해 수명 균등화. 안 쓰는 데이터도 가끔 옮겨 마모를 평준화.

## TRIM: OS가 SSD를 돕기

파일 삭제 시 OS는 블록을 "안 씀"으로 표시하지만 SSD는 모름 (파일시스템 개념이 없음). **TRIM** 명령으로 OS가 "이 LBA는 이제 무효"라고 알림:

- SSD가 그 페이지를 GC 대상으로 → 불필요한 복사 회피 → write amplification 감소
- TRIM 없으면 SSD가 죽은 데이터를 계속 옮김 → 성능 저하

## 설계에 주는 함의

- **큰 순차 쓰기 선호**: GC 부담 적음. 작은 랜덤 쓰기는 write amplification 유발
- **읽기는 걱정 없음**: 랜덤 읽기 빠르고 마모 없음
- **로그 구조 자료구조 유리**: LSM-tree([[lsm-tree]])가 SSD에서 뜬 이유 - 순차 쓰기 + 덮어쓰기 회피
- over-provisioning: SSD가 여유 공간을 숨겨둬 GC 원활하게

## 셀프 체크

> [!question]- SSD는 왜 "제자리 덮어쓰기"가 불가능한가?
> 플래시의 물리적 제약 때문. 읽기/쓰기 단위는 페이지(4~16KB)지만 지우기 단위는 블록(수백 페이지)이라 크기가 비대칭이다. 이미 쓴 페이지를 다시 쓰려면 그 페이지가 속한 블록 전체를 먼저 지워야 한다. 그래서 FTL이 out-of-place로 새 페이지에 쓰고 옛 페이지는 무효로 표시한다.

> [!question]- write amplification은 왜 생기고 왜 문제인가?
> Garbage Collection이 무효 페이지가 섞인 블록을 회수할 때, 그 안의 유효 페이지를 다른 블록으로 복사해야 하기 때문이다. 사용자가 1을 쓰려 해도 GC 복사분까지 실제 플래시엔 더 많이 쓰인다. 이게 쓰기 성능을 깎고 셀 마모를 앞당겨 수명을 갉아먹는다.

> [!question]- TRIM이 없으면 왜 성능이 떨어지나?
> SSD는 파일시스템 개념이 없어 OS가 파일을 삭제해도 그 페이지가 죽었는지 모른다. TRIM으로 "이 LBA는 이제 무효"라고 알려주지 않으면, SSD는 이미 삭제된 죽은 데이터까지 GC 때마다 유효한 줄 알고 계속 복사한다. 곧 불필요한 write amplification과 성능 저하로 이어진다.

> [!question]- wear leveling은 왜 필요하고 무엇을 하나?
> 각 블록은 지우기 횟수 제한(수천~수만)이 있어, 특정 블록에만 쓰기가 몰리면 거기만 빨리 죽고 SSD 전체가 못 쓰게 된다. FTL이 쓰기를 모든 블록에 고루 분산하고, 안 쓰는 데이터도 가끔 옮겨 마모를 평준화해 수명을 균등하게 만든다.

> [!question]- SSD가 큰 순차 쓰기를 선호하는 이유는?
> 큰 순차 쓰기는 블록을 통째로 채우고 통째로 무효화하는 경향이라 GC가 복사할 유효 페이지가 적어 부담이 작다. 반대로 작은 랜덤 쓰기는 유효/무효 페이지를 블록마다 섞어 놓아 GC 복사량을 늘리고 write amplification을 유발한다.

## 연습문제

> [!example]- 문제: GC로 인한 write amplification factor(WAF)를 계산하라
> 블록당 64페이지인 SSD에서 빈 블록이 바닥나 GC가 발동한다. GC 대상 블록은 유효 40페이지, 무효 24페이지다. 호스트는 24페이지 분량의 새 데이터를 쓰려 한다. 이때 WAF를 구하라.
> **풀이**
> WAF = (실제 플래시 쓰기) / (호스트 요청 쓰기).
> - 빈 공간 확보를 위해 GC가 유효 40페이지를 다른 블록으로 복사 → 40페이지 쓰기(호스트가 요청하지 않은 쓰기)
> - 확보된 자리에 호스트의 새 데이터 24페이지 쓰기
> - 실제 플래시 쓰기 = 40 + 24 = 64페이지, 호스트 요청 = 24페이지
> - WAF = 64 / 24 ≈ 2.67
> 유효 페이지 비율이 높은(꽉 찬) 블록을 GC할수록 복사량이 늘어 WAF가 커진다. over-provisioning으로 여유 공간을 늘리면 더 비어 있는 블록을 골라 GC할 수 있어 WAF가 내려간다.

> [!example]- 문제: SSD 수명을 계산하라
> 512GB SSD, 셀 P/E(program/erase) 한계 3000회, WAF = 2, 하루 호스트 쓰기 100GB. wear leveling이 완벽해 모든 셀이 균등 마모된다고 가정할 때 예상 수명(년)을 구하라.
> **풀이**
> - 전체 셀이 견디는 총 플래시 쓰기 = 용량 × P/E = 512GB × 3000 = 1,536,000GB = 1536TB
> - WAF=2이므로 호스트 쓰기 1당 플래시 쓰기 2 → 견딜 수 있는 호스트 쓰기 = 1536TB / 2 = 768TB
> - 하루 100GB → 768,000GB / 100GB = 7680일 ≈ 21년
> WAF가 2배가 되거나 하루 쓰기가 2배면 수명은 절반으로 준다. wear leveling이 없으면 특정 블록이 먼저 3000회에 도달해 훨씬 일찍 죽는다.

> [!example]- 문제: TRIM 유무에 따른 동작 차이를 설명하라
> 사용자가 큰 파일을 삭제한 직후, TRIM을 지원하는 경우와 안 하는 경우 SSD 내부에서 무슨 일이 벌어지는지 대조하라.
> **풀이**
> - TRIM 있음: OS가 삭제된 파일의 LBA들을 "무효"로 SSD에 통보 → FTL이 해당 페이지를 stale로 표시 → 다음 GC 때 그 블록의 유효 페이지가 줄어드니 복사량 감소 → WAF 감소, 성능 유지.
> - TRIM 없음: SSD는 파일이 삭제된 걸 모름 → 그 페이지들을 여전히 유효로 간주 → GC가 죽은 데이터를 계속 다른 블록으로 복사 → 헛된 write amplification 누적 → 쓰기 성능 서서히 저하.
> 결론: TRIM은 OS가 파일시스템 지식을 SSD에 넘겨주어 GC를 돕는 협력 메커니즘이다.

## 파인만

> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> - 플래시의 "쓰기 단위는 페이지, 지우기 단위는 블록" 비대칭에서 왜 out-of-place 쓰기, GC, write amplification이 순서대로 따라 나오는지 인과로 이을 수 있는가.
> - FTL이 감추는 세 가지 일(LBA↔물리 매핑, garbage collection, wear leveling)을 각각 한 문장으로 설명할 수 있는가.
> - 이 하드웨어 제약이 왜 LSM-tree 같은 로그 구조 자료구조와 큰 순차 쓰기 설계를 유리하게 만드는지 말할 수 있는가.
