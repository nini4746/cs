# CPU 스케줄링 (CPU Scheduling)

## 한 줄 요약

준비된 프로세스가 여러 개일 때 누구를 CPU에 올릴지 정하는 정책. 반환 시간(빨리 끝내기)과 응답 시간(빨리 반응하기)은 상충하고, 실전 스케줄러(MLFQ, CFS)는 이 둘을 타협한다.

## 왜 필요한가

- 여러 프로그램이 어떻게 "동시에" 도는 것처럼 보이나
- 대화형 앱이 왜 배치 작업보다 우선되나
- 리눅스가 실제로 쓰는 스케줄러의 원리

## 두 지표의 상충

- **반환 시간(turnaround)** = 완료 시각 − 도착 시각. 작업을 빨리 끝내는 관점 (배치)
- **응답 시간(response)** = 첫 실행 시각 − 도착 시각. 빨리 반응하는 관점 (대화형)

이 둘은 상충한다. 짧은 작업을 먼저 처리하면 반환은 좋지만, 한 작업을 끝까지 돌리면 다른 작업의 응답이 나빠짐. 스케줄링 = 이 상충의 관리.

## 기본 정책들 (점진적 개선)

| 정책 | 규칙 | 문제 |
|---|---|---|
| **FIFO** | 도착 순 | 긴 작업이 앞이면 뒤가 다 기다림 (convoy effect) |
| **SJF** | 짧은 작업 먼저 | 반환 최적이지만 실행 시간을 미리 알아야 함, 긴 작업 기아 |
| **STCF** | 남은 시간 짧은 것 먼저 (선점형 SJF) | 여전히 실행 시간 예측 필요 |
| **RR** | 타임슬라이스 돌아가며 | 응답 좋음, 반환은 나쁨. 슬라이스 크기 트레이드오프 |

**RR의 슬라이스**: 짧으면 응답 좋지만 컨텍스트 스위치([[limited-direct-execution]]) 오버헤드↑. 길면 반대. 보통 수~수십 ms.

## MLFQ (다단계 피드백 큐)

실행 시간을 모르면서 SJF의 이점(짧은/대화형 작업 우대)을 근사. 핵심 아이디어: **과거 행동을 관찰해 미래를 추측**.

규칙:
1. 여러 우선순위 큐. 높은 큐부터 실행
2. 새 작업은 최상위 큐에서 시작
3. **타임슬라이스를 다 쓰면 강등** (CPU 집약 → 아래로), **I/O로 양보하면 유지** (대화형 → 위 유지)
4. 주기적으로 전부 최상위로 리셋 (기아 방지 + 행동 변화 대응)

결과: 대화형/짧은 작업은 자연히 위에 머물러 빠른 응답, CPU 집약 작업은 아래로 내려가지만 결국 실행됨. 실행 시간을 몰라도 SJF 비슷하게 동작.

## 리눅스 CFS (Completely Fair Scheduler)

오랫동안 리눅스 기본. 아이디어: **모든 프로세스가 CPU를 공평하게 나눠 가지도록**.

- 각 프로세스의 **vruntime**(가상 실행 시간) 추적 → 가장 적게 실행된(vruntime 최소) 프로세스를 다음에 실행
- 우선순위(nice)는 vruntime이 쌓이는 속도를 조절 (높은 우선순위 = 천천히 쌓임 = 더 자주 선택)
- red-black tree로 vruntime 최소값을 O(log n)에 찾음 ([[balanced-trees]])
- 고정 타임슬라이스 대신 "목표 지연 시간"을 프로세스 수로 나눔

(2023년 커널 6.6부터 **EEVDF** 스케줄러로 교체 - 지연 민감 작업을 더 잘 다루는 방향)

## 선점 vs 비선점

- **비선점(non-preemptive)**: 프로세스가 자발적으로 놓을 때만 전환 (블록/종료)
- **선점(preemptive)**: 타이머 인터럽트로 강제 전환 ([[limited-direct-execution]]). 현대 OS 표준
- 선점 덕에 한 프로세스가 CPU를 독점 못 함

## I/O 고려

프로세스는 CPU 버스트와 I/O 버스트를 번갈아 함. 좋은 스케줄러는:

- I/O 시작하면 CPU를 다른 작업에 (I/O와 CPU 겹치기 → 활용률↑)
- I/O 집약(대화형) 작업을 우대해 응답성 유지
- MLFQ의 "I/O 양보 시 우선순위 유지"가 이걸 반영

## 셀프 체크

> [!question]- 반환 시간(turnaround)과 응답 시간(response)이 상충하는 이유는?
> 반환 시간은 완료 시각−도착 시각으로 작업을 빨리 끝내는 관점(배치)이고, 응답 시간은 첫 실행 시각−도착 시각으로 빨리 반응하는 관점(대화형)이다. 한 작업을 끝까지 돌리면 반환은 좋아지지만 다른 작업의 첫 실행이 늦어져 응답이 나빠진다. 그래서 둘을 동시에 최적화할 수 없다.

> [!question]- MLFQ가 실행 시간을 모르면서도 SJF 비슷하게 동작하는 원리는?
> "과거 행동을 관찰해 미래를 추측"한다. 타임슬라이스를 다 쓰면 CPU 집약으로 보고 강등하고, I/O로 양보하면 대화형으로 보고 우선순위를 유지한다. 그 결과 짧은/대화형 작업은 자연히 위 큐에 머물러 빠른 응답을 얻고, CPU 집약 작업은 아래로 내려간다 - 실행 시간을 몰라도 짧은 작업을 우대하는 효과.

> [!question]- CFS에서 nice(우선순위)는 스케줄링에 어떻게 작동하나?
> CFS는 vruntime(가상 실행 시간)이 가장 적은 프로세스를 다음에 실행한다. nice는 vruntime이 쌓이는 속도를 조절해, 높은 우선순위일수록 vruntime이 천천히 쌓인다. 천천히 쌓이면 최소값이 될 기회가 많아 더 자주 선택된다. red-black tree로 최소 vruntime을 O(log n)에 찾는다.

> [!question]- 선점(preemptive)이 없으면 어떤 문제가 생기나?
> 비선점에서는 프로세스가 스스로 블록하거나 종료할 때만 CPU를 놓는다. 그러면 CPU 집약 작업 하나가 계속 돌며 CPU를 독점해 다른 작업이 전혀 실행되지 못한다. 선점은 타이머 인터럽트로 강제 전환해, 한 프로세스의 독점을 막고 응답성을 보장한다.

## 연습문제

> [!example]- 문제: 작업 A(도착 0, 길이 8), B(도착 0, 길이 4), C(도착 0, 길이 2)를 FIFO(A→B→C)와 SJF로 스케줄링해 평균 반환 시간을 비교하라
> **풀이**
> FIFO (A→B→C):
> - A 완료 8, B 완료 12, C 완료 14 → 반환 8, 12, 14 → 평균 (8+12+14)/3 = **11.33**
> SJF (짧은 것 먼저 C→B→A):
> - C 완료 2, B 완료 6, A 완료 14 → 반환 2, 6, 14 → 평균 (2+6+14)/3 = **7.33**
> 짧은 작업을 먼저 처리하니 짧은 것들의 대기가 줄어 평균 반환이 크게 개선된다. 이것이 SJF가 반환 최적인 이유다. 다만 실행 시간을 미리 알아야 하고 긴 작업 A는 기아 위험이 있다.

> [!example]- 문제: A(길이 5), B(길이 5)가 시각 0에 함께 도착. RR(슬라이스 1)과 FIFO의 평균 응답 시간과 평균 반환 시간을 비교하라
> **풀이**
> FIFO (A→B):
> - 응답: A=0, B=5 → 평균 2.5
> - 반환: A 완료 5, B 완료 10 → 평균 7.5
> RR (슬라이스 1, A B A B …):
> - 응답: A 첫 실행 0, B 첫 실행 1 → 평균 0.5
> - 반환: 번갈아 돌아 A는 9에, B는 10에 완료 → 평균 9.5
> RR은 응답(0.5)이 FIFO(2.5)보다 훨씬 좋지만, 번갈아 도느라 둘 다 늦게 끝나 반환(9.5)은 FIFO(7.5)보다 나쁘다. 슬라이스를 키우면 응답은 나빠지고 반환·컨텍스트 스위치 오버헤드는 개선되는 트레이드오프.
