# 크래시 일관성 (Crash Consistency)

## 한 줄 요약

파일 하나 쓰기가 여러 블록 갱신(inode, 비트맵, 데이터)을 요구하는데, 그 중간에 전원이 나가면 파일시스템이 모순 상태가 된다. 저널링(먼저 로그에 쓰기)이 이를 원자적으로 만든다.

## 왜 필요한가

- 전원이 나가도 파일시스템이 왜 안 망가지나 (혹은 망가지나)
- `fsync`가 실제로 뭘 보장하나
- 데이터베이스 WAL([[recovery]])과 같은 원리

## 문제: 하나의 논리적 작업 = 여러 물리적 쓰기

파일에 데이터 블록 하나 추가하려면 최소 **세 곳**을 갱신 ([[file-system-basics]]):

1. **데이터 비트맵**: 새 블록을 "사용 중"으로
2. **inode**: 파일 크기 + 새 블록 포인터
3. **데이터 블록**: 실제 내용

이 셋은 디스크에 **따로따로** 쓰임. 중간에 크래시(전원, 커널 패닉)나면 일부만 반영된 모순 상태:

| 크래시 시점 | 결과 |
|---|---|
| 데이터만 씀 | inode가 모름 → 데이터 유실 (그나마 일관성은 OK) |
| inode만 씀 | inode가 쓰레기 블록 가리킴 → 잘못된 데이터 |
| inode+데이터, 비트맵 안 씀 | 블록이 "빈 것"으로 표시 → 나중에 재할당 → **손상** |
| 비트맵만 씀 | 블록 영구 누출 (아무도 안 씀) |

부분 갱신이 파일시스템 무결성을 깨뜨림. 원자성이 필요.

## 해법 1: fsck (사후 검사)

크래시 후 부팅 시 **전체 파일시스템을 스캔**해 모순을 찾아 고침:

- 비트맵과 실제 inode 사용을 대조해 복구
- 링크 수 재계산, 미아 inode를 lost+found로

문제: **디스크 전체 스캔** → 대용량이면 몇 시간. 느려서 현대엔 최후 수단.

## 해법 2: 저널링 (Write-Ahead Logging) - 표준

핵심 아이디어: **실제 위치에 쓰기 전에, 하려는 일을 로그(저널)에 먼저 기록.** 크래시 후 로그를 보고 완성하거나 버림.

```
1. 저널에 "이 블록들을 이렇게 바꿀 것" 기록 (TxBegin, 변경들, TxEnd)
2. 저널 쓰기 완료 확인 (커밋)
3. 실제 위치에 반영 (checkpoint)
4. 저널 엔트리 해제
```

크래시 복구:
- 저널에 **완전한 트랜잭션**(TxEnd까지)이 있으면 → 재실행(redo)해 완성
- **불완전**하면 → 무시 (아직 커밋 안 된 것으로 취급)

어느 시점에 크래시나도 "전부 반영" 또는 "전혀 반영 안 됨" → **원자성**. 부분 갱신 불가능.

### 순서가 생명

저널이 안전하려면 **저널 쓰기가 실제 쓰기보다 먼저** 완료돼야 함 (Write-Ahead). 그래서 커밋 전 저널을 디스크에 강제(배리어). 순서 안 지키면 저널 자체가 손상.

### 메타데이터 저널링 (성능 타협)

데이터까지 저널하면 **모든 것을 두 번 쓰기** (저널 + 실제) → 느림. 절충:

- **메타데이터만 저널링**(ext4 기본): inode/비트맵만 로그, 데이터는 바로 씀. 파일시스템 구조는 항상 일관, 데이터는 약간 위험 (덜 안전, 더 빠름)
- **전체 저널링**: 데이터도 로그. 안전하지만 2배 쓰기
- 순서 모드(ordered): 데이터를 먼저 쓰고 메타데이터 저널 → 흔한 타협

## 해법 3: copy-on-write (ZFS, btrfs)

기존 블록을 덮어쓰지 않고 **새 위치에 쓴 뒤 포인터를 원자적으로 교체**:

- 옛 데이터는 새 것이 완전히 쓰일 때까지 그대로 → 크래시나도 옛 상태로 일관
- 스냅샷이 공짜 (옛 블록을 안 지우면 됨)
- 저널의 이중 쓰기 회피. 대신 단편화·포인터 관리 복잡

## fsync: 앱의 보장 요청

쓰기는 보통 페이지 캐시([[page-cache]])에 머물다 나중에 디스크로 (write-back). 앱이 "지금 디스크에 확실히"를 원하면 `fsync(fd)`:

- 해당 파일의 더티 페이지를 디스크로 강제하고 완료까지 블록
- DB가 커밋 시 fsync로 내구성 보장 ([[transactions-acid]]의 D)
- fsync 없이 "썼다"고 믿으면 크래시 시 유실. 많은 데이터 손실 버그의 원인

## 셀프 체크

> [!question]- 파일에 데이터 블록 하나 추가하는 게 왜 크래시에 취약한가?
> 하나의 논리적 작업이 세 곳의 물리적 쓰기(데이터 비트맵, inode, 데이터 블록)를 요구하는데, 이 셋은 디스크에 따로따로 쓰인다. 중간에 전원이 나가면 일부만 반영된 모순 상태가 된다. 원자성이 보장되지 않아 파일시스템 무결성이 깨진다.

> [!question]- 저널링에서 "순서(Write-Ahead)"가 생명인 이유는?
> 저널(로그)에 하려는 일을 먼저 완전히 기록한 뒤에야 실제 위치에 반영해야, 크래시 후 저널을 보고 완성(redo)하거나 버릴 수 있다. 저널 쓰기가 실제 쓰기보다 먼저 끝나도록 커밋 전 배리어로 강제한다. 순서가 뒤바뀌면 저널 자체가 손상돼 복구 근거가 사라진다.

> [!question]- 메타데이터만 저널링하는 ext4 기본 모드가 전체 저널링보다 빠른 이유와 그 대가는?
> 전체 저널링은 데이터까지 저널+실제 위치에 두 번 쓴다. 메타데이터만 저널링하면 inode/비트맵만 로그하고 데이터는 바로 쓰므로 쓰기량이 준다. 대가는 파일시스템 구조는 항상 일관하지만 데이터는 약간 위험하다는 점 - 덜 안전하고 더 빠른 타협이다.

> [!question]- copy-on-write 파일시스템(ZFS, btrfs)이 크래시에 강한 이유는?
> 기존 블록을 덮어쓰지 않고 새 위치에 완전히 쓴 뒤 포인터를 원자적으로 교체한다. 새 데이터가 다 쓰일 때까지 옛 데이터가 그대로 남아, 중간에 크래시나도 옛 상태로 일관성이 유지된다. 저널의 이중 쓰기도 피하고 스냅샷이 공짜다. 대신 단편화·포인터 관리가 복잡하다.

## 연습문제

> [!example]- 문제: 데이터 블록 추가 중 (a) inode만, (b) 비트맵만 쓰고 크래시났을 때 각각 어떤 문제가 생기는지 판정하라
> **풀이**
> (a) inode만 씀: inode는 새 블록을 가리키는데 비트맵은 그 블록을 "빈 것"으로 표시하고 데이터도 안 쓰였다. inode가 쓰레기 블록을 가리켜 잘못된 데이터를 읽고, 그 블록이 나중에 재할당되면 손상이 된다.
> (b) 비트맵만 씀: 블록이 "사용 중"으로 표시됐지만 아무 inode도 그 블록을 가리키지 않는다. 그 블록은 영구히 누출되어(아무도 안 쓰고 아무도 못 회수) 공간만 낭비된다. 일관성 자체는 덜 치명적이지만 공간 누수.

> [!example]- 문제: 저널에 [TxBegin, 변경들] 까지만 쓰이고 TxEnd 전에 크래시났다. 복구 시 이 트랜잭션을 어떻게 처리해야 하며 그 이유는?
> **풀이**
> TxEnd(커밋)가 없으므로 **불완전한 트랜잭션**이다. 복구 시 이 엔트리는 무시(버림)한다 - 아직 커밋 안 된 것으로 취급.
> 이유: TxEnd까지 완전한 트랜잭션만 실제 위치에 반영을 시작했다고 보장할 수 있다. 불완전한 로그를 redo하면 절반만 적용된 변경을 강제해 오히려 모순을 만든다. "완전하면 재실행, 불완전하면 무시" 규칙 덕에 어느 시점에 크래시나도 전부 반영 또는 전혀 반영 안 됨 = 원자성이 성립한다.
