# 파일 시스템 기초 (File System Basics)

## 한 줄 요약

파일 시스템은 블록 장치([[io-devices]]) 위에 파일과 디렉토리라는 추상을 세운다. inode가 파일의 메타데이터+블록 위치를 담고, 디렉토리는 "이름 → inode 번호" 매핑일 뿐이다. 하드링크가 이 구조를 그대로 드러낸다.

## 왜 필요한가

- 파일이 디스크에 실제로 어떻게 저장되나
- 하드링크와 심볼릭 링크가 왜 다르게 동작하나
- `rm`이 실제로 뭘 지우나 (파일? 이름?)

## 온디스크 구조

디스크(블록의 배열)를 이렇게 나눔:

```
[슈퍼블록][inode 비트맵][데이터 비트맵][inode 테이블][........데이터 블록........]
```

- **슈퍼블록**: 파일시스템 전체 정보 (크기, inode 개수 등)
- **비트맵**: 어느 inode/블록이 사용 중인지 (0/1)
- **inode 테이블**: 모든 inode 배열
- **데이터 블록**: 실제 파일 내용

## inode = 파일의 정체

**inode**(index node)가 파일 하나를 대표. 담는 것:

- 메타데이터: 크기, 소유자, 권한, 타임스탬프, **링크 수**
- **데이터 블록 위치**: 파일 내용이 어느 블록들에 있나

주목: **inode에 파일 이름은 없다.** 이름은 디렉토리에 있음. inode는 번호로 식별.

### 블록 위치를 담는 법: 간접 블록

작은 파일: inode가 직접 블록 포인터 몇 개(direct) 보유. 큰 파일:

- **간접 블록(indirect)**: 블록 포인터들을 담은 블록을 가리킴
- **이중/삼중 간접**: 포인터의 포인터... → 큰 파일 지원하면서 작은 파일은 오버헤드 없음
- 멀티레벨 페이지 테이블([[page-tables]])과 같은 트리 아이디어

## 디렉토리 = 이름→inode 매핑

디렉토리도 파일(inode 있음). 그 내용은 **엔트리 목록**:

```
디렉토리 내용:
  "original.txt" → inode 45180616
  "report.pdf"   → inode 45180620
  ...
```

파일 열기 `open("/a/b/c")`:
1. 루트 디렉토리 inode에서 "a"의 inode 찾기
2. a 디렉토리에서 "b" 찾기
3. b에서 "c" 찾기 → c의 inode
경로 순회 = 디렉토리를 따라 inode를 찾아가는 것.

## 하드링크: 이름과 inode의 분리 - 실증

**하드링크** = 같은 inode를 가리키는 또 다른 이름. `ln original hardlink`:

```
inode 번호: original=45180616  hardlink=45180616   ← 같은 inode!
links=2                                             ← 이 inode를 가리키는 이름 2개
```

두 이름이 완전히 동등 (누가 원본인지 구분 없음). inode의 **링크 수**가 가리키는 이름 개수. 실증:

```
원본 rm 후: cat hardlink → "hello"   ← 여전히 살아있음!
```

`rm original`은 파일을 지운 게 아니라 **디렉토리 엔트리 하나를 지우고 링크 수를 감소**시킬 뿐. 링크 수가 0이 돼야 실제 블록 회수. 그래서 하드링크가 남아있으면 데이터는 생존. **rm = 이름 제거, 링크 수 0일 때만 데이터 삭제.**

## 심볼릭 링크: 경로를 담은 파일

**심볼릭(soft) 링크** = 다른 경로를 텍스트로 담은 별도 파일 (자기 inode 가짐):

```
symlink.txt → "original.txt" (경로 문자열)
```

차이:
- 하드링크: 같은 inode. 원본/링크 구분 없음. 같은 파일시스템 내만
- 심볼릭: 경로를 가리킴. 원본 지우면 **깨짐**(dangling). 파일시스템/디렉토리 넘어 가능

실증: 원본 rm 후 `cat symlink` → "No such file or directory" (경로가 이제 없는 곳을 가리킴).

## 파일 디스크립터

프로세스가 `open`하면 **파일 디스크립터**(정수)를 받음. 프로세스별 테이블이 이를 열린 파일(inode + 오프셋)에 매핑:

- 0/1/2 = stdin/stdout/stderr (관례)
- fork하면 자식이 fd 복사 → 부모/자식이 오프셋 공유 ([[process]])
- 리다이렉션(`>`)/파이프(`|`)가 이 fd를 조작하는 것 ([[io-devices]])

## 셀프 체크

> [!question]- inode에는 파일 이름이 없다. 그럼 이름은 어디에 있나?
> 이름은 디렉토리에 있다. inode는 크기·소유자·권한·타임스탬프·링크 수 같은 메타데이터와 데이터 블록 위치만 담고, 번호로 식별된다. 디렉토리가 "이름 → inode 번호" 매핑을 보관하므로, 같은 inode를 여러 이름이 가리킬 수 있다(하드링크).

> [!question]- `rm original`은 실제로 무엇을 지우나?
> 파일 자체가 아니라 디렉토리 엔트리(이름) 하나를 지우고 inode의 링크 수를 1 감소시킨다. 링크 수가 0이 될 때만 실제 데이터 블록을 회수한다. 그래서 하드링크가 남아 있으면 원본을 rm해도 데이터는 생존한다. 즉 rm = 이름 제거, 링크 수 0일 때만 데이터 삭제.

> [!question]- 하드링크와 심볼릭 링크는 왜 다르게 동작하나?
> 하드링크는 같은 inode를 가리키는 또 다른 이름이라 원본/링크 구분이 없고 원본을 지워도 살아있지만, 같은 파일시스템 내에서만 가능하다. 심볼릭 링크는 다른 경로를 텍스트로 담은 별도 inode의 파일이라 원본을 지우면 깨지고(dangling), 파일시스템·디렉토리 경계를 넘을 수 있다.

> [!question]- 작은 파일과 큰 파일의 블록 위치를 어떻게 같은 inode 구조로 다 담나?
> inode는 직접(direct) 블록 포인터 몇 개를 가진다. 작은 파일은 이것만으로 충분해 오버헤드가 없다. 큰 파일은 간접 블록(포인터를 담은 블록)을 가리키고, 필요하면 이중·삼중 간접으로 확장한다. 멀티레벨 페이지 테이블과 같은 트리 아이디어로, 작은 파일 오버헤드 없이 큰 파일을 지원한다.

## 연습문제

> [!example]- 문제: `ln original hardlink` 후 `rm original`을 했는데 `cat hardlink`가 여전히 내용을 출력한다. 링크 수 변화로 설명하라
> **풀이**
> 1. `ln original hardlink` 직후: 두 이름이 같은 inode(예: 45180616)를 가리키고 links=2.
> 2. `rm original`: 디렉토리에서 "original" 엔트리만 제거하고 링크 수를 2→1로 감소. 데이터 블록은 그대로.
> 3. 링크 수가 아직 1(>0)이므로 inode와 데이터 블록은 회수되지 않음.
> 4. 따라서 `cat hardlink`는 살아있는 inode를 따라가 내용을 그대로 출력. 반면 심볼릭 링크였다면 경로 "original"이 사라져 "No such file or directory"가 났을 것.

> [!example]- 문제: `open("/a/b/c")`가 c의 inode를 찾기까지의 경로 순회 과정을 단계로 써라
> **풀이**
> 경로 순회 = 디렉토리를 따라 inode를 찾아가는 것.
> 1. 루트 디렉토리 inode를 읽어 그 엔트리 목록에서 "a"의 inode 번호를 찾음.
> 2. a 디렉토리 inode를 읽어 엔트리에서 "b"의 inode 번호를 찾음.
> 3. b 디렉토리 inode를 읽어 엔트리에서 "c"의 inode 번호를 찾음 → c의 inode.
> 각 단계마다 디렉토리 inode를 읽고, 그 안의 "이름 → inode 번호" 매핑을 조회하는 것을 반복한다.

> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 온디스크 레이아웃(슈퍼블록/비트맵/inode 테이블/데이터 블록)의 역할을 각각 말할 수 있는가.

> 3. 하드링크 vs 심볼릭 링크의 차이(같은 inode vs 경로, 파일시스템 경계, dangling)를 대비할 수 있는가.
