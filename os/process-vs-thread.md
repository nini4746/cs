# 프로세스 vs 스레드 (Process vs Thread)

## 한 줄 요약

스레드는 한 프로세스 안의 여러 실행 흐름이다. 주소 공간(코드/힙/전역)은 공유하고 스택과 레지스터만 각자 가진다. 이 공유가 스레드를 가볍고 빠르게 만들지만 동시에 경쟁 조건의 원천이 된다.

## 왜 필요한가

- 병렬성을 프로세스로 할까 스레드로 할까
- 스레드가 뭘 공유하고 뭘 안 하나 (버그의 근원)
- 왜 스레드는 락이 필요하고 프로세스는 덜 필요한가

## 스레드 = 프로세스 내 실행 흐름

프로세스([[process]])는 주소 공간 + 실행 상태. **멀티스레드 프로세스**는 하나의 주소 공간에 **여러 실행 흐름(스레드)**:

```
프로세스
├─ 코드 (공유)
├─ 전역/데이터 (공유)
├─ 힙 (공유)
├─ 스레드1: 스택1 + 레지스터1(PC,SP)
├─ 스레드2: 스택2 + 레지스터2
└─ 스레드3: 스택3 + 레지스터3
```

## 공유하는 것 vs 안 하는 것

| | 공유 | 각자 |
|---|---|---|
| 코드 | ✓ | |
| 전역 변수 | ✓ | |
| 힙 (malloc) | ✓ | |
| 열린 파일 | ✓ | |
| **스택** | | ✓ (지역 변수) |
| **레지스터** (PC, SP) | | ✓ |

핵심: **전역/힙은 공유, 스택은 각자**. 실측:

```c
int global = 0;              // 공유
void* worker(void* arg){
    global += 1000;          // 이 변경이 메인 스레드에 보임
    int local = *(int*)arg;  // 스택은 스레드마다 독립
}
```

```
before: global=0
  thread sees global=1000, local=7
after : global=1000 (thread's change visible!)
```

스레드가 바꾼 `global`이 메인에 그대로 보인다. **fork한 프로세스라면 안 보였을 것** ([[process]]의 주소 공간 분리). 이 공유가 스레드의 정체.

## 왜 스레드를 쓰나

1. **병렬성**: 멀티코어에서 한 작업을 여러 코어로 ([[multicore-and-numa]])
2. **I/O 중첩**: 한 스레드가 I/O 대기하는 동안 다른 스레드가 CPU 사용
3. **공유의 편의**: 같은 메모리를 직접 공유 → 프로세스 간 통신(IPC)보다 쉽고 빠름
4. **가벼움**: 생성/컨텍스트 스위치가 프로세스보다 쌈 (주소 공간 안 바꿈 → TLB 유지)

## 프로세스 vs 스레드 - 트레이드오프

| | 멀티프로세스 | 멀티스레드 |
|---|---|---|
| 메모리 | 분리 (안전) | 공유 (위험하지만 빠름) |
| 통신 | IPC (파이프, 공유메모리, 소켓) | 그냥 공유 변수 |
| 생성 비용 | 비쌈 (fork) | 쌈 |
| 격리/안정성 | 하나 죽어도 나머지 생존 | 하나 크래시 → 프로세스 전체 죽음 |
| 컨텍스트 스위치 | 비쌈 (주소 공간+TLB) | 쌈 (같은 주소 공간) |

**선택 기준**: 격리·안정성 중요(브라우저 탭, 서버 워커) → 프로세스. 긴밀한 데이터 공유 + 성능 → 스레드. 크롬이 탭을 프로세스로 나눈 이유 = 한 탭 크래시가 전체를 안 죽이도록.

## 공유의 대가: 경쟁 조건

공유 메모리 = 편리하지만 위험. 두 스레드가 같은 변수를 동시에 수정하면 **경쟁 조건(race condition)** → 예측 불가 결과. 위 예제는 스레드가 하나라 괜찮지만, 둘이 `global += 1000`을 동시에 하면 갱신이 유실될 수 있음.

이것이 [[threads-and-races]], [[locks]] 등 동시성 전체의 출발점. 스레드의 강점(공유)이 곧 약점(동기화 필요).

## 스레드 모델

- **커널 스레드**: OS가 알고 스케줄링. 진짜 병렬 (멀티코어)
- **유저 스레드**: 라이브러리가 관리, OS는 모름. 가볍지만 하나가 블록하면 전부 블록
- **M:N**: 유저 스레드 M개를 커널 스레드 N개에 매핑 (Go의 goroutine)
- 현대 언어의 경량 스레드(goroutine, async task)는 이 위에 세운 추상 → [[io-multiplexing]]

## 셀프 체크

> [!question]- 스레드끼리 공유하는 것과 각자 갖는 것은 무엇이고, 그 경계가 왜 중요한가?
> 코드/전역/힙/열린 파일은 공유하고, 스택과 레지스터(PC, SP)는 각자 가진다. 전역·힙이 공유되기 때문에 한 스레드의 변경이 다른 스레드에 그대로 보이고, 이게 곧 경쟁 조건의 원천이자 동기화가 필요한 이유다. 반대로 지역 변수(스택)는 독립이라 안전하다.

> [!question]- 왜 스레드의 컨텍스트 스위치가 프로세스보다 싼가?
> 같은 프로세스 내 스레드는 주소 공간을 공유하므로 스위치할 때 주소 공간을 바꾸지 않는다. 그래서 TLB를 그대로 유지할 수 있고(플러시 불필요), 생성 비용도 낮다. 프로세스 스위치는 주소 공간과 TLB까지 갈아끼워야 해서 비싸다.

> [!question]- 크롬이 탭을 스레드가 아니라 프로세스로 나눈 이유는?
> 격리·안정성 때문이다. 멀티스레드는 하나가 크래시하면 프로세스 전체가 죽지만, 프로세스로 분리하면 한 탭이 죽어도 나머지 탭은 생존한다. 긴밀한 데이터 공유·성능보다 안정성이 우선인 경우의 선택이다.

> [!question]- fork한 프로세스와 새 스레드는 공유 메모리 관점에서 어떻게 다른가?
> fork한 자식은 부모와 주소 공간이 분리되어 부모의 전역 변수 변경이 서로 보이지 않는다(copy-on-write). 반면 같은 프로세스의 스레드는 전역·힙을 공유해 변경이 즉시 보인다. 이 차이가 통신 방식(IPC vs 공유 변수)을 가른다.

## 연습문제

> [!example]- 문제: 두 스레드가 공유 전역 `int counter`에 각각 `counter += 1`을 100만 번 수행하는데 최종값이 200만보다 작게 나온다. 원인과 해결을 설명하라.
> **풀이**
> 원인: `counter += 1`은 원자적이지 않다. 읽기→증가→쓰기 세 단계로 나뉘고, 두 스레드가 같은 값을 읽은 뒤 각자 +1해서 쓰면 한 번의 증가가 유실된다(lost update, 경쟁 조건). counter는 공유 힙/전역이라 두 스레드가 동시에 건드릴 수 있다.
> 해결: 임계 영역을 락으로 보호하거나(뮤텍스로 read-modify-write를 원자화), 원자 연산(atomic add)을 사용한다. 그러면 매 증가가 직렬화되어 200만이 보장된다.

> [!example]- 문제: "이미지 100장을 다운로드(I/O 대기 많음)한 뒤 각 이미지를 필터링(CPU 집약)"하는 작업을 설계한다. 어느 단계에 스레드를 왜 쓰는가?
> **풀이**
> 다운로드 단계: I/O 대기가 지배적이므로 여러 스레드로 요청을 중첩시킨다. 한 스레드가 네트워크 응답을 기다리는 동안 다른 스레드가 CPU를 쓰거나 다른 요청을 진행해 대기 시간을 겹친다(I/O 중첩).
> 필터링 단계: CPU 집약이므로 멀티코어에서 스레드를 코어 수만큼 두어 진짜 병렬성을 얻는다(커널 스레드여야 실제 병렬). 이미지끼리 독립이면 공유 상태가 없어 락도 거의 불필요하다. 공유 결과 집계 지점만 동기화하면 된다.

> [!example]- 문제: 유저 스레드만 쓰는 라이브러리에서 한 스레드가 blocking read를 호출하자 프로그램 전체가 멈췄다. 왜인가? 어떻게 고치나?
> **풀이**
> 원인: 유저 스레드는 OS가 존재를 모르고 라이브러리가 관리한다. 커널 입장에선 스레드가 하나뿐이라 blocking 시스템 콜로 그 하나가 블록되면 프로세스 전체(모든 유저 스레드)가 함께 멈춘다.
> 해결: 커널 스레드(또는 M:N 매핑)를 써서 OS가 스케줄링하게 하거나, 논블로킹 I/O + 이벤트 루프로 블록 자체를 피한다. Go의 goroutine처럼 M:N 런타임은 블로킹 콜을 감지해 다른 goroutine을 별 커널 스레드로 돌린다.
