# 빠른 파일 시스템 (FFS와 지역성)

## 한 줄 요약

초기 유닉스 파일시스템은 디스크 위에 데이터를 아무 데나 흩어놔서 느렸다. FFS는 관련 데이터를 디스크에서 가까이 배치(지역성)해 탐색 시간을 줄였다. 이 "디스크 인식 배치" 아이디어가 이후 모든 파일시스템의 기반.

## 왜 필요한가

- 파일시스템 성능이 왜 배치(layout)에 달렸나
- 디스크(HDD)의 물리 특성이 소프트웨어 설계를 어떻게 강제했나
- 단편화가 파일시스템에서 왜 문제인가

## 문제: 옛 파일시스템은 느렸다

초기 유닉스 FS는 디스크를 단순한 블록 배열로 보고 데이터를 **아무 빈 곳에나** 배치. 결과:

- 한 파일의 블록들이 디스크 전체에 흩어짐
- inode는 디스크 앞쪽, 데이터는 뒤쪽 → 파일 하나 읽는데 디스크 헤드가 왕복
- 시간이 지나며 단편화 심화 → 성능이 원래의 몇 %로 추락

핵심 원인: **HDD의 물리 특성**을 무시. HDD는 헤드를 움직이는 **탐색 시간(seek)**이 지배적 ([[memory-hierarchy]]의 디스크 지연). 멀리 흩어진 블록 = 헤드가 계속 이동 = 느림.

## HDD 물리 복습

```
디스크 접근 시간 = 탐색(seek) + 회전 지연(rotation) + 전송(transfer)
```

- **탐색**: 헤드를 목표 트랙으로 (수 ms, 가장 비쌈)
- **회전 지연**: 목표 섹터가 헤드 밑으로 올 때까지 (~수 ms)
- **전송**: 실제 읽기 (빠름)

→ **가까운 블록끼리 = 탐색 최소 = 빠름.** 지역성이 디스크에서도 핵심.

## FFS의 해법: 실린더 그룹

FFS(Fast File System, 1984)는 디스크를 **블록 그룹(실린더 그룹)**으로 나누고, 관련 데이터를 같은 그룹에 모음:

- **파일의 inode와 데이터 블록을 같은 그룹에** → 파일 읽을 때 탐색 최소
- **한 디렉토리의 파일들을 같은 그룹에** → 디렉토리 순회(`ls -l`)가 빠름
- 각 그룹에 inode·데이터·비트맵을 분산 배치 → 지역성

핵심 원칙: **관련된 것은 가까이(keep related stuff together).** 이 한 문장이 FFS의 전부.

### 배치 정책 예외

- 새 디렉토리는 여유 있는 다른 그룹에 (부하 분산)
- 아주 큰 파일은 여러 그룹에 나눔 (한 그룹 독점 방지)
- 지역성과 균형 사이 타협

## 그 밖의 개선

- **더 큰 블록**: 작은 블록은 오버헤드, 큰 블록은 내부 단편화 → FFS는 큰 블록 + 작은 파일용 조각(fragment)
- **비트맵 기반 할당**: 연속 빈 공간 찾기 쉽게

## 유산

FFS의 지역성 원칙은 ext2/3/4, XFS 등 거의 모든 후속 FS의 기반. "디스크를 인식하고 배치하라"가 표준이 됨.

### SSD 시대의 변화

FFS는 HDD의 탐색 시간을 전제. **SSD는 탐색 시간이 없음**([[ssd-internals]]) → 지역성의 의미가 달라짐:

- 물리적 인접성은 SSD에서 덜 중요 (어디든 균일 접근)
- 대신 SSD는 **쓰기 패턴**이 중요 (wear leveling, 큰 순차 쓰기 선호) → LSM-tree, 로그 구조 FS가 부상
- 그래도 순차 접근은 SSD에서도 프리페치·큰 전송으로 유리
