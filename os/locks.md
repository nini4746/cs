# 락 (Locks)

## 한 줄 요약

임계 구역을 한 번에 한 스레드만 실행하도록 강제하는 도구. 내부적으로 하드웨어 원자 명령(test-and-set, CAS)으로 구현되고, 경합 시 스핀(계속 시도)이냐 슬립(양보)이냐가 성능을 가른다.

## 왜 필요한가

- [[threads-and-races]]의 갱신 유실을 실제로 어떻게 막나
- 뮤텍스가 내부에서 뭘 하나
- 스핀락 vs 뮤텍스 언제 뭘 쓰나

## 락으로 경쟁 해결 - 실측

[[threads-and-races]]의 카운터를 세 방식으로. 4스레드 × 100만:

```
plain  : 1771651   0.004s   ← 락 없음: 빠르지만 틀림 (유실)
mutex  : 4000000   0.093s   ← 뮤텍스: 정확, 느림
atomic : 4000000   0.067s   ← 원자 연산: 정확, 뮤텍스보다 빠름
```

관찰:
- 락/원자 둘 다 정확히 400만 (유실 0)
- **정확성의 대가는 속도**: plain의 20배 느림. 상호 배제는 병렬성을 직렬화하니까
- 단순 카운터엔 **원자 연산이 뮤텍스보다 빠름** (락 자료구조 오버헤드 없이 하드웨어 명령 하나)

## 락의 인터페이스

```c
lock(&m);      // 획득 (남이 쥐고 있으면 대기)
critical_section();
unlock(&m);    // 해제
```

보장: `lock`과 `unlock` 사이엔 최대 한 스레드. 나머지는 대기.

## 어떻게 구현하나: 하드웨어 원자 명령

락 자체가 공유 변수라 `if(free) take` 식으로 짜면 그것도 경쟁 조건. 그래서 **하드웨어 원자 명령**이 필요:

### test-and-set

"값을 읽고 + 1로 세팅"을 원자적으로:

```c
int test_and_set(int *p){ int old=*p; *p=1; return old; }  // 원자적으로
// 스핀락:
void lock(int *l){ while(test_and_set(l)==1); }  // 0이 될 때까지 스핀
void unlock(int *l){ *l=0; }
```

`old==0`이었으면 내가 획득(0→1), `old==1`이면 남이 쥔 것 → 계속 시도.

### compare-and-swap (CAS)

"기댓값과 같으면 새 값으로 교체"를 원자적으로. 더 범용 → [[lock-free-basics]]:

```c
bool CAS(int *p, int expected, int new){
    if(*p==expected){ *p=new; return true; } return false;  // 원자적
}
```

이 원자 명령들은 [[cache-coherence]]에서 캐시 라인을 독점(M 상태)해서 구현 → 경합 심하면 코어 간 트래픽 폭발.

## 스핀 vs 슬립

락을 못 얻었을 때 뭘 하나:

| | 스핀락 | 뮤텍스(슬립) |
|---|---|---|
| 대기 방식 | 계속 시도 (CPU 태움) | 블록, OS가 재웠다 깨움 |
| 짧은 임계구역 | 유리 (컨텍스트 스위치보다 쌈) | 오버헤드 큼 |
| 긴 임계구역 | 낭비 (CPU 헛돔) | 유리 (CPU 양보) |
| 사용처 | 커널, 초경량 임계구역 | 일반 앱 |

- **스핀락**: 임계 구역이 아주 짧고 스핀 시간 < 컨텍스트 스위치 비용일 때. CPU를 태우며 기다림
- **뮤텍스**: 못 얻으면 OS에 양보하고 잠듦 → 깨우는 비용(시스템 콜, [[limited-direct-execution]])이 있지만 CPU 낭비 없음
- **하이브리드**: 잠깐 스핀하다 안 되면 슬립 (adaptive mutex, 실제 pthread_mutex 다수)

### futex (리눅스)

빠른 유저공간 뮤텍스: 경합 없으면 유저공간에서 원자 연산만(시스템 콜 0), 경합 시에만 커널로 가서 슬립. 무경합 케이스를 싸게 → 실전 뮤텍스의 기반.

## 락 사용의 함정

- **데드락**: 락을 순환 대기 → [[deadlock]]
- **과도한 임계 구역**: 너무 크면 병렬성 죽음 (직렬화). 최소로
- **경합(contention)**: 한 락에 스레드 몰림 → 확장성 붕괴. 락 쪼개기(fine-grained), 스레드별 데이터 후 병합
- **우선순위 역전**: 낮은 우선순위가 락 쥐고 높은 걸 막음

## 성능 원칙

1. 임계 구역을 **짧게** (락 잡은 시간 최소)
2. 락 **경합을 줄여라** (스레드별 데이터, 락 분할)
3. 단순 카운터/플래그는 **원자 연산** (락 없이)
4. 읽기 많으면 **reader-writer 락** → [[semaphores]]
