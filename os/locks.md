# 락 (Locks)

## 한 줄 요약

임계 구역을 한 번에 한 스레드만 실행하도록 강제하는 도구. 내부적으로 하드웨어 원자 명령(test-and-set, CAS)으로 구현되고, 경합 시 스핀(계속 시도)이냐 슬립(양보)이냐가 성능을 가른다.

## 왜 필요한가

- [[threads-and-races]]의 갱신 유실을 실제로 어떻게 막나
- 뮤텍스가 내부에서 뭘 하나
- 스핀락 vs 뮤텍스 언제 뭘 쓰나

## 락으로 경쟁 해결 - 실측

[[threads-and-races]]의 카운터를 세 방식으로. 4스레드 × 100만:

```
plain  : 1771651   0.004s   ← 락 없음: 빠르지만 틀림 (유실)
mutex  : 4000000   0.093s   ← 뮤텍스: 정확, 느림
atomic : 4000000   0.067s   ← 원자 연산: 정확, 뮤텍스보다 빠름
```

관찰:
- 락/원자 둘 다 정확히 400만 (유실 0)
- **정확성의 대가는 속도**: plain의 20배 느림. 상호 배제는 병렬성을 직렬화하니까
- 단순 카운터엔 **원자 연산이 뮤텍스보다 빠름** (락 자료구조 오버헤드 없이 하드웨어 명령 하나)

## 락의 인터페이스

```c
lock(&m);      // 획득 (남이 쥐고 있으면 대기)
critical_section();
unlock(&m);    // 해제
```

보장: `lock`과 `unlock` 사이엔 최대 한 스레드. 나머지는 대기.

## 어떻게 구현하나: 하드웨어 원자 명령

락 자체가 공유 변수라 `if(free) take` 식으로 짜면 그것도 경쟁 조건. 그래서 **하드웨어 원자 명령**이 필요:

### test-and-set

"값을 읽고 + 1로 세팅"을 원자적으로:

```c
int test_and_set(int *p){ int old=*p; *p=1; return old; }  // 원자적으로
// 스핀락:
void lock(int *l){ while(test_and_set(l)==1); }  // 0이 될 때까지 스핀
void unlock(int *l){ *l=0; }
```

`old==0`이었으면 내가 획득(0→1), `old==1`이면 남이 쥔 것 → 계속 시도.

### compare-and-swap (CAS)

"기댓값과 같으면 새 값으로 교체"를 원자적으로. 더 범용 → [[lock-free-basics]]:

```c
bool CAS(int *p, int expected, int new){
    if(*p==expected){ *p=new; return true; } return false;  // 원자적
}
```

이 원자 명령들은 [[cache-coherence]]에서 캐시 라인을 독점(M 상태)해서 구현 → 경합 심하면 코어 간 트래픽 폭발.

## 스핀 vs 슬립

락을 못 얻었을 때 뭘 하나:

| | 스핀락 | 뮤텍스(슬립) |
|---|---|---|
| 대기 방식 | 계속 시도 (CPU 태움) | 블록, OS가 재웠다 깨움 |
| 짧은 임계구역 | 유리 (컨텍스트 스위치보다 쌈) | 오버헤드 큼 |
| 긴 임계구역 | 낭비 (CPU 헛돔) | 유리 (CPU 양보) |
| 사용처 | 커널, 초경량 임계구역 | 일반 앱 |

- **스핀락**: 임계 구역이 아주 짧고 스핀 시간 < 컨텍스트 스위치 비용일 때. CPU를 태우며 기다림
- **뮤텍스**: 못 얻으면 OS에 양보하고 잠듦 → 깨우는 비용(시스템 콜, [[limited-direct-execution]])이 있지만 CPU 낭비 없음
- **하이브리드**: 잠깐 스핀하다 안 되면 슬립 (adaptive mutex, 실제 pthread_mutex 다수)

### futex (리눅스)

빠른 유저공간 뮤텍스: 경합 없으면 유저공간에서 원자 연산만(시스템 콜 0), 경합 시에만 커널로 가서 슬립. 무경합 케이스를 싸게 → 실전 뮤텍스의 기반.

## 락 사용의 함정

- **데드락**: 락을 순환 대기 → [[deadlock]]
- **과도한 임계 구역**: 너무 크면 병렬성 죽음 (직렬화). 최소로
- **경합(contention)**: 한 락에 스레드 몰림 → 확장성 붕괴. 락 쪼개기(fine-grained), 스레드별 데이터 후 병합
- **우선순위 역전**: 낮은 우선순위가 락 쥐고 높은 걸 막음

## 성능 원칙

1. 임계 구역을 **짧게** (락 잡은 시간 최소)
2. 락 **경합을 줄여라** (스레드별 데이터, 락 분할)
3. 단순 카운터/플래그는 **원자 연산** (락 없이)
4. 읽기 많으면 **reader-writer 락** → [[semaphores]]

## 셀프 체크

> [!question]- 락 자체가 공유 변수인데 왜 `if(free) take` 식 소프트웨어 락은 안 되나?
> 락 변수를 읽고("비었나?") 세팅하는("내가 쥔다") 두 단계 사이에 다른 스레드가 끼어들 수 있어 그것 자체가 경쟁 조건이다. 두 스레드가 동시에 free를 보고 둘 다 획득할 수 있다. 그래서 읽기+세팅을 원자적으로 묶는 하드웨어 명령(test-and-set, CAS)이 필요하다.

> [!question]- 단순 카운터에서 원자 연산이 뮤텍스보다 빠른 이유는?
> 뮤텍스는 락 자료구조 관리와 (경합 시) OS 슬립/웨이크업 오버헤드가 있지만, 원자 연산은 하드웨어 명령 하나로 끝난다. 실측에서 atomic 0.067s < mutex 0.093s였다. 락이 감싸는 임계 구역이 연산 하나뿐이라 락의 부가 비용이 상대적으로 크게 드러난다.

> [!question]- 짧은 임계 구역엔 스핀락, 긴 임계 구역엔 뮤텍스가 유리한 이유는?
> 스핀락은 CPU를 태우며 계속 시도하므로 대기 시간이 컨텍스트 스위치 비용보다 짧을 때만 이득이다. 임계 구역이 길면 스핀이 CPU를 낭비한다. 뮤텍스는 못 얻으면 OS에 양보해 잠들어 CPU 낭비가 없지만, 깨우는 시스템 콜 비용이 있어 짧은 구역엔 오버헤드가 과하다.

> [!question]- futex가 무경합 뮤텍스를 싸게 만드는 방법은?
> 경합이 없으면 유저공간에서 원자 연산만으로 획득/해제해 시스템 콜을 아예 안 한다. 실제 락을 두고 다투는 경합 상황에서만 커널로 진입해 슬립한다. 대부분의 락 획득이 무경합이므로 이 fast-path가 실전 뮤텍스를 빠르게 만든다.

> [!question]- 원자 명령이 경합 심할 때 코어 간 트래픽을 폭발시키는 이유는?
> test-and-set/CAS는 캐시 라인을 독점 상태(M)로 가져와야 실행된다. 여러 코어가 같은 락을 다투면 그 캐시 라인 소유권이 코어 사이를 계속 오가며(coherence 트래픽) 확장성이 무너진다. 그래서 락 쪼개기나 스레드별 데이터로 경합을 줄여야 한다.

## 연습문제

> [!example]- 문제: 아래 스핀락 unlock에 숨은 정확성 문제가 있는지, test_and_set 기반이라면 왜 괜찮은지 설명하라.
> ```c
> void lock(int *l){ while(test_and_set(l)==1); }
> void unlock(int *l){ *l=0; }
> ```
> **풀이**
> unlock의 `*l=0`은 단순 쓰기지만, 획득은 test_and_set이 원자적으로 0→1 전환을 보장하므로 정확하다. 여러 스레드가 스핀 중일 때 해제되면 그중 test_and_set이 old==0을 받는 단 하나만 획득(0→1)하고 나머지는 old==1을 받아 계속 스핀한다. 단, 실전에선 unlock의 쓰기가 임계 구역 내 쓰기보다 먼저 보이지 않도록 release 순서 보장이 필요하다(약한 메모리 모델에서).

> [!example]- 문제: 4스레드가 한 락에 몰려 확장성이 붕괴한다. 세 가지 완화책과 각 원리를 제시하라.
> **풀이**
> 1. 임계 구역 축소: 락 잡은 시간을 최소화해 다른 스레드의 대기·경합 자체를 줄인다.
> 2. 락 쪼개기(fine-grained): 하나의 큰 락을 데이터 부분별 여러 락으로 나눠 서로 다른 부분을 만지는 스레드가 안 부딪히게 한다.
> 3. 스레드별 데이터 후 병합: 각 스레드가 로컬에 누적하고 마지막에 한 번만 합쳐, 공유 캐시 라인 경합(및 false sharing)을 제거한다. 단순 카운터라면 아예 원자 연산으로 대체.

> [!example]- 문제: 낮은 우선순위 스레드 L이 락을 쥔 채, 중간 우선순위 M이 CPU를 계속 점유하고, 높은 우선순위 H가 그 락을 기다린다. 무슨 문제이며 왜 H가 막히나?
> **풀이**
> 우선순위 역전이다. H는 L이 락을 놔야 진행할 수 있는데, L은 M보다 우선순위가 낮아 스케줄되지 못해 락을 놓지 못한다. 결국 M이 H를 간접적으로 막는다. 해결은 우선순위 상속(락 쥔 L을 H의 우선순위로 임시 승격)으로 L을 빨리 실행시켜 락을 놓게 하는 것이다.
