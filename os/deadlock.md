# 데드락 (Deadlock)

## 한 줄 요약

여러 스레드가 서로가 쥔 자원을 기다리며 영원히 멈추는 상황. 네 조건이 동시에 성립할 때만 발생하고, 락을 항상 같은 순서로 잡으면 예방할 수 있다.

## 왜 필요한가

- 멀티스레드 프로그램이 "멈춰버리는" 대표 원인
- 락을 여러 개 쓸 때의 위험
- 예방/회피/탐지 중 실전에서 뭘 쓰나

## 실증: 락 순서 꼬임

두 스레드가 두 락을 **반대 순서**로 잡음:

```c
// 스레드1: A 잡고 → B
lock(&A); usleep(1000); lock(&B); ...

// 스레드2: B 잡고 → A
lock(&B); usleep(1000); lock(&A); ...
```

이 머신 실측: "t1 got both" / "t2 got both"가 **영원히 안 나옴**. 3초 후에도 두 스레드 살아있음 = 데드락.

무슨 일:
```
스레드1: A 획득 → B 대기 (2가 B 쥠)
스레드2: B 획득 → A 대기 (1이 A 쥠)
→ 서로 상대가 놓기를 영원히 기다림
```

## 데드락의 4조건 (Coffman conditions)

**네 개가 동시에** 성립해야 데드락. 하나만 깨도 예방:

1. **상호 배제(mutual exclusion)**: 자원을 한 번에 하나만 (락의 본질)
2. **점유 대기(hold and wait)**: 자원 쥔 채 다른 자원 대기
3. **비선점(no preemption)**: 남의 자원을 강제로 뺏을 수 없음
4. **순환 대기(circular wait)**: 대기 사슬이 원을 이룸 (1→2→1)

위 예제는 넷 다 성립. 특히 순환 대기(1은 B를, 2는 A를)가 핵심.

## 예방: 조건 하나 깨기

### 순환 대기 제거 (가장 실용적)

**모든 락을 항상 같은 전역 순서로 획득.** 위 예제에서 두 스레드 모두 "A 먼저, B 나중"으로 하면:

```
스레드1: lock(A); lock(B);   // 같은 순서
스레드2: lock(A); lock(B);   // 같은 순서 → 순환 불가
```

순환이 생길 수 없음. 락에 순서(주소나 ID)를 정하고 항상 오름차순 획득 → 데드락 원천 차단. **실전 표준 규칙.**

### 다른 조건 깨기

- **점유 대기 제거**: 필요한 락을 전부 한 번에 획득(all-or-nothing), 하나라도 못 얻으면 전부 놓기. 구현 복잡
- **비선점 제거**: `trylock` - 못 얻으면 쥔 것 놓고 재시도. 라이브락 위험
- **상호 배제 제거**: 락 없는 자료구조 → [[lock-free-basics]]

## 회피 (avoidance)

실행 중 자원 요청을 보고 "이걸 주면 데드락 가능성 있나" 판단해 안전할 때만 허용. **은행원 알고리즘(Banker's)**이 고전. 최대 요구량을 미리 알아야 하고 비싸서 실전 범용 OS엔 거의 안 씀.

## 탐지 + 복구

데드락을 막지 않고 **생기면 감지해서 처리**:

- 자원 할당 그래프에서 사이클 탐지
- 발견 시 한 스레드를 죽이거나(희생) 롤백
- **데이터베이스**가 이 방식 ([[concurrency-control]]): 트랜잭션 데드락을 감지해 하나를 abort하고 재시도. 락 순서를 강제하기 어려운 환경이라

## 라이브락과 기아

데드락의 사촌들:

- **라이브락(livelock)**: 멈추진 않는데 진전이 없음. 서로 양보하다 계속 충돌 (복도에서 마주쳐 같은 방향으로 계속 비키는 두 사람)
- **기아(starvation)**: 특정 스레드가 계속 자원을 못 얻음 (우선순위 낮아서, 다른 스레드가 계속 선점). 데드락은 아니지만 그 스레드는 진전 없음

## 실전 지침

1. **락 순서 규칙**: 여러 락은 항상 정해진 순서로. 문서화
2. **락 최소화**: 락을 적게, 잡는 시간 짧게 → 데드락 표면 축소
3. **trylock + 타임아웃**: 못 얻으면 포기하고 재시도
4. **계층화**: 락에 레벨을 매기고 낮은→높은만 허용
5. 도구: TSan, 락 순서 검증기(lockdep, 리눅스 커널)
