# 데드락 (Deadlock)

## 한 줄 요약

여러 스레드가 서로가 쥔 자원을 기다리며 영원히 멈추는 상황. 네 조건이 동시에 성립할 때만 발생하고, 락을 항상 같은 순서로 잡으면 예방할 수 있다.

## 왜 필요한가

- 멀티스레드 프로그램이 "멈춰버리는" 대표 원인
- 락을 여러 개 쓸 때의 위험
- 예방/회피/탐지 중 실전에서 뭘 쓰나

## 실증: 락 순서 꼬임

두 스레드가 두 락을 **반대 순서**로 잡음:

```c
// 스레드1: A 잡고 → B
lock(&A); usleep(1000); lock(&B); ...

// 스레드2: B 잡고 → A
lock(&B); usleep(1000); lock(&A); ...
```

이 머신 실측: "t1 got both" / "t2 got both"가 **영원히 안 나옴**. 3초 후에도 두 스레드 살아있음 = 데드락.

무슨 일:
```
스레드1: A 획득 → B 대기 (2가 B 쥠)
스레드2: B 획득 → A 대기 (1이 A 쥠)
→ 서로 상대가 놓기를 영원히 기다림
```

## 데드락의 4조건 (Coffman conditions)

**네 개가 동시에** 성립해야 데드락. 하나만 깨도 예방:

1. **상호 배제(mutual exclusion)**: 자원을 한 번에 하나만 (락의 본질)
2. **점유 대기(hold and wait)**: 자원 쥔 채 다른 자원 대기
3. **비선점(no preemption)**: 남의 자원을 강제로 뺏을 수 없음
4. **순환 대기(circular wait)**: 대기 사슬이 원을 이룸 (1→2→1)

위 예제는 넷 다 성립. 특히 순환 대기(1은 B를, 2는 A를)가 핵심.

## 예방: 조건 하나 깨기

### 순환 대기 제거 (가장 실용적)

**모든 락을 항상 같은 전역 순서로 획득.** 위 예제에서 두 스레드 모두 "A 먼저, B 나중"으로 하면:

```
스레드1: lock(A); lock(B);   // 같은 순서
스레드2: lock(A); lock(B);   // 같은 순서 → 순환 불가
```

순환이 생길 수 없음. 락에 순서(주소나 ID)를 정하고 항상 오름차순 획득 → 데드락 원천 차단. **실전 표준 규칙.**

### 다른 조건 깨기

- **점유 대기 제거**: 필요한 락을 전부 한 번에 획득(all-or-nothing), 하나라도 못 얻으면 전부 놓기. 구현 복잡
- **비선점 제거**: `trylock` - 못 얻으면 쥔 것 놓고 재시도. 라이브락 위험
- **상호 배제 제거**: 락 없는 자료구조 → [[lock-free-basics]]

## 회피 (avoidance)

실행 중 자원 요청을 보고 "이걸 주면 데드락 가능성 있나" 판단해 안전할 때만 허용. **은행원 알고리즘(Banker's)**이 고전. 최대 요구량을 미리 알아야 하고 비싸서 실전 범용 OS엔 거의 안 씀.

## 탐지 + 복구

데드락을 막지 않고 **생기면 감지해서 처리**:

- 자원 할당 그래프에서 사이클 탐지
- 발견 시 한 스레드를 죽이거나(희생) 롤백
- **데이터베이스**가 이 방식 ([[concurrency-control]]): 트랜잭션 데드락을 감지해 하나를 abort하고 재시도. 락 순서를 강제하기 어려운 환경이라

## 라이브락과 기아

데드락의 사촌들:

- **라이브락(livelock)**: 멈추진 않는데 진전이 없음. 서로 양보하다 계속 충돌 (복도에서 마주쳐 같은 방향으로 계속 비키는 두 사람)
- **기아(starvation)**: 특정 스레드가 계속 자원을 못 얻음 (우선순위 낮아서, 다른 스레드가 계속 선점). 데드락은 아니지만 그 스레드는 진전 없음

## 실전 지침

1. **락 순서 규칙**: 여러 락은 항상 정해진 순서로. 문서화
2. **락 최소화**: 락을 적게, 잡는 시간 짧게 → 데드락 표면 축소
3. **trylock + 타임아웃**: 못 얻으면 포기하고 재시도
4. **계층화**: 락에 레벨을 매기고 낮은→높은만 허용
5. 도구: TSan, 락 순서 검증기(lockdep, 리눅스 커널)

## 셀프 체크

> [!question]- 데드락의 네 조건(Coffman)이 "동시에" 성립해야 한다는 말이 예방에서 왜 중요한가?
> 상호 배제, 점유 대기, 비선점, 순환 대기 네 개가 모두 성립할 때만 데드락이 생긴다. 따라서 이 중 하나만 깨도 데드락을 원천 예방할 수 있다. 실전에서는 순환 대기를 깨는(락을 항상 같은 순서로 획득) 방식이 가장 실용적이다.

> [!question]- 락을 항상 같은 전역 순서로 획득하면 데드락이 사라지는 이유는?
> 순환 대기 조건이 성립할 수 없기 때문이다. 모든 스레드가 "A 먼저, B 나중"처럼 정해진 순서로만 잡으면, "1은 B 기다리고 2는 A 기다리는" 식의 대기 사이클이 만들어질 수 없다. 락에 주소나 ID로 순서를 정해 항상 오름차순 획득하는 것이 실전 표준 규칙이다.

> [!question]- 회피(banker's)가 예방보다 실전 범용 OS에서 거의 안 쓰이는 이유는?
> 은행원 알고리즘은 각 프로세스의 최대 자원 요구량을 미리 알아야 하고, 매 요청마다 안전 상태인지 판정하는 비용이 크다. 범용 OS에서는 프로세스가 얼마나 자원을 요구할지 미리 알기 어렵고 오버헤드도 부담이라, 순서 규칙 같은 예방이나 탐지+복구를 택한다.

> [!question]- 데드락, 라이브락, 기아의 차이는?
> 데드락은 서로가 쥔 자원을 기다리며 완전히 멈춤. 라이브락은 멈추진 않지만 서로 양보하며 계속 충돌해 진전이 없음(복도에서 계속 같은 방향으로 비키는 두 사람). 기아는 특정 스레드가 우선순위나 선점 때문에 계속 자원을 못 얻어 진전이 없음. 셋 다 진전 없음이지만 원인과 활동 여부가 다르다.

## 연습문제

> [!example]- 문제: 스레드1이 lock(A)→lock(B), 스레드2가 lock(B)→lock(A) 순으로 잡는다. 데드락을 4조건으로 판정하고 최소 수정으로 없애라
> **풀이**
> 판정 - 네 조건 모두 성립:
> - 상호 배제: A, B 락은 한 번에 하나만 소유.
> - 점유 대기: 1은 A 쥔 채 B 대기, 2는 B 쥔 채 A 대기.
> - 비선점: 상대의 락을 뺏을 수 없음.
> - 순환 대기: 1→(B 대기)→2→(A 대기)→1 사이클.
> 넷 다 성립하므로 데드락. 특히 두 스레드의 획득 순서가 반대라 순환이 생긴다.
> 수정: 순환 대기만 깨면 됨. 스레드2도 `lock(A); lock(B);`로 바꿔 전역 순서를 통일하면 사이클이 불가능해져 데드락 소멸.

> [!example]- 문제: 자원 할당 그래프에서 P1→R1, R1→P2, P2→R2, R2→P1 (→는 대기/할당 엣지)일 때 데드락 여부와 탐지 후 처리 방법을 서술하라
> **풀이**
> 그래프를 따라가면 P1이 R1을 요청 → R1은 P2가 보유 → P2가 R2 요청 → R2는 P1이 보유 → 다시 P1. **사이클이 존재**한다. 단일 인스턴스 자원이면 사이클 = 데드락이므로 데드락이 맞다.
> 탐지+복구 처리: 자원 할당 그래프에서 사이클을 감지한 뒤, 사이클에 속한 스레드 하나를 희생(죽이거나 롤백)해 사이클을 끊는다. 데이터베이스가 이 방식으로, 트랜잭션 데드락을 감지해 하나를 abort하고 재시도시킨다(락 순서를 강제하기 어려운 환경이라).

> 1. 4조건을 나열하고 각 조건을 깨는 예방 기법을 하나씩 짝지을 수 있는가
> 2. 왜 "락 순서 통일"이 순환 대기를 깨는 실전 표준인지, 반대 순서 예제로 설명할 수 있는가
> 3. 예방/회피/탐지+복구가 각각 언제 쓰이는지(특히 DB가 왜 탐지+복구인지) 구분할 수 있는가
