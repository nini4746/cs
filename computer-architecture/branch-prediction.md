# 분기 예측 (Branch Prediction)

## 한 줄 요약

파이프라인 CPU는 분기 결과가 나오기 전에 다음 명령을 미리 가져와야 한다. 그래서 "분기할지"를 예측하고 투기적으로 진행한다. 맞으면 공짜, 틀리면 파이프라인을 통째로 비운다. 예측 가능한 분기와 불가능한 분기의 성능 차이가 여기서 난다.

## 왜 필요한가

- 제어 해저드([[hazards]])의 실제 해결책
- "정렬된 배열이 정렬 안 된 배열보다 빨리 처리된다"는 유명한 현상의 정체
- 왜 예측하기 좋은 코드를 짜야 하나

## 문제: 분기는 늦게 결정된다

[[pipelining]]에서 분기 명령의 결과(점프 여부/목적지)는 EX 단계쯤에야 확정. 하지만 IF 단계는 매 사이클 명령을 가져와야 함 → 분기 뒤에 뭘 넣을지 모른 채 진행해야 함.

- 그냥 멈춰 기다리면(스톨): 분기마다 파이프라인 깊이만큼 손해. 코드의 ~20%가 분기라 치명적
- 그래서 **예측하고 투기적으로 실행**(speculative execution)

## 예측기 (predictor)

분기 이력을 기록해 다음을 예측:

- **정적 예측**: "뒤로 가는 분기(루프)는 taken" 같은 고정 규칙
- **동적 예측**: 각 분기의 과거 행동을 하드웨어 테이블에 저장. 2비트 포화 카운터, 이력 기반(gshare), 현대는 TAGE 같은 정교한 예측기 (95%+ 적중)

**맞으면**: 투기적으로 실행한 결과를 커밋. 손실 0
**틀리면**: 투기 실행한 명령 전부 폐기(flush), 올바른 경로부터 다시 채움. 파이프라인 깊이(14~19)만큼 사이클 손실

## 정렬된 배열 현상 - 실측

가장 유명한 데모. 0~255 랜덤 값 배열에서 `if (a[i] >= 128) sum += a[i]`:

```c
for (long i = 0; i < N; i++)
    if (a[i] >= 128) s += a[i];
```

- **정렬 안 됨**: `a[i] >= 128`이 랜덤 → 예측기가 못 맞춤 (50% 실패). 매 실패마다 flush
- **정렬됨**: 앞부분은 전부 false, 어느 지점부터 전부 true → 거의 항상 맞음

이 머신 실측 (`-O0`, N=5천만):

```
unsorted: 0.229s
sorted  : 0.067s     ← 3.4배
```

데이터만 정렬했을 뿐 코드는 동일. 차이 전부가 분기 예측 성패에서 나온다.

## 반전: 컴파일러가 분기를 없애버린다

같은 코드를 `-O1`로 컴파일하면 차이가 크게 줄어든다. 왜? 컴파일러가 **if를 조건부 실행(branchless)으로 변환**:

```asm
csel  ...   ; 분기 대신 조건부 선택 - 분기 자체가 없음
```

분기가 없으면 예측할 것도 없다 → 예측 실패 비용 소멸. 이 머신 `-O1`은 `csel` 2개로 변환, 위 3.4배가 ~1.8배로 축소.

**교훈 두 개**:
1. 예측 불가 분기는 비싸다 (예측 효과 실증)
2. 그래서 컴파일러는 짧은 조건부를 아예 branchless로 바꾼다 ([[assembly-basics]]의 `cneg`, [[compiler-optimization-limits]]와 같은 맥락). 현대 컴파일러에서 이 고전 데모가 `-O2`에선 잘 재현 안 되는 이유

## 실전 함의

- **예측 가능하게**: 루프 조건, 정렬된 데이터, 편향된 분기(거의 항상 한 쪽)는 예측기가 좋아함
- **branchless 기법**: 예측 불가 분기는 산술/비트 연산으로 분기를 제거 (조건부 이동, 마스킹)
- **Spectre**: 투기적 실행이 보안 취약점이 된 사건 - 예측 실패로 폐기됐어야 할 명령이 캐시에 흔적을 남김 → [[cache-coherence]] 곁가지, [[memory-safety]]

## 셀프 체크

> [!question]- 파이프라인 CPU에서 분기 예측이 왜 필요한가?
> 분기 명령의 결과(점프 여부/목적지)는 EX 단계쯤에야 확정되지만, IF 단계는 매 사이클 명령을 가져와야 한다. 그냥 멈춰 기다리면(스톨) 분기마다 파이프라인 깊이만큼 손해고, 코드의 약 20%가 분기라 치명적이다. 그래서 결과가 나오기 전에 예측하고 투기적으로 실행한다.

> [!question]- 예측이 맞을 때와 틀릴 때 각각 어떤 비용이 드나?
> 맞으면 투기 실행한 결과를 그대로 커밋하므로 손실 0(공짜)이다. 틀리면 투기 실행한 명령을 전부 폐기(flush)하고 올바른 경로부터 파이프라인을 다시 채워야 해서, 파이프라인 깊이(대략 14~19)만큼의 사이클을 손해 본다.

> [!question]- 정렬된 배열이 정렬 안 된 배열보다 빠르게 처리되는 이유는?
> `if (a[i] >= 128)` 같은 분기에서, 정렬 안 된 배열은 조건이 랜덤(50% 실패)이라 예측기가 못 맞추고 매번 flush가 난다. 정렬된 배열은 앞부분이 전부 false, 어느 지점부터 전부 true라 거의 항상 맞는다. 데이터만 정렬했을 뿐 코드는 동일하며, 차이 전부가 분기 예측 성패에서 나온다.

> [!question]- 같은 코드를 `-O1`로 컴파일하면 정렬/미정렬 차이가 왜 줄어드나?
> 컴파일러가 짧은 `if`를 조건부 실행(branchless, `csel`)으로 변환하기 때문이다. 분기 자체가 없으면 예측할 것도 없어서 예측 실패 비용이 소멸한다. 그래서 이 고전 데모가 `-O2`에서는 잘 재현되지 않는다.

## 연습문제

> [!example]- 문제: 파이프라인 깊이 15, 전체 명령 중 분기 20%, 분기 예측 실패율 10%일 때, 명령당 평균 추가 사이클(CPI 증가분)을 구하라. 예측 실패 페널티는 파이프라인 깊이와 같다고 본다.
> **풀이**
> 예측 실패 1건당 페널티 = 15 사이클.
> 명령당 예측 실패 확률 = 0.20(분기 비율) × 0.10(실패율) = 0.02.
> 명령당 평균 추가 사이클 = 0.02 × 15 = **0.3 CPI**.
> 명령 100개당 2번 실패하고 각각 15사이클을 버리므로 30사이클, 즉 명령당 0.3사이클의 오버헤드다.

> [!example]- 문제: 위와 같은 조건(깊이 15, 분기 20%)에서 예측기를 개선해 실패율을 10% → 2%로 낮추면 명령당 추가 사이클은 얼마가 되고, 절감량은 몇 %인가?
> **풀이**
> 새 실패 확률 = 0.20 × 0.02 = 0.004. 추가 사이클 = 0.004 × 15 = **0.06 CPI**.
> 기존 0.3에서 0.06으로 줄었으므로 절감 = (0.3 − 0.06) / 0.3 = 0.24 / 0.3 = **80% 감소**. 실패율을 1/5로 낮췄으니 페널티도 정확히 1/5이 된다.

> [!example]- 문제: `if (a[i] >= 128) s += a[i];`를 예측 실패에 취약하지 않도록 branchless로 바꾸는 아이디어를 하나 제시하라.
> **풀이**
> 조건을 산술/마스크로 바꿔 분기를 제거한다. 예: `int mask = -(a[i] >= 128); s += a[i] & mask;` - 조건이 참이면 mask가 전부 1비트(−1), 거짓이면 0이 되어 더할 값을 마스킹한다. 또는 조건부 이동(`csel`/cmov)을 유도한다. 분기가 사라지면 예측할 대상이 없어져 미정렬 데이터에서도 페널티가 없다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: (1) 파이프라인에서 분기 결과가 늦게 확정되어 예측 없이는 스톨해야 하는 이유. (2) 예측 성공은 공짜, 실패는 파이프라인 깊이만큼 flush 비용 - 그래서 예측 불가 분기가 비싸다. (3) 컴파일러가 짧은 조건부를 branchless(`csel`)로 바꿔 예측 자체를 없애는 반전.
