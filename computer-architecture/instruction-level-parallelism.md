# 명령어 수준 병렬성 (ILP)

## 한 줄 요약

현대 CPU는 한 사이클에 여러 명령어를 동시에, 순서까지 바꿔가며 실행한다. 하지만 의존 체인(앞 결과를 뒤가 기다림)이 있으면 이 병렬성이 막힌다. 의존 체인을 끊으면 같은 코드가 몇 배 빨라진다.

## 왜 필요한가

- IPC(사이클당 명령어)가 1을 넘는 이유 - 파이프라인은 1이 이론 최대인데
- 같은 연산량인데 누적 변수 개수만 늘려도 빨라지는 현상
- 컴파일러 `-O3`의 루프 언롤링이 노리는 것

## 슈퍼스칼라 + 비순차 실행

[[pipelining]]은 명령어를 겹쳤지만 한 번에 하나씩 발행. 현대 CPU는 더 나감:

- **슈퍼스칼라 (superscalar)**: 한 사이클에 여러 명령어 발행. 실행 유닛(ALU, load/store, FP)이 여러 개 → IPC > 1 가능. Apple Silicon은 8-wide 수준
- **비순차 실행 (out-of-order)**: 프로그램 순서를 무시하고 준비된 명령부터 실행, 결과는 원래 순서로 커밋(retire). 앞 명령이 메모리 대기 중이면 뒤 독립 명령을 먼저
- **레지스터 리네이밍**: 이름만 같고 실제 독립인 의존(WAR/WAW)을 물리 레지스터를 늘려 제거

이 모든 하드웨어가 "동시에 실행 가능한 독립 명령어"를 찾아 채운다. 그게 **ILP**.

## 병목: 의존 체인

ILP의 적은 **데이터 의존**. 각 연산이 이전 결과를 기다리면 아무리 실행 유닛이 많아도 놀 수밖에 없다.

```c
double s = 0;
for (i = 0; i < N; i++)
    s += v[i];      // 각 덧셈이 이전 s를 기다림 - 체인 하나
```

`s`에 대한 덧셈이 한 줄로 사슬처럼 묶임. FP 덧셈 지연시간이 ~4사이클이면, 처리량이 아무리 높아도 4사이클마다 하나씩만 진행.

### 의존 끊기: 다중 누적

독립 누적 변수로 사슬을 여러 개로 쪼개면 병렬 실행:

```c
double a=0,b=0,c=0,d=0;
for (i = 0; i < N; i += 4) {
    a += v[i];    b += v[i+1];   // 네 사슬이 서로 독립
    c += v[i+2];  d += v[i+3];   // → 동시에 진행
}
// 마지막에 a+b+c+d
```

이 머신 실측 (`-O1 -fno-tree-vectorize`, N=1억):

```
1 accumulator : 0.090s
4 accumulators: 0.022s   ← 4배
```

같은 덧셈 횟수인데 4배. 실행 유닛이 놀지 않고 4개 사슬을 병렬 처리한 것.

### 왜 컴파일러가 자동으로 안 하나

정수라면 하지만, **FP 덧셈은 결합법칙이 안 성립**([[floating-point]]) → 컴파일러가 `(a+b)+c`를 `a+(b+c)`로 재배열하면 결과가 미세하게 달라질 수 있어 금지. `-ffast-math`를 켜면 컴파일러가 알아서 다중 누적/벡터화. 정확성을 포기하는 대가.

## 루프 언롤링

루프 본체를 여러 번 복제해 한 반복에 여러 원소 처리:

- 루프 오버헤드(카운터 증가, 분기) 상각
- 독립 명령어를 노출해 ILP 재료 제공
- 컴파일러가 `-O3`에서 자동으로, 위 다중 누적과 결합

## 한계: ILP는 무한하지 않다

- 프로그램의 의존 구조가 본질적으로 순차면 (pointer chasing, 다음 주소가 이전 로드 결과) ILP 없음 → 링크드 리스트가 배열보다 느린 또 하나의 이유 ([[memory-hierarchy]])
- 분기 예측 실패는 파이프라인을 비움 → [[branch-prediction]]
- 실행 유닛 개수, 명령어 윈도우 크기가 물리적 상한

## 셀프 체크

> [!question]- IPC가 파이프라인의 이론 최대 1을 넘을 수 있는 이유는?
> 슈퍼스칼라 CPU는 한 사이클에 여러 명령어를 동시에 발행하고, ALU·load/store·FP 등 실행 유닛을 여러 벌 갖는다. 독립적인 명령이 충분하면 한 사이클에 여러 개가 완료되므로 IPC가 1을 넘는다. Apple Silicon은 8-wide 수준까지 발행한다.

> [!question]- 의존 체인이 왜 ILP를 막나?
> 각 연산이 직전 결과를 입력으로 기다리면, 실행 유닛이 아무리 많아도 한 번에 하나씩만 진행할 수 있다. FP 덧셈 지연이 4사이클이면 한 사슬은 4사이클마다 하나씩만 나아가 처리량을 다 못 쓴다. 병렬 하드웨어가 놀게 된다.

> [!question]- 다중 누적 변수가 왜 같은 덧셈 횟수인데 몇 배 빨라지나?
> 하나의 누적 변수는 덧셈을 한 사슬로 묶어 지연시간에 묶인다. 독립 누적 변수 여러 개로 쪼개면 사슬이 여러 개가 되어 서로 독립이므로 실행 유닛이 이들을 병렬로 처리한다. 지연시간에 묶이던 것이 처리량 한계로 옮겨가 빨라진다.

> [!question]- 컴파일러가 FP 덧셈을 자동으로 다중 누적/재배열하지 않는 이유는?
> FP 덧셈은 결합법칙이 성립하지 않는다. `(a+b)+c`와 `a+(b+c)`가 반올림 때문에 미세하게 다른 값이 될 수 있어, 컴파일러가 순서를 바꾸면 결과가 달라질 수 있다. 그래서 기본적으로 금지되고, `-ffast-math`로 정확성을 포기해야 자동으로 해준다.

> [!question]- pointer chasing에 ILP가 없는 이유는?
> 다음에 접근할 주소가 직전 로드의 결과일 때(링크드 리스트 순회 등), 각 로드가 이전 로드 완료를 기다려야 한다. 본질적으로 순차 의존이라 비순차 실행이 앞질러 채울 독립 명령이 없다. 배열보다 링크드 리스트가 느린 한 이유다.

## 연습문제

> [!example]- 문제: 누적 변수 개수에 따른 소요 사이클을 계산하라
> **풀이**
> 조건: FP 덧셈 지연 L = 4사이클, 처리량 1개/사이클(완전 파이프라인), 덧셈 N회.
> 누적 1개: 각 덧셈이 이전 결과 대기 → 사슬이 지연에 묶임 → 약 **L × N = 4N 사이클**.
> 누적 k개: 사슬이 k개로 독립. k가 지연을 다 덮으면(k ≥ L) 매 사이클 하나씩 완료 → 약 **N 사이클**(처리량 한계).
> 최적 누적 개수 = 지연 × 처리량 = 4 × 1 = **4개**. 이 지점에서 4N → N으로 약 4배. 노트 실측(1 acc 0.090s → 4 acc 0.022s ≈ 4배)과 일치한다.

> [!example]- 문제: 누적 변수를 8개로 늘려도 8배가 안 나오는 상한을 계산하라
> **풀이**
> 유용한 누적 개수의 상한 = FP 덧셈 지연 × FP 덧셈 유닛 수. 지연 4, FP 덧셈 유닛이 2개라면 상한 = 4 × 2 = **8개**. 다만 지연만 덮는 데 필요한 최소는 4개(유닛 1개 기준)이고, 그 이상은 유닛 개수만큼만 추가 처리량이 붙는다.
> 유닛이 1개뿐이면 4개를 넘겨도 처리량이 이미 포화라 더 빨라지지 않는다. 즉 4개에서 처리량 한계(1개/사이클)에 도달했다면 8개는 개선이 없다. 실행 유닛 개수와 명령어 윈도우 크기가 물리적 상한이다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 슈퍼스칼라·비순차 실행·레지스터 리네이밍이 각각 무엇을 하고, 왜 셋이 함께 ILP를 늘리는지 말할 수 있는가.
> 2. 의존 체인이 왜 병목이고, 다중 누적이 지연 한계를 처리량 한계로 어떻게 바꾸는지 설명할 수 있는가.
> 3. 최적 누적 개수를 지연·처리량·유닛 수로 유도할 수 있고, ILP의 물리적 상한을 셋 이상 댈 수 있는가.
