# 파이프라인 해저드 (Hazards)

## 한 줄 요약

파이프라인이 이상적인 1 IPC를 못 내는 이유. 자원 충돌(구조), 데이터 의존(데이터), 분기 불확실(제어) 세 종류가 파이프라인을 멈추게 하고, 포워딩·스톨·예측으로 완화한다.

## 왜 필요한가

- [[pipelining]]이 "이상적으로 5배"라 했는데 실제로 왜 덜 나오나
- 코드가 왜 어떤 순서에서 빠르고 어떤 순서에서 느린가 (데이터 의존)
- 컴파일러가 명령어 순서를 왜 재배열하나

## 1. 구조 해저드 (Structural Hazard)

**하드웨어 자원이 한 사이클에 두 곳에서 필요.**

예: 명령어와 데이터가 같은 메모리에 있으면, IF(명령어 인출)와 MEM(데이터 접근)이 같은 사이클에 충돌.

해결: 자원 분리. **명령어 캐시와 데이터 캐시를 나눔** (하버드 구조) → [[memory-hierarchy]]에서 L1이 I/D로 분리된 이유. 현대 CPU는 설계로 대부분 제거.

## 2. 데이터 해저드 (Data Hazard)

**앞 명령의 결과를 뒷 명령이 필요한데 아직 안 나옴.**

```asm
add x1, x2, x3    ; x1을 WB에서 씀 (사이클 5)
sub x4, x1, x5    ; x1을 ID/EX에서 읽으려 함 (사이클 3) - 아직 없다!
```

x1은 사이클 5에 써지는데 sub는 사이클 3에 읽으려 함 → 잘못된 값.

### 해결 1: 포워딩 (forwarding / bypassing)

결과를 WB까지 기다리지 않고 **EX 단계 출력에서 바로 다음 명령의 EX 입력으로 전달**. ALU가 계산을 끝낸 순간(사이클 3 EX 끝) 그 값을 바로 넘김 → 스톨 없이 해결.

대부분의 데이터 해저드는 포워딩으로 거품 없이 해결됨.

### 해결 2: load-use 해저드는 스톨 강제

```asm
ldr x1, [x2]      ; 메모리 값은 MEM 단계(사이클 4)에야 나옴
add x3, x1, x4    ; EX(사이클 3)에서 x1 필요 - 포워딩해도 1사이클 부족
```

load 결과는 MEM 단계에야 준비되므로 아무리 포워딩해도 1사이클을 못 당김 → **1사이클 스톨(거품) 강제**. 컴파일러가 load 뒤에 무관한 명령을 끼워넣어(스케줄링) 이 거품을 채우려는 이유.

## 3. 제어 해저드 (Control Hazard)

**분기 결과가 나오기 전에 다음 명령을 가져와야 함.**

```asm
cmp x1, #0
b.eq LABEL        ; 분기할지 말지 EX에서 결정 - 그 전에 IF는 뭘 가져오나?
...               ; 분기 결과 나올 때까지 파이프라인에 넣을 명령이 불확실
```

분기 명령이 "다음 주소"를 확정하기 전에 파이프라인은 이미 뒤 명령들을 인출하기 시작함. 분기가 예상과 다르면 그 명령들을 버려야 함(flush).

### 해결: 분기 예측

과거 이력으로 "분기할 것 같다/아니다"를 예측하고 미리 진행. 맞으면 손실 0, 틀리면 파이프라인 flush (파이프라인 깊이만큼 손실). 상세는 [[branch-prediction]].

## 정리 표

| 해저드 | 원인 | 주 해결책 | 잔여 비용 |
|---|---|---|---|
| 구조 | 자원 충돌 | 자원 분리 (I/D 캐시) | 보통 0 |
| 데이터 | 결과 미준비 | 포워딩 | load-use는 1사이클 |
| 제어 | 분기 불확실 | 분기 예측 | 예측 실패 시 flush |

## 컴파일러의 역할

컴파일러는 이 해저드를 알고 명령어를 재배열(instruction scheduling):

- load 뒤에 의존 명령을 바로 안 놓고 무관한 명령을 끼움 (load-use 거품 채우기)
- 독립적인 연산들을 섞어 데이터 의존 체인을 끊음

이것이 `-O2`에서 명령어 순서가 소스와 크게 달라지는 이유 중 하나 ([[assembly-basics]]에서 본 재배열).

## 셀프 체크

> [!question]- 포워딩이 있어도 load-use 해저드는 왜 최소 1사이클 스톨이 강제되나?
> 포워딩은 값이 준비된 단계의 출력을 다음 명령의 EX 입력으로 넘긴다. 하지만 load 결과는 MEM 단계(사이클 4)에야 나오는데, 바로 뒤 명령은 EX(사이클 3)에서 그 값을 필요로 한다. 시간상 1사이클 앞선 값을 뒤로 넘길 수는 없으므로 포워딩으로도 못 메우고 1사이클 거품이 강제된다.

> [!question]- 구조 해저드를 하버드 구조(I/D 캐시 분리)가 어떻게 제거하나?
> 구조 해저드는 한 사이클에 같은 하드웨어 자원이 두 곳에서 필요할 때 생긴다. 명령어와 데이터가 한 메모리에 있으면 IF(명령어 인출)와 MEM(데이터 접근)이 충돌한다. L1을 명령어 캐시와 데이터 캐시로 물리적으로 나누면 두 접근이 서로 다른 포트를 쓰므로 충돌이 사라진다.

> [!question]- 대부분의 데이터 해저드가 스톨 없이 해결되는 이유는?
> ALU 연산 결과는 EX 단계 끝에 이미 확정된다. 이 값을 WB까지 기다리지 않고 EX 출력에서 다음 명령의 EX 입력으로 바로 포워딩하면 거품 없이 이어진다. load처럼 결과가 뒤 단계(MEM)에야 나오는 경우만 예외적으로 스톨이 남는다.

> [!question]- 제어 해저드에서 분기 예측 실패 손실이 왜 파이프라인 깊이에 비례하나?
> 분기 방향이 확정되기 전에 파이프라인은 이미 예측한 경로의 명령들을 인출·진행시킨다. 예측이 틀리면 그때까지 잘못 넣은 명령을 전부 flush해야 하는데, 그 개수가 분기 확정 시점까지 채워진 파이프라인 단계 수 - 즉 파이프라인 깊이에 비례한다.

> [!question]- 컴파일러가 load 바로 뒤에 무관한 명령을 끼우는 이유는?
> load-use 거품을 채우기 위해서다. load 결과에 의존하는 명령을 바로 놓으면 1사이클 스톨이 생기는데, 그 자리에 load 결과와 무관한 독립 명령을 넣으면 CPU가 거품 대신 유용한 일을 한다. instruction scheduling의 핵심 동기다.

## 연습문제

> [!example]- 문제: 5단계 파이프라인의 실효 CPI를 계산하라
> **풀이**
> 조건: 기본 CPI 1. 명령어 중 load 20%, 그 중 절반이 바로 뒤에서 결과를 써 load-use 스톨 1사이클. 분기 15%, 예측 실패율 10%, 실패 시 flush 3사이클.
> load-use 기여 = 0.20 × 0.5 × 1 = 0.10
> 분기 실패 기여 = 0.15 × 0.10 × 3 = 0.045
> 실효 CPI = 1 + 0.10 + 0.045 = **1.145**
> 이상적 1 IPC 대비 약 14.5% 손실. 대부분이 load-use 거품에서 온다.

> [!example]- 문제: 다음 코드의 load-use 스톨을 세고, 재배열로 제거하라
> **풀이**
> 원본:
> ```asm
> ldr x1, [x2]
> add x3, x1, x5    ; x1 사용 → load-use 스톨 1
> ldr x6, [x7]
> add x8, x6, x9    ; x6 사용 → load-use 스톨 1
> ```
> load 바로 뒤에 그 결과를 쓰는 의존이 두 번 → **총 2사이클 스톨**.
> 재배열(두 load를 앞으로 모음):
> ```asm
> ldr x1, [x2]
> ldr x6, [x7]     ; x1의 MEM이 진행되는 동안 독립 load 수행
> add x3, x1, x5   ; 이제 x1 준비됨, 스톨 0
> add x8, x6, x9   ; x6도 준비됨, 스톨 0
> ```
> 두 add가 각자 의존하는 load에서 2명령 이상 떨어져 결과가 준비된다 → **스톨 0**. 연산 순서는 바뀌었지만 결과는 동일(모두 독립).

> 3. 컴파일러의 instruction scheduling이 어떤 거품을 어떻게 채우는지 예로 들 수 있는가.
