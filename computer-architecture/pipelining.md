# 파이프라이닝 (Pipelining)

## 한 줄 요약

명령어 하나를 여러 단계로 쪼개고, 세탁기-건조기처럼 단계를 겹쳐 실행한다. 명령어 하나의 지연시간은 그대로지만 처리량(throughput)이 단계 수만큼 오른다.

## 왜 필요한가

- CPU가 클럭을 어떻게 올릴 수 있었나의 핵심 (한 사이클에 할 일을 줄이면 클럭 ↑)
- 분기 예측([[branch-prediction]])·해저드([[hazards]])가 왜 존재하는지의 전제
- "명령어 하나가 여러 사이클 걸리는데 어떻게 IPC가 1에 가깝나"의 답

## 비유: 빨래

빨래 4더미, 각 더미가 세탁(30분)→건조(30분)→개기(30분):

- **순차**: 한 더미 끝내고 다음 → 4 × 90분 = 6시간
- **파이프라인**: 1더미가 건조 중일 때 2더미 세탁 시작 → 겹침. 4더미 = 90분 + 3×30분 = 3시간

단계를 겹치면 각 더미의 소요는 그대로(90분)인데 전체 처리량이 오른다.

## 고전 5단계 파이프라인 (RISC 교과서 모델)

```
IF  → ID  → EX  → MEM → WB
```

1. **IF** (Instruction Fetch): 명령어 메모리에서 가져오기
2. **ID** (Instruction Decode): 해독 + 레지스터 읽기
3. **EX** (Execute): ALU 연산 / 주소 계산
4. **MEM** (Memory): 메모리 접근 (load/store만)
5. **WB** (Write Back): 결과를 레지스터에 쓰기

각 단계 사이에 **파이프라인 레지스터**가 있어 중간 상태를 다음 사이클로 전달.

## 겹침의 그림

```
사이클:    1    2    3    4    5    6    7    8
명령1:    IF   ID   EX   MEM  WB
명령2:         IF   ID   EX   MEM  WB
명령3:              IF   ID   EX   MEM  WB
명령4:                   IF   ID   EX   MEM  WB
```

사이클 5부터는 매 사이클 명령어 하나씩 완료 → 정상 상태 처리량 = 명령어/사이클 = 1 (이상적).

## 성능: 지연시간 vs 처리량

- **지연시간(latency)**: 명령어 하나가 5단계 다 거치는 시간 = 5 사이클. **안 줄어듦** (오히려 단계 레지스터 오버헤드로 살짝 늘 수도)
- **처리량(throughput)**: 이상적으로 5배. 실제로는 해저드 때문에 덜

왜 클럭이 오르나: 5단계로 쪼개면 한 단계가 하는 일이 1/5 → 사이클 시간을 1/5로 줄일 수 있음 → 클럭 5배. 이게 파이프라이닝의 진짜 이득.

### 더 깊은 파이프라인의 함정

단계를 더 쪼개면(예: 20단계, Pentium 4) 클럭은 더 오르지만:

- 분기 예측 실패 시 버리는 명령어가 많아짐 (파이프라인이 길수록 손실 큼)
- 단계 레지스터 오버헤드 증가
- Pentium 4가 31단계까지 갔다가 전력·발열로 실패 → 현대는 14~19단계 수준에서 균형

## 해저드: 이상이 깨지는 지점

파이프라인이 항상 1 IPC를 못 내는 이유. 세 종류 (상세는 [[hazards]]):

- **구조 해저드**: 하드웨어 자원 부족 (한 사이클에 IF와 MEM이 같은 메모리를 원함)
- **데이터 해저드**: 앞 명령의 결과를 뒷 명령이 아직 못 받음 (`add x1,...; sub x2,x1,...`)
- **제어 해저드**: 분기 결과가 나오기 전에 다음 명령을 가져와야 함 → [[branch-prediction]]

해결책 맛보기: 포워딩(결과를 WB 전에 앞당겨 전달), 스톨(거품 삽입), 분기 예측. 전부 [[hazards]]에서.

## 현대 CPU와의 거리

교과서 5단계는 개념 모델. 실제는:

- 14~19단계 파이프라인
- **슈퍼스칼라**: 한 사이클에 여러 명령어 발행 (IPC > 1) → [[instruction-level-parallelism]]
- **비순차 실행**: 순서 바꿔 실행 후 재정렬
- 하지만 이 모든 것의 기반 개념이 파이프라이닝

## 셀프 체크

> [!question]- 파이프라이닝은 명령어 하나의 지연시간을 줄이는가, 처리량을 늘리는가? 그리고 클럭이 오르는 진짜 이유는?
> 지연시간(한 명령어가 5단계를 다 거치는 시간)은 줄지 않고 오히려 단계 레지스터 오버헤드로 살짝 늘 수도 있다. 늘어나는 것은 처리량(단위 시간당 완료 명령어 수)이다. 클럭이 오르는 이유는 명령어를 5단계로 쪼개면 한 단계가 하는 일이 1/5로 줄어 사이클 시간을 1/5로 줄일 수 있기 때문이다. 이것이 파이프라이닝의 진짜 이득이다.

> [!question]- 고전 5단계 파이프라인의 각 단계(IF/ID/EX/MEM/WB)는 무엇을 하며, 파이프라인 레지스터의 역할은?
> IF는 명령어 메모리에서 명령어를 가져오고, ID는 해독 + 레지스터 읽기, EX는 ALU 연산 / 주소 계산, MEM은 메모리 접근(load/store만), WB는 결과를 레지스터에 쓴다. 각 단계 사이의 파이프라인 레지스터가 중간 상태를 다음 사이클로 전달해, 여러 명령어가 서로 다른 단계에 동시에 머무를 수 있게 한다.

> [!question]- 해저드 3종(구조/데이터/제어)은 각각 무엇이며 대표 해결책은?
> 구조 해저드는 하드웨어 자원 부족(한 사이클에 IF와 MEM이 같은 메모리를 원함)으로, 자원 분리로 해결한다. 데이터 해저드는 앞 명령의 결과를 뒷 명령이 아직 못 받는 경우로, 포워딩(WB 전에 결과를 앞당겨 전달)과 스톨로 해결한다. 제어 해저드는 분기 결과가 나오기 전에 다음 명령을 가져와야 하는 문제로, 분기 예측으로 완화한다.

> [!question]- Pentium 4처럼 파이프라인을 더 깊게(예: 31단계) 쪼개면 왜 이득만 있는 게 아닌가?
> 단계를 더 쪼개면 한 단계의 일이 줄어 클럭은 더 오르지만, 분기 예측 실패 시 버려야 하는 in-flight 명령어가 많아져 손실이 커지고, 단계 레지스터 오버헤드도 누적된다. Pentium 4는 31단계까지 갔다가 전력·발열로 실패했고, 현대 CPU는 14~19단계 수준에서 균형을 잡는다.

## 연습문제

> [!example]- 문제: 고전 5단계 파이프라인에서 해저드가 전혀 없을 때 명령어 100개를 완료하는 데 걸리는 사이클 수와 CPI를 구하고, 비파이프라인(명령어당 5사이클) 대비 speedup을 계산하라.
> **풀이**
> 파이프라인 총 사이클 = 채우는 시간(depth − 1) + 명령어 수 = (5 − 1) + 100 = 104사이클.
> CPI = 104 / 100 = 1.04. 명령어 수가 커질수록 초기 채움 4사이클이 희석되어 CPI는 이상값 1에 수렴한다.
> 비파이프라인 = 100 × 5 = 500사이클. speedup = 500 / 104 ≈ 4.81배. 이상적 상한 5배에 근접하며, 부족분은 파이프라인 채움 오버헤드 때문이다.

> [!example]- 문제: 명령어 중 20%가 분기이고 분기의 30%에서 예측이 빗나가며, 예측 실패마다 3사이클 페널티가 붙는다. 기본 CPI가 1일 때 유효 CPI를 구하라.
> **풀이**
> 분기당 평균 페널티 = 예측 실패 확률 × 페널티 = 0.30 × 3 = 0.9사이클.
> 전체 명령어 기준 추가 CPI = 분기 비율 × 분기당 평균 페널티 = 0.20 × 0.9 = 0.18.
> 유효 CPI = 기본 CPI + 추가 CPI = 1 + 0.18 = 1.18.
> 즉 분기 예측 실패만으로 CPI가 18% 나빠진다. 파이프라인이 깊어 페널티가 3 대신 15가 되면 추가 CPI = 0.20 × 0.30 × 15 = 0.9로 급증해, 깊은 파이프라인일수록 정확한 분기 예측이 결정적임을 보여준다.
