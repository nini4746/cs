# 가상 메모리 - 하드웨어 관점 (Virtual Memory)

## 한 줄 요약

프로그램이 보는 주소(가상)는 실제 물리 주소가 아니다. CPU의 MMU가 페이지 테이블로 매번 변환하고, TLB가 이 변환을 캐싱한다. 이 한 겹의 간접이 격리·보호·스와핑을 전부 가능케 한다.

## 왜 필요한가

- 모든 프로세스가 왜 같은 주소(0x...)를 쓰는데 안 충돌하나
- 포인터가 가리키는 주소가 "진짜" 주소가 아닌 이유
- [[cache-misses]] 곡선에 TLB 미스가 어떻게 겹치나
- OS 관점(페이지 교체, 할당)은 [[page-tables]]/[[swapping]], 여기선 하드웨어 변환에 집중

## 핵심 아이디어: 한 겹의 간접

프로그램의 모든 메모리 접근은 **가상 주소(VA)**를 쓴다. 하드웨어(MMU)가 이를 **물리 주소(PA)**로 변환:

```
CPU가 VA 발행 → MMU 변환 → PA로 실제 메모리 접근
```

이 간접 덕분에:

- **격리**: 프로세스마다 다른 매핑 → 서로의 메모리 못 봄. 같은 VA라도 다른 PA로 감
- **보호**: 페이지마다 권한(읽기/쓰기/실행) → NX 비트가 [[buffer-overflow]] 방어에 쓰이는 그것
- **가상 > 물리**: 물리 RAM보다 큰 주소 공간. 안 쓰는 페이지는 디스크로 → [[swapping]]

## 페이지와 페이지 테이블

메모리를 고정 크기 **페이지**(보통 4KB, Apple Silicon 16KB)로 나눔. VA → PA 변환은 페이지 단위:

```
VA = [ 가상 페이지 번호 (VPN) | 페이지 오프셋 ]
     페이지 테이블[VPN] = 물리 페이지 번호 (PPN)
PA = [ PPN | 같은 오프셋 ]
```

오프셋(페이지 내 위치)은 그대로, 페이지 번호만 테이블로 변환. **페이지 테이블**이 VPN → PPN 매핑을 저장 (프로세스마다 하나, 커널이 관리).

### 멀티레벨 페이지 테이블

64비트 주소 공간의 페이지 테이블을 통짜로 만들면 거대함. 그래서 **여러 단계로 쪼갬**(예: 4레벨) → 실제 쓰는 부분만 테이블 할당. 트리 구조로 희소하게. 상세는 [[page-tables]].

## TLB: 변환의 캐시

문제: 매 메모리 접근마다 페이지 테이블을 또 읽어야 함 → 접근 하나가 여러 메모리 접근으로 폭발.

해결: **TLB (Translation Lookaside Buffer)** - 최근 VPN→PPN 변환을 캐싱하는 작은 초고속 캐시 (완전 연관, 수십~수천 엔트리).

- **TLB 히트**: 변환 즉시 (1사이클)
- **TLB 미스**: 페이지 테이블 순회(page walk) → 느림. 하드웨어(x86) 또는 소프트웨어가 처리 후 TLB 채움

TLB는 [[memory-hierarchy]]의 또 다른 캐시. 캐시 계층이 데이터에만 있는 게 아니라 주소 변환에도 있다.

## 성능 함의: TLB도 지역성

- 각 TLB 엔트리가 페이지 하나(4KB/16KB)를 커버 → 수천 엔트리면 수십 MB만 커버
- working set이 이를 넘으면 **TLB 미스**가 [[cache-misses]]의 capacity miss와 별개로 발생
- 랜덤 접근이 큰 영역을 훑으면 캐시 미스 + TLB 미스 이중고 → pointer chasing이 더 느린 숨은 이유
- **huge page**(2MB): 엔트리 하나가 큰 영역 커버 → TLB 압박 완화. DB/JVM이 쓰는 이유

## 변환 + 캐시 순서

실제 접근: VA → TLB로 PA 얻기 → 캐시(L1) 조회. 현대 CPU는 이를 겹침(VIPT 캐시 - 가상 인덱스, 물리 태그)해서 TLB와 L1을 병렬 조회. 하드웨어 세부지만 요점은 **변환과 캐싱이 매 접근마다 함께** 일어난다는 것.

## 페이지 폴트 (하드웨어 → OS)

VA의 페이지가 물리 메모리에 없으면(테이블 엔트리 무효) → **페이지 폴트**([[exceptions-and-interrupts]]) → 커널이 디스크에서 로드하거나(디맨드 페이징) 잘못된 접근이면 종료(segfault). 하드웨어가 감지, OS가 처리 → [[swapping]].

## 셀프 체크

> [!question]- VA→PA 변환에서 페이지 오프셋은 왜 변환되지 않고 그대로 통과하는가?
> 변환이 페이지 단위로 이루어지기 때문이다. VA는 [VPN | 오프셋], PA는 [PPN | 오프셋]으로 나뉘고, 페이지 테이블은 VPN→PPN 매핑만 담당한다. 오프셋은 페이지 내부 위치라서 같은 페이지 안에서는 가상이든 물리든 위치가 동일하므로 그대로 복사된다. 그래서 오프셋 비트 수가 페이지 크기를 결정한다(4KB → 12비트).

> [!question]- 한 겹의 간접(MMU 변환)이 격리·보호·가상>물리를 각각 어떻게 가능하게 하나?
> 격리: 프로세스마다 다른 페이지 테이블을 두어 같은 VA라도 다른 PA로 매핑하므로 서로의 메모리를 볼 수 없다. 보호: 페이지마다 읽기/쓰기/실행 권한 비트를 두어 위반 시 폴트를 낸다(NX 비트가 buffer overflow 방어에 쓰이는 그것). 가상>물리: 안 쓰는 페이지를 디스크로 내려 물리 RAM보다 큰 주소 공간을 제공한다(스와핑).

> [!question]- TLB가 필요한 이유와, TLB 히트/미스 시 각각 무슨 일이 일어나는가?
> TLB가 없으면 매 메모리 접근마다 페이지 테이블을 또 읽어야 해서 접근 하나가 여러 메모리 접근으로 폭발한다. TLB는 최근 VPN→PPN 변환을 캐싱하는 작고 빠른 완전 연관 캐시다. TLB 히트면 변환이 즉시(약 1사이클) 끝나고, TLB 미스면 페이지 테이블을 순회(page walk)해 변환을 구한 뒤 TLB를 채우는데 이 순회가 느리다.

> [!question]- huge page(2MB)가 TLB 압박을 완화하는 원리는?
> TLB 엔트리 하나가 페이지 하나를 커버하는데, 일반 4KB 페이지에서는 수천 개 엔트리로도 수십 MB만 커버해 큰 working set에서 TLB 미스가 잦다. huge page는 엔트리 하나가 2MB를 커버하므로 같은 엔트리 수로 수백 배 넓은 영역을 매핑한다. 그래서 대용량 메모리를 훑는 DB/JVM이 TLB 미스를 줄이려 huge page를 쓴다.

## 연습문제

> [!example]- 문제: 48비트 가상 주소, 16KB 페이지(Apple Silicon)를 쓸 때 페이지 오프셋 비트 수와 VPN 비트 수를 구하라.
> **풀이**
> 페이지 크기 16KB = 2^14바이트 → 오프셋 = log2(16384) = 14비트.
> VPN = 전체 주소 비트 − 오프셋 = 48 − 14 = 34비트.
> 즉 VPN 공간이 2^34개나 되어 단일 페이지 테이블을 통짜로 만들면 거대하다. 이 때문에 멀티레벨(여러 단계) 페이지 테이블로 실제 쓰는 부분만 희소하게 할당한다.

> [!example]- 문제: TLB 히트율 98%, TLB 히트 시 변환 1사이클, TLB 미스 시 page walk 100사이클이라 하자. 변환의 평균 비용(사이클)을 구하고, huge page로 히트율이 99.8%로 오르면 얼마나 개선되는지 계산하라.
> **풀이**
> 평균 변환 비용 = 히트시간 + 미스율 × page walk 페널티.
> 히트율 98%: 1 + 0.02 × 100 = 1 + 2 = 3사이클.
> 히트율 99.8%: 1 + 0.002 × 100 = 1 + 0.2 = 1.2사이클.
> 개선폭 = (3 − 1.2) / 3 = 60% 감소. TLB 미스는 드물어도 페널티가 커서 평균 변환 비용을 좌우하며, huge page로 미스율을 10배 줄이면 변환 오버헤드가 극적으로 준다.

> [!example]- 문제: 4KB 페이지 머신에서 TLB 엔트리가 512개다. TLB가 커버하는 총 메모리 크기를 구하고, 이보다 큰 배열을 랜덤 접근할 때 왜 캐시 미스와 별개의 비용이 추가되는지 설명하라.
> **풀이**
> TLB 커버 범위 = 엔트리 수 × 페이지 크기 = 512 × 4KB = 2048KB = 2MB.
> working set이 2MB를 넘으면 접근하는 페이지가 TLB에 다 못 들어가 TLB 미스(capacity 성격)가 데이터 캐시 미스와 별개로 발생한다. 랜덤/pointer chasing 접근은 인접성이 없어 매 접근이 새 페이지를 건드리기 쉬워, 데이터 캐시 미스(라인 로드) + TLB 미스(page walk)의 이중 페널티를 함께 맞는다. 이것이 링크드 리스트 추적이 더 느린 숨은 이유다.
