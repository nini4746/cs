# 함수 호출과 스택 (Procedures and the Stack)

## 한 줄 요약

함수 호출은 "인자를 약속된 레지스터에 넣고, 복귀 주소를 저장하고, 점프"다. 스택은 이 호출들이 쌓이고 풀리는 LIFO 구조고, 재귀가 되는 이유 전부가 여기 있다.

## 왜 필요한가

- 재귀가 "그냥 되는" 게 아니라 **왜** 되는지 - 호출마다 지역변수가 독립인 메커니즘
- 디버거의 backtrace가 어떻게 만들어지는지
- 스택 오버플로우가 뭘 넘친 건지
- [[buffer-overflow]] 공격 = 이 노트의 구조를 악용하는 것. 전제 지식

## 스택: 아래로 자라는 LIFO

```
높은 주소  ┌──────────────┐
          │ main의 프레임  │
          ├──────────────┤
          │ f의 프레임     │   f가 g를 호출하면
          ├──────────────┤ ← sp
          │ g의 프레임     │   ↓ 이 방향으로 자람 (주소 감소)
낮은 주소  └──────────────┘
```

- **sp (스택 포인터)**: 현재 스택 꼭대기. push = sp 감소, pop = sp 증가
- 함수 호출의 수명이 정확히 LIFO라서 (나중에 불린 게 먼저 반환) 스택과 완벽하게 맞음
- 프로세스당 크기 제한 있음 - 이 머신 실측: `ulimit -s` = **8176KB (~8MB)**. 무한 재귀가 segfault 나는 이유

## 호출 규약 (calling convention)

"인자는 어디에, 반환값은 어디에, 어느 레지스터는 누가 지키나"의 약속. ISA마다 표준 존재 (ARM64: AAPCS64, x86-64: System V ABI).

| | ARM64 | x86-64 (System V) |
|---|---|---|
| 인자 1~N | x0~x7 (8개) | rdi, rsi, rdx, rcx, r8, r9 (6개) |
| 그 이상 | 스택으로 | 스택으로 |
| 반환값 | x0 | rax |
| 복귀 주소 | **lr (x30) 레지스터** | **스택에 push** |
| caller-saved | x0~x17 (호출하면 깨질 수 있음) | rax, rcx, rdx, rsi, rdi, r8~r11 |
| callee-saved | x19~x28 (쓰려면 복원 책임) | rbx, rbp, r12~r15 |

**caller-saved vs callee-saved**: 함수 호출 후에도 살아야 할 값을 caller-saved 레지스터에 두면 호출자가 백업해야 하고, callee-saved에 두면 피호출자가 백업 책임. 컴파일러의 레지스터 배분 전략이 여기서 갈림.

### call/return 메커니즘 - ARM과 x86의 큰 차이

- **ARM64**: `bl f` = 복귀 주소를 **lr 레지스터**에 넣고 점프. `ret` = lr로 점프. 메모리 접근 0회
- **x86-64**: `call f` = 복귀 주소를 **스택에 push**하고 점프. `ret` = 스택에서 pop해서 점프

ARM64에서도 f가 다른 함수를 부르면 lr이 덮이니까 결국 lr을 스택에 저장 - 그래서 leaf가 아닌 함수는 프롤로그에서 `stp x29, x30, [sp, ...]`가 나온다.

## 프레임 해부 - 실물

`fact` 재귀를 이 머신에서 `gcc -O0 -S`로 컴파일한 실제 출력:

```c
long fact(long n) {
    if (n <= 1) return 1;
    return n * fact(n - 1);
}
```

```asm
_fact:
	sub	sp, sp, #48             ; ① 프레임 48바이트 확보
	stp	x29, x30, [sp, #32]     ; ② 이전 fp(x29)와 복귀주소 lr(x30) 백업
	add	x29, sp, #32            ; ③ 새 프레임 포인터 설정
	str	x0, [sp, #16]           ; ④ 인자 n을 지역 슬롯에 저장
	ldr	x8, [sp, #16]
	subs	x8, x8, #1
	b.gt	LBB1_2                  ; n > 1 이면 재귀 경로로
LBB1_1:                             ; 기저: 반환값 1
	mov	x8, #1
	stur	x8, [x29, #-8]
	b	LBB1_3
LBB1_2:                             ; 재귀 경로
	ldr	x8, [sp, #16]
	str	x8, [sp, #8]            ; n을 스택에 백업 (호출하면 레지스터 깨지니까)
	ldr	x8, [sp, #16]
	subs	x0, x8, #1              ; 인자 = n - 1
	bl	_fact                   ; ⑤ 재귀 호출! lr에 복귀주소 넣고 점프
	ldr	x8, [sp, #8]            ; 백업해둔 n 복원
	mul	x8, x8, x0              ; n * fact(n-1)  (x0 = 반환값)
	stur	x8, [x29, #-8]
LBB1_3:
	ldur	x0, [x29, #-8]          ; 반환값을 x0에
	ldp	x29, x30, [sp, #32]     ; ⑥ fp, lr 복원
	add	sp, sp, #48             ; ⑦ 프레임 해제
	ret                             ; ⑧ lr로 복귀
```

①~④ = **프롤로그**, ⑥~⑧ = **에필로그**. 재귀 호출마다 이 48바이트 프레임이 새로 쌓임 → `fact(5)`는 프레임 5개가 동시에 존재, 각각 자기 `n`을 가짐. **재귀의 지역변수가 독립인 이유가 이것 전부다.**

### fp 체인 = backtrace의 원리

각 프레임의 x29(fp)는 이전 프레임의 fp를 가리킴 → 연결 리스트. 디버거는 이 체인을 따라가며 backtrace 출력. `-fomit-frame-pointer`가 이걸 끊어서 프로파일링을 방해하는 이유.

## 최적화가 이 그림을 바꾼다

같은 코드를 `-O1`로 컴파일하면:

```asm
_fact:                              ; 프레임 없음!
	mov	x8, x0
	mov	w0, #1
	cmp	x8, #2
	b.lt	LBB1_2
LBB1_1:
	mul	x0, x0, x8              ; 재귀가 사라지고
	sub	x8, x8, #1              ; 그냥 루프가 됨
	cmp	x8, #2
	b.ge	LBB1_1
LBB1_2:
	ret
```

관찰 (전부 이 머신 실제 출력):

1. **재귀 → 루프 변환**: `bl`이 없다. 스택 사용 0, 오버플로우 불가능
2. **leaf 함수는 프레임 생략**: `add3` 같은 단순 함수는 `-O1`에서 프롤로그/에필로그 자체가 없음 (lr 안 건드리니까)
3. **인라인**: `caller(x) { return add3(x,10,20)*2; }`가 `lsl x8, x0, #1; add x0, x8, #60`으로 - 호출이 아예 사라지고 산술로 접힘

"스택 프레임은 개념 모델, 컴파일러는 의미만 지키면 뭐든 한다."

## 인자가 레지스터를 넘치면

인자 10개 함수의 `-O1` 실물:

```asm
_many_args:
	ldp	x9, x8, [sp]            ; 9, 10번째 인자는 스택에서 로드
	add	x10, x1, x0             ; 1~8번째는 x0~x7 그대로
	...
```

ARM64는 8개까지 레지스터, 초과분은 호출자가 스택에 놓고 부름. 인자 적게 유지하라는 조언의 하드웨어적 근거.

## 직접 해보기

```bash
gcc -O0 -S proc_demo.c -o O0.s   # 프레임 풀 버전

```

## 셀프 체크

> [!question]- 재귀 호출에서 각 호출의 지역변수가 서로 독립인 이유는 무엇인가?
> 함수를 호출할 때마다 스택에 새 프레임이 쌓이고, 그 프레임 안에 해당 호출만의 지역변수 슬롯(인자 n 등)이 별도로 존재하기 때문이다. `fact(5)`는 프레임 5개가 동시에 스택에 존재하며 각각 자기 `n`을 가진다. 프레임이 물리적으로 분리된 메모리 영역이라 서로 덮어쓰지 않는 것이 재귀가 되는 근본 이유다.

> [!question]- caller-saved와 callee-saved 레지스터의 차이, 그리고 컴파일러가 이를 어떻게 활용하나?
> caller-saved 레지스터에 둔 값은 함수를 호출하면 피호출자가 마음대로 덮어쓸 수 있으므로, 호출 후에도 살아야 하는 값이면 호출자가 백업 책임을 진다. callee-saved 레지스터는 피호출자가 쓰기 전에 저장하고 반환 전에 복원할 책임을 진다. 컴파일러는 호출을 넘어 오래 살아야 하는 값은 callee-saved에, 짧게 쓰고 버릴 값은 caller-saved에 배분해 백업 비용을 최소화한다.

> [!question]- ARM64의 `bl`/`ret`와 x86-64의 `call`/`ret`이 복귀 주소를 다루는 방식은 어떻게 다른가?
> ARM64는 `bl f`가 복귀 주소를 lr(x30) 레지스터에 넣고 점프하며 `ret`은 lr로 점프한다 - 메모리 접근이 0회다. x86-64는 `call f`가 복귀 주소를 스택에 push하고 점프하며 `ret`은 스택에서 pop한다. 다만 ARM64도 함수가 다른 함수를 호출하면 lr이 덮이므로, leaf가 아닌 함수는 프롤로그에서 lr을 스택에 저장(`stp x29, x30, ...`)한다.

> [!question]- 디버거의 backtrace는 어떤 자료구조를 따라가며 만들어지고, `-fomit-frame-pointer`가 이를 왜 방해하나?
> 각 프레임의 fp(ARM64의 x29)가 이전 프레임의 fp를 가리키는 연결 리스트(fp 체인)를 이룬다. 디버거는 이 체인을 한 칸씩 따라가며 각 프레임의 복귀 주소를 읽어 호출 스택을 복원한다. `-fomit-frame-pointer`는 fp를 일반 용도 레지스터로 풀어 이 체인을 끊어버리므로, 프레임 포인터에 의존하는 backtrace/프로파일링이 어려워진다.

## 연습문제

> [!example]- 문제: `fact` 재귀가 프레임당 48바이트를 쓰고, 이 머신의 스택 한계가 `ulimit -s` = 8176KB다. 프레임 크기만으로 재귀가 스택을 넘치기까지 가능한 대략적 호출 깊이를 계산하라.
> **풀이**
> 스택 한계 = 8176KB = 8176 × 1024 = 8,372,224바이트.
> 호출 깊이 ≈ 스택 한계 / 프레임 크기 = 8,372,224 / 48 ≈ 174,421.
> 즉 약 17만 단계 깊이의 재귀에서 스택 오버플로우(segfault)가 난다. 실제로는 main과 라이브러리 프레임, 정렬 등으로 한계가 조금 더 낮다. `-O1`로 재귀가 루프로 변환되면 `bl`이 사라져 프레임이 안 쌓이므로 이 한계 자체가 사라진다.

> [!example]- 문제: ARM64(인자 레지스터 8개)와 x86-64 System V(인자 레지스터 6개)에서 인자 10개짜리 함수를 호출할 때, 각각 몇 개의 인자가 스택을 거쳐 전달되는지 구하고 그 함의를 설명하라.
> **풀이**
> ARM64: 앞 8개(x0~x7)는 레지스터, 나머지 10 − 8 = 2개는 호출자가 스택에 놓고 부른다.
> x86-64 SysV: 앞 6개(rdi, rsi, rdx, rcx, r8, r9)는 레지스터, 나머지 10 − 6 = 4개는 스택.
> 스택 전달 인자는 메모리 store/load를 유발해 레지스터 전달보다 느리다. 그래서 인자가 레지스터 개수를 넘기면 호출 비용이 늘고, "인자 개수를 적게 유지하라"는 조언에는 이런 하드웨어적 근거가 있다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지. (1) 함수 호출이 프레임 push → 실행 → 프레임 pop으로 진행되며 왜 LIFO 스택과 완벽히 맞는가, 그리고 재귀의 지역변수 독립성이 여기서 나오는 이유. (2) 호출 규약(인자/반환값/복귀주소 위치, caller vs callee-saved)과 ARM64 lr 방식 대 x86 스택 push 방식의 차이. (3) fp 체인이 backtrace를 만드는 원리와, 최적화(재귀→루프, leaf 프레임 생략, 인라인)가 이 그림을 어떻게 바꾸나.
