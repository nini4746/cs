# 부동소수점 (IEEE 754)

## 한 줄 요약

부동소수점은 실수가 아니라 유한개의 이진 분수로 실수를 근사하는 체계다. "상대 정밀도 고정"이라는 설계를 이해하면 0.1 + 0.2 문제부터 결합법칙 실패까지 전부 필연이 된다.

## 왜 필요한가

- `0.1 + 0.2 == 0.3`이 거짓인 이유를 설명 못 하면 부동소수점을 모르는 것
- 실전 사고: 패트리엇 미사일 오차 누적 (1991, 0.1초의 이진 표현 오차가 100시간 누적 → 요격 실패), 금융 계산에서 float 쓰다 잔액 안 맞는 사고
- JS의 `Number`는 전부 double → 정수도 2⁵³ 넘으면 깨짐

## IEEE 754 구조

값 = (−1)ˢ × M × 2ᴱ. 비트를 세 필드로 분할:

```
[ sign | exponent | fraction ]
float  (32비트):  1 | 8  | 23
double (64비트):  1 | 11 | 52
```

### 지수는 bias 표기

exponent 필드는 2의 보수가 아니라 **bias를 더한 unsigned** (float 127, double 1023). E = exp − bias.

이유: 비트 패턴을 unsigned 정수로 비교해도 크기 순서가 맞도록 (정렬·비교 하드웨어 단순화).

### 값의 세 가지 종류 (exp 필드 값으로 구분)

| exp | 종류 | 값 |
|---|---|---|
| 1 ~ 최대−1 | **정규화 (normalized)** | (−1)ˢ × **1**.f × 2^(exp−bias). 맨 앞 1은 저장 안 함 (implicit leading 1 - 공짜 1비트) |
| 전부 0 | **비정규화 (denormalized)** | (−1)ˢ × **0**.f × 2^(1−bias). 0과 0 근처를 촘촘히 (gradual underflow) |
| 전부 1 | **특수값** | f=0이면 ±Inf, f≠0이면 NaN |

### 특수값 규칙

- `1.0/0.0 == Inf`, `-1.0/0.0 == -Inf` (예외 아님, 값임)
- `0.0/0.0`, `Inf - Inf`, `sqrt(-1)` → NaN
- **NaN은 자기 자신과도 다르다**: `NaN != NaN`이 참. `x != x`가 NaN 판별법인 이유 (그래서 `isnan()` 쓰라는 것)
- ±0 두 개 존재: `0.0 == -0.0`이지만 `1/0.0 != 1/-0.0`

## 핵심 직관: 수직선 위 밀도

표현 가능한 수는 균등하지 않다. **2ᴱ 구간마다 같은 개수(2⁵²개)의 수가 배치** → 큰 수일수록 간격이 넓다:

- [1, 2) 구간: 간격 2⁻⁵²
- [2⁵², 2⁵³) 구간: 간격 1
- **2⁵³부터는 간격 2 → 홀수 정수 표현 불가**. `2⁵³ + 1 == 2⁵³`이 참

JS `Number.MAX_SAFE_INTEGER == 2⁵³ − 1`인 이유가 정확히 이것.

절대 오차는 커지지만 **상대 오차는 일정** (~2⁻⁵², "machine epsilon"). 부동(floating)소수점이라는 이름의 의미: 소수점이 떠다니며 상대 정밀도를 유지.

## 왜 0.1은 표현 불가인가

이진 분수는 분모가 2의 거듭제곱인 수만 정확히 표현. 0.1 = 1/10, 분모에 5가 있음 → 이진수로 무한 순환:

```
0.1 = 0.0001100110011001100...₂ (0011 순환)
```

double은 52비트에서 잘라 반올림 → 실제 저장값은 0.1000000000000000055511151231257827... 0.2도 마찬가지. 둘을 더하면 오차가 합쳐져 0.3의 근사값과 다른 근사값이 나옴.

십진수에서 1/3 = 0.333...이 유한 표기 불가인 것과 같은 현상. 진법의 문제지 컴퓨터의 버그가 아님.

## 반올림과 연산 성질

- 기본 반올림: **round-to-nearest-even** (동점이면 짝수 쪽 - 통계적 편향 제거)
- 연산 정의: "무한 정밀도로 계산한 뒤 반올림" - 개별 연산은 최선을 다함
- 그런데 **결합법칙이 깨진다**: `(a + b) + c != a + (b + c)` 가능
  - 예: `(1e20 + -1e20) + 1 == 1`이지만 `1e20 + (-1e20 + 1) == 0` (1이 1e20에 흡수됨)
  - 컴파일러가 부동소수점 연산 순서를 함부로 못 바꾸는 이유. `-ffast-math`가 하는 일 = "재배열 허용, 정확성 포기"
- **큰 수 + 작은 수 = 작은 수 소실** (swamping), **비슷한 수 빼기 = 유효자리 붕괴** (catastrophic cancellation)

## 실전 규칙

1. **`==` 비교 금지** - 허용 오차(epsilon) 비교. 절대 오차보다 상대 오차 기준이 보통 맞음
2. **돈은 float 금지** - 정수 (센트/원 단위) 또는 decimal 타입
3. **합산 순서 신경** - 작은 수부터 더하기, 정밀도 중요하면 Kahan summation
4. **정수를 double에 넣을 거면 2⁵³ 한계 인지** - JS에서 ID를 Number로 받다 깨지는 사고 (Twitter ID가 string인 이유)
5. NaN 전파 주의 - 한 번 NaN이면 이후 연산 전부 NaN, 비교는 전부 거짓

## 코드로 확인

```c
#include <stdio.h>
#include <stdint.h>
#include <string.h>
#include <math.h>

int main(void) {
    // 1. 0.1 + 0.2의 실체 (%.17g = double 구분에 충분한 자릿수)
    printf("0.1 + 0.2        = %.17g\n", 0.1 + 0.2);
    printf("0.3              = %.17g\n", 0.3);
    printf("0.1+0.2 == 0.3   : %d\n", 0.1 + 0.2 == 0.3);

    // 2. 0.1의 비트 패턴 (sign 0 | exp 01111111011 | frac ...1010 반복)
    double d = 0.1;
    uint64_t bits;
    memcpy(&bits, &d, sizeof bits);
    printf("bits of 0.1      : %016llx\n", (unsigned long long)bits);

    // 3. 2^53 벽: 홀수 정수 표현 불가
    double big = 9007199254740992.0;  // 2^53
    printf("2^53 + 1 == 2^53 : %d\n", big + 1.0 == big);

    // 4. 결합법칙 실패
    printf("(1e20 + -1e20)+1 = %g\n", (1e20 + -1e20) + 1.0);
    printf("1e20 + (-1e20+1) = %g\n", 1e20 + (-1e20 + 1.0));

    // 5. NaN은 자기 자신과 다르다
    double nan = 0.0 / 0.0;
    printf("nan == nan       : %d\n", nan == nan);
    printf("nan != nan       : %d\n", nan != nan);

    // 6. ±0
    printf("0.0 == -0.0      : %d\n", 0.0 == -0.0);
    printf("1/0.0, 1/-0.0    : %g, %g\n", 1 / 0.0, 1 / -0.0);
    return 0;
}
```

```bash
gcc -O1 fp.c -o fp && ./fp
```

실측 (Apple M계열, clang):

```
0.1 + 0.2        = 0.30000000000000004
0.3              = 0.29999999999999999
0.1+0.2 == 0.3   : 0
bits of 0.1      : 3fb999999999999a
2^53 + 1 == 2^53 : 1
(1e20 + -1e20)+1 = 1
1e20 + (-1e20+1) = 0
nan == nan       : 0
nan != nan       : 1
0.0 == -0.0      : 1
1/0.0, 1/-0.0    : inf, -inf
```

비트 패턴 해독: `3fb999999999999a` → exp = 0x3fb = 1019 → E = 1019 − 1023 = −4, frac `99999...a` = 1001 1001... 반복 → 1.1001100110011...₂ × 2⁻⁴ = 0.0001100110011...₂ ✓ (위에서 유도한 순환과 일치, 마지막 `a`는 반올림 흔적)

## 셀프 체크

> [!question]- 0.1을 double로 정확히 표현할 수 없는 이유는?
> 이진 분수는 분모가 2의 거듭제곱인 수만 정확히 표현한다. 0.1 = 1/10이고 분모에 5가 있어 이진수로 무한 순환한다(0.0001100110011...₂, 0011 반복). double은 52비트에서 잘라 반올림하므로 실제 저장값은 0.1000000000000000055...가 된다. 십진수에서 1/3이 유한 표기 불가인 것과 같은 진법의 문제지 버그가 아니다.

> [!question]- `NaN != NaN`이 참인 이유와 이것의 실용적 쓸모는?
> IEEE 754가 NaN을 "자기 자신과도 같지 않다"로 정의했기 때문이다. 덕분에 `x != x`가 참이면 x가 NaN이라는 판별법이 성립한다(그래서 `isnan()`을 쓴다). 전부 1인 exponent에 f≠0이면 NaN이고, 한 번 NaN이 나오면 이후 연산이 전부 NaN으로 전파되며 비교는 모두 거짓이 된다.

> [!question]- 부동소수점 덧셈에서 결합법칙이 깨지는 이유는?
> 각 연산이 "무한 정밀도로 계산한 뒤 반올림"으로 정의되어, 큰 수에 작은 수를 더하면 작은 수가 흡수(swamping)되기 때문이다. 예: `(1e20 + -1e20) + 1 == 1`이지만 `1e20 + (-1e20 + 1) == 0`이다(1이 1e20에 흡수됨). 그래서 컴파일러가 순서를 함부로 못 바꾸고, `-ffast-math`가 "재배열 허용, 정확성 포기"를 한다.

## 연습문제

> [!example]- 문제: float 비트 패턴에서 값을 계산하라
> 32비트 float에서 sign=0, exponent 필드=10000001₂, fraction 필드=01000...0₂(맨 앞 비트만 1)일 때 값을 구하라. (float bias = 127)
> **풀이**
> exponent 필드 10000001₂ = 129. E = exp − bias = 129 − 127 = 2.
> 정규화 값이므로 가수 = 1.f = 1.01₂ = 1 + 0.25 = 1.25.
> 값 = (−1)⁰ × 1.25 × 2² = 1.25 × 4 = 5.0.

> [!example]- 문제: 2⁵³ 벽과 표현 간격을 설명하라
> double에서 `2^53 + 1 == 2^53`이 참인 이유를, [2⁵², 2⁵³)와 [2⁵³, 2⁵⁴) 구간의 표현 간격을 계산해 설명하라. 그리고 `Number.MAX_SAFE_INTEGER`가 2⁵³ − 1인 것과 연결하라.
> **풀이**
> double의 fraction은 52비트라 각 2ᴱ 구간에 2⁵²개의 수가 균등 배치된다.
> [2⁵², 2⁵³) 구간: 폭 2⁵², 간격 = 2⁵²/2⁵² = 1 → 모든 정수 표현 가능.
> [2⁵³, 2⁵⁴) 구간: 폭 2⁵³, 간격 = 2⁵³/2⁵² = 2 → 짝수만 표현, 홀수 불가.
> 따라서 2⁵³ 바로 위는 2⁵³ + 2가 다음 표현값이고 2⁵³ + 1은 2⁵³으로 반올림된다.
> 모든 정수를 정확히 담을 수 있는 최대 경계가 2⁵³ − 1이라 `MAX_SAFE_INTEGER = 2⁵³ − 1`이다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: (1) sign/exponent(bias)/fraction 세 필드로 값 = (−1)ˢ × 1.f × 2^(exp−bias)를 세우고 비트에서 값을 역산할 수 있는가. (2) "상대 정밀도 고정, 절대 간격은 2ᴱ마다 벌어짐"이라는 밀도 그림으로 2⁵³ 벽을 설명할 수 있는가. (3) 0.1 표현 불가·NaN 자기부등·결합법칙 실패가 모두 같은 설계에서 나오는 필연임을 연결할 수 있는가.
