# 행렬곱 블로킹 (Cache Blocking / Tiling)

## 한 줄 요약

행렬곱은 capacity miss의 교과서 사례다. 전체를 한 번에 훑으면 캐시에서 계속 밀려나지만, 캐시에 들어가는 작은 타일 단위로 쪼개 처리하면 같은 연산이 몇 배 빨라진다. 직접 측정하면 8배 이상.

## 왜 필요한가

- [[cache-misses]]의 capacity miss를 알고리즘으로 잡는 대표 기법
- 같은 O(n³) 연산인데 왜 구현에 따라 성능이 10배 나나
- BLAS/딥러닝 커널이 실제로 하는 것의 축소판

## 순진한 행렬곱의 문제

C = A × B, `C[i][j] = Σ A[i][k] × B[k][j]`:

```c
for (i) for (j) { double s=0;
    for (k) s += A[i][k] * B[k][j];   // B[k][j]: k가 증가하면 행을 건너뜀
    C[i][j] = s;
}
```

- `A[i][k]`: k 순차 → 행 우선, 캐시 친화적
- `B[k][j]`: k 순차인데 **열 접근** → 매번 한 행(N×8B)만큼 점프 → 캐시 라인 낭비 ([[memory-hierarchy]]의 열 우선 순회 문제)

N이 크면 B를 한 번 훑는 동안 앞서 캐시에 올린 B가 이미 밀려남(capacity miss). 다음 i에서 B를 처음부터 다시 - **B 전체를 N번 캐시 미스하며 재로드**.

## 블로킹: working set을 캐시에 맞추기

행렬을 작은 타일(예: 64×64)로 쪼개고, 타일끼리 곱함:

```c
for (ii; ii+=Bs) for (kk; kk+=Bs) for (jj; jj+=Bs)   // 타일 순회
    for (i=ii..ii+Bs) for (k=kk..kk+Bs) {
        double aik = A[i][k];
        for (j=jj..jj+Bs) C[i][j] += aik * B[k][j];   // 타일 내부는 행 접근
    }
```

핵심: **한 번 캐시에 올린 타일(64×64×8B = 32KB)을 밀려나기 전에 재사용을 다 끝냄**. B 타일을 여러 번 쓰는 동안 그게 캐시에 머묾 → capacity miss 급감.

타일 크기 선택: 세 타일(A,B,C)이 함께 캐시(L1/L2)에 들어갈 만큼. 너무 크면 안 들어가고, 너무 작으면 오버헤드.

## 실측

이 머신, N=1024 double, `-O2`:

```
naive ijk : 1.74 GFLOPS
blocked64 : 15.09 GFLOPS    ← 8.7배
```

연산 횟수(2N³)는 동일. 8.7배 전부가 캐시 재사용에서 나온다. 순진한 버전이 메모리 대기로 대부분 시간을 낭비한 것.

## 더 나아가면

이건 시작일 뿐. 실제 고성능 커널(OpenBLAS, MKL)은 여기에 더:

- **다층 블로킹**: L1/L2/L3 각 계층에 맞춘 중첩 타일
- **SIMD 벡터화**: 타일 내부를 벡터 명령으로 → [[simd]]
- **레지스터 블로킹**: 가장 안쪽을 레지스터에 유지
- **패킹**: 타일을 연속 메모리로 재배치해 접근 규칙화
- 멀티스레딩

이 조합으로 이론 피크(수백 GFLOPS)의 90%+ 달성. 순진한 1.74에서 갈 길이 멀지만 블로킹이 첫 관문.

## 일반 원리: cache blocking

행렬곱만이 아니라 **큰 데이터를 여러 번 훑는 모든 연산**에 적용:

- 이미지 컨볼루션, FFT, 데이터베이스 조인([[query-execution]]의 hash join 블로킹), 정렬(external sort)
- 규칙: "데이터를 한 번 올렸을 때 최대한 재사용하고 넘어가라"
- cache-oblivious 알고리즘은 타일 크기를 몰라도 재귀 분할로 자동으로 이 효과 (궁금한 것 참고)

## 셀프 체크

> [!question]- naive ijk 행렬곱이 왜 느린가?
> 안쪽 루프에서 `B[k][j]`를 k 순차로 접근하는데, B가 행 우선 저장이면 이는 열 접근이라 매번 한 행(N×8B)만큼 점프해 캐시 라인을 낭비한다. 게다가 N이 크면 B를 한 번 훑는 동안 앞서 올린 부분이 이미 밀려나(capacity miss), 다음 i마다 B 전체를 다시 로드한다.

> [!question]- 블로킹이 무엇을 캐시에 맞추는가?
> working set을 캐시 용량에 맞춘다. 행렬을 작은 타일로 쪼개, 한 번 캐시에 올린 타일을 밀려나기 전에 재사용을 다 끝낸다. B 타일을 여러 번 쓰는 동안 그것이 캐시에 머무르므로 capacity miss가 급감한다.

> [!question]- 타일 크기는 무슨 기준으로 고르나?
> A·B·C 세 타일이 함께 대상 캐시(L1/L2)에 동시에 들어갈 만큼 커야 하되, 그 안에 들어가는 최대 크기에 가깝게 잡는다. 너무 크면 캐시에 안 들어가 재사용 효과가 사라지고, 너무 작으면 루프 오버헤드가 커진다.

> [!question]- 실측 8.7배(1.74 → 15.09 GFLOPS)가 순전히 무엇에서 나오나?
> 연산 횟수(2N³)는 두 버전이 동일하다. 8.7배 전부가 캐시 재사용에서 온다. naive 버전은 메모리 대기(capacity miss로 인한 재로드)로 대부분의 시간을 낭비했고, 블로킹이 그 대기를 없앤 것이다.

> [!question]- cache blocking의 일반 원리는 무엇이며 어디에 적용되나?
> "데이터를 한 번 캐시에 올렸을 때 최대한 재사용하고 넘어가라"는 것이다. 행렬곱뿐 아니라 큰 데이터를 여러 번 훑는 모든 연산 - 이미지 컨볼루션, FFT, DB의 hash join, external sort 등 - 에 적용된다.

## 연습문제

> [!example]- 문제: L1 32KB에서 세 타일이 동시에 들어가는 최대 타일 크기 b를 계산하라
> **풀이**
> 조건: L1 = 32KB = 32768B, double = 8B, A·B·C 세 개의 b×b 타일이 동시에 L1에 상주해야 함.
> 세 타일이 차지하는 바이트 = 3 × b² × 8 = 24b².
> 조건: 24b² ≤ 32768 → b² ≤ 1365 → b ≤ 36.9.
> 따라서 안전한 최대 타일은 약 **36×36**. 실무에서는 여유와 정렬을 고려해 **32×32**를 자주 쓴다(3 × 32² × 8 = 24576B ≈ 24KB로 여유 확보). 노트의 64×64는 32KB로 L1엔 빠듯하고 보통 L2를 겨냥한 크기다.

> [!example]- 문제: N=1024에서 두 버전의 실행 시간을 GFLOPS로부터 계산하라
> **풀이**
> 총 부동소수점 연산 = 2N³ (곱셈·덧셈 각 N³). N=1024:
> 2 × 1024³ = 2 × 1,073,741,824 ≈ 2.147 × 10⁹ FLOP.
> naive (1.74 GFLOPS): 2.147e9 / 1.74e9 ≈ **1.23초**.
> blocked64 (15.09 GFLOPS): 2.147e9 / 15.09e9 ≈ **0.142초**.
> 속도비 = 1.23 / 0.142 ≈ **8.7배**. GFLOPS 비(15.09/1.74)와 정확히 일치한다 - 같은 연산량이므로 처리율 비가 곧 시간 절감비다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. naive 행렬곱에서 어느 접근(B의 열 접근)이 왜 capacity miss를 일으키는지 코드로 짚을 수 있는가.
> 2. 블로킹이 working set을 캐시에 맞춰 재사용을 끝내는 원리와, 타일 크기를 캐시 용량에서 유도하는 법을 설명할 수 있는가.
> 3. 8.7배가 연산량이 아닌 캐시 재사용에서 나온다는 점, 그리고 cache blocking이 행렬곱 밖의 어떤 연산에 일반화되는지 말할 수 있는가.
