# 캐시 일관성 (Cache Coherence)

## 한 줄 요약

각 코어가 자기 캐시를 가지면, 한 코어가 값을 바꿔도 다른 코어의 캐시엔 옛 값이 남는다. 이를 막는 하드웨어 프로토콜(MESI)이 있고, 그 부작용이 false sharing이다.

## 왜 필요한가

- [[memory-hierarchy]]의 false sharing이 왜 일어나는지의 근본 원인
- 멀티스레드 프로그램이 왜 코어를 늘려도 안 빨라지는 경우가 있나
- 원자적 연산(CAS)과 락이 하드웨어에서 어떻게 비싼지 → [[locks]]

## 문제: 캐시가 코어마다 있다

L1/L2는 코어 전용([[memory-hierarchy]]). 코어 A와 B가 같은 변수 x를 각자 캐시에 로드한 상태에서 A가 x를 바꾸면:

- A의 캐시: x = 새 값
- B의 캐시: x = 옛 값 ← 문제. B가 이걸 읽으면 틀림

**일관성(coherence)** = "어느 코어가 읽어도 최신 값을 본다"는 보장. 소프트웨어가 아니라 하드웨어가 책임진다.

## MESI 프로토콜

각 캐시 라인에 4상태 중 하나를 태그:

| 상태 | 의미 |
|---|---|
| **M** (Modified) | 이 캐시만 가짐 + 수정됨 (메모리와 다름). 유일한 최신본 |
| **E** (Exclusive) | 이 캐시만 가짐 + 메모리와 동일 |
| **S** (Shared) | 여러 캐시가 가짐 + 메모리와 동일 |
| **I** (Invalid) | 무효 (없는 것과 같음) |

핵심 규칙:

- 쓰려면 라인을 **M 상태로** 만들어야 함 → 다른 모든 캐시의 그 라인을 **I(무효화)** 시킴 (invalidate 메시지 브로드캐스트)
- 그래서 한 코어의 쓰기는 다른 코어들에게 "네 사본 버려" 신호를 보냄
- 다음에 그 코어가 읽으면 미스 → 최신본을 M 가진 캐시나 메모리에서 가져옴

코어들은 서로의 메모리 트래픽을 감시(**snooping**)하거나 디렉토리로 관리. 이 조율이 코어 간 통신 비용을 만든다.

## false sharing - MESI의 부작용

[[memory-hierarchy]]에서 측정한 현상의 원인이 이제 보인다.

두 스레드가 **다른 변수**를 쓰는데 그 변수들이 **같은 캐시 라인(64B)**에 있으면:

1. 코어 A가 변수1 씀 → 라인을 M으로, 코어 B의 라인 무효화
2. 코어 B가 변수2 씀 → 라인을 다시 뺏어옴(M), 코어 A 무효화
3. A가 또 씀 → 다시 뺏김... **라인이 코어 사이를 핑퐁**

논리적으로 두 스레드는 아무것도 공유하지 않는데, 물리적으로 같은 라인이라 MESI가 라인을 계속 주고받게 만듦. 각 쓰기가 코어 간 무효화 왕복(수십~수백 사이클)을 유발 → [[memory-hierarchy]]에서 padding으로 6배 빨라진 그 현상.

**해결**: 스레드별 데이터를 캐시 라인 단위로 분리 (`alignas(64)`, padding).

## 메모리 순서 (memory ordering) - 곁가지

일관성(한 주소의 최신값)과 별개로, **여러 주소에 대한 쓰기가 다른 코어에 어떤 순서로 보이나**는 또 다른 문제 (memory consistency).

- x86: 강한 순서(TSO) - 대부분 순서 유지
- ARM: 약한 순서 - 컴파일러/CPU가 재배열 가능 → 명시적 배리어(`dmb`) 필요
- 그래서 락프리 코드가 아키텍처마다 다르게 동작. Apple의 Rosetta가 x86 번역 시 겪는 난제 → [[isa-design]], [[lock-free-basics]]

## 실전 함의

- 원자적 연산(CAS)·락은 라인을 M으로 확보 + 무효화 → 경합 심하면 코어 간 트래픽 폭발 ([[locks]])
- 공유 카운터를 여러 스레드가 증가시키면 그 라인이 핑퐁 → 코어 늘려도 안 빨라짐. 스레드별 카운터 후 합산이 정석
- 코어 수 확장성의 숨은 벽이 대부분 여기

## 셀프 체크

> [!question]- 캐시 일관성 문제는 왜 생기나?
> L1/L2 캐시가 코어마다 전용으로 있기 때문이다. 코어 A와 B가 같은 변수 x를 각자 캐시에 로드한 상태에서 A가 x를 바꾸면, A의 캐시에는 새 값이지만 B의 캐시에는 옛 값이 남는다. B가 그걸 읽으면 틀린 값을 본다. "어느 코어가 읽어도 최신 값을 본다"는 보장(coherence)을 소프트웨어가 아니라 하드웨어가 책임진다.

> [!question]- MESI의 네 상태는 각각 무엇을 뜻하나?
> M(Modified): 이 캐시만 가지고 수정됨(메모리와 다름), 유일한 최신본. E(Exclusive): 이 캐시만 가지고 메모리와 동일. S(Shared): 여러 캐시가 가지고 메모리와 동일. I(Invalid): 무효(없는 것과 같음). 쓰려면 라인을 M으로 만들어야 하고, 그때 다른 모든 캐시의 그 라인을 I로 무효화한다.

> [!question]- false sharing이란 무엇이고 왜 성능을 떨어뜨리나?
> 두 스레드가 서로 다른 변수를 쓰는데 그 변수들이 같은 64B 캐시 라인에 있으면, 논리적으로는 공유하지 않아도 물리적으로 같은 라인이라 MESI가 라인을 계속 주고받게 만든다. 한 코어가 쓸 때마다 다른 코어의 라인을 무효화해서 라인이 코어 사이를 핑퐁하고, 매 쓰기가 코어 간 무효화 왕복(수십~수백 사이클)을 유발한다.

> [!question]- 일관성(coherence)과 메모리 순서(consistency)는 어떻게 다른가?
> 일관성은 "한 주소의 최신값을 모든 코어가 본다"는 보장이다. 메모리 순서는 "여러 주소에 대한 쓰기가 다른 코어에 어떤 순서로 보이나"라는 별개 문제다. x86은 강한 순서(TSO)로 대부분 순서를 유지하고, ARM은 약한 순서라 재배열이 가능해 명시적 배리어(`dmb`)가 필요하다.

## 연습문제

> [!example]- 문제: 64B 캐시 라인에 8바이트 `long` 배열 `a[8]`이 정확히 한 라인에 들어차 있다. 스레드 T0은 `a[0]`을, T1은 `a[7]`을 반복해서 쓴다. false sharing이 발생하는가? 발생한다면 왜인가?
> **풀이**
> 발생한다. `a[0]`과 `a[7]`은 서로 다른 변수(다른 메모리 위치)지만, 8개 × 8B = 64B로 배열 전체가 하나의 캐시 라인에 담긴다. MESI는 라인 단위로 상태를 관리하므로, T0이 `a[0]`을 쓰면 라인을 M으로 만들며 T1의 라인을 무효화하고, T1이 `a[7]`을 쓰면 다시 뺏어온다. 논리적 공유가 없어도 라인이 핑퐁한다.

> [!example]- 문제: 위 상황을 false sharing 없이 만들려면 T0/T1이 쓰는 데이터를 어떻게 배치해야 하나? 각 스레드 카운터를 64B 라인 크기에 맞춰 분리할 때 8바이트 카운터 하나에 필요한 padding 크기를 구하라.
> **풀이**
> 각 스레드 카운터를 서로 다른 캐시 라인에 놓으면 된다. 8B 카운터를 64B 라인 하나가 독점하게 하려면 8B(실데이터) + **56B padding** = 64B로 맞춘다(`alignas(64)` 구조체에 카운터 + 56바이트 패딩). 그러면 각 카운터가 자기 라인을 가져 한 코어의 쓰기가 다른 코어의 라인을 건드리지 않는다.

> [!example]- 문제: 코어 A만 변수 x를 캐시에 두고 아직 안 고친 상태다. 이후 (1) 코어 B가 x를 읽고, (2) 코어 A가 x를 쓴다. 각 시점에서 A와 B의 캐시 라인 MESI 상태를 추적하라.
> **풀이**
> 초기: A = E(자기만 가짐, 메모리와 동일), B = I.
> (1) B가 x를 읽음: 두 캐시가 공유하게 되어 A = S, B = S로 전이(메모리와 동일).
> (2) A가 x를 씀: A는 라인을 M으로 만들며 invalidate를 브로드캐스트, B의 라인은 I가 된다. 결과 A = M, B = I. 이후 B가 x를 읽으면 미스가 나 A의 M 사본에서 최신값을 가져온다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: (1) 코어별 전용 캐시 때문에 같은 주소의 사본이 어긋나는 것이 일관성 문제이고, 이를 하드웨어가 MESI로 해결한다. (2) 쓰기는 라인을 M으로 만들며 다른 캐시를 무효화한다는 핵심 규칙. (3) false sharing이 이 규칙의 부작용(라인 단위 관리 + 물리적 같은 라인)으로 생기고 padding으로 분리해 푼다.
