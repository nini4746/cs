# 스킵 리스트 (Skip List)

## 한 줄 요약

정렬된 링크드 리스트에 "고속도로" 층을 확률적으로 얹어 O(log n) 탐색을 낸다. 균형 트리와 성능은 비슷하지만 구현이 단순하고 락 없는 동시성에 유리해 Redis가 채택했다.

## 왜 필요한가

- 균형 트리([[balanced-trees]]) 없이 O(log n) 정렬 구조를 만드는 다른 길
- Redis sorted set이 왜 스킵 리스트인가
- 확률적 균형이라는 아이디어

## 문제: 정렬된 리스트는 탐색이 O(n)

정렬된 링크드 리스트([[linked-lists]])는 삽입 위치를 알면 O(1) 삽입이지만, **탐색이 O(n)** (순차 순회밖에). 이진 탐색을 못 함 (임의 접근 불가).

스킵 리스트: 리스트에 **여러 층의 지름길**을 추가해 이진 탐색처럼 건너뛰기.

## 구조: 확률적 다층 리스트

```
레벨3: HEAD ──────────────→ 30 ──────────→ NULL
레벨2: HEAD ──→ 10 ───────→ 30 ────→ 50 → NULL
레벨1: HEAD → 5→10→20→30→40→50→60 → NULL   (전체 리스트)
```

- 레벨 1 = 모든 노드 (완전한 정렬 리스트)
- 위 레벨 = 일부 노드만 (고속도로) - 노드를 **확률적으로 승격** (동전 던져 앞면이면 위 레벨에도, 1/2 확률)
- 탐색: 위 레벨에서 최대한 건너뛰다 목표 넘으면 한 층 내려감 → 이진 탐색처럼 절반씩

## 왜 O(log n)인가

- 각 노드가 레벨 L에 있을 확률 = 1/2^L → 레벨 L 노드 수 ≈ n/2^L
- 레벨 개수 ≈ log n (기댓값)
- 각 레벨에서 평균 상수 개 이동 → 총 O(log n) **기댓값**

균형 트리처럼 회전으로 강제하는 게 아니라 **확률적으로** 균형. 최악은 O(n)이지만 확률적으로 극히 드묾 (동전이 계속 한쪽 나올 확률).

## 연산

| 연산 | 방법 | 복잡도 |
|---|---|---|
| 탐색 | 위 레벨부터 건너뛰며 하강 | O(log n) 기댓값 |
| 삽입 | 위치 찾고, 레벨을 랜덤 결정해 삽입 | O(log n) |
| 삭제 | 각 레벨에서 노드 제거 | O(log n) |
| 범위 질의 | 시작 찾고 레벨1 순회 | O(log n + 결과) |

## 균형 트리 대비 장점

| | 스킵 리스트 | 균형 트리 |
|---|---|---|
| 성능 | O(log n) 기댓값 | O(log n) 보장 |
| 구현 | **단순** (회전 없음) | 복잡 (회전, 색) |
| 동시성 | **락프리 쉬움** | 어려움 |
| 메모리 | 포인터 여럿/노드 | 보통 2 포인터 |
| 캐시 | 나쁨 (링크드) | 나쁨 |

핵심 매력:
1. **구현 단순**: 회전/재색 로직 없음, 확률로 균형
2. **동시성 친화**: 노드별 부분 갱신이라 락프리/세밀한 락이 트리보다 쉬움 ([[lock-free-basics]])

## 어디에 쓰나

- **Redis sorted set (ZSET)**: 스킵 리스트 + 해시 조합. 범위 질의(순위, score 범위)에 스킵 리스트, O(1) 조회에 해시
- **LevelDB memtable**: 정렬 유지 인메모리 구조
- **동시성 맵**: Java ConcurrentSkipListMap (락프리 정렬 맵)
- **커널**: 일부 스케줄러/타이머

## Redis가 트리 대신 스킵 리스트를 쓴 이유

Redis 저자(antirez)의 설명: 균형 트리와 성능 비슷한데 **구현이 훨씬 단순**하고, 범위 질의가 자연스럽고(레벨1 순회), ZSET의 요구(순위, score 범위)에 잘 맞아서. 복잡한 red-black tree를 피한 실용적 선택.

## 셀프 체크

> [!question]- 정렬된 링크드 리스트의 무엇이 문제이고 스킵 리스트는 이를 어떻게 푸나?
> 정렬된 링크드 리스트는 임의 접근이 안 돼 이진 탐색을 못 하므로 탐색이 O(n)(순차 순회)이다. 스킵 리스트는 리스트 위에 여러 층의 지름길(고속도로)을 얹어, 위 레벨에서 최대한 건너뛰다 목표를 넘으면 한 층 내려가는 식으로 이진 탐색처럼 절반씩 좁혀 O(log n)을 낸다.

> [!question]- 스킵 리스트가 O(log n)인 이유는 (균형 트리와 다른 점 포함)?
> 노드가 레벨 L에 있을 확률이 1/2^L라 레벨 L 노드 수 ≈ n/2^L, 레벨 개수 ≈ log n이고 각 레벨에서 평균 상수 개만 이동해 총 O(log n) 기댓값이다. 회전으로 균형을 강제하는 게 아니라 동전 던지기(승격)로 확률적으로 균형을 이룬다. 최악은 O(n)이지만 확률적으로 극히 드물다.

> [!question]- 균형 트리 대비 스킵 리스트의 두 가지 핵심 매력은?
> 첫째, 회전이나 재색 로직이 없고 확률로 균형을 이뤄 구현이 단순하다. 둘째, 노드별 부분 갱신이라 락프리/세밀한 락이 트리보다 쉬워 동시성에 친화적이다. 대신 성능은 보장이 아니라 기댓값이고 캐시는 링크드라 나쁘다.

> [!question]- Redis가 sorted set에 red-black tree 대신 스킵 리스트를 택한 이유는?
> 저자(antirez)에 따르면 균형 트리와 성능이 비슷한데 구현이 훨씬 단순하고, 범위 질의가 레벨1 순회로 자연스러우며, ZSET의 요구(순위, score 범위)에 잘 맞아서다. 복잡한 red-black tree를 피한 실용적 선택이다. Redis ZSET은 스킵 리스트(범위 질의)와 해시(O(1) 조회)를 함께 쓴다.

## 연습문제

> [!example]- 문제: 스킵 리스트의 탐색(search)과 삽입 시 레벨 결정 로직을 직접 구현(의사코드)하라.
> **풀이**
> - search(key): cur = head. 최상위 레벨부터 아래로, 각 레벨에서 cur.next[L]의 값이 key보다 작으면 오른쪽으로 전진, 아니면 한 레벨 하강. 레벨1에서 멈춘 다음 노드가 후보.
> - randomLevel(): lvl = 1; while(rand() < p && lvl < maxLevel) lvl++; return lvl. (p=1/2면 동전 던지기)
> - insert(key): 각 레벨에서 삽입 위치 직전 노드(update[L])를 탐색 중 기록 → randomLevel로 새 노드 레벨 결정 → 그 레벨 수만큼 update[L].next와 새 노드를 이어붙임.
> - 회전이 없어 트리보다 삽입 로직이 단순한 점이 드러난다.

> [!example]- 문제: 승격 확률 p=1/2일 때 탐색이 O(log n) 기댓값임을 노드 수·레벨 수로 분석하라.
> **풀이**
> - 노드가 레벨 L 이상에 있을 확률 = p^(L-1) = (1/2)^(L-1). 레벨 L의 기대 노드 수 ≈ n·(1/2)^(L-1).
> - 최상위 레벨(노드 1개 정도)까지 레벨 수 ≈ log_{1/p} n = log2 n (기댓값).
> - 각 레벨에서 오른쪽 이동 횟수의 기댓값은 상수(약 1/p = 2). 상승은 없고 레벨당 상수 이동.
> - 총 비용 = (레벨 수) × (레벨당 상수) = O(log n) 기댓값. 최악은 모든 승격이 몰릴 때 O(n)이나 확률이 지수적으로 작다.

> [!example]- 문제: 순위·score 범위 질의가 잦은 정렬 셋(예: 리더보드)에 균형 트리 대신 스킵 리스트를 택하는 근거를 대라.
> **풀이**
> - 범위 질의: 스킵 리스트는 시작점을 O(log n)에 찾고 레벨1을 순차 순회하면 되어 자연스럽다(O(log n + 결과 수)). 트리는 중위 순회에 상태 관리가 더 번거롭다.
> - 구현: 회전/재색이 없어 삽입/삭제 코드가 단순 → 버그·유지보수 비용이 낮다.
> - 동시성: 노드별 부분 갱신이라 락프리/세밀한 락이 트리보다 쉬워 다중 쓰기 리더보드에 유리.
> - 대가: 성능이 보장이 아니라 기댓값이고 노드당 포인터가 여럿이라 메모리가 더 든다. 리더보드처럼 범위·순위가 핵심이고 확률적 보장으로 충분하면 이 트레이드오프가 맞다(Redis ZSET의 선택).

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 정렬 링크드 리스트의 O(n) 탐색 문제를 다층 지름길이 어떻게 이진 탐색처럼 O(log n)으로 바꾸는지.
> 2. 레벨 확률 1/2^L과 레벨 수 ≈ log n으로부터 O(log n) 기댓값이 나오는 논리, 그리고 최악 O(n)이 왜 드문지.
> 3. 회전 없는 확률적 균형이 주는 두 실익(구현 단순, 락프리 동시성)과 그 대가(기댓값 보장, 나쁜 캐시).
