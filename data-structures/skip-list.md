# 스킵 리스트 (Skip List)

## 한 줄 요약

정렬된 링크드 리스트에 "고속도로" 층을 확률적으로 얹어 O(log n) 탐색을 낸다. 균형 트리와 성능은 비슷하지만 구현이 단순하고 락 없는 동시성에 유리해 Redis가 채택했다.

## 왜 필요한가

- 균형 트리([[balanced-trees]]) 없이 O(log n) 정렬 구조를 만드는 다른 길
- Redis sorted set이 왜 스킵 리스트인가
- 확률적 균형이라는 아이디어

## 문제: 정렬된 리스트는 탐색이 O(n)

정렬된 링크드 리스트([[linked-lists]])는 삽입 위치를 알면 O(1) 삽입이지만, **탐색이 O(n)** (순차 순회밖에). 이진 탐색을 못 함 (임의 접근 불가).

스킵 리스트: 리스트에 **여러 층의 지름길**을 추가해 이진 탐색처럼 건너뛰기.

## 구조: 확률적 다층 리스트

```
레벨3: HEAD ──────────────→ 30 ──────────→ NULL
레벨2: HEAD ──→ 10 ───────→ 30 ────→ 50 → NULL
레벨1: HEAD → 5→10→20→30→40→50→60 → NULL   (전체 리스트)
```

- 레벨 1 = 모든 노드 (완전한 정렬 리스트)
- 위 레벨 = 일부 노드만 (고속도로) - 노드를 **확률적으로 승격** (동전 던져 앞면이면 위 레벨에도, 1/2 확률)
- 탐색: 위 레벨에서 최대한 건너뛰다 목표 넘으면 한 층 내려감 → 이진 탐색처럼 절반씩

## 왜 O(log n)인가

- 각 노드가 레벨 L에 있을 확률 = 1/2^L → 레벨 L 노드 수 ≈ n/2^L
- 레벨 개수 ≈ log n (기댓값)
- 각 레벨에서 평균 상수 개 이동 → 총 O(log n) **기댓값**

균형 트리처럼 회전으로 강제하는 게 아니라 **확률적으로** 균형. 최악은 O(n)이지만 확률적으로 극히 드묾 (동전이 계속 한쪽 나올 확률).

## 연산

| 연산 | 방법 | 복잡도 |
|---|---|---|
| 탐색 | 위 레벨부터 건너뛰며 하강 | O(log n) 기댓값 |
| 삽입 | 위치 찾고, 레벨을 랜덤 결정해 삽입 | O(log n) |
| 삭제 | 각 레벨에서 노드 제거 | O(log n) |
| 범위 질의 | 시작 찾고 레벨1 순회 | O(log n + 결과) |

## 균형 트리 대비 장점

| | 스킵 리스트 | 균형 트리 |
|---|---|---|
| 성능 | O(log n) 기댓값 | O(log n) 보장 |
| 구현 | **단순** (회전 없음) | 복잡 (회전, 색) |
| 동시성 | **락프리 쉬움** | 어려움 |
| 메모리 | 포인터 여럿/노드 | 보통 2 포인터 |
| 캐시 | 나쁨 (링크드) | 나쁨 |

핵심 매력:
1. **구현 단순**: 회전/재색 로직 없음, 확률로 균형
2. **동시성 친화**: 노드별 부분 갱신이라 락프리/세밀한 락이 트리보다 쉬움 ([[lock-free-basics]])

## 어디에 쓰나

- **Redis sorted set (ZSET)**: 스킵 리스트 + 해시 조합. 범위 질의(순위, score 범위)에 스킵 리스트, O(1) 조회에 해시
- **LevelDB memtable**: 정렬 유지 인메모리 구조
- **동시성 맵**: Java ConcurrentSkipListMap (락프리 정렬 맵)
- **커널**: 일부 스케줄러/타이머

## Redis가 트리 대신 스킵 리스트를 쓴 이유

Redis 저자(antirez)의 설명: 균형 트리와 성능 비슷한데 **구현이 훨씬 단순**하고, 범위 질의가 자연스럽고(레벨1 순회), ZSET의 요구(순위, score 범위)에 잘 맞아서. 복잡한 red-black tree를 피한 실용적 선택.
