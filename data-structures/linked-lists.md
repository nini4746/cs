# 링크드 리스트 (Linked Lists)

## 한 줄 요약

각 노드가 다음 노드의 포인터를 담는 구조. 이론상 O(1) 삽입/삭제가 강점이지만, 실전에선 pointer chasing으로 캐시를 못 써서 배열보다 200배 느릴 수 있다. 교과서의 총아지만 실무에선 드물다.

## 왜 필요한가

- 왜 교과서는 링크드 리스트를 강조하는데 실무는 배열을 쓰나
- Big-O가 같아도 성능이 왜 극단적으로 다른가
- 언제 링크드 리스트가 진짜 답인가

## 구조

```
head → [val|next] → [val|next] → [val|next] → NULL
```

- 각 노드 = 값 + 다음 포인터 (이중 연결은 prev도)
- 연속 메모리 아님 - 노드가 힙 여기저기 흩어짐

## 이론적 강점

| 연산 | 링크드 리스트 | 배열 |
|---|---|---|
| 인덱스 접근 `a[i]` | **O(n)** (순회) | O(1) |
| 맨 앞 삽입/삭제 | **O(1)** | O(n) (밀기) |
| 노드 위치 알 때 삽입/삭제 | **O(1)** (포인터만 조정) | O(n) |
| 순회 | O(n) | O(n) |
| 크기 조정 | 자동 (재할당 없음) | 재할당 필요 |

이론상: 중간 삽입/삭제가 잦고 위치를 이미 알면 O(1). 배열은 밀어야 해서 O(n). 그래서 교과서가 강조.

## 실전의 배신: pointer chasing

같은 O(n) 순회인데 실측 (1000만 노드, 랜덤 힙 배치):

```
array  sum: 0.004s
list   sum: 0.953s  (217.8x slower)
```

**217배 느림.** 이유:

- 배열: 연속 → 순차 접근 → 캐시 라인·프리페처 최대 활용 ([[memory-hierarchy]])
- 리스트: 다음 노드 주소가 현재 노드를 읽어야 나옴 → **pointer chasing** → 프리페처 무력, 매 노드가 캐시 미스 → RAM 지연시간(~100ns) 그대로 ([[cache-misses]]의 그 곡선)

Big-O는 같은데 상수가 200배. **캐시가 Big-O를 이기는 대표 사례.**

(주의: 노드를 순차로 할당하면 우연히 인접해 빨라 보임. 실제 앱은 노드가 시간에 걸쳐 흩어지므로 셔플한 위 결과가 현실적.)

## 그래서 실무는 배열

- 인덱스 접근·순회가 대부분 → 배열/동적 배열([[arrays-and-dynamic-arrays]])이 압도
- "중간 삽입 O(1)"도 실은 그 위치를 **찾는** 데 O(n)이 들면 의미 없음
- 삽입/삭제가 잦아도 배열 밀기(memmove)가 캐시 덕에 종종 더 빠름
- Rust는 아예 `LinkedList`를 "대부분 Vec을 쓰라"고 문서에 경고

## 링크드 리스트가 진짜 답인 경우

드물지만 존재:

1. **위치를 이미 쥐고 있고** 삽입/삭제가 빈번 (LRU 캐시의 노드 이동 - 해시가 노드 포인터를 줌)
2. **노드를 옮기면 안 될 때** (다른 곳에서 노드를 직접 가리킴 - intrusive list, 커널)
3. **splice**: 두 리스트 잇기 O(1)
4. 큰 원소라 이동 비용이 크고 안정적 주소가 필요할 때

리눅스 커널이 intrusive linked list를 많이 쓰는 이유 = 노드가 여러 리스트에 동시에 속하고 위치를 항상 쥐고 있음.

## 변형

- **단일 연결**: next만. 앞으로만
- **이중 연결**: prev+next. 양방향, 삭제 시 이전 노드 O(1) 접근
- **원형**: 꼬리가 머리를 가리킴
- **intrusive**: 노드 포인터를 데이터 구조체 안에 박음 (별도 할당 회피)

## 셀프 체크

> [!question]- 같은 O(n) 순회인데 링크드 리스트가 배열보다 200배 느릴 수 있는 이유는?
> pointer chasing 때문이다. 다음 노드 주소가 현재 노드를 읽어야 나오므로 프리페처가 무력화되고 매 노드가 캐시 미스가 되어 RAM 지연시간(~100ns)을 그대로 문다. 배열은 연속 메모리라 순차 접근으로 캐시 라인과 프리페처를 최대 활용한다. Big-O는 같아도 상수가 200배 벌어지는, 캐시가 Big-O를 이기는 대표 사례다.

> [!question]- "중간 삽입 O(1)"이라는 이론적 강점이 실무에서 종종 무의미한 이유는?
> 삽입/삭제가 O(1)인 건 그 위치를 이미 쥐고 있을 때뿐이고, 위치를 찾는 데 O(n) 순회가 들면 의미가 없기 때문이다. 게다가 삽입/삭제가 잦아도 배열 밀기(memmove)가 캐시 덕에 종종 더 빠르다. 그래서 인덱스 접근·순회가 대부분인 실무는 배열을 쓴다.

> [!question]- 링크드 리스트가 진짜 답인 경우는?
> 위치를 이미 쥐고 있고 삽입/삭제가 빈번할 때(LRU 캐시의 노드 이동, 해시가 노드 포인터를 줌), 노드를 옮기면 안 될 때(다른 곳에서 노드를 직접 가리키는 intrusive list, 커널), 두 리스트를 O(1)에 잇는 splice가 필요할 때다. 리눅스 커널이 intrusive list를 많이 쓰는 건 노드가 여러 리스트에 동시에 속하고 위치를 항상 쥐고 있어서다.

> [!question]- 벤치마크에서 노드를 순차로 할당하면 결과가 왜곡되는 이유는?
> 순차로 할당하면 노드들이 우연히 메모리상 인접해 캐시가 잘 들어 빨라 보이기 때문이다. 실제 앱은 노드가 시간에 걸쳐 힙 여기저기 흩어지므로, 셔플한(랜덤 힙 배치) 측정이 현실적이다.

## 연습문제

> [!example]- 문제: 단일 연결 리스트를 제자리(in-place)에서 뒤집는 함수를 직접 구현하고 복잡도를 분석하라.
> **풀이**
> - 포인터 3개(prev=NULL, cur=head, next)로 순회.
> - 각 스텝: next = cur.next; cur.next = prev; prev = cur; cur = next. 끝나면 prev가 새 head.
> - 시간 O(n) 한 번 순회, 추가 공간 O(1)(노드 새로 안 만듦).
> - 재귀 버전은 코드가 짧지만 호출 스택 O(n)을 써서 반복 버전이 공간상 우월하다.

> [!example]- 문제: 같은 원소 순회인데 배열과 링크드 리스트의 실측 성능 차이를 캐시 관점에서 복잡도·상수로 분석하라.
> **풀이**
> - 둘 다 순회는 O(n)으로 점근 복잡도는 동일.
> - 배열: 연속 메모리라 캐시 라인 하나에 여러 원소가 실리고 프리페처가 다음 라인을 미리 당김 → 대부분 L1/L2 히트, 원소당 몇 ns.
> - 리스트(힙에 흩어진 경우): 다음 주소를 알려면 현재 노드를 먼저 읽어야 함(pointer chasing) → 프리페처 무력, 노드마다 캐시 미스 → 원소당 ~100ns.
> - 결론: Big-O가 같아도 캐시 미스 비율이 상수를 200배까지 벌린다. 성능 비교는 점근식만이 아니라 메모리 접근 패턴까지 봐야 한다.

> [!example]- 문제: LRU 캐시를 설계하라. 왜 이중 연결 리스트 + 해시맵 조합이 배열보다 적합한지 근거를 대라.
> **풀이**
> - 요구: get/put 모두 O(1), 가장 오래 안 쓴 항목을 O(1)에 제거.
> - 구조: 해시맵(key → 노드 포인터) + 이중 연결 리스트(사용 순서, head=최신 tail=오래됨).
> - get: 해시맵으로 노드 O(1) 접근 → 그 노드를 리스트에서 떼어(prev/next O(1)) head로 이동.
> - put: 새 노드 head 삽입, 초과 시 tail 제거. 해시맵이 노드 위치를 쥐어주므로 "위치 탐색 O(n)" 문제가 사라진다.
> - 이 사례가 링크드 리스트의 진짜 강점(위치를 이미 쥐고 있을 때 O(1) 이동, splice)을 정확히 활용한다. 배열이면 중간 이동에 memmove O(n)이 든다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. Big-O가 같은 순회인데 링크드 리스트가 배열보다 수백 배 느린 이유(pointer chasing과 캐시 미스).
> 2. "중간 삽입 O(1)"이 실무에서 종종 무의미한 이유(위치 탐색 O(n), memmove가 더 빠름).
> 3. 링크드 리스트가 진짜 답인 세 경우(위치를 쥔 잦은 삽입/삭제, 노드 이동 금지 intrusive, O(1) splice).
