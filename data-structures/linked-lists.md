# 링크드 리스트 (Linked Lists)

## 한 줄 요약

각 노드가 다음 노드의 포인터를 담는 구조. 이론상 O(1) 삽입/삭제가 강점이지만, 실전에선 pointer chasing으로 캐시를 못 써서 배열보다 200배 느릴 수 있다. 교과서의 총아지만 실무에선 드물다.

## 왜 필요한가

- 왜 교과서는 링크드 리스트를 강조하는데 실무는 배열을 쓰나
- Big-O가 같아도 성능이 왜 극단적으로 다른가
- 언제 링크드 리스트가 진짜 답인가

## 구조

```
head → [val|next] → [val|next] → [val|next] → NULL
```

- 각 노드 = 값 + 다음 포인터 (이중 연결은 prev도)
- 연속 메모리 아님 - 노드가 힙 여기저기 흩어짐

## 이론적 강점

| 연산 | 링크드 리스트 | 배열 |
|---|---|---|
| 인덱스 접근 `a[i]` | **O(n)** (순회) | O(1) |
| 맨 앞 삽입/삭제 | **O(1)** | O(n) (밀기) |
| 노드 위치 알 때 삽입/삭제 | **O(1)** (포인터만 조정) | O(n) |
| 순회 | O(n) | O(n) |
| 크기 조정 | 자동 (재할당 없음) | 재할당 필요 |

이론상: 중간 삽입/삭제가 잦고 위치를 이미 알면 O(1). 배열은 밀어야 해서 O(n). 그래서 교과서가 강조.

## 실전의 배신: pointer chasing

같은 O(n) 순회인데 실측 (1000만 노드, 랜덤 힙 배치):

```
array  sum: 0.004s
list   sum: 0.953s  (217.8x slower)
```

**217배 느림.** 이유:

- 배열: 연속 → 순차 접근 → 캐시 라인·프리페처 최대 활용 ([[memory-hierarchy]])
- 리스트: 다음 노드 주소가 현재 노드를 읽어야 나옴 → **pointer chasing** → 프리페처 무력, 매 노드가 캐시 미스 → RAM 지연시간(~100ns) 그대로 ([[cache-misses]]의 그 곡선)

Big-O는 같은데 상수가 200배. **캐시가 Big-O를 이기는 대표 사례.**

(주의: 노드를 순차로 할당하면 우연히 인접해 빨라 보임. 실제 앱은 노드가 시간에 걸쳐 흩어지므로 셔플한 위 결과가 현실적.)

## 그래서 실무는 배열

- 인덱스 접근·순회가 대부분 → 배열/동적 배열([[arrays-and-dynamic-arrays]])이 압도
- "중간 삽입 O(1)"도 실은 그 위치를 **찾는** 데 O(n)이 들면 의미 없음
- 삽입/삭제가 잦아도 배열 밀기(memmove)가 캐시 덕에 종종 더 빠름
- Rust는 아예 `LinkedList`를 "대부분 Vec을 쓰라"고 문서에 경고

## 링크드 리스트가 진짜 답인 경우

드물지만 존재:

1. **위치를 이미 쥐고 있고** 삽입/삭제가 빈번 (LRU 캐시의 노드 이동 - 해시가 노드 포인터를 줌)
2. **노드를 옮기면 안 될 때** (다른 곳에서 노드를 직접 가리킴 - intrusive list, 커널)
3. **splice**: 두 리스트 잇기 O(1)
4. 큰 원소라 이동 비용이 크고 안정적 주소가 필요할 때

리눅스 커널이 intrusive linked list를 많이 쓰는 이유 = 노드가 여러 리스트에 동시에 속하고 위치를 항상 쥐고 있음.

## 변형

- **단일 연결**: next만. 앞으로만
- **이중 연결**: prev+next. 양방향, 삭제 시 이전 노드 O(1) 접근
- **원형**: 꼬리가 머리를 가리킴
- **intrusive**: 노드 포인터를 데이터 구조체 안에 박음 (별도 할당 회피)
