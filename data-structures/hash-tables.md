# 해시 테이블 (Hash Tables)

## 한 줄 요약

키를 해시 함수로 배열 인덱스로 바꿔 평균 O(1) 삽입/조회를 낸다. 충돌은 불가피하며 체이닝이나 오픈 어드레싱으로 해결하고, 로드 팩터가 오르면 성능이 나빠져 리사이징으로 관리한다.

## 왜 필요한가

- 딕셔너리/맵/셋의 내부 - 가장 많이 쓰는 O(1) 구조
- 왜 "평균 O(1)"이고 최악은 O(n)인가
- 로드 팩터가 뭐고 왜 리사이징하나

## 기본 아이디어

키 → 해시 함수 → 배열 인덱스 → 그 자리에 저장:

```
index = hash(key) % table_size
table[index] = value
```

배열 인덱싱이 O(1)이니 해시가 좋으면 조회도 O(1). 문제는 **두 키가 같은 인덱스로 가는 충돌**.

## 해시 함수

좋은 해시 함수의 조건:

- **균등 분포**: 키를 버킷에 고루 (몰리면 성능 저하)
- **결정적**: 같은 키 → 같은 해시
- **빠름**: 매 연산마다 계산
- 예: FNV, MurmurHash, SipHash(공격 저항 → [[hash-in-practice]])

## 충돌 해결 1: 체이닝

각 버킷이 **연결 리스트**(또는 다른 구조)를 가짐. 충돌하면 그 리스트에 추가:

```
버킷[3] → [keyA|v] → [keyD|v]    ← 둘 다 hash%size==3
```

- 조회: 버킷 찾고 → 리스트 순회하며 키 비교
- 평균 체인 길이 = 로드 팩터 → 조회 O(1 + 로드팩터)

## 로드 팩터 - 실측

**로드 팩터 α = 원소 수 / 버킷 수.** 평균 체인 길이를 결정. 100개 키를 다른 버킷 수에:

```
buckets= 16  load=6.25  max_chain=9   ← 붐빔
buckets= 64  load=1.56  max_chain=4
buckets=256  load=0.39  max_chain=2   ← 여유
```

버킷을 늘려(로드 팩터 낮춤) 최대 체인이 9→2로. 로드 팩터가 낮을수록 충돌이 적어 조회가 빠름. 대신 메모리 낭비. **트레이드오프의 핵심 다이얼.**

## 리사이징 (rehashing)

로드 팩터가 임계값(보통 0.7~0.75) 넘으면 **테이블을 2배로 키우고 전부 재해시**:

- 동적 배열([[arrays-and-dynamic-arrays]])의 2배 성장과 같은 amortized O(1) 논리
- 재해시는 O(n)이지만 드물게 → 삽입당 amortized O(1)
- 이때 모든 키의 인덱스가 바뀜 (`% 새크기`)

이것이 삽입이 보통 O(1)이지만 가끔 O(n)인 이유 (리사이징 순간).

## 충돌 해결 2: 오픈 어드레싱

리스트 없이 **빈 버킷을 찾아** 저장 (탐사):

- **선형 탐사**: 충돌 시 다음 칸, 다음 칸... 순차. 캐시 친화적이지만 clustering(뭉침)
- **이차 탐사**: 1, 4, 9칸 뛰며 탐사
- **더블 해싱**: 두 번째 해시로 간격 결정

- 장점: 포인터 없음, 연속 메모리 → **캐시 친화적** ([[memory-hierarchy]]). 체이닝보다 빠를 수 있음
- 단점: 로드 팩터 한계 낮음(꽉 차면 못 넣음, ~0.7에서 급격히 나빠짐), 삭제가 까다로움(tombstone)
- 현대 고성능 맵(Google SwissTable, Rust HashMap)은 오픈 어드레싱 + SIMD

## 복잡도 정리

| | 평균 | 최악 |
|---|---|---|
| 조회/삽입/삭제 | O(1) | O(n) |

**최악 O(n)**: 모든 키가 한 버킷에 몰릴 때 (나쁜 해시 or 악의적 입력 → HashDoS, [[hash-in-practice]]). 그래서 "평균 O(1)"이라 하지 보장이 아님.

## 해시 vs 균형 트리

| | 해시 테이블 | 균형 트리 (map) |
|---|---|---|
| 조회 | O(1) 평균 | O(log n) |
| 순서 | 없음 | 정렬 유지 |
| 범위 질의 | 불가 | 가능 |
| 최악 | O(n) | O(log n) 보장 |

순서·범위가 필요하면 트리([[balanced-trees]]), 단순 조회면 해시.

## 셀프 체크

> [!question]- 해시 테이블이 "평균 O(1)"이면서 최악 O(n)인 이유는?
> 좋은 해시가 키를 버킷에 고루 분산하면 배열 인덱싱이 O(1)이라 조회도 평균 O(1)이다. 하지만 나쁜 해시나 악의적 입력(HashDoS)으로 모든 키가 한 버킷에 몰리면 하나의 긴 체인이 되어 조회가 O(n)이 된다. 그래서 보장이 아니라 평균이라 한다.

> [!question]- 로드 팩터가 무엇이고 왜 성능의 핵심 다이얼인가?
> 로드 팩터 α = 원소 수 / 버킷 수로, 평균 체인 길이를 결정한다. 낮을수록 충돌이 적어 조회가 빠르지만(예: 버킷을 늘리면 최대 체인이 9→2로) 메모리를 낭비한다. 속도와 메모리의 트레이드오프를 조절하는 값이다.

> [!question]- 리사이징(rehashing)이 왜 필요하고 삽입 복잡도에 어떤 영향을 주나?
> 로드 팩터가 임계값(보통 0.7~0.75)을 넘으면 테이블을 2배로 키우고 전부 재해시해 충돌을 낮춘다. 재해시 자체는 O(n)이지만 동적 배열의 2배 성장처럼 드물게 일어나 삽입당 amortized O(1)이다. 이것이 삽입이 보통 O(1)이면서 가끔 O(n)인 이유다.

> [!question]- 체이닝과 오픈 어드레싱의 트레이드오프는?
> 체이닝은 각 버킷이 리스트를 가져 로드 팩터 한계가 높지만 포인터 추적으로 캐시가 나쁘다. 오픈 어드레싱은 빈 버킷을 탐사해 저장하므로 포인터 없이 연속 메모리라 캐시 친화적이지만, 로드 팩터가 ~0.7 넘으면 급격히 나빠지고 삭제가 까다롭다(tombstone). 현대 고성능 맵은 오픈 어드레싱 + SIMD를 쓴다.

## 연습문제

> [!example]- 문제: 체이닝 기반 해시 테이블을 리사이징 포함해 직접 구현하고, 삽입이 amortized O(1)임을 논증하라.
> **풀이**
> 버킷 배열(각 원소는 연결 리스트)과 원소 수 카운터를 둔다.
> - put(k,v): `i = hash(k) % size`, 버킷[i] 리스트에서 k 찾으면 갱신, 없으면 추가하고 count++. 이후 `count/size > 0.75`면 resize.
> - resize(): 배열을 2배로 만들고 모든 원소를 `% 새 크기`로 재삽입.
> amortized 논증: n번 삽입 동안 리사이징은 크기 1,2,4,...,n에서만 발생하고 재해시 비용 합은 1+2+4+...+n < 2n = O(n). 이를 n번 삽입에 분할 상환하면 삽입당 O(1). 동적 배열의 2배 성장과 동일한 논리다([[arrays-and-dynamic-arrays]]).

> [!example]- 문제: 로드 팩터 임계값을 0.75가 아니라 0.5 또는 0.95로 잡으면 시간/공간이 어떻게 달라지는지 복잡도를 분석하라.
> **풀이**
> - 0.5(낮게): 평균 체인 길이 ≈ 0.5로 충돌·조회 시간이 줄지만 버킷의 절반이 비어 공간 낭비가 크고 리사이징이 더 자주 일어난다.
> - 0.95(높게): 공간은 알차지만 평균 체인이 길어져 조회 상수가 커진다. 특히 오픈 어드레싱이면 0.7 부근부터 탐사 길이가 급격히 폭증(clustering)해 사실상 사용 불가.
> 결론: 체이닝은 임계값에 상대적으로 관대하나 오픈 어드레싱은 ~0.7이 실질 상한. 0.75는 시간·공간의 경험적 절충점이다.

> [!example]- 문제: 키가 자주 삭제되고 조회가 지배적인 워크로드에서 체이닝과 오픈 어드레싱 중 무엇을 고를지 근거를 대라.
> **풀이**
> 잦은 삭제가 핵심 변수다.
> - 오픈 어드레싱: 삭제 시 실제로 비우면 탐사 사슬이 끊겨 tombstone(삭제 표시)을 써야 하는데, tombstone이 쌓이면 탐사 길이가 늘어 조회가 느려지고 주기적 정리(rehash)가 필요하다.
> - 체이닝: 리스트에서 노드만 제거하면 끝이라 삭제가 깔끔하고 tombstone 부담이 없다.
> 결론: 삭제가 잦으면 체이닝이 유리. 다만 삭제가 드물고 조회 캐시 성능이 최우선이면 오픈 어드레싱 + SIMD([[hash-in-practice]])가 더 빠를 수 있어, "삭제 빈도"로 갈린다.

## 파인만

> [!note]- 백지에 해시 테이블의 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 왜 평균 O(1)이고 최악은 O(n)인가, 그 최악은 언제 발생하는가.
> 2. 로드 팩터가 무엇이고 리사이징이 어떻게 amortized O(1)을 만드는가.
> 3. 체이닝 vs 오픈 어드레싱의 캐시·삭제·로드 팩터 한계 트레이드오프.
