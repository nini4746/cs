# 이진 탐색 트리 (Binary Search Trees)

## 한 줄 요약

각 노드의 왼쪽 서브트리는 작은 값, 오른쪽은 큰 값. 이 불변식이 O(log n) 탐색/삽입/삭제를 주지만, 균형이 깨지면 O(n)으로 퇴화한다. 그래서 균형 트리가 필요해진다.

## 왜 필요한가

- 정렬 유지 + 빠른 탐색이 동시에 필요할 때 (해시는 순서 없음)
- 왜 BST가 O(log n)이면서 최악 O(n)인가
- 균형 트리([[balanced-trees]])가 왜 필요한지의 동기

## BST 불변식

모든 노드에서: **왼쪽 서브트리 < 노드 < 오른쪽 서브트리**.

```
        8
      /   \
     3     10
    / \      \
   1   6     14
```

이 순서 덕에:
- **탐색**: 루트에서 시작, 작으면 왼쪽·크면 오른쪽 → 매 단계 절반씩 (이진 탐색 → algorithms/[[binary-search]])
- **중위 순회(in-order)**: 왼쪽→노드→오른쪽 방문 = **정렬된 순서** 출력. BST의 공짜 정렬

## 연산

| 연산 | 방법 | 복잡도 |
|---|---|---|
| 탐색 | 비교하며 내려감 | O(높이) |
| 삽입 | 탐색 후 리프에 붙임 | O(높이) |
| 삭제 | 3경우 (아래) | O(높이) |
| 최소/최대 | 왼쪽/오른쪽 끝까지 | O(높이) |
| 후속자(successor) | 중위 순서 다음 | O(높이) |

**전부 O(높이).** 균형 잡히면 높이 = log n → O(log n). 문제는 높이.

### 삭제의 세 경우

1. **리프**: 그냥 제거
2. **자식 하나**: 자식으로 대체
3. **자식 둘**: 후속자(오른쪽 서브트리의 최소)로 대체 후 그 후속자 삭제

## 문제: 균형이 깨지면 O(n)

BST 성능은 전부 **높이**에 달림. 최선은 log n이지만:

**정렬된 데이터를 순서대로 삽입**하면:

```
1 → 2 → 3 → 4 → 5 삽입:
1
 \
  2
   \
    3
     \
      4    ← 사실상 링크드 리스트! 높이 = n
```

모든 삽입이 오른쪽 끝 → **한쪽으로 치우친 트리** = 높이 n = 탐색 O(n). BST의 O(log n)이 완전히 무너짐.

이게 실전에서 흔함 (정렬된 입력, 순차 ID). 순수 BST를 실무에서 잘 안 쓰는 이유.

## 해결의 방향: 균형 유지

높이를 항상 O(log n)으로 강제하려면 삽입/삭제 시 트리를 재조정:

- **자가 균형 트리**: AVL, Red-Black → [[balanced-trees]]. 회전으로 균형 유지
- **랜덤화**: treap, skip list → [[skip-list]]. 확률적으로 균형
- 이들이 최악을 O(log n)으로 보장

## BST가 여전히 유용한 곳

균형 버전으로:

- **정렬 유지 맵/셋**: C++ std::map, Java TreeMap (red-black tree)
- **범위 질의**: "10~20 사이 키 전부" - 해시로는 불가, BST는 O(log n + 결과수)
- **순서 통계**: k번째 원소, 순위
- **후속자/선행자**: 정렬 순서 이웃

해시 테이블([[hash-tables]])이 순수 조회는 더 빠르지만(O(1)), **순서·범위**가 필요하면 트리.

## 셀프 체크

> [!question]- BST 불변식이 주는 두 가지 이점은?
> 왼쪽 서브트리 < 노드 < 오른쪽 서브트리라는 순서 덕에, 탐색은 매 단계 절반씩 좁혀가는 이진 탐색이 되고, 중위 순회(왼쪽→노드→오른쪽)는 정렬된 순서로 출력된다. 즉 빠른 탐색과 공짜 정렬을 동시에 얻는다.

> [!question]- BST의 모든 연산이 O(높이)인데 왜 최악 O(n)이 되나?
> 성능이 전부 높이에 달려 있는데, 정렬된 데이터를 순서대로 삽입하면 모든 삽입이 한쪽 끝으로 가 사실상 링크드 리스트(높이 n)가 되기 때문이다. 이러면 탐색이 O(n)으로 퇴화한다. 정렬된 입력이나 순차 ID에서 흔해 순수 BST를 실무에서 잘 안 쓰는 이유다.

> [!question]- 자식이 둘인 노드를 삭제하는 방법은?
> 후속자(오른쪽 서브트리의 최소값)로 그 노드를 대체한 뒤, 그 후속자를 삭제한다. 후속자는 자식이 최대 하나뿐이라 삭제가 단순한 경우로 귀결된다.

> [!question]- 조회가 O(1)인 해시 테이블이 있는데도 BST(균형 버전)를 쓰는 경우는?
> 순서나 범위가 필요할 때다. 범위 질의("10~20 사이 키 전부")는 해시로 불가하지만 트리는 O(log n + 결과수)로 가능하고, 순서 통계(k번째 원소)나 후속자/선행자 같은 정렬 순서 이웃 탐색도 트리만 할 수 있다.

## 연습문제

> [!example]- 문제: 재귀로 BST 탐색(search)과 삽입(insert)을 직접 구현하라.
> **풀이**
> 불변식(왼쪽 < 노드 < 오른쪽)을 따라 한쪽으로만 내려간다.
> 
> ```
> search(node, k):
>     if node == null: return null
>     if k == node.key: return node
>     if k < node.key: return search(node.left, k)
>     else:            return search(node.right, k)
> 
> insert(node, k):
>     if node == null: return new Node(k)
>     if k < node.key:  node.left  = insert(node.left, k)
>     elif k > node.key: node.right = insert(node.right, k)
>     return node        # 중복은 무시
> ```
> 둘 다 루트에서 리프까지 한 경로만 타므로 O(높이). 균형이면 O(log n).

> [!example]- 문제: 같은 키 집합 {1,2,3,4,5,6,7}을 (a) 정렬 순서대로, (b) 균형이 되도록 삽입할 때 높이를 각각 구하고 왜 차이가 나는지 분석하라.
> **풀이**
> (a) 1,2,3,4,5,6,7을 순서대로 삽입하면 매 삽입이 오른쪽 끝에 붙어 한쪽 사슬이 된다. 높이 = 7(=n), 탐색 O(n).
> (b) 중앙값을 루트로 삽입(4 먼저, 그다음 2,6, 그다음 1,3,5,7)하면 완전 균형 트리가 된다. 높이 = 3 ≈ log₂7, 탐색 O(log n).
> 차이의 원인: BST 성능은 전적으로 높이에 달려 있고, 높이는 삽입 순서에 좌우된다. 정렬 입력은 최악의 편향을, 중앙값 우선 삽입은 최선의 균형을 만든다. 실전에서 삽입 순서를 통제할 수 없기에 자가 균형 트리([[balanced-trees]])가 필요하다.

> [!example]- 문제: 어떤 이진 트리가 유효한 BST인지 검사하는 알고리즘을 설계하라.
> **풀이**
> 흔한 오답은 각 노드에서 left.key < key < right.key만 보는 것 - 이는 손자 레벨의 위반을 놓친다. 올바른 방법은 각 노드에 허용 범위 (min, max)를 전파하며 재귀하는 것이다.
> 
> ```
> isBST(node, min, max):
>     if node == null: return true
>     if not (min < node.key < max): return false
>     return isBST(node.left,  min, node.key)
>        and isBST(node.right, node.key, max)
> # 최초 호출: isBST(root, -∞, +∞)
> ```
> 또는 중위 순회 결과가 순증가하는지 확인해도 된다(BST면 중위 순회 = 정렬). 둘 다 O(n).

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. BST 불변식과 그로부터 나오는 두 이점 (이진 탐색 + 중위 순회 = 정렬)
> 2. 모든 연산이 O(높이)인 이유와, 정렬 입력에서 높이=n으로 퇴화하는 메커니즘
> 3. 자식 둘인 노드 삭제(후속자로 대체)와, 균형이 왜 필요한지의 동기 → [[balanced-trees]]
