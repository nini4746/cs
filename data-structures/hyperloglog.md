# 하이퍼로그로그 (HyperLogLog)

## 한 줄 요약

거대한 스트림의 고유 원소 개수(카디널리티)를 아주 적은 메모리로 추정한다. 수십억 개의 유니크 값을 ~12KB로 ~2% 오차에 센다. 원리는 "해시값에서 연속된 0의 최대 길이가 개수를 암시한다"는 통찰.

## 왜 필요한가

- "고유 방문자 수", "유니크 검색어 수"를 어떻게 세나
- 정확히 세려면 모든 값을 저장해야 하는데(수 GB), 왜 12KB로 되나
- 확률적 카운팅의 대표

## 문제: 정확한 카디널리티는 비싸다

"고유 원소 몇 개?"를 정확히 세려면 본 것을 전부 기억해야 함 (셋/해시). 10억 개 유니크면 수 GB. 스트림/분산 환경에선 감당 불가.

HyperLogLog: **원소를 저장하지 않고** 통계적 흔적만 남겨 개수를 추정.

## 핵심 통찰: 연속된 0

원소를 해시하면 균등 랜덤 비트열이 됨. 그 비트열의 **앞쪽 연속된 0의 개수**를 보면:

- 앞에 0이 1개일 확률 = 1/2
- 0이 2개 연속일 확률 = 1/4
- **0이 k개 연속**일 확률 = 1/2^k

거꾸로: "최대 k개의 연속 0을 봤다" → 대략 **2^k개의 서로 다른 값**을 봤을 것. 드문 이벤트(많은 0)를 봤다는 건 그만큼 많이 시도했다는 뜻.

동전으로 비유: 앞면 10개 연속을 본 사람은 대략 2^10번 던졌을 것.

## 정확도 개선: 버킷 평균

하나의 최대값은 분산이 큼 (운). 그래서 **여러 버킷으로 나눠 평균**:

- 해시의 앞 몇 비트로 버킷 선택 (예: 16384개 버킷)
- 나머지 비트로 연속 0 개수 측정, 버킷별 최대값 저장
- 버킷들의 (조화)평균으로 추정 → 분산 감소

이것이 "LogLog" → "HyperLogLog"의 개선 (조화 평균 + 보정). 각 버킷은 작은 카운터(6비트면 충분)만 저장.

## 메모리 vs 정확도

- 버킷 m개 → 표준 오차 ≈ **1.04/√m**
- 16384 버킷 → 오차 ~0.8%, 메모리 ~12KB
- **10억 개든 1조 개든 12KB 고정** (카디널리티가 커도 메모리 불변)
- 정확한 카운팅의 수 GB vs HLL의 12KB → 수십만 배 절약, 대가는 ~1% 오차

## 어디에 쓰나

- **유니크 방문자/사용자 수**: 웹 분석 (Google Analytics 등)
- **Redis PFCOUNT**: HyperLogLog 내장 명령 (PFADD/PFCOUNT)
- **데이터베이스 `APPROX_COUNT_DISTINCT`**: BigQuery, Presto, Redshift
- **네트워크 모니터링**: 고유 IP/플로우 수
- **분산 집계**: HLL은 **병합 가능**(mergeable) - 여러 노드의 HLL을 합쳐 전체 카디널리티. 분산 환경의 결정적 장점

## 병합 가능성 (분산의 핵심)

각 서버가 자기 스트림의 HLL을 유지하고, 나중에 **버킷별 최대값을 취해** 합침:

```
merged[i] = max(hll_A[i], hll_B[i])
```

정확한 셋이라면 합집합 계산이 비싸지만, HLL은 버킷 max만 → O(m). 맵리듀스/스트리밍 집계에 완벽.

## 확률적 구조 계열 정리

| 구조 | 답하는 질문 | 메모리 |
|---|---|---|
| Bloom filter | 소속 (있나?) | 원소당 몇 비트 |
| **HyperLogLog** | 카디널리티 (몇 개?) | 고정 (~12KB) |
| Count-Min Sketch | 빈도 (몇 번?) | 고정 |

전부 "정확성을 메모리와 맞바꾼다"는 같은 철학 → [[bloom-filter]].

## 셀프 체크

> [!question]- "연속된 0의 최대 길이"가 어떻게 고유 원소 개수를 암시하나?
> 원소를 해시하면 균등 랜덤 비트열이 되는데, 앞쪽에 0이 k개 연속일 확률은 1/2^k이다. 거꾸로 "최대 k개의 연속 0을 봤다"면 대략 2^k개의 서로 다른 값을 봤을 것이다. 드문 이벤트를 봤다는 건 그만큼 많이 시도했다는 뜻이다(동전 앞면 10연속을 본 사람은 약 2^10번 던졌을 것).

> [!question]- 하나의 최대값만 쓰지 않고 여러 버킷으로 나눠 평균 내는 이유는?
> 단일 최대값은 운에 좌우돼 분산이 크기 때문이다. 해시 앞 몇 비트로 버킷을 고르고 나머지로 연속 0을 측정해 버킷별 최대값을 저장한 뒤, 조화 평균으로 추정하면 분산이 줄어 정확도가 오른다. 이 개선이 LogLog에서 HyperLogLog로 이어진다.

> [!question]- 카디널리티가 10억이든 1조든 메모리가 ~12KB로 고정인 이유는?
> 원소를 저장하지 않고 각 버킷에 연속 0 개수(6비트면 충분)라는 통계적 흔적만 남기기 때문이다. 표준 오차는 버킷 수 m에 대해 ≈ 1.04/√m로, 버킷 수만 정하면 카디널리티 크기와 무관하게 메모리가 결정된다(16384 버킷 → 오차 ~0.8%, ~12KB).

> [!question]- HLL이 분산 환경에서 결정적으로 유리한 성질은?
> 병합 가능성(mergeable)이다. 각 서버가 자기 스트림의 HLL을 유지하고 나중에 버킷별 최대값을 취해(merged[i] = max(hll_A[i], hll_B[i])) 합치면 전체 카디널리티가 나온다. 정확한 셋은 합집합 계산이 비싸지만 HLL은 O(m)이라 맵리듀스/스트리밍 집계에 완벽하다.

## 연습문제

> [!example]- 문제: 원소를 저장하지 않는 기본형 HLL의 추정 로직을 직접 구현하라 (버킷 배열, add, estimate).
> **풀이**
> - 버킷 m개(예: 2^14), 각 버킷은 6비트 카운터 배열로 표현.
> - add(x): h = hash(x). 앞 log2(m)비트로 버킷 인덱스 j 결정, 나머지 비트에서 첫 1이 나오기까지의 연속 0 개수 + 1을 rank로 계산 → register[j] = max(register[j], rank).
> - estimate(): 각 버킷 register[j]에 대해 2^(-register[j])의 합을 구하고, 조화 평균 형태 E = α_m · m² / Σ 2^(-register[j]) 로 추정.
> - α_m은 편향 보정 상수. 이 골격이 "저장 없이 통계적 흔적만"이라는 핵심을 그대로 구현한다.

> [!example]- 문제: 정확한 카운팅(HashSet)과 HLL의 메모리·정확도 트레이드오프를 복잡도로 분석하라.
> **풀이**
> - HashSet: n개 유니크면 공간 O(n) (원소당 수십 바이트). 10억 유니크 → 수 GB. 카운트는 정확(오차 0).
> - HLL: 공간 O(m)로 카디널리티와 무관하게 고정. 표준 오차 ≈ 1.04/√m.
> - 즉 HLL은 공간을 n에서 상수로 낮추는 대신 상대오차 ~1%를 받는다. 정확도를 2배로 올리려면(오차 절반) 버킷을 4배로 늘려야 함(1/√m 관계).
> - "정확성을 메모리와 맞바꾼다"가 수식으로 드러난다.

> [!example]- 문제: 유니크 방문자 집계에 HashSet, 정렬 후 dedup, HLL 중 무엇을 왜 고르는지 근거를 대라.
> **풀이**
> - 정확한 수가 반드시 필요하고 데이터가 메모리에 들어가면 HashSet.
> - 스트림/분산이고 ~1% 오차 허용, 메모리 예산이 빡빡하면 HLL. 이유: 메모리 고정(~12KB)에 병합 가능해 노드별 부분 집계를 O(m)에 합칠 수 있음.
> - 여러 날/샤드의 유니크를 합산해야 하면 HLL이 결정적. HashSet 합집합은 O(n)에 원소 전송까지 필요하지만 HLL은 버킷 max만 주고받으면 됨.
> - 결론: "근사 허용 + 병합 필요 + 메모리 제약" 세 조건이 겹치면 HLL.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 해시값 앞쪽 연속 0의 최대 길이가 왜 카디널리티(2^k)를 암시하는가.
> 2. 단일 최대값의 분산 문제를 버킷 분할 + 조화 평균이 어떻게 줄이며, 오차가 왜 1.04/√m인가.
> 3. HLL이 카디널리티와 무관하게 ~12KB 고정이고 병합 가능(버킷 max)한 이유.
