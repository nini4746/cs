# 세그먼트 트리와 펜윅 트리 (Segment Tree & BIT)

## 한 줄 요약

"구간 합/최솟값을 구하고 원소를 갱신하는" 작업을 둘 다 O(log n)에 하는 구조. 세그먼트 트리는 범용, 펜윅 트리(BIT)는 합 전용으로 더 간결하고 빠르다.

## 왜 필요한가

- 배열 구간 질의 + 갱신을 동시에 빠르게
- 단순 배열의 한계 (질의 O(n) 또는 갱신 O(n))
- 경쟁 프로그래밍 필수 도구

## 문제: 구간 질의 vs 갱신

배열에서 "구간 [l, r]의 합"과 "원소 갱신"을 반복. 순진한 방법들은 하나가 느림:

| 방법 | 구간 합 | 원소 갱신 |
|---|---|---|
| 원본 배열 | O(n) (순회) | O(1) |
| 누적 합(prefix sum) | O(1) | **O(n)** (뒤 전부 갱신) |
| **세그먼트/펜윅** | O(log n) | O(log n) |

누적 합은 질의는 빠르지만 갱신이 느림. 둘 다 필요하면 세그먼트/펜윅 트리.

## 세그먼트 트리

각 노드가 **한 구간의 집계값**(합/최소/최대)을 저장하는 이진 트리:

```
             [0..7]합
          /          \
      [0..3]        [4..7]
      /    \        /    \
   [0..1] [2..3] [4..5] [6..7]
    ...
```

- 리프 = 원소 하나, 내부 노드 = 자식 두 구간의 합
- **구간 질의**: 질의 구간을 O(log n)개 노드로 분해해 합침
- **갱신**: 리프부터 루트까지 O(log n)개 노드 갱신
- 배열로 표현 (힙처럼, [[heaps]]): 노드 i의 자식 = 2i, 2i+1

### 범용성

세그먼트 트리는 **결합법칙 있는 어떤 연산이든** 가능: 합, 최소, 최대, GCD, XOR... 노드에 저장하는 집계만 바꾸면 됨. lazy propagation으로 **구간 갱신**(구간 전체에 +5)도 O(log n).

## 펜윅 트리 (BIT, Binary Indexed Tree)

**합 전용**의 더 간결한 구조. 비트 트릭 기반:

```c
// 갱신: i부터 위로 (i += i & -i)
void update(int i, int v){ for(;i<=n;i+=i&-i) tree[i]+=v; }
// 누적합 [1..i]: i에서 아래로 (i -= i & -i)
int query(int i){ int s=0; for(;i>0;i-=i&-i) s+=tree[i]; return s; }
// 구간 [l..r] = query(r) - query(l-1)
```

- `i & -i` = 최하위 1비트 ([[bits-and-integers]]의 그 트릭) → 각 인덱스가 담당하는 구간 결정
- 세그먼트 트리보다 **코드 짧고, 메모리 절반, 상수 빠름**
- 대신 합/XOR 같은 **역연산 있는 연산만** (최소/최대는 안 됨 - 뺄셈 불가)

## 세그먼트 트리 vs 펜윅

| | 세그먼트 트리 | 펜윅 (BIT) |
|---|---|---|
| 연산 | 임의 결합 연산 (min/max/gcd...) | 역연산 있는 것 (합/XOR) |
| 구간 갱신 | lazy로 가능 | 제한적 |
| 코드/메모리 | 길고 2배 | 짧고 절반 |
| 상수 | 큼 | 작음 |

**합만 필요하면 펜윅**(간결·빠름), **최소/최대/구간갱신이면 세그먼트 트리**.

## 어디에 쓰나

- 구간 합/최소 질의 (통계, 랭킹)
- 2D 확장 (격자 구간 질의)
- inversion count (정렬 관련)
- 좌표 압축 + BIT (구간 개수 세기)
- 경쟁 프로그래밍의 핵심 도구

## 셀프 체크

> [!question]- 누적 합(prefix sum)이 있는데도 세그먼트/펜윅 트리가 필요한 이유는?
> 누적 합은 구간 합 질의는 O(1)이지만 원소를 갱신하면 뒤의 누적값을 전부 다시 계산해야 해 갱신이 O(n)이다. 구간 질의와 갱신을 둘 다 반복해야 할 때 세그먼트/펜윅 트리는 둘 다 O(log n)에 처리한다.

> [!question]- 세그먼트 트리가 어떤 연산까지 지원하며 왜 그런가?
> 결합법칙이 있는 어떤 연산이든 지원한다(합, 최소, 최대, GCD, XOR 등). 각 노드에 저장하는 집계값만 바꾸면 되기 때문이다. 또 lazy propagation으로 구간 전체 갱신도 O(log n)에 가능하다.

> [!question]- 펜윅 트리(BIT)가 세그먼트 트리보다 간결/빠른 대신 제약이 있는 이유는?
> 펜윅은 `i & -i`(최하위 1비트) 트릭으로 각 인덱스가 담당하는 구간을 정해 코드가 짧고 메모리가 절반, 상수가 작다. 대신 구간 [l..r]을 query(r) - query(l-1)로 계산하므로 뺄셈 같은 역연산이 있는 연산(합/XOR)만 되고 최소/최대는 안 된다.

> [!question]- 합만 필요할 때와 최소/최대·구간 갱신이 필요할 때 각각 무엇을 쓰나?
> 합만 필요하면 펜윅(간결하고 빠름), 최소/최대나 구간 갱신이 필요하면 세그먼트 트리를 쓴다. 펜윅은 역연산이 없는 min/max를 지원하지 못하고 구간 갱신도 제한적이기 때문이다.

## 연습문제

> [!example]- 문제: 합 전용 펜윅 트리(update, query)를 직접 구현하고 update가 O(log n)인 이유를 설명하라.
> **풀이**
> - 배열 tree[1..n], 1-인덱스.
> - update(i, v): for(; i<=n; i += i & -i) tree[i] += v;
> - query(i) = Σ[1..i]: for(int s=0; i>0; i -= i & -i) s += tree[i]; return s;
> - 구간 [l..r] = query(r) - query(l-1).
> - i & -i는 i의 최하위 1비트. update는 그만큼씩 더해 위로 올라가는데, 각 스텝이 최하위 1비트를 제거/이동시켜 2진 표현의 1의 개수(최대 log n)만큼만 방문 → O(log n). query도 대칭적으로 O(log n).

> [!example]- 문제: 세그먼트 트리와 펜윅 트리 중 무엇을 쓸지 결정하는 근거를 세 시나리오로 대라. (a) 구간 합+점 갱신, (b) 구간 최솟값+점 갱신, (c) 구간 전체에 +5 하고 구간 합.
> **풀이**
> - (a) 펜윅. 합은 역연산(뺄셈)이 있어 query(r)-query(l-1)이 되고, 펜윅이 코드·메모리·상수 모두 우월.
> - (b) 세그먼트 트리. min은 역연산이 없어 펜윅으로 구간 min이 불가. 세그먼트 트리는 노드에 min을 저장해 결합.
> - (c) lazy propagation 세그먼트 트리. 구간 갱신을 O(log n)에 하려면 lazy 태그가 필요. 순수 펜윅은 구간 갱신이 제한적.
> - 결론: 합만이면 펜윅, 역연산 없는 집계나 구간 갱신이면 세그먼트 트리.

> [!example]- 문제: 배열, 누적 합, 세그먼트 트리 세 방식의 구간 합/점 갱신 복잡도를 표로 분석하고 언제 어느 것이 최선인지 논하라.
> **풀이**
> - 원본 배열: 구간 합 O(n), 갱신 O(1). 갱신만 많고 질의는 드물면 최선.
> - 누적 합: 구간 합 O(1), 갱신 O(n)(뒤 전부 재계산). 갱신이 거의 없고 질의만 많은 정적 데이터에 최선.
> - 세그먼트/펜윅: 둘 다 O(log n). 질의와 갱신이 뒤섞여 반복되는 동적 상황에 최선.
> - 핵심: 워크로드의 질의:갱신 비율이 자료구조 선택을 지배한다. "둘 다 자주"면 log n 구조가 유일하게 균형 잡힌 선택.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 누적 합이 구간 질의 O(1)인데도 왜 갱신이 O(n)이라 부족하고, 세그먼트/펜윅이 둘 다 O(log n)을 내는지.
> 2. 세그먼트 트리가 결합법칙 있는 임의 연산을 지원하는 이유, 펜윅은 왜 역연산 있는 합/XOR로 제한되는지.
> 3. i & -i(최하위 1비트) 트릭이 펜윅의 update/query를 어떻게 O(log n)으로 만드는지.
