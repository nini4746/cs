# B-트리 (B-Trees)

## 한 줄 요약

한 노드가 수백 개의 키와 자식을 갖는 다분기 균형 트리. 노드 하나 = 디스크 블록 하나로 맞춰, 디스크 접근 횟수를 트리 높이(2~4)로 최소화한다. 데이터베이스 인덱스와 파일시스템의 표준.

## 왜 필요한가

- 왜 DB 인덱스가 이진 트리가 아니라 B-tree인가
- 디스크/SSD 특성이 자료구조를 어떻게 바꾸나
- database/[[btree-index]]의 기반

## 이진 트리의 디스크 문제

균형 이진 트리([[balanced-trees]])는 O(log₂ n). n=10억이면 높이 ~30. **각 노드 방문 = 디스크 접근**이면:

- 디스크 접근 30번 × ~100μs~ms([[memory-hierarchy]]) = 치명적
- 노드 하나(키+포인터 몇십 바이트)를 읽으려고 블록 하나(4~16KB)를 통째 읽음 → 대부분 낭비

문제의 핵심: 이진 트리는 분기가 2 → 높이가 큼 → 디스크 접근 많음. 디스크는 **접근 횟수**가 지배적.

## B-트리: 분기를 키운다

한 노드에 **많은 키(수백~수천)와 자식**을 담음:

```
          [30 | 70]                    ← 루트: 키 2개, 자식 3개
         /    |    \
   [10|20] [40|50|60] [80|90]          ← 각 노드도 여러 키
```

- 노드 하나 = **디스크 블록 하나**(4~16KB)에 꽉 차게. 한 번 읽으면 키 수백 개
- 분기 계수(fanout) B가 크면 높이 = log_B(n) → **매우 낮음**
- n=10억, B=100이면 높이 ≈ log₁₀₀(10⁹) ≈ **4~5**. 디스크 접근 4~5번으로 10억 개 탐색

이진 트리 30번 → B-tree 4번. **접근 횟수를 fanout으로 줄인 것.**

## B-트리 성질

- 모든 리프가 같은 깊이 (균형)
- 각 노드는 최소 절반 이상 채움 (공간 효율)
- 노드 내 키는 정렬 → 노드 안에서 이진 탐색
- 삽입/삭제 시 노드 분할(split)/병합(merge)으로 균형 유지

## B+트리 (실전 변형)

DB가 실제 쓰는 것은 대개 **B+tree**:

- **모든 실제 데이터는 리프에만**, 내부 노드는 인덱스(키+포인터)만
- **리프들이 링크드 리스트로 연결** → 범위 스캔(range scan)이 리프 순회로 빠름
- 내부 노드가 데이터를 안 담아 fanout 더 큼 → 높이 더 낮음

```
내부:     [30 | 70]
리프: [10 20 30]→[40 50 60 70]→[80 90]   ← 연결됨, 범위 질의 O(범위)
```

DB에서 `WHERE age BETWEEN 20 AND 40`이 빠른 이유 = 시작 리프 찾고 링크 따라 순회 → database/[[btree-index]].

## 왜 여전히 B-tree인가 (SSD 시대)

SSD는 탐색 시간이 없지만([[ssd-internals]]) B-tree는 여전히 표준:

- SSD도 **블록 단위 접근**([[ssd-internals]]) → 블록에 꽉 채우는 B-tree가 여전히 맞음
- 캐시 계층에도 유리 (노드 = 캐시 라인 여러 개, 순차)
- 다만 **쓰기 집약** 워크로드는 B-tree의 in-place 갱신이 SSD와 안 맞아 **LSM-tree**가 대안 → database/[[lsm-tree]]

## B-tree vs 이진 균형 트리

| | B-tree | AVL/Red-Black |
|---|---|---|
| 분기 | 수백~수천 | 2 |
| 높이 | 매우 낮음 | log₂ n |
| 최적 매체 | 디스크/SSD (블록) | 메모리 |
| 노드 크기 | 블록 하나 | 작음 |
| 용도 | DB 인덱스, 파일시스템 | 인메모리 맵 |

**메모리 안이면 이진 트리, 디스크면 B-tree.** 매체가 자료구조를 정한다.

## 셀프 체크

<details>
<summary>DB 인덱스는 왜 균형 이진 트리가 아니라 B-tree를 쓰나?</summary>

이진 트리는 분기가 2라 n=10억이면 높이가 ~30이고, 각 노드 방문이 디스크 접근이면 30번 × ~100μs~ms로 치명적이다. 게다가 수십 바이트 노드를 읽으려 블록(4~16KB)을 통째 읽어 대부분 낭비한다. 디스크는 접근 횟수가 지배적이므로 분기를 키워 접근 횟수를 줄여야 한다.
</details>

<details>
<summary>B-tree의 fanout이 크면 왜 디스크 접근이 줄어드나?</summary>

노드 하나를 디스크 블록 하나에 꽉 채워 한 번에 키 수백 개를 읽고, 높이 = log_B(n)이 된다. n=10억, B=100이면 높이 ≈ log₁₀₀(10⁹) ≈ 4~5로, 디스크 접근 4~5번에 10억 개를 탐색한다. 이진 트리 30번을 fanout으로 4번으로 줄인 것이다.
</details>

<details>
<summary>B+tree가 순수 B-tree와 다른 점과 그 이점은?</summary>

B+tree는 실제 데이터를 리프에만 두고 내부 노드는 인덱스(키+포인터)만 담는다. 그래서 내부 노드 fanout이 더 커져 높이가 더 낮다. 또 리프들이 링크드 리스트로 연결돼 `WHERE age BETWEEN 20 AND 40` 같은 범위 스캔이 시작 리프를 찾고 링크를 따라 순회하면 돼 빠르다.
</details>

<details>
<summary>SSD는 탐색 시간이 없는데 왜 여전히 B-tree가 표준인가?</summary>

SSD도 블록 단위로 접근하므로 블록에 꽉 채우는 B-tree가 여전히 맞고, 캐시 계층에도 유리하다. 다만 쓰기 집약 워크로드는 B-tree의 in-place 갱신이 SSD와 잘 안 맞아 LSM-tree가 대안이 된다.
</details>
