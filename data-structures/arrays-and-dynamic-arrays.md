# 배열과 동적 배열 (Arrays and Dynamic Arrays)

## 한 줄 요약

배열은 연속 메모리라 인덱싱이 O(1)이고 캐시 친화적이다. 동적 배열(vector, list, ArrayList)은 꽉 차면 2배로 키우는데, 이 "2배 성장"이 삽입을 amortized O(1)로 만든다.

## 왜 필요한가

- 가장 많이 쓰는 자료구조. 다른 것의 기반
- `append`가 왜 보통 빠른데 가끔 느린가
- 왜 2배로 키우나 (+1이 아니라)
- 배열이 왜 실전에서 링크드 리스트보다 빠른가 → [[linked-lists]]

## 배열: 연속 메모리

`T a[N]` = `sizeof(T)×N` 연속 블록 ([[data-layout]]):

- **인덱싱 O(1)**: `a[i]` 주소 = `base + i×sizeof(T)`, 곱셈 하나 (스케일 인덱스, [[assembly-basics]])
- **캐시 친화적**: 순차 접근이 캐시 라인·프리페처를 최대 활용 ([[memory-hierarchy]])
- **삽입/삭제 느림**: 중간에 넣으려면 뒤를 다 밀어야 → O(n)
- **고정 크기**: 컴파일/할당 시 크기 결정

## 동적 배열: 자라는 배열

크기를 미리 모를 때. 내부는 배열 + (용량 capacity, 크기 size):

```
[■■■■□□□□]   size=4, capacity=8
 ↑쓴 것  ↑여유
```

- **append**: 여유 있으면 그냥 씀 O(1)
- **꽉 차면**: 더 큰 배열 할당 → 기존 것 복사 → 새 원소 추가. 이때만 O(n)

## 왜 2배로 키우나 - amortized 분석

꽉 찰 때마다 **용량을 2배로**. 실측 (100만 append):

```
2x growth: 21 reallocs, ~1048575 total copies (1.0 per elem)
```

100만 삽입에 재할당은 **21번**뿐 (2²¹ ≈ 200만), 총 복사는 약 100만 = **원소당 평균 1회 복사**. 개별 append는 O(1)이거나 가끔 O(n)이지만, **평균 내면 O(1)** = amortized O(1).

### 왜 amortized O(1)인가 (수학)

용량 n까지 키우며 든 총 복사 = 1 + 2 + 4 + ... + n/2 + n ≈ **2n**. n개 삽입에 총 2n 작업 → 삽입당 평균 2 = O(1).

핵심: 비싼 재할당이 **드물게**(지수적으로 성장) 일어나 자주 일어나는 싼 삽입에 분산됨.

### +1씩 키우면? (재앙)

꽉 찰 때마다 1칸씩만 키우면 매번 재할당 → 복사 = 1+2+...+n = **n²/2**. 100만이면 5000억 복사. O(n) per append, O(n²) 전체. 2배 성장이 이걸 O(n)으로 바꿈.

### 성장 배수 트레이드오프

- **2배**: 재할당 드묾. 하지만 최악 메모리 낭비 ~2배, 옛 공간 재사용 불가(1+2+4 < 8)
- **1.5배**(일부 구현): 메모리 낭비 적음, 옛 공간 재사용 가능. 재할당 조금 더 잦음
- Java ArrayList 1.5배, Go slice·C++ vector 대략 2배·1.5~2배

## 삭제와 축소

- 끝에서 삭제(pop): O(1)
- 중간 삭제: 뒤를 당김 O(n)
- 축소: size가 capacity의 1/4로 떨어지면 절반으로 (1/2 아님 - 경계에서 반복 재할당 방지, hysteresis)

## 캐시 관점의 우위

동적 배열이 실전 최강인 이유는 Big-O만이 아님:

- 연속 메모리 → 순회 시 캐시 미스 최소 ([[cache-misses]])
- 프리페처가 다음 원소 미리 로드
- 같은 O(1) 접근이라도 링크드 리스트(pointer chasing)보다 상수가 훨씬 작음 → [[linked-lists]]

"의심스러우면 배열 써라"의 근거.

## 셀프 체크

> [!question]- 동적 배열은 꽉 차면 왜 +1이 아니라 2배로 키우나?
> +1씩 키우면 매번 재할당이 일어나 총 복사가 1+2+...+n = n²/2, 즉 append당 O(n)에 전체 O(n²)이 된다. 2배로 키우면 재할당이 지수적으로 드물게 일어나(n까지 총 복사 ≈ 2n) append당 평균 2회 = amortized O(1)이 된다.

> [!question]- append가 amortized O(1)인 이유를 총 복사량으로 설명하면?
> 용량 n까지 키우며 든 총 복사는 1 + 2 + 4 + ... + n/2 + n ≈ 2n이다. n개 삽입에 총 2n 작업이므로 삽입당 평균 2회 = O(1). 비싼 재할당이 드물게 일어나 자주 일어나는 싼 삽입에 분산되는 것이 핵심이다.

> [!question]- 축소할 때 왜 1/2이 아니라 capacity의 1/4로 떨어져야 절반으로 줄이나?
> 경계에서 반복 재할당(진동)을 막기 위한 hysteresis다. 1/2 지점에서 축소하면 그 근처에서 삽입/삭제가 오가며 계속 재할당이 일어날 수 있다. 축소 임계(1/4)와 성장 임계 사이에 여유를 둬 이를 방지한다.

> [!question]- Big-O가 같은 O(1) 접근인데도 동적 배열이 링크드 리스트보다 실전에서 빠른 이유는?
> 동적 배열은 연속 메모리라 순회 시 캐시 미스가 최소이고 프리페처가 다음 원소를 미리 로드한다. 링크드 리스트는 pointer chasing으로 상수가 훨씬 크다. Big-O가 같아도 캐시 친화성이 상수 배수를 크게 벌린다.

> [!question]- 성장 배수 2배와 1.5배의 트레이드오프는?
> 2배는 재할당이 드물지만 최악 메모리 낭비가 ~2배이고 옛 공간을 재사용할 수 없다(1+2+4 < 8). 1.5배는 메모리 낭비가 적고 옛 공간을 재사용할 수 있지만 재할당이 조금 더 잦다. Java ArrayList는 1.5배, C++ vector는 대략 2배다.

## 연습문제

> [!example]- 문제: 재할당을 직접 하는 동적 배열(push_back, capacity 2배 성장)을 구현하라.
> **풀이**
> size와 capacity를 분리해 관리한다. push_back: size == capacity이면 capacity를 2배(초기엔 0→1)로 잡아 새 버퍼를 할당하고 기존 원소를 복사한 뒤 포인터를 교체한다. 그다음 버퍼[size++] = x.
> 
> ```
> void push_back(x):
>     if size == capacity:
>         new_cap = capacity == 0 ? 1 : capacity * 2
>         new_buf = allocate(new_cap)
>         copy old_buf[0..size) into new_buf
>         free old_buf; buf = new_buf; capacity = new_cap
>     buf[size++] = x
> ```
> 재할당 시에만 O(n), 그 외 O(1)이라 amortized O(1)이다.

> [!example]- 문제: n번 push_back의 총 복사 횟수가 2배 성장에서 Θ(n)임을 복잡도로 분석하라.
> **풀이**
> 재할당은 capacity가 1, 2, 4, ..., n/2, n일 때 일어난다. 각 재할당의 복사량은 그 시점 size(≈ 직전 capacity)이므로 총 복사 = 1 + 2 + 4 + ... + n ≈ 2n = Θ(n). n번 삽입에 총 Θ(n) 작업이므로 삽입당 amortized Θ(1). 만약 +1 성장이면 총 복사 = 1 + 2 + ... + n = n²/2 = Θ(n²)로 삽입당 Θ(n)이 된다.

> [!example]- 문제: 리스트를 순회하며 합만 구하는 작업에서 동적 배열과 [[linked-lists]] 중 무엇을 쓸지 근거를 대라.
> **풀이**
> 동적 배열을 쓴다. 두 자료구조 모두 순회는 O(n)이지만, 동적 배열은 연속 메모리라 순차 접근이 캐시 라인과 프리페처를 최대로 활용해 캐시 미스가 최소다. 링크드 리스트는 노드가 흩어져 있어 pointer chasing으로 매 원소마다 캐시 미스가 나 상수가 훨씬 크다. Big-O가 같아도 실측은 배열이 크게 빠르다. "의심스러우면 배열 써라".

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 배열의 인덱싱이 왜 O(1)이고 캐시 친화적인지 (base + i×sizeof(T), 연속 메모리)
> 2. 2배 성장이 왜 append를 amortized O(1)로 만드는지 (총 복사 ≈ 2n vs +1의 n²/2)
> 3. 축소 임계를 1/4로 두는 이유(hysteresis)와 성장 배수 2 vs 1.5의 트레이드오프
