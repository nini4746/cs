# 트라이 (Tries)

## 한 줄 요약

문자열을 문자 단위로 트리에 저장. 각 경로가 하나의 키(접두사)를 나타내 접두사 탐색이 자연스럽다. 자동완성, IP 라우팅의 기반. 메모리를 많이 먹어 압축 변형(radix tree)이 실전에서 쓰인다.

## 왜 필요한가

- 자동완성/접두사 검색이 어떻게 O(키길이)로 되나
- 해시맵이 못 하는 것 (접두사 질의)
- IP 라우팅 테이블의 구조

## 구조

각 노드가 한 문자, 루트에서 노드까지 경로가 문자열:

```
       (root)
      /   |   \
     c    t    b
     |    |    |
     a    o    e
    / \       / \
   t   r     e   d
  (cat)(car)(bee)(bed)
```

- 키를 문자 단위로 쪼개 트리에 배치
- 각 노드는 자식들(다음 문자 → 노드)의 맵을 가짐
- 단어 끝 표시 (이 노드에서 끝나는 키가 있나)

## 연산

| 연산 | 방법 | 복잡도 |
|---|---|---|
| 삽입 | 문자 따라 내려가며 노드 생성 | O(키 길이) |
| 탐색 | 문자 따라 내려감 | O(키 길이) |
| 접두사 탐색 | 접두사까지 내려간 뒤 서브트리 | O(접두사 + 결과) |

**해시맵과 결정적 차이**: 복잡도가 **키 개수(n)와 무관**, 키 길이(k)에만 의존. 해시맵은 O(1) 평균이지만 해시 계산에 전체 키를 봐야 하고 **접두사 질의 불가**.

## 트라이만의 강점: 접두사

- **자동완성**: "cat" 입력 → "cat" 노드로 가서 서브트리 전체가 후보 (cats, catch, category...)
- **접두사 매칭**: 사전 검색, 명령어 완성
- **정렬 순회**: 자식을 알파벳 순으로 방문하면 사전순 출력
- 해시맵은 이 중 아무것도 못 함 (해시가 순서/접두사를 파괴)

## 약점: 메모리

- 노드마다 자식 맵 (알파벳 26개면 포인터 배열, 유니코드면 큼)
- 키가 짧고 많으면 노드 오버헤드가 데이터보다 큼
- 포인터 추적 → 캐시 안 좋음 ([[linked-lists]] 문제)

그래서 압축 변형이 실전 표준:

## Radix Tree (압축 트라이)

**자식이 하나뿐인 연쇄를 하나의 간선으로 압축**:

```
일반 트라이:  t→e→s→t (4노드)
radix:        [test] (1간선)
```

- 분기 없는 경로를 합침 → 노드 수 급감, 메모리 절약
- **IP 라우팅**: 라우팅 테이블이 radix tree (longest prefix match) → network/[[routing]]
- 리눅스 커널 페이지 캐시 인덱스, Redis 스트림 ID

## 관련 구조

- **PATRICIA trie**: 이진 radix tree (비트 단위)
- **suffix trie/tree**: 문자열의 모든 접미사 → 부분 문자열 검색 O(패턴)
- **DAWG**: 접미사도 공유해 더 압축
- **ternary search tree**: 트라이+BST 하이브리드, 메모리 절약

## 어디에 쓰나

- 자동완성/검색 제안
- 사전, 맞춤법 검사
- IP 라우팅 (longest prefix match) → network/[[routing]]
- 문자열 집합의 접두사 압축
- algorithms/[[string-matching]]의 일부 (Aho-Corasick = 트라이 + 실패 링크)

## 셀프 체크

> [!question]- 트라이의 연산 복잡도가 해시맵과 결정적으로 다른 점은?
> 트라이는 삽입/탐색이 O(키 길이)로 키 개수 n과 무관하고 키 길이 k에만 의존한다. 해시맵은 O(1) 평균이지만 해시 계산에 전체 키를 봐야 하고, 무엇보다 접두사 질의를 못 한다.

> [!question]- 트라이만이 할 수 있고 해시맵은 못 하는 것들은?
> 접두사 관련 기능이다. 자동완성("cat" 노드로 가면 서브트리 전체가 후보), 접두사 매칭(사전 검색, 명령어 완성), 자식을 알파벳 순으로 방문하는 정렬 순회가 가능하다. 해시맵은 해시가 순서와 접두사를 파괴해 이 중 아무것도 못 한다.

> [!question]- 트라이의 약점은 무엇이고 radix tree가 이를 어떻게 개선하나?
> 약점은 메모리다. 노드마다 자식 맵이 있어(유니코드면 큼) 키가 짧고 많으면 노드 오버헤드가 데이터보다 커지고, 포인터 추적으로 캐시도 나쁘다. radix tree는 자식이 하나뿐인 연쇄(분기 없는 경로)를 하나의 간선으로 압축해(t→e→s→t를 [test] 하나로) 노드 수를 급감시키고 메모리를 절약한다.

> [!question]- IP 라우팅 테이블이 radix tree를 쓰는 이유와 관련 개념은?
> longest prefix match(가장 긴 접두사 일치)로 라우팅하기 때문이다. radix tree는 접두사 구조를 그대로 담아 이 매칭에 자연스럽게 맞는다. 리눅스 커널 페이지 캐시 인덱스, Redis 스트림 ID도 radix tree를 쓴다.

## 연습문제

> [!example]- 문제: 소문자 알파벳 트라이를 직접 구현하라 - insert와 startsWith(접두사 존재) 연산을.
> **풀이**
> 각 노드는 자식 배열 `children[26]`과 단어 끝 플래그 `isEnd`를 가진다.
> - insert(word): 루트에서 시작해 각 문자 c마다 `children[c-'a']`가 없으면 노드 생성 후 내려간다. 마지막 노드의 isEnd=true.
> - startsWith(prefix): 각 문자를 따라 내려가다 중간에 자식이 없으면 false, 끝까지 내려가면 true(isEnd는 볼 필요 없음).
> 둘 다 O(키 길이). search(정확 일치)는 startsWith와 같되 마지막에 isEnd까지 확인하면 된다.

> [!example]- 문제: 자동완성에서 트라이 대신 정렬된 배열+이진 탐색을 쓸 때의 트레이드오프로 자료구조 선택 근거를 대라.
> **풀이**
> 정렬 배열에서 접두사 P로 시작하는 항목은 연속 구간을 이루므로, 이진 탐색으로 하한/상한을 찾으면 O(log n + 결과)에 접두사 후보를 뽑을 수 있다(트라이의 O(P+결과)와 비슷). 배열은 캐시 친화적이고 메모리도 작다. 그러나 삽입/삭제가 O(n)(원소 밀기)이라 사전이 자주 바뀌면 불리하다. 트라이는 동적 갱신이 O(P)로 싸고 공통 접두사를 공유하지만 노드 오버헤드와 포인터 추적 캐시 손해가 있다. 즉 정적/읽기 위주면 정렬 배열, 갱신이 잦으면 트라이가 유리하다.

> [!example]- 문제: 일반 트라이와 radix tree의 노드 수와 메모리를 복잡도로 분석하라.
> **풀이**
> 키 집합의 총 문자 수를 M이라 하면 일반 트라이는 최악 O(M) 노드(공유 접두사만큼만 절약)이고, 노드마다 자식 맵(알파벳 σ면 포인터 σ개)을 가져 메모리가 노드 수 × σ에 비례해 부풀 수 있다. radix tree는 분기 없는 연쇄를 한 간선으로 합치므로 노드 수가 키 개수 n에 비례하는 O(n) 수준으로 줄고(내부 노드는 분기점에서만 생김), 각 간선이 문자열 조각을 담아 포인터 오버헤드가 급감한다. 시간 복잡도는 둘 다 O(키 길이)로 같고 차이는 상수/메모리다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 트라이 구조(노드=문자, 경로=키, 단어 끝 표시)를 그리고 삽입/탐색이 왜 O(키 길이)이며 키 개수 n과 무관한지 설명할 수 있는가.
> 2. 트라이가 해시맵과 달리 할 수 있는 접두사 기능(자동완성, 접두사 매칭, 정렬 순회)과 그 이유를 댈 수 있는가.
> 3. 트라이의 약점(메모리/캐시)과 radix tree가 분기 없는 연쇄를 압축해 이를 개선하는 원리, 그리고 IP 라우팅의 longest prefix match와의 연결을 말할 수 있는가.
