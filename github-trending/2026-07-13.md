오늘 트렌딩은 "에이전트에게 어디까지 맡길 것인가"라는 질문이 여러 층위에서 반복해서 등장하는 하루입니다. 위험한 명령을 막는 가드레일(destructive_command_guard)은 여전히 상위권에 남아 있고, 여기에 더해 "AI 지원 개발로 Postgres를 통째로 다시 짠다"는 대담한 시스템 프로그래밍 프로젝트(pgrust), 실제 자산을 굴리는 트레이딩 에이전트(Vibe-Trading), 그리고 에이전트가 만든 산출물의 "티"를 없애려는 디자인 스킬(hallmark)까지 겹치면서, 에이전트가 코드를 짜는 단계를 넘어 인프라·자산·디자인 감각까지 대신하기 시작한 흐름이 뚜렷합니다. 그 사이에서 Prefect처럼 사람이 오래 다져온 워크플로 오케스트레이션 도구가 꾸준히 트렌딩에 오르는 것도, 자동화의 신뢰성을 어떻게 구조적으로 보장할지에 대한 관심이 여전하다는 신호로 읽힙니다.

## Dicklesworthstone/destructive_command_guard

- <https://github.com/Dicklesworthstone/destructive_command_guard>
- 언어: Rust
- 오늘 스타: 444 (총 3,141)

AI 에이전트가 실행하려는 위험한 git·셸 명령을 사전에 가로채 차단하는 가드레일 도구로, 어제에 이어 오늘도 상위권을 지켰습니다. 하루 만에 다시 400개가 넘는 스타가 붙은 것을 보면 단발성 화제가 아니라 "에이전트에게 셸 실행 권한을 주는 것이 일상이 된" 개발 환경에서 실질적인 안전장치로 자리잡고 있다는 뜻으로 보입니다. 어떤 명령 패턴을 되돌리기 어려운 것으로 분류하고 어떻게 가로채는지, 방어적 셸 래퍼를 설계할 때 참고하기 좋은 사례입니다.

## malisper/pgrust

- <https://github.com/malisper/pgrust>
- 언어: Rust
- 오늘 스타: 518 (총 2,568)

PostgreSQL 18.3을 Rust로 통째로 재구현하는 프로젝트로, 이미 4만 6천 개가 넘는 공식 회귀 테스트 쿼리를 통과했고 PostgreSQL 데이터 디렉터리와 디스크 호환까지 확보했다고 밝히고 있습니다. "AI 지원 개발을 활용해 Postgres를 더 수정하기 쉬운 형태로 다시 만든다"는 목표를 내세우는데, 프로세스당 커넥션 모델을 스레드 기반으로 바꿔 트랜잭션 처리 속도를 높이겠다는 로드맵까지 보면, 에이전트 시대에 대형 C 코드베이스를 안전한 언어로 재작성하는 접근법을 구체적으로 관찰할 수 있는 프로젝트입니다. 실제 프로덕션 DB를 통째로 재작성한다는 발상 자체와, 그 검증 기준으로 원본의 회귀 테스트 스위트를 그대로 쓰는 방법론 둘 다 공부할 가치가 있습니다.

## HKUDS/Vibe-Trading

- <https://github.com/HKUDS/Vibe-Trading>
- 언어: Python
- 오늘 스타: 768 (총 20,769)

"개인용 트레이딩 에이전트"를 표방하며 이틀 연속 트렌딩 전체에서 가장 큰 폭으로 스타가 늘고 있는 프로젝트입니다. LLM이 시장 데이터를 해석해 매매 판단까지 내리는 구조로 보이는데, 손실이 실제로 발생하는 금융 도메인에 자율 에이전트를 투입할 때 리스크 한도나 실행 승인 단계 같은 안전장치를 어떻게 설계하는지 살펴보기 좋은 사례입니다. 오늘 함께 트렌딩에 오른 가드레일 도구(destructive_command_guard)와 나란히 놓고 보면, "에이전트에게 무엇을 자율적으로 맡길 수 있는가"라는 질문이 코드 실행뿐 아니라 자산 운용까지 확장되고 있음을 보여줍니다.

## PrefectHQ/prefect

- 언어: Python
- 오늘 스타: 66 (총 23,216)

파이썬 스크립트를 프로덕션급 데이터 파이프라인으로 끌어올려주는 워크플로 오케스트레이션 프레임워크입니다. `@task`, `@flow` 데코레이터만으로 작업 단위를 정의하고, 스케줄링·캐싱·재시도·장애 복구를 프레임워크가 대신 처리해주는 구조가 핵심입니다. 매달 2억 건이 넘는 데이터 태스크를 처리하며 실무에서 검증된 성숙한 도구인 만큼, 자동화 파이프라인에 신뢰성(재시도, 관측 가능성, 이벤트 기반 트리거)을 어떻게 구조적으로 녹여내는지 참고하기 좋습니다. 오늘 트렌딩의 여러 자율 에이전트 프로젝트와 대비해서 보면, "실패를 전제로 설계된 오케스트레이션"이 왜 필요한지 감이 잡히는 프로젝트이기도 합니다.

## 오늘의 히든잼

- <https://github.com/philipl/inferencefs>

2015년에 나온 농담 프로젝트 πfs("π의 자릿수 속에 모든 파일이 이미 존재하므로 저장할 필요가 없다")의 2026년판 패러디입니다. FUSE로 마운트되는 이 파일시스템은 파일을 읽으면 실제로 디스크에서 데이터를 꺼내는 대신 LLM에게 "이 경로의 파일에 가장 그럴듯한 내용은 무엇인가"를 물어보고, 그 답을 파일 내용으로 돌려줍니다. 쓰기 연산은 접수되지만 그냥 버려지는데, README는 이를 "모델이 항상 마지막 발언권을 가진다"고 설명합니다. 스타 137개짜리 만우절 장난 프로젝트지만, FUSE로 파일시스템을 직접 구현하는 법과 LLM의 환각(confabulation) 문제를 동시에 웃음 포인트로 압축해낸 점에서, 오늘 트렌딩 전반에 흐르는 "에이전트를 얼마나 믿을 것인가"라는 주제를 가장 날카롭게 풍자하는 자료입니다.
