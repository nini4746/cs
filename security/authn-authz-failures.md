# 인증·인가 실패 (Authentication/Authorization Failures)

## 한 줄 요약

인증(누구인가)과 인가(무엇을 할 수 있나)의 실패 - 세션 하이재킹, IDOR(남의 데이터 접근), 권한 상승. OWASP 상위 취약점이며, 대부분 "권한을 서버에서 매번 확인 안 함"이 원인.

## 왜 필요한가

- 인증과 인가 실패의 구체적 패턴
- IDOR가 뭐고 왜 흔한가
- 접근 제어를 어디서 해야 하나

## 인증 vs 인가 (재확인)

web/[[web-auth]]:
- **인증(authentication)**: **누구인가** (로그인) - 이 사람이 진짜 alice인가
- **인가(authorization)**: **무엇을 할 수 있나** (권한) - alice가 이 자원에 접근 권한 있나

둘 다 실패 지점. 로그인은 됐어도(인증) 남의 데이터 접근(인가 실패) 가능.

## 인증 실패 패턴

### 세션 하이재킹

세션 ID/토큰을 훔쳐 남으로 위장 (web/[[web-auth]]):
- **XSS로 탈취**([[xss-csrf]]): HttpOnly로 방어
- **네트워크 도청**: HTTPS로 방어 (network/[[tls]])
- **세션 고정(fixation)**: 공격자가 정한 세션 ID를 피해자가 쓰게 → 로그인 시 세션 ID 재발급으로 방어
- **예측 가능한 세션 ID**: 랜덤·충분한 엔트로피 필요

### 약한 인증

- 무차별 대입: rate limiting, 계정 잠금 ([[password-storage]])
- 약한 비밀번호: 정책 (하지만 UX 균형)
- MFA 없음: 2FA로 비번 유출 방어
- 크리덴셜 스터핑: 유출된 비번 재사용 공격 → 유출 비번 차단

## 인가 실패 패턴

### IDOR (Insecure Direct Object Reference)

**남의 자원 ID로 직접 접근** - 가장 흔한 인가 실패:

```
GET /orders/123    → 내 주문 (OK)
GET /orders/124    → 남의 주문 (권한 확인 안 하면 접근됨!)
```

- URL/파라미터의 ID를 바꿔 남의 데이터 접근
- **원인**: 서버가 "이 사용자가 이 자원 소유자인가"를 **확인 안 함**
- 방어: 매 요청마다 **소유권/권한 검증** (`WHERE order.user_id = 현재_사용자`)
- 흔한 이유: 인증만 하고 인가를 빠뜨림 (로그인했으니 됐다고 착각)

### 권한 상승 (privilege escalation)

- **수직**: 일반 사용자 → 관리자 (관리자 기능에 권한 확인 없음)
- **수평**: 사용자 A → 사용자 B (IDOR)
- **함수 레벨 접근 제어 누락**: 관리자 API를 일반 사용자가 호출 (UI엔 안 보이지만 API는 열림)

### 클라이언트 신뢰

**클라이언트를 믿으면 안 됨**:
- UI에서 버튼 숨긴다고 보안 아님 (API 직접 호출 가능)
- 클라이언트 검증은 UX용, **서버에서 반드시 재검증**
- 가격·수량 등을 클라이언트가 보내면 서버가 검증 (조작 가능)

## 핵심 원칙: 서버에서 매번 확인

인가 실패의 근본 = **접근 제어를 서버에서 일관되게 안 함**:

```
매 요청마다:
1. 인증 확인 (누구?)
2. 인가 확인 (이 자원에 이 동작 권한?)
```

- **모든 엔드포인트**에서 (하나 빠뜨리면 그게 구멍)
- **자원 소유권** 확인 (IDOR 방어)
- **기본 거부(deny by default)**: 명시적으로 허용된 것만
- 중앙화된 접근 제어 (곳곳에 흩어지면 빠뜨림)

## 접근 제어 모델

- **RBAC**(역할 기반): 역할에 권한, 사용자에 역할 (admin, user)
- **ABAC**(속성 기반): 속성으로 세밀하게 (부서, 시간, 위치)
- **ACL**(접근 제어 목록): 자원마다 누가 뭘 할 수 있나
- **최소 권한**: 필요한 것만 → [[least-privilege]]

## 실무 체크리스트

1. **모든 엔드포인트에 인가 검사** (빠진 곳 없나)
2. **자원 소유권 확인** (IDOR)
3. **서버에서 재검증** (클라이언트 불신)
4. **기본 거부**
5. **세션 관리**: 재발급, 만료, HttpOnly/Secure ([[password-storage]], web/[[web-auth]])
6. **관리자 기능 보호** (함수 레벨)

## 셀프 체크

> [!question]- 인증과 인가의 차이는? 로그인에 성공해도 생길 수 있는 실패는?
> 인증은 "누구인가"(신원 확인, 로그인), 인가는 "무엇을 할 수 있나"(권한). 로그인(인증)에 성공해도 남의 자원 소유권을 확인 안 하면 인가 실패(IDOR)로 남의 데이터에 접근할 수 있다.

> [!question]- IDOR가 발생하는 근본 원인과 방어책은?
> 서버가 "이 사용자가 이 자원의 소유자인가"를 확인하지 않고 요청받은 ID로 바로 자원을 반환하기 때문. 방어는 매 요청마다 소유권/권한을 검증하는 것(예: `WHERE order.user_id = 현재_사용자`).

> [!question]- UI에서 관리자 버튼을 숨기면 안전한가?
> 아니다. 클라이언트는 신뢰할 수 없다. API를 직접 호출할 수 있으므로 함수 레벨 접근 제어가 서버에 없으면 그대로 뚫린다. 클라이언트 검증은 UX용이고 서버에서 반드시 재검증해야 한다.

> [!question]- 세션 고정(fixation) 공격과 그 방어는?
> 공격자가 미리 정한 세션 ID를 피해자가 쓰게 만든 뒤, 피해자가 로그인하면 그 세션으로 위장하는 공격. 로그인 시 세션 ID를 새로 재발급하면 막힌다.

## 연습문제

> [!example]- 다음 엔드포인트의 취약점을 진단하고 고쳐라: `GET /api/invoices/:id` 가 로그인만 확인하고 `SELECT * FROM invoices WHERE id = :id` 를 실행한다.
> **풀이**
> 로그인(인증)만 하고 소유권(인가)을 확인하지 않는 전형적 IDOR. 다른 사용자의 invoice id를 넣으면 그대로 조회된다.
> 수정: 쿼리에 소유자 조건을 추가한다. `SELECT * FROM invoices WHERE id = :id AND user_id = :현재사용자`. 결과가 없으면 404/403. 관리자는 별도 역할 검사로 예외 처리. 모든 엔드포인트에 이 소유권 검사를 일관되게 적용한다.

> [!example]- 크리덴셜 스터핑 공격에 대한 다층 방어를 설계하라.
> **풀이**
> 크리덴셜 스터핑은 다른 사이트에서 유출된 (아이디, 비번) 쌍을 그대로 시도하는 공격. 방어는 계층으로: (1) rate limiting과 계정 잠금으로 대량 시도 차단, (2) 유출된 비번 차단(haveibeenpwned 등), (3) MFA로 비번만으로는 못 들어오게, (4) 비정상 로그인 탐지(새 기기/위치 알림). 단일 방어가 아니라 조합이 핵심.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지. (1) 인증 vs 인가의 차이와 각각의 실패 패턴, (2) IDOR의 원인과 소유권 검증 방어, (3) "서버에서 매번, 모든 엔드포인트에서, 기본 거부로 확인"이 왜 근본 원칙인가.

## 연결

- 인증 기초 → web/[[web-auth]]
- 세션 탈취 (XSS) → [[xss-csrf]]
- 도청 방어 → network/[[tls]]
- 비밀번호·rate limit → [[password-storage]]
- 최소 권한 → [[least-privilege]]
- ARP/네트워크 MITM → network/[[ethernet-and-arp]]
