# 위협 모델링 (Threat Modeling)

## 한 줄 요약

시스템의 공격 표면을 체계적으로 분석해 "무엇을, 누구로부터, 어떻게 지킬까"를 설계 단계에서 정하는 것. STRIDE 같은 프레임워크로 위협을 열거하고 우선순위를 매긴다.

## 왜 필요한가

- 보안을 나중에 붙이지 말고 설계부터
- 어떤 위협에 집중할지 (전부는 불가능)
- 공격자 관점으로 생각하는 법

## 왜 모델링인가

보안은 무한 - 모든 것을 막을 수 없음. **위협 모델링 = 무엇이 중요하고 어디가 취약한지 체계적으로 파악**:

- 설계 단계에서 (나중에 붙이면 비쌈)
- 자원을 어디에 쓸지 우선순위
- "공격자라면 어떻게?" 관점 전환

## 네 가지 질문

위협 모델링의 뼈대 (Shostack):

1. **무엇을 만드나?** - 시스템 구조 파악 (데이터 흐름도)
2. **무엇이 잘못될 수 있나?** - 위협 열거 (STRIDE 등)
3. **어떻게 대응하나?** - 완화책
4. **잘 했나?** - 검증·반복

## 자산과 공격 표면

먼저 **무엇을 지키나**:

- **자산(asset)**: 보호할 것 (사용자 데이터, 비밀번호, 결제 정보, 가용성)
- **공격 표면(attack surface)**: 공격자가 접근 가능한 지점 (API, 입력, 네트워크, 의존성)
- **신뢰 경계(trust boundary)**: 신뢰 수준이 바뀌는 곳 (사용자 입력 → 서버, 인터넷 → 내부망)
- 데이터 흐름도(DFD)로 시각화 → 신뢰 경계를 넘는 데이터에 집중

## STRIDE 프레임워크

위협을 여섯 범주로 열거 (마이크로소프트):

| 위협 | 의미 | 대응 |
|---|---|---|
| **S**poofing | 신원 위조 | 인증 → web/[[web-auth]] |
| **T**ampering | 변조 | 무결성 → [[hashing]], [[digital-signatures]] |
| **R**epudiation | 부인 | 감사 로그, 서명 → [[digital-signatures]] |
| **I**nformation disclosure | 정보 유출 | 암호화 → [[crypto-basics]] |
| **D**enial of service | 서비스 거부 | rate limit, 확장 |
| **E**levation of privilege | 권한 상승 | 인가 → [[authn-authz-failures]], [[least-privilege]] |

각 구성요소·데이터 흐름에 STRIDE를 적용해 위협을 빠짐없이 열거.

## 위험 우선순위

열거한 위협을 다 막을 순 없음 → **우선순위**:

- **위험 = 가능성 × 영향(피해)**
- 높은 위험(쉽고 피해 큰 것)부터
- **DREAD**(옛), **CVSS**(취약점 점수) 등으로 정량화
- 비용 대비 효과 (완화 비용 vs 위험 감소)

## 공격 트리 (attack tree)

목표를 루트로, 공격 방법을 가지로 (algorithms/[[graph-traversal]]의 트리):

```
목표: 계정 탈취
├─ 비밀번호 훔치기
│  ├─ 피싱
│  ├─ 무차별 대입 → [[password-storage]]
│  └─ DB 유출 → [[injection]]
├─ 세션 하이재킹 → [[authn-authz-failures]]
└─ MFA 우회
```

공격 경로를 체계적으로 → 각 경로에 완화책.

## 언제 하나

- **설계 단계**: 가장 효과적 (구조 바꾸기 쉬움)
- **주요 변경 시**: 새 기능, 아키텍처 변경
- **정기적**: 위협은 진화 (새 공격, 새 의존성)
- 완벽한 한 번보다 지속적 반복

## 실무 관점

- **과하지 않게**: 시스템 규모에 맞게 (스타트업 vs 은행)
- **개발자 참여**: 보안팀만이 아니라 (설계 이해)
- **문서화**: 결정과 근거 기록
- **자동화 보조**: 위협 모델링 도구 (하지만 사고가 핵심)
- 심층 방어([[least-privilege]], [[xss-csrf]])와 결합

## 셀프 체크

> [!question]- 왜 "모든 것을 막기"가 아니라 위협 모델링을 하는가?
> 보안은 무한이라 모든 것을 막을 수 없다. 위협 모델링은 무엇이 중요하고 어디가 취약한지 체계적으로 파악해 한정된 자원을 우선순위대로 투입하기 위한 것. 설계 단계에서 하면 구조를 바꾸기 쉬워 비용도 적다.

> [!question]- 위협 모델링의 네 가지 질문은?
> (1) 무엇을 만드나(구조·데이터 흐름도), (2) 무엇이 잘못될 수 있나(STRIDE로 위협 열거), (3) 어떻게 대응하나(완화책), (4) 잘 했나(검증·반복). Shostack의 뼈대.

> [!question]- 신뢰 경계(trust boundary)가 왜 위협 분석의 초점인가?
> 신뢰 수준이 바뀌는 지점(사용자 입력→서버, 인터넷→내부망)에 위협이 집중되기 때문. 데이터 흐름도로 신뢰 경계를 넘는 데이터를 식별해 거기에 STRIDE를 적용한다.

> [!question]- STRIDE 여섯 범주와 각 대응 방향을 짝지으면?
> Spoofing→인증, Tampering→무결성(해시/서명), Repudiation→감사 로그·서명, Information disclosure→암호화, Denial of service→rate limit·확장, Elevation of privilege→인가·최소 권한.

> [!question]- 열거한 위협의 우선순위는 무엇으로 정하는가?
> 위험 = 가능성 × 영향(피해). 쉽고 피해 큰 것부터 다루며, DREAD·CVSS 등으로 정량화하고 완화 비용 대비 위험 감소를 따진다.

## 연습문제

> [!example]- 사용자 파일 업로드·공유 기능에 STRIDE를 적용해 위협을 각 범주별로 하나씩 열거하고 대응을 붙여라.
> **풀이**
> S(Spoofing): 남의 계정으로 위장 업로드 → 인증·세션 보호. T(Tampering): 저장된 파일 변조 → 해시/서명 무결성. R(Repudiation): "내가 안 올렸다" 부인 → 감사 로그. I(Information disclosure): 공유 링크로 남의 파일 유출(IDOR) → 소유권 인가 검사·암호화. D(DoS): 대용량 업로드 폭주 → rate limit·크기 제한·확장. E(EoP): 일반 사용자가 관리자 파일 접근 → 최소 권한·함수 레벨 접근 제어.

> [!example]- "계정 탈취"를 루트로 공격 트리를 그리고, 각 경로에 완화책을 매핑하라.
> **풀이**
> 계정 탈취 ← (1) 비밀번호 훔치기: 피싱→사용자 교육·MFA, 무차별 대입→rate limit·느린 해시([[password-storage]]), DB 유출→인젝션 방어·해시 저장. (2) 세션 하이재킹→HttpOnly/Secure·HTTPS·세션 재발급([[authn-authz-failures]]). (3) MFA 우회→피싱 저항 MFA(WebAuthn). 각 잎(경로)마다 완화책을 붙여 가장 값싸고 확실한 경로부터 차단.

> **점검 포인트**: (1) 자산·공격 표면·신뢰 경계의 의미, (2) STRIDE 여섯 범주와 대응, (3) 위험=가능성×영향으로 우선순위를 정하는 이유.

## 연결

- 각 위협의 대응이 다른 노트들:
  - 인증 → web/[[web-auth]], [[authn-authz-failures]]
  - 암호화 → [[crypto-basics]], [[tls-deep-dive]]
  - 무결성 → [[hashing]], [[digital-signatures]]
  - 최소 권한 → [[least-privilege]]
  - 입력 검증 → [[injection]], [[xss-csrf]]
- 공격 트리 = 그래프 → algorithms/[[graph-traversal]]
- 위험=가능성×영향 정량화 → math/[[probability-basics]]
