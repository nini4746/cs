# 디지털 서명 (Digital Signatures)

## 한 줄 요약

개인키로 서명하고 공개키로 검증해 "누가 보냈나 + 변조 안 됐나"를 증명한다. 비대칭 암호의 반대 방향 사용. 인증서(PKI)로 공개키 신뢰를 세우고, MAC과는 용도가 다르다.

## 왜 필요한가

- 메시지가 진짜 그 사람에게서 왔는지 증명
- TLS 인증서·JWT·코드 서명의 기반
- MAC과 서명의 차이

## 서명 = 비대칭의 반대 방향

[[crypto-basics]]의 비대칭 암호를 반대로:

- **암호화**: 공개키로 암호화 → 개인키로 복호화 (기밀성)
- **서명**: **개인키로 서명 → 공개키로 검증** (인증 + 무결성)

```
서명 (보내는 쪽):
  1. 메시지 해시 계산 ([[hashing]])
  2. 해시를 개인키로 서명 → 서명값
  3. 메시지 + 서명 전송

검증 (받는 쪽):
  1. 메시지 해시 계산
  2. 서명을 공개키로 검증 → 해시 복원
  3. 두 해시 일치? → 진짜 + 변조 없음
```

- **개인키는 소유자만** → 서명은 그 사람만 만들 수 있음 = **인증**
- 메시지 바뀌면 해시가 달라짐 = **무결성**
- **부인 방지(non-repudiation)**: 서명자가 "내가 안 했다" 못 함 (개인키는 자기만)

## 왜 해시를 서명하나

메시지 전체가 아니라 **해시([[hashing]])를 서명**:

- 비대칭 연산은 느림([[crypto-basics]]) → 큰 메시지 전체 서명은 비쌈
- 해시(고정 짧은 길이)를 서명 → 효율
- 해시의 충돌 저항이 서명 안전성 보장 (다른 메시지가 같은 해시 못 냄)

## 서명 알고리즘

- **RSA 서명**: RSA 기반
- **ECDSA**: 타원곡선 ([[crypto-basics]]의 ECC) - 짧은 키, 빠름
- **EdDSA (Ed25519)**: 현대 권장 (빠르고 안전, 구현 실수 적음)

## MAC vs 서명 (혼동 주의)

둘 다 무결성+인증이지만 **키가 다름**:

| | MAC (HMAC) | 디지털 서명 |
|---|---|---|
| 키 | 대칭 (공유 비밀) | 비대칭 (개인/공개) |
| 검증자 | 같은 키 아는 사람 | 누구나 (공개키) |
| 부인 방지 | ✗ (양쪽이 키 공유) | ✓ (개인키는 하나) |
| 속도 | 빠름 | 느림 |
| 용도 | 양쪽이 신뢰, 빠름 (API 토큰) | 공개 검증, 부인 방지 |

- **MAC**: 공유 키라 양쪽 다 만들 수 있음 → 부인 방지 안 됨. 하지만 빠름
- **서명**: 개인키는 한 명만 → 누구나 검증, 부인 방지. 하지만 느림
- JWT([[web/web-auth]])는 상황에 따라 HMAC(대칭) 또는 서명(비대칭)

## 공개키 신뢰: PKI

문제: 공개키가 **진짜 그 사람 것인지** 어떻게 믿나? (공격자가 자기 공개키를 "은행 것"이라 주장 가능) → **PKI(공개키 기반구조)**:

```
인증서 = 공개키 + 신원 + CA의 서명
CA(인증기관)가 신원 확인 후 서명 → "이 공개키는 이 도메인 것"
루트 CA는 브라우저/OS에 내장 (신뢰 앵커)
```

- 인증서 체인을 따라 검증 → network/[[tls]]의 인증서
- CA가 신뢰의 뿌리 → CA 침해 시 큰 문제 (실제 사고들)
- **web of trust**(PGP): CA 없이 사용자끼리 서명 (대안)

## 용도

- **TLS 인증서**: 서버 신원 증명 → network/[[tls]], [[tls-deep-dive]]
- **코드 서명**: 소프트웨어가 변조 안 됐나 (OS가 검증)
- **JWT 서명**: 토큰 위조 방지 → web/[[web-auth]]
- **이메일 서명**: DKIM → network/[[email-protocols]]
- **Git 커밋 서명**: 작성자 증명 → devops/[[git-internals]]
- **블록체인**: 트랜잭션 서명
- **패키지 서명**: 공급망 보안 → [[supply-chain]]

## 셀프 체크

> [!question]- 디지털 서명은 비대칭 암호를 어느 방향으로 쓰는가?
> 암호화와 반대. 암호화는 공개키로 암호화→개인키로 복호화(기밀성). 서명은 개인키로 서명→공개키로 검증(인증+무결성). 개인키는 소유자만 갖기에 그 사람만 서명을 만들 수 있다.

> [!question]- 메시지 전체가 아니라 해시를 서명하는 이유는?
> 비대칭 연산은 느려서 큰 메시지 전체 서명은 비싸다. 고정 길이의 짧은 해시를 서명하면 효율적이고, 해시의 충돌 저항이 서명 안전성을 보장한다(다른 메시지가 같은 해시를 못 냄).

> [!question]- MAC과 디지털 서명의 차이, 그리고 부인 방지가 서명에만 있는 이유는?
> MAC은 대칭(공유 비밀) 키, 서명은 비대칭 키를 쓴다. MAC은 양쪽이 같은 키를 공유해 둘 다 만들 수 있어 부인 방지가 안 된다. 서명은 개인키가 한 명에게만 있어 서명자가 "안 했다"고 못 한다(부인 방지).

> [!question]- PKI가 해결하는 문제는?
> 어떤 공개키가 진짜 그 주체의 것인지 신뢰하는 문제. CA가 신원 확인 후 (공개키+신원)에 서명해 인증서를 발급하고, 루트 CA는 브라우저/OS에 내장되어 신뢰 앵커가 된다.

## 연습문제

> [!example]- 웹훅 페이로드의 출처·무결성을 보장해야 한다. 외부 파트너가 다수라 각자 검증만 가능해야 하고 부인 방지가 필요하다. MAC과 서명 중 무엇을 택하고 왜인가?
> **풀이**
> 디지털 서명. 파트너별로 각자 공개키로 검증만 하면 되고(공유 비밀 배포 불필요), 개인키가 발신자 한 곳에만 있어 부인 방지가 성립한다. HMAC(대칭)이면 검증자가 키를 알아야 하므로 그 검증자도 위조가 가능해 부인 방지가 깨진다. 서버 간 신뢰가 이미 있고 속도가 중요하면 HMAC도 가능하지만, 다자 검증+부인 방지 요구엔 서명이 맞다.

> [!example]- CA 하나가 침해되면 어떤 위험이 생기고, 이를 감지/완화하는 장치는?
> **풀이**
> 침해된 CA가 임의 도메인에 대한 위조 인증서를 발급할 수 있어 중간자 공격이 가능해진다. 완화: (1) 인증서 투명성(CT) 로그 - 발급된 인증서가 공개 로그에 기록되어 부정 발급을 도메인 소유자가 감지, (2) 인증서 고정(pinning), (3) 폐지(OCSP/CRL). 신뢰의 뿌리인 CA는 단일 실패점이므로 다층 감시가 필요하다.
