# 의미 분석 (Semantic Analysis)

## 한 줄 요약

파싱(구문)이 끝난 뒤 "의미가 맞는가"를 검사하는 단계. 변수 선언-사용 매칭, 타입 검사, 스코프 해석 - 문법(CFG)으로는 못 잡는 것들을 처리한다. 계산 이론의 CFG 한계가 여기서 실무로 드러난다.

## 왜 필요한가

- 구문은 맞는데 의미가 틀린 오류 (선언 안 한 변수, 타입 불일치)
- 왜 파서(CFG)로 이걸 못 하나 → automata/[[cfl-pumping]]
- 컴파일러 프론트엔드의 마지막 검증

## 구문 vs 의미

- **구문(syntax)**: 문법에 맞나 (파서가 검사) → [[parsing]]
- **의미(semantics)**: 말이 되나

```
구문 OK, 의미 오류:
  int x = "hello";     # 타입 불일치
  y = 5;               # y 선언 안 됨
  foo(1, 2);           # foo가 인자 3개 요구
```

파서는 이걸 못 잡음 - 문법적으론 완벽. 의미 분석이 잡음.

## 왜 CFG로 못 하나

핵심: **변수 선언-사용 매칭은 문맥 자유를 넘어섬** → automata/[[cfl-pumping]]:

- "선언된 변수만 사용" = 같은 이름을 매칭 = {ww} 형태 → CFL 아님
- 파서(PDA, 스택 하나)는 무한한 이름 집합을 추적 못 함
- 그래서 **파싱 후 별도 단계**로 처리 (심볼 테이블 사용)

이게 automata/[[cfl-pumping]]에서 "aⁿbⁿcⁿ은 CFL 아님"이 실무로 나타나는 지점. 계산 이론의 계층이 컴파일러 단계를 강제.

## 심볼 테이블 (symbol table)

이름 → 정보(타입, 스코프, 위치) 매핑:

```
심볼 테이블:
  "x" → {type: int, scope: 함수, 선언 위치: 3줄}
  "foo" → {type: 함수(int,int)→int, ...}
```

- **선언**: 심볼 테이블에 추가 (중복 선언 검사)
- **사용**: 테이블 조회 (없으면 "선언 안 됨" 오류)
- data-structures/[[hash-tables]]로 구현 (이름 조회 O(1))

## 스코프 해석 (scope resolution)

각 이름 사용이 **어느 선언을 가리키나** ([[scope-and-closures]]의 렉시컬 스코프):

- 스코프마다 심볼 테이블 (중첩 → 테이블 스택/체인)
- 이름 조회: 현재 스코프 → 바깥 스코프 → ... (렉시컬 체인)
- 섀도잉(안쪽이 바깥 이름 가림), 클로저 캡처 결정 → [[scope-and-closures]]
- **변수 해석 최적화**: 이름을 슬롯 인덱스로 미리 변환 → 런타임 조회를 배열 인덱싱으로 (tree-walking 가속)

## 타입 검사 (type checking)

정적 타입 언어에서 타입 규칙 검증 ([[static-vs-dynamic-typing]]):

- **타입 추론**: 표현식의 타입 계산 (`2 + 3` → int)
- **타입 일치**: 대입·인자·연산의 타입 확인
- **타입 오류**: 불일치를 컴파일 타임에 보고
- 동적 언어는 이 단계 생략 (런타임에 검사)

AST를 순회하며 각 노드의 타입을 아래에서 위로 계산 (bottom-up).

## 그 밖의 의미 검사

- **제어 흐름**: 모든 경로가 값 반환하나, 도달 불가 코드
- **상수성**: const에 대입 안 하나
- **초기화**: 변수를 쓰기 전 초기화했나 (Rust의 엄격한 검사)
- **접근 제어**: private 멤버 접근 규칙
- **빌림 검사**: Rust의 소유권/수명 → [[memory-management-models]]

## 정적 분석의 한계

의미 분석은 **컴파일 타임에 결정 가능한 것만**:

- "이 변수가 null일 수 있나"는 일반적으로 **결정 불가** (정지 문제 → automata/[[decidability]])
- 그래서 보수적 근사 (거짓 양성 허용) 또는 타입 시스템으로 제약 (Option → [[error-handling-models]])
- 완벽한 정적 검사는 원리적으로 불가 → automata/[[decidability]]의 Rice 정리

## 파이프라인 위치

```
소스 → 렉싱 → 파싱 → AST → [의미 분석] → 중간 표현 → 최적화 → 코드 생성
       DFA    PDA           심볼테이블+타입    → [[codegen-and-optimization]]
                            CFG 넘어섬
```

여기까지가 **프론트엔드**(언어 의존). 이후 **백엔드**(타겟 의존) → [[codegen-and-optimization]].

## 셀프 체크

> [!question]- 구문(syntax)과 의미(semantics)의 차이를, 파서가 통과시키지만 의미 분석이 잡는 예로 설명하라.
> 구문은 문법에 맞는가(파서 검사), 의미는 말이 되는가이다. `int x = "hello";`(타입 불일치), `y = 5;`(y 미선언), `foo(1,2);`(인자 개수 불일치)는 전부 문법적으로 완벽해 파서를 통과하지만 의미가 틀렸다. 의미 분석 단계가 심볼 테이블과 타입 규칙으로 이를 잡는다.

> [!question]- "선언된 변수만 사용"을 왜 파서(CFG)로 검사할 수 없나?
> 같은 이름의 선언과 사용을 매칭하는 것은 {ww} 형태로, 문맥 자유 언어(CFL)를 넘어선다. PDA는 스택 하나뿐이라 무한한 이름 집합을 추적하지 못한다. 그래서 파싱 후 별도 단계에서 심볼 테이블로 처리한다. 이것이 `aⁿbⁿcⁿ`이 CFL이 아니라는 이론이 실무로 드러나는 지점이다.

> [!question]- 심볼 테이블은 무엇을 저장하며 선언과 사용에서 각각 어떻게 쓰이나?
> 이름 → 정보(타입, 스코프, 선언 위치) 매핑이다. 선언 시 테이블에 추가하며 중복 선언을 검사하고, 사용 시 테이블을 조회해 없으면 "선언 안 됨" 오류를 낸다. 해시 테이블로 구현해 조회가 O(1)이다.

> [!question]- 중첩 스코프에서 이름 조회는 어떻게 이루어지나? 섀도잉은 무엇인가?
> 스코프마다 심볼 테이블을 두고 중첩을 테이블 스택/체인으로 표현한다. 이름 조회는 현재 스코프 → 바깥 스코프 → ... 순으로 렉시컬 체인을 따라간다. 섀도잉은 안쪽 스코프의 이름이 바깥의 같은 이름을 가리는 것이다.

> [!question]- 정적 분석이 원리적으로 완벽할 수 없는 이유는?
> "이 변수가 null일 수 있나" 같은 질문은 일반적으로 결정 불가능하다(정지 문제). Rice 정리에 따라 프로그램의 의미적 성질을 완벽히 판정하는 것은 불가능하므로, 보수적 근사(거짓 양성 허용)를 쓰거나 타입 시스템(Option 등)으로 애초에 제약한다.

## 연습문제

> [!example]- 문제: 아래 코드를 위에서 아래로 훑으며 심볼 테이블 상태 변화를 추적하고, 어느 줄에서 어떤 의미 오류가 나는지 밝혀라.
> ```
> 1: int a = 1;
> 2: int a = 2;
> 3: b = a + 1;
> ```
> **풀이**
> - 1줄: 선언 `a`. 테이블 = `{a: int}`. OK
> - 2줄: 선언 `a`. 이미 테이블에 `a` 존재 → **중복 선언 오류**
> - 3줄: `b`에 대입하는데 `b`가 테이블에 없음 → **미선언 사용 오류**(`a`는 조회 성공). 
> 파서는 세 줄 다 문법적으로 통과시키지만, 심볼 테이블 기반 의미 분석이 2줄의 중복 선언과 3줄의 미선언을 잡는다.

> [!example]- 문제: `int x = 2 + 3;`에서 우변의 타입을 AST 순회로 bottom-up 계산하고, 대입이 타입 검사를 통과하는지 판정하라.
> **풀이**
> AST: `Assign(int x, BinOp('+', Number(2), Number(3)))`.
> - 잎부터: `Number(2)` → int, `Number(3)` → int
> - `BinOp('+', int, int)`: `+`의 타입 규칙 int+int→int → 우변 타입 = int
> - 대입: 선언 타입 int == 우변 타입 int → **타입 일치, 통과**
> 만약 우변이 `"hi"`(string)였다면 int ≠ string으로 타입 오류. 타입은 이렇게 잎에서 위로 계산되어 각 노드에 부여된다. 동적 언어는 이 단계를 생략하고 런타임에 검사한다.

## 파인만

> [!note]- 백지에 "왜 의미 분석이 파싱과 별도의 단계로 존재해야 하는가"를 계산 이론과 엮어 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: (1) 변수 선언-사용 매칭이 CFL을 넘어서므로 PDA(파서)로는 불가능하다는 것, (2) 심볼 테이블이 선언/사용/스코프를 어떻게 추적하는지, (3) 정적 분석은 결정 가능한 것만 다루고 나머지는 보수적 근사나 타입 시스템으로 제약한다는 것 - 이 셋을 설명할 수 있어야 한다.
