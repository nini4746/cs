# 렉싱 (Lexing / Tokenization)

## 한 줄 요약

소스 문자 스트림을 의미 있는 토큰(숫자, 연산자, 키워드)으로 쪼개는 컴파일러 첫 단계. 토큰은 정규 언어라서 렉서는 본질적으로 DFA다. 파서가 문자가 아닌 토큰을 다루게 해준다.

## 왜 필요한가

- 컴파일러/인터프리터의 첫 단계
- 왜 렉서가 오토마타(DFA)인가 → automata/[[dfa-nfa]]
- 파싱을 단순하게 만드는 전처리

## 역할: 문자 → 토큰

렉서(scanner/tokenizer)는 **원시 문자열을 토큰 열로**:

```
"12 + 34 * (5 - 6)"
→ [NUM(12), OP(+), NUM(34), OP(*), OP((), NUM(5), OP(-), NUM(6), OP())]
```

실측 (간단한 산술식 렉서):
```
[('NUM','12'), ('OP','+'), ('NUM','34'), ('OP','*'), ('OP','('),
 ('NUM','5'), ('OP','-'), ('NUM','6'), ('OP',')')]
```

- 공백·주석 버림
- 숫자 여러 글자를 하나의 NUM 토큰으로 묶음
- 각 토큰 = (종류, 값) - 파서가 다루기 쉬운 단위

## 토큰 종류

- **리터럴**: 숫자(123, 3.14), 문자열("hi"), 불리언
- **식별자**: 변수·함수 이름 (foo, count)
- **키워드**: 예약어 (if, while, return) - 식별자와 구분
- **연산자/구두점**: `+ - * / ( ) { } ; ,`
- 각 토큰에 위치 정보(줄, 열) 첨부 → 오류 메시지용

## 왜 렉서는 DFA인가

토큰의 패턴은 **정규 언어** (automata/[[dfa-nfa]], [[regex-and-finite-automata]]):

- 숫자 = `[0-9]+`, 식별자 = `[a-zA-Z_][a-zA-Z0-9_]*`, 연산자 = 고정 문자
- 이들은 전부 정규식으로 표현 → DFA로 인식
- 렉서 = 각 문자를 읽으며 상태 전이하는 유한 오토마타

```
숫자 인식 DFA: 시작 →(digit)→ 숫자상태 →(digit, 루프)→ ... →(non-digit)→ 토큰 완성
```

렉서 생성기(lex/flex)는 정규식 명세를 받아 **DFA를 자동 생성** → automata/[[regex-and-finite-automata]]의 정규식→NFA→DFA. 계산 이론이 실무 도구가 되는 지점.

## 최장 일치 (maximal munch)

렉서의 핵심 규칙: **가능한 가장 긴 토큰을 취함**:

```
"123" → NUM(123)  (1, 12, 123 중 가장 긴 것)
">=" → GTE 하나  (> 와 = 로 안 쪼갬)
"foreach" → 식별자 하나  (for + each 아님)
```

DFA가 계속 진행하다 더 못 가면(non-accepting) 마지막 수락 지점까지가 토큰. 이 규칙이 대부분 애매함을 해결.

## 키워드 vs 식별자

`if`, `while`은 식별자 패턴에도 맞음. 처리:
- 식별자로 렉싱한 뒤 **키워드 테이블 조회** → 키워드면 키워드 토큰으로
- 또는 렉서 DFA에 키워드를 직접 인코딩 (상태 폭발)
- 보통 전자 (식별자 인식 후 해시 조회 → data-structures/[[hash-tables]])

## 렉서가 처리하는 성가신 것들

- **공백·주석**: 버림 (하지만 위치 추적은 유지)
- **문자열 이스케이프**: `"a\"b"`의 `\"` 처리
- **숫자 형식**: 정수, 실수(computer-architecture/[[floating-point]]), 16진수, 지수
- **오류**: 알 수 없는 문자를 위치와 함께 보고
- **문맥 의존**: 일부 언어는 렉서가 파서 정보 필요 (C의 typedef, 나머지는 렉서/파서 분리 깔끔)

## 파이프라인에서의 위치

```
소스 문자열
  → [렉서: DFA]        ← 여기 (문자 → 토큰)
  → [파서: PDA]        → compilers/[[parsing]]
  → AST
  → ...
```

렉서(정규/DFA)와 파서(문맥자유/PDA)의 분리 = automata 계층의 실무 반영. 렉서가 문자 잡음을 제거해 파서가 구조에 집중.

## 셀프 체크

> [!question]- 렉서가 본질적으로 DFA인 이유는? 어떤 종류의 토큰까지 이 방식으로 인식할 수 있나?
> 토큰의 패턴(숫자, 식별자, 연산자)이 전부 정규 언어이기 때문. 정규식으로 표현되는 것은 NFA를 거쳐 DFA로 변환 가능하므로, 렉서는 각 문자를 읽으며 상태 전이하는 유한 오토마타로 구현된다. 중첩 괄호처럼 스택이 필요한 문맥 자유 구조는 렉서가 못 잡고 파서(PDA)로 넘긴다.

> [!question]- 최장 일치(maximal munch) 규칙이란 무엇이고, `>=`나 `foreach` 같은 입력을 어떻게 처리하나?
> 가능한 가장 긴 토큰을 취하는 규칙. DFA가 계속 전이하다 더 못 가면 마지막 수락 지점까지를 하나의 토큰으로 확정한다. `>=`는 `>`와 `=`로 쪼개지 않고 GTE 하나로, `foreach`는 `for`+`each`가 아닌 식별자 하나로 처리된다.

> [!question]- `if`, `while` 같은 키워드는 식별자 패턴에도 맞는데 어떻게 구분하나?
> 보통 먼저 식별자로 렉싱한 뒤 키워드 테이블(해시 조회)을 참조해, 예약어면 키워드 토큰으로 바꾼다. DFA에 키워드를 직접 인코딩할 수도 있으나 상태가 폭발하므로 대개 테이블 조회 방식을 쓴다.

> [!question]- 렉서가 버리는 것과 유지하는 것은 각각 무엇인가?
> 공백과 주석은 버린다. 하지만 각 토큰의 위치 정보(줄, 열)는 유지해 오류 메시지에 쓴다.

## 연습문제

> [!example]- 문제: 식별자와 정수를 인식하는 렉서의 DFA 상태 전이를, 입력 `foo42 7`을 따라가며 추적하라. 최장 일치가 어디서 적용되는지 표시할 것.
> **풀이**
> 상태: START, IDENT(식별자 진행), NUM(숫자 진행).
> - `f`: START →(letter)→ IDENT
> - `o`,`o`: IDENT →(letter/digit, 루프)→ IDENT
> - `4`,`2`: IDENT →(digit, 루프)→ IDENT (식별자 패턴 `[a-zA-Z_][a-zA-Z0-9_]*`는 뒤에 숫자 허용)
> - ` `(공백): IDENT에서 더 못 감 → 마지막 수락 지점까지 = `foo42`를 IDENT 토큰으로 확정 (최장 일치)
> - 공백 버림, START로
> - `7`: START →(digit)→ NUM
> - 입력 끝: NUM 수락 → `7`을 NUM 토큰으로 확정
> 결과: `[IDENT("foo42"), NUM("7")]`. `foo42`가 `foo`와 `42`로 안 쪼개진 게 최장 일치의 효과.

> [!example]- 문제: `12 + 34 * (5 - 6)`를 (종류, 값) 토큰 열로 손으로 렉싱하고, 각 단계에서 공백 처리와 숫자 묶기가 어떻게 일어나는지 서술하라.
> **풀이**
> 왼쪽부터: `1`,`2`를 연속 digit으로 하나의 NUM으로 묶음 → NUM(12). 공백 버림. `+` → OP(+). 공백. `3`,`4` → NUM(34). 공백. `*` → OP(*). 공백. `(` → OP((). `5` → NUM(5). 공백. `-` → OP(-). 공백. `6` → NUM(6). `)` → OP()).
> 결과: `[('NUM','12'),('OP','+'),('NUM','34'),('OP','*'),('OP','('),('NUM','5'),('OP','-'),('NUM','6'),('OP',')')]`. 여러 글자 숫자는 최장 일치로 한 토큰이 되고, 공백은 토큰 경계 역할만 하고 버려진다.

## 파인만

> [!note]- 백지에 "왜 렉서와 파서를 분리하고, 렉서가 왜 DFA로 충분한가"를 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: (1) 토큰 패턴이 정규 언어라서 DFA로 인식 가능하다는 것, (2) 최장 일치 규칙으로 애매함을 해소한다는 것, (3) 렉서가 문자 잡음(공백/주석)을 제거해 파서가 구조에만 집중하게 한다는 것 - 이 셋을 스스로 설명할 수 있어야 한다.
