# 계층형 아키텍처 (Layered Architecture)

## 한 줄 요약

시스템을 책임별 계층(표현·비즈니스·데이터)으로 나누고 의존성을 한 방향으로 흐르게 한다. 헥사고날/클린 아키텍처는 의존성을 안쪽(도메인)으로 향하게 뒤집어 비즈니스 로직을 인프라에서 격리한다.

## 왜 필요한가

- 큰 시스템을 어떻게 구조화하나
- 의존성 방향이 왜 중요한가
- 클린 아키텍처가 뭘 뒤집나

## 계층형 (layered / n-tier)

시스템을 **책임별 수평 계층**으로:

```
표현(Presentation) - UI, API 엔드포인트 (web/[[rest-design]])
    ↓
비즈니스(Business/Domain) - 핵심 로직, 규칙
    ↓
데이터(Data/Persistence) - DB 접근 (database/)
    ↓
데이터베이스 / 외부
```

- 각 계층이 아래 계층에만 의존 (단방향)
- 관심사 분리 (표현이 DB를 직접 안 봄)
- network/[[internet-overview]]의 계층 모델과 같은 발상 (추상화 계층)

문제: **의존성이 아래(인프라)로** 흐름 → 비즈니스 로직이 DB·프레임워크에 의존 → 인프라 바뀌면 도메인도 영향.

## 의존성 방향의 문제

전통 계층형의 약점:
- 비즈니스 로직이 데이터 계층(구체 DB)에 의존 → DB 교체 어려움
- 프레임워크·인프라가 도메인에 스며듦
- 도메인 로직 테스트에 DB 필요 ([[testing-strategy]])
- **핵심 자산(비즈니스 규칙)이 세부(인프라)에 종속** = 거꾸로

## 헥사고날 / 클린 아키텍처

의존성을 **뒤집어** 도메인을 중심에:

```
        [UI]  [DB]  [외부 API]   ← 세부(어댑터)
          ↓     ↓      ↓
       ┌─────────────────┐
       │   비즈니스 도메인   │  ← 중심 (아무것도 의존 안 함)
       └─────────────────┘
    의존성이 안쪽(도메인)을 향함
```

- **의존성 역전**([[solid]]의 DIP): 도메인이 인프라 인터페이스(포트)를 정의, 인프라(어댑터)가 구현
- **도메인이 중심, 아무것도 의존 안 함**: 프레임워크·DB·UI를 몰라도 됨
- 세부(DB·UI)는 **교체 가능한 어댑터** ([[structural-patterns]]의 어댑터)
- 이름: 헥사고날(ports & adapters), 클린 아키텍처(Uncle Bob), 오니언

이점:
- **비즈니스 로직 격리**: 인프라 없이 테스트 (DB 목 → [[testing-strategy]])
- **교체 자유**: DB·프레임워크·UI 바꿔도 도메인 무관
- **핵심을 세부에서 보호**: 가장 중요한 것(도메인)이 가장 안정적

## 트레이드오프

클린 아키텍처가 항상 정답은 아님:

- **비용**: 포트·어댑터·인터페이스 추가 → 보일러플레이트 ([[deep-modules]]의 얕은 모듈 위험)
- **작은 앱엔 과함**: CRUD 앱에 클린 아키텍처는 오버엔지니어링
- **적합**: 복잡한 도메인 로직, 오래 갈 시스템, 인프라 교체 가능성
- 단순 앱은 계층형(또는 그냥 MVC)으로 충분

## MVC / MVVM

UI 쪽 계층 분리 패턴:
- **MVC**: Model(데이터)-View(표현)-Controller(조율)
- **MVVM**: Model-View-ViewModel (반응형 바인딩, web 프론트엔드)
- 표현 계층 내부의 관심사 분리

## 계층화의 일반 원리

network/[[internet-overview]], os와 공통:
- **관심사 분리**: 각 계층이 하나의 책임 ([[coupling-cohesion]])
- **추상화**: 위 계층이 아래 세부를 몰라도 됨
- **교체 가능**: 계층 내부 바꿔도 인터페이스 유지하면 무관
- 계층화는 복잡도 관리의 보편 도구 ([[deep-modules]])
