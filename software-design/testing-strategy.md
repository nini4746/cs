# 테스트 전략 (Testing Strategy)

## 한 줄 요약

단위·통합·E2E 테스트를 피라미드로 균형 잡는다 - 빠르고 많은 단위 테스트 아래, 느리고 적은 E2E 위. 무엇을 테스트하고 무엇을 안 할지가 핵심이며, 테스트는 리팩토링의 안전망이다.

## 왜 필요한가

- 무엇을 어떻게 테스트하나
- 테스트 피라미드가 뭔가
- 과도한 테스트를 피하는 법

## 왜 테스트

- **회귀 방지**: 고친 게 다시 안 깨지게 (algorithms/[[correctness-proofs]]의 "테스트는 버그 존재를 보임")
- **리팩토링 안전망**([[refactoring-catalog]]): 동작 보존 확인
- **문서**: 테스트가 사용 예시
- **설계 압박**: 테스트하기 쉬운 코드 = 결합 낮은 코드 ([[coupling-cohesion]])

주의: 테스트는 **버그의 존재만 증명**, 부재는 못 함 (algorithms/[[correctness-proofs]]) → 증명은 별개.

## 테스트 피라미드

테스트를 세 층으로, 비율을 피라미드로:

```
        /\  E2E (적게, 느림, 비쌈)
       /  \
      /────\ 통합 (중간)
     /      \
    /────────\ 단위 (많이, 빠름, 쌈)
```

### 단위 테스트 (unit)

- **한 함수·클래스를 격리** 테스트
- 빠름 (ms), 많이 (수천 개), 정확한 실패 위치
- 외부 의존(DB·네트워크)을 **목(mock)**으로 대체 → DI 필요 ([[solid]], [[creational-patterns]]의 싱글턴 문제)
- 순수 함수(programming-languages/[[functional-programming]])가 테스트 쉬움 (입력→출력)

### 통합 테스트 (integration)

- **여러 컴포넌트가 함께** 동작하는지 (DB, API 연동)
- 실제 의존 일부 사용 (테스트 DB 등)
- 단위보다 느리고 적게
- 단위가 못 잡는 연동 버그 (인터페이스 불일치)

### E2E 테스트 (end-to-end)

- **전체 시스템을 사용자처럼** (브라우저 클릭 → DB까지)
- 가장 현실적이지만 느리고(초~분), 불안정(flaky), 실패 위치 불명확
- 핵심 시나리오만 (전부 하면 느리고 깨지기 쉬움)

## 피라미드의 논리

- **아래로 갈수록 많이**: 단위가 빠르고 정확 → 대부분을 단위로
- **위로 갈수록 적게**: E2E는 비싸니 핵심만
- 거꾸로(E2E 많고 단위 적음) = **아이스크림콘 안티패턴** (느리고 불안정)
- 균형: 빠른 피드백(단위) + 현실성(E2E 일부)

## 테스트 더블 (test doubles)

외부 의존을 대체:
- **목(mock)**: 호출 검증 (이 메서드가 불렸나)
- **스텁(stub)**: 정해진 값 반환
- **페이크(fake)**: 간단한 실제 구현 (인메모리 DB)
- 과도한 목 주의: 구현에 묶여 리팩토링 시 다 깨짐 (동작 아니라 구현 테스트)

## 무엇을 테스트하지 말 것 (중요)

과도한 테스트도 나쁨:

- **구현 세부**: private 메서드, 내부 상태 → 리팩토링 시 깨짐 (동작을 테스트해야)
- **자명한 것**: getter/setter, 프레임워크 코드
- **외부 라이브러리**: 이미 테스트됨 (내 코드만)
- **100% 커버리지 집착**: 커버리지는 지표지 목표 아님 (의미 없는 테스트 양산)
- 테스트도 유지보수 비용 → 가치 있는 것만

## 좋은 테스트

- **빠름**: 자주 돌릴 수 있게
- **독립**: 순서·서로 무관 (병렬 가능)
- **결정적**: 항상 같은 결과 (flaky 금지 - 랜덤·시간·네트워크 의존 제거)
- **명확**: 실패 시 뭐가 문제인지 (좋은 이름 [[naming-and-readability]])
- **동작 테스트**: 무엇을 하나 (어떻게 아니라)

## TDD와의 관계

테스트 먼저 쓰기 → [[tdd]] (별도 노트). 테스트 전략은 TDD 여부와 별개로 필요.

## CI 통합

- 테스트를 CI에서 자동 실행 (devops/[[ci-cd-principles]])
- push·PR마다 → 회귀 즉시 발견
- 빠른 단위는 항상, 느린 E2E는 선택적

## 셀프 체크

> [!question]- 테스트 피라미드의 세 층과 "아래로 갈수록 많이"의 논리를 설명하라.
> 아래부터 단위(많고 빠르고 쌈), 통합(중간), E2E(적고 느리고 비쌈). 단위는 빠르고 실패 위치가 정확해 대부분을 여기서 잡고, E2E는 비싸니 핵심 시나리오만 둔다. 빠른 피드백과 현실성의 균형이다.

> [!question]- 아이스크림콘 안티패턴이란?
> 피라미드가 뒤집혀 E2E가 많고 단위가 적은 형태다. 느리고 flaky하며 실패 위치가 불명확해 피드백이 나쁘다. 단위로 잡을 수 있는 걸 비싼 E2E로 잡는 낭비다.

> [!question]- 무엇을 테스트하지 말아야 하나?
> 구현 세부(private 메서드·내부 상태 - 리팩토링 시 깨짐), 자명한 것(getter/setter·프레임워크 코드), 이미 테스트된 외부 라이브러리, 100% 커버리지 집착(의미 없는 테스트 양산). 테스트도 유지보수 비용이라 가치 있는 것만 한다.

## 연습문제

> [!example]- 문제: 어떤 팀의 테스트가 대부분 브라우저 E2E라 CI가 40분 걸리고 자주 flaky하게 실패한다. 재균형 전략을 세워라.
> **풀이**
> 아이스크림콘 안티패턴.
> 로직·규칙 검증은 빠른 단위 테스트로 대거 내린다(외부 의존은 목/페이크로 격리 → DI 필요). 컴포넌트 연동은 통합 테스트로 다룬다. E2E는 결제·로그인 같은 핵심 사용자 시나리오 소수만 남긴다. flaky 원인(시간·랜덤·네트워크 의존)을 제거해 결정적으로 만든다. 결과적으로 빠르고 안정적인 피라미드가 된다.

> [!example]- 문제: 목을 많이 쓴 단위 테스트들이 내부 구현을 그대로 검증해, 동작을 안 바꾼 리팩토링에도 대량으로 깨진다. 원인과 개선책은?
> **풀이**
> 동작이 아니라 구현을 테스트하고 있다(과도한 목 - 호출 순서·내부 협력을 검증).
> 개선: 입력→출력의 관찰 가능한 동작을 검증하도록 바꾼다. 목 대신 상태 검증이나 인메모리 페이크를 쓰고, 정말 상호작용이 계약인 경계에서만 목을 쓴다. 그러면 내부를 리팩토링해도 테스트가 살아남는다.

## 파인만

> [!note]- 백지에 테스트 피라미드를 그리고 각 층의 역할·비율과 "무엇을 테스트 안 할지"까지 설명해보라. 막히면 그 부분만 다시.
> **점검 포인트**: (1) 단위·통합·E2E의 특성과 비율 논리, (2) 테스트 더블과 과도한 목의 위험, (3) 좋은 테스트의 조건(빠름·독립·결정적·동작 테스트)과 테스트하지 말 것.
