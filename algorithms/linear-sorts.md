# 선형 시간 정렬 (Linear Sorts)

## 한 줄 요약

비교를 안 하고 키의 값 자체를 인덱스로 쓰면 O(n)에 정렬할 수 있다. counting/radix/bucket 정렬이 그것. 단, 키의 범위나 분포에 제약이 있어 만능은 아니다.

## 왜 필요한가

- 비교 정렬의 Ω(n log n) 하한([[comparison-sorts]])을 어떻게 뚫나
- 언제 O(n) 정렬이 가능한가
- 정수/문자열 대량 정렬 최적화

## 하한을 뚫는 법: 비교를 안 한다

[[comparison-sorts]]의 Ω(n log n)은 **비교 기반**에만 적용. 비교 대신 **키 값을 직접 인덱스로** 쓰면 그 하한이 무의미:

- 비교 정렬: "a[i] < a[j]?"를 물음 → 결정 트리 → log(n!)
- 선형 정렬: 키를 세거나 배열 위치로 매핑 → 비교 없음

대가: 키에 제약이 생김 (범위, 정수성).

## counting sort (계수 정렬)

키 범위가 작을 때. 각 값의 **개수를 세서** 배치:

```
1. 각 값의 등장 횟수를 세기 (count[값]++)
2. 누적합으로 각 값의 최종 위치 계산
3. 원소를 그 위치에 배치
```

- **O(n + k)** (k = 값의 범위). k가 n 규모면 O(n)
- 조건: 키가 **작은 정수 범위** (0~k). k가 크면(예: 32비트 전체) 메모리 폭발
- **안정적** (같은 값 순서 유지) → radix의 부품으로 쓰임
- 예: 시험 점수(0~100), 나이 정렬

## radix sort (기수 정렬)

큰 정수/문자열을 **자릿수별로** 정렬. counting sort를 자릿수마다:

```
LSD (최하위 자리부터):
  1의 자리로 안정 정렬 → 10의 자리로 → 100의 자리로 → ...
```

- 각 자릿수를 counting sort(안정)로 → 전체 **O(d·(n+b))** (d=자릿수, b=진법)
- d가 상수면 O(n). 32비트 정수를 8비트씩 4패스 → 4×O(n) = O(n)
- 조건: 고정 길이 키 (정수, 고정 문자열)
- **안정성이 핵심**: 상위 자리 정렬이 하위 자리 순서를 보존해야 (그래서 counting sort 사용)

대용량 정수 정렬에서 퀵정렬보다 빠를 수 있음 (비교 없이 O(n)).

## bucket sort (버킷 정렬)

키가 **균등 분포**일 때. 구간별 버킷에 나눠 담고 각 버킷 정렬:

```
1. [0,1) 구간을 n개 버킷으로
2. 각 원소를 해당 버킷에 (값 × n)
3. 각 버킷 내부 정렬 (삽입 정렬 등)
4. 버킷 순서대로 이어붙임
```

- 균등 분포면 버킷당 원소 ~1개 → **평균 O(n)**
- 분포가 치우치면 한 버킷에 몰려 O(n²)로 퇴화
- 조건: 분포를 알고 균등해야

## 선형 정렬의 제약 정리

| 정렬 | 조건 | 복잡도 |
|---|---|---|
| counting | 작은 정수 범위 k | O(n+k) |
| radix | 고정 길이 정수/문자열 | O(d(n+b)) |
| bucket | 균등 분포 | O(n) 평균 |

**만능 아님**: 임의 비교 가능 객체(커스텀 비교 함수)엔 못 씀 → 그땐 비교 정렬. 키가 정수/고정길이/균등이라는 **구조를 알 때만** O(n) 가능.

## 언제 쓰나

- **정수 대량 정렬**: radix가 퀵정렬 이길 수 있음
- **작은 범위 키**: counting (점수, 나이, 색상)
- **DB/빅데이터**: 정수 키 정렬, 기수 기반 파티셔닝
- 실무 기본은 여전히 하이브리드 비교 정렬(범용성), 선형 정렬은 조건 맞을 때 특화 최적화

## 셀프 체크

> [!question]- 비교 정렬의 Ω(n log n) 하한을 선형 정렬이 어떻게 우회하나?
> Ω(n log n) 하한은 "a[i] < a[j]?" 같은 비교 기반 정렬에만 적용된다. 선형 정렬은 비교 대신 키 값을 직접 세거나 배열 위치로 매핑하므로 비교를 하지 않고, 그 하한이 무의미해진다. 대가로 키의 범위나 정수성 같은 제약이 생긴다.

> [!question]- counting sort의 복잡도가 O(n+k)인데 k가 커지면 왜 문제가 되나?
> k는 값의 범위이고 개수를 세는 배열의 크기다. k가 n 규모면 O(n)이지만, 32비트 전체처럼 k가 크면 count 배열이 그만큼 커져 메모리가 폭발한다. 그래서 counting sort는 작은 정수 범위(0~k)에서만 실용적이다.

> [!question]- radix sort에서 안정성이 왜 핵심인가?
> LSD radix는 하위 자리부터 정렬하면서 상위 자리로 올라간다. 상위 자리를 정렬할 때 하위 자리에서 만든 순서가 보존되어야 최종 결과가 맞다. 안정 정렬(같은 값의 상대 순서 유지)이 이 보존을 보장하므로 각 자릿수 정렬에 안정적인 counting sort를 쓴다.

> [!question]- bucket sort는 언제 O(n)이고 언제 O(n²)로 퇴화하나?
> 키가 균등 분포이면 각 버킷에 원소가 평균 1개씩 들어가 평균 O(n)이다. 분포가 치우치면 한 버킷에 원소가 몰려 그 버킷 내부 정렬이 O(n²)가 되어 전체가 퇴화한다. 그래서 분포가 균등하다는 것을 알 때만 쓴다.

## 연습문제

> [!example]- 문제: counting sort가 안정 정렬이 되도록 하는 의사코드를 작성하고 왜 안정적인지 설명하라.
> **풀이**
> ```
> count[0..k] = 0
> for x in A: count[x]++            # 등장 횟수
> for i in 1..k: count[i] += count[i-1]  # 누적합 = 최종 위치
> for i from n-1 downto 0:          # 뒤에서부터 순회가 핵심
>   out[count[A[i]] - 1] = A[i]
>   count[A[i]]--
> ```
> 입력을 뒤에서부터 순회하고 누적합 위치에서 감소시키므로, 같은 값이면 원래 뒤에 있던 원소가 뒤 위치에 배치된다. 즉 원래 순서가 보존되어 안정적이다.

> [!example]- 문제: 32비트 정수를 8비트씩 4패스로 radix sort할 때 전체 복잡도가 O(n)임을 유도하라.
> **풀이**
> 각 패스는 8비트 자릿수에 대한 counting sort로, 진법 b=2^8=256이므로 O(n+b)=O(n+256).
> 자릿수 수 d = 32/8 = 4 (상수).
> 전체 = O(d·(n+b)) = 4·O(n+256) = O(n) (d와 b가 상수이므로).
> 따라서 자릿수가 고정이고 진법이 상수면 radix는 선형이다.

> [!example]- 문제: "모든 정렬은 선형 정렬로 대체할 수 있다"는 주장의 반례를 들어라.
> **풀이**
> 커스텀 비교 함수로만 순서가 정의되는 객체를 정렬하는 경우가 반례다. 예를 들어 임의의 비교 규칙을 가진 객체 집합은 키를 정수 인덱스나 자릿수로 매핑할 방법이 없다.
> 선형 정렬은 키가 정수/고정 길이/균등 분포라는 구조를 알 때만 가능하다. 그런 구조가 없으면 비교 정렬(Ω(n log n))을 써야 하므로 주장은 거짓이다.
