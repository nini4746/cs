# 비교 정렬 (Comparison Sorts)

## 한 줄 요약

원소를 비교해서 정렬하는 알고리즘. merge/quick/heap이 O(n log n)이고, 비교만으로는 Ω(n log n)보다 빠를 수 없다는 하한이 증명된다. 실무 정렬은 이들을 조합한다.

## 왜 필요한가

- 가장 많이 쓰는 알고리즘. 다른 것의 부품
- O(n log n)과 O(n²)의 실전 차이
- 왜 비교 정렬은 O(n log n)이 한계인가

## O(n log n) 삼총사

| 정렬 | 방식 | 최악 | 평균 | 공간 | 안정성 |
|---|---|---|---|---|---|
| **병합(merge)** | 분할 후 병합 | O(n log n) | O(n log n) | O(n) | 안정 |
| **퀵(quick)** | 피벗 분할 | **O(n²)** | O(n log n) | O(log n) | 불안정 |
| **힙(heap)** | 힙 추출 | O(n log n) | O(n log n) | O(1) | 불안정 |

### 병합 정렬

절반씩 나눠 각각 정렬 후 병합. T(n)=2T(n/2)+O(n) → O(n log n) ([[recurrences]]):
- **안정적**, 최악도 O(n log n) 보장
- O(n) 추가 공간 필요 (병합 버퍼)
- 외부 정렬(디스크), 연결 리스트 정렬에 적합

### 퀵 정렬

피벗을 골라 작은 것/큰 것으로 분할, 재귀:
- **평균 O(n log n)에 상수가 작아 실전 최고속** (제자리, 캐시 친화 → [[memory-hierarchy]])
- **최악 O(n²)**: 피벗이 계속 최악(정렬된 입력 + 끝 피벗). 랜덤 피벗/median-of-3로 회피
- 불안정

### 힙 정렬

힙 만들고 최댓값 반복 추출 → [[heaps]]:
- 최악도 O(n log n) 보장, 제자리 O(1) 공간
- 캐시 지역성 나빠 퀵보다 느린 편

## 하한: Ω(n log n) 증명

**비교만으로는 O(n log n)보다 빠를 수 없다.** 결정 트리(decision tree) 논증:

- n개 원소의 가능한 순서 = n! 가지
- 각 비교는 이진 결정 (≤ 또는 >) → 결정 트리의 한 분기
- n!개 잎을 구분하려면 트리 높이 ≥ log₂(n!)
- log₂(n!) = Θ(n log n) (스털링 근사)
- 트리 높이 = 최악 비교 횟수 → **최소 Ω(n log n)**

그래서 어떤 비교 정렬도 이 벽을 못 넘음. 넘으려면 비교를 안 해야 함 → [[linear-sorts]].

## 실측: O(n log n) vs O(n²)

```
qsort   2000000 elems: 0.155s
insort   40000 elems: 0.242s  (n²)
insort을 2000000개로 외삽: ~604초
```

퀵정렬은 200만 개를 0.155초. 삽입 정렬(O(n²))은 4만 개에 0.242초 - 200만 개로 외삽하면 **~600초(10분)**. 4000배 차이. [[asymptotic-analysis]]의 성장률 격차가 실전에서 이렇게 나타남.

## 그런데 작은 n에선 삽입 정렬이 빠르다

O(n²)인데도 삽입 정렬은 **작은 배열(~수십 개)에서 O(n log n)보다 빠름**:
- 상수가 작음 (비교/이동만, 재귀 오버헤드 없음)
- 캐시 친화, 거의 정렬된 데이터에 O(n)

그래서 실무 정렬은 **하이브리드**:
- **introsort** (C++ std::sort): 퀵 → 깊이 깊어지면 힙(최악 O(n²) 회피) → 작은 조각은 삽입
- **Timsort** (Python, Java): 병합 + 삽입, 이미 정렬된 런(run) 활용 → 실데이터에 빠름, 안정적

[[asymptotic-analysis]]의 "상수가 중요해지는 경우"의 대표.

## 안정성 (stability)

같은 키의 원소가 **원래 순서를 유지**하나:
- 안정: 병합, 삽입, Timsort
- 불안정: 퀵, 힙
- 다중 키 정렬(이름순 후 나이순)에 중요. DB `ORDER BY`도

## 셀프 체크

> [!question]- 퀵 정렬은 왜 최악 O(n²)인데도 실전에서 가장 빠른 경우가 많나?
> 평균 O(n log n)에 상수가 작기 때문이다. 제자리 정렬이라 캐시 친화적이고 추가 공간이 O(log n)뿐이다. 최악은 피벗이 계속 극단(정렬된 입력 + 끝 피벗)일 때인데 랜덤 피벗이나 median-of-3로 회피한다.

> [!question]- 병합 정렬과 힙 정렬 모두 최악 O(n log n)인데 실무에서 힙보다 병합/퀵을 자주 쓰는 이유는?
> 힙 정렬은 O(1) 제자리라는 장점이 있지만 힙 추출이 메모리를 널뛰어 캐시 지역성이 나빠 퀵보다 느린 편이다. 병합은 안정 정렬이고 최악도 보장되어 외부 정렬/연결 리스트에 적합하다.

> [!question]- 비교 정렬의 하한이 Ω(n log n)인 논증을 요약하라.
> 결정 트리 논증이다. n개 원소의 가능한 순서는 n!가지이고 각 비교는 이진 분기라, n!개 잎을 구분하려면 트리 높이가 log₂(n!) 이상이어야 한다. 스털링 근사로 log₂(n!) = Θ(n log n)이고, 트리 높이가 최악 비교 횟수이므로 최소 Ω(n log n)이다.

> [!question]- O(n²)인 삽입 정렬이 작은 배열에서 오히려 빠른 이유는?
> 상수가 작기 때문이다. 비교/이동만 하고 재귀 오버헤드가 없으며 캐시 친화적이고, 거의 정렬된 데이터에는 O(n)에 가깝다. 그래서 introsort/Timsort 같은 하이브리드가 작은 조각을 삽입 정렬로 처리한다.

> [!question]- 정렬의 안정성이란 무엇이고 언제 중요한가?
> 같은 키를 가진 원소가 원래 순서를 유지하는 성질이다. 병합/삽입/Timsort는 안정, 퀵/힙은 불안정이다. 다중 키 정렬(이름순 후 나이순)이나 DB의 ORDER BY처럼 이전 정렬 순서를 보존해야 할 때 중요하다.

## 연습문제

> [!example]- 문제: 결정 트리 논증으로 비교 정렬의 하한이 Ω(n log n)임을 증명하라.
> **풀이**
> 어떤 비교 정렬이든 실행을 이진 결정 트리로 나타낼 수 있다: 각 내부 노드는 두 원소의 비교, 각 잎은 하나의 순열(정렬 결과)에 대응.
> 정답을 내려면 n!개의 서로 다른 순열이 각각 잎으로 도달 가능해야 하므로 잎의 수 ≥ n!.
> 높이 h인 이진 트리의 잎은 최대 2^h개이므로 2^h ≥ n!, 즉 h ≥ log₂(n!).
> 스털링 근사로 log₂(n!) = Θ(n log n). 최악의 비교 횟수는 트리 높이 h이므로 어떤 비교 정렬도 최악 Ω(n log n). ∎

> [!example]- 문제: 병합 정렬의 복잡도를 점화식으로 유도하라.
> **풀이**
> 크기 n을 반으로 나눠 각각 정렬(2개의 T(n/2))하고 병합은 O(n)이므로 T(n) = 2T(n/2) + O(n).
> 재귀 트리로 풀면 각 레벨의 총 병합 비용이 O(n)이고 레벨 수가 log₂n이므로 T(n) = O(n log n).
> (마스터 정리로도 a=2, b=2, f(n)=n → n^(log_b a)=n과 같은 차수라 case 2 → Θ(n log n).)

> [!example]- 문제: 퀵 정렬을 최악 O(n²)로 만드는 입력을 하나 제시하고, 이를 막는 방법과 실무 하이브리드가 취하는 대비책을 설명하라.
> **풀이**
> 이미 정렬된 배열 [1,2,3,...,n]에서 항상 마지막 원소를 피벗으로 고르면, 분할이 (n-1)개와 0개로 극도로 불균형해져 T(n)=T(n-1)+O(n)=O(n²)가 된다.
> 회피: 랜덤 피벗 또는 median-of-3(첫/중간/끝의 중앙값)로 극단 피벗 확률을 낮춘다.
> introsort는 재귀 깊이가 임계치를 넘으면 힙 정렬로 전환해 최악을 O(n log n)으로 강제하고, 작은 조각은 삽입 정렬로 마무리한다.
