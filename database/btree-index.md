# B+tree 인덱스 (B+tree Index)

## 한 줄 요약

DB 인덱스의 표준 자료구조. B+tree([[b-trees]])로 키를 정렬 저장해 O(log n) 조회와 범위 스캔을 준다. 인덱스 없으면 풀스캔 O(n), 있으면 트리 탐색. 클러스터드/논클러스터드로 데이터 배치가 갈린다.

## 왜 필요한가

- 인덱스가 왜 조회를 빠르게 하나
- 인덱스가 왜 B+tree인가 → data-structures/[[b-trees]]
- 클러스터드 vs 논클러스터드 차이

## 인덱스 = 빠른 조회 자료구조

인덱스 없으면 조건 조회가 **풀스캔** O(n) (모든 행 검사). 인덱스는 **정렬된 자료구조**로 O(log n):

실측 (SQLite EXPLAIN, 10만 행):
```
인덱스 있음: SEARCH t USING COVERING INDEX idx_val   ← 인덱스 탐색
인덱스 없음: SCAN t2                                  ← 풀스캔
```

같은 `WHERE val=50000`인데 인덱스가 SCAN(전체)을 SEARCH(트리 탐색)로. 10만 행에서 17회(log₂) vs 10만 회.

## 왜 B+tree인가

DB 인덱스 = 거의 항상 **B+tree** → data-structures/[[b-trees]]:

- **디스크 친화**: 노드 = 페이지([[db-storage]]), 높은 fanout → 적은 디스크 접근 (10억 행이 3~4 페이지 읽기)
- **범위 스캔**: 리프가 연결 리스트 → `WHERE age BETWEEN 20 AND 40`이 리프 순회로 O(범위)
- **정렬 유지**: ORDER BY 공짜
- 해시 인덱스는 등호(`=`)만, 범위·정렬 불가 → 그래서 기본은 B+tree

data-structures/[[b-trees]]에서 "왜 이진 트리가 아니라 B-tree인가"의 실전.

## 클러스터드 vs 논클러스터드

인덱스와 실제 데이터의 관계로 갈림:

### 클러스터드 인덱스 (clustered)

**데이터 자체가 인덱스 순서로 저장**:

```
인덱스 = 데이터 (리프에 실제 행)
```

- 테이블당 하나만 (물리적 순서는 하나)
- 보통 기본 키 → 기본 키 순으로 물리 저장 (MySQL InnoDB)
- 기본 키 조회·범위가 매우 빠름 (리프가 곧 데이터)

### 논클러스터드 인덱스 (secondary)

**인덱스는 별도, 리프가 행 위치를 가리킴**:

```
인덱스 리프 → 행 ID(또는 기본 키) → 실제 행 조회
```

- 여러 개 가능 (열마다 인덱스)
- 인덱스 찾은 뒤 실제 행을 또 조회 (**세컨더리 조회, bookmark lookup**) → 추가 비용
- PostgreSQL은 모든 인덱스가 논클러스터드 (heap + 인덱스 분리)

## 커버링 인덱스

실측의 "COVERING INDEX"가 핵심 최적화:

- 쿼리에 필요한 **모든 열이 인덱스에** 있으면 → 실제 행 조회 불필요 (인덱스만으로 답)
- 세컨더리 조회 회피 → 빠름
- `SELECT val FROM t WHERE val=x`에서 인덱스가 val을 담으니 테이블 안 봄
- 자주 함께 조회하는 열을 인덱스에 포함시켜 유도

## 인덱스의 대가

인덱스는 공짜가 아님:

- **쓰기 느려짐**: INSERT/UPDATE/DELETE 시 인덱스도 갱신 (B+tree 재조정)
- **공간**: 인덱스가 디스크 차지
- **너무 많으면**: 쓰기 부담↑, 최적화기 혼란
- → 조회 패턴에 맞는 인덱스만 (무작정 다 걸면 역효과) → [[index-usage]]

읽기 vs 쓰기 트레이드오프. 읽기 많으면 인덱스 이득, 쓰기 많으면 부담.

## LSM-tree 대안

쓰기 많은 워크로드는 B+tree 대신 LSM-tree → [[lsm-tree]]:
- B+tree는 제자리 갱신 → SSD에 부담 (os/[[ssd-internals]])
- LSM은 순차 쓰기 → 쓰기 최적. Cassandra, RocksDB

## 셀프 체크

> [!question]- 인덱스는 왜 조건 조회를 O(n)에서 O(log n)으로 바꾸나?
> 인덱스 없으면 모든 행을 검사하는 풀스캔(SCAN) O(n)이다. 인덱스는 키를 정렬 저장한 자료구조라 트리 탐색(SEARCH)으로 대상을 좁힌다. 10만 행이면 풀스캔 10만 회 vs B+tree 약 17회(log₂).

> [!question]- DB 인덱스가 이진 트리가 아니라 B+tree인 이유 세 가지는?
> 첫째, 노드가 페이지 크기라 fanout이 높아 디스크 접근이 적다(10억 행이 3~4 페이지 읽기). 둘째, 리프가 연결 리스트로 이어져 범위 스캔·ORDER BY가 순회로 해결된다. 셋째, 정렬이 유지돼 범위·정렬 쿼리에 강하다. 해시 인덱스는 등호만 되고 범위·정렬이 안 된다.

> [!question]- 클러스터드와 논클러스터드 인덱스의 차이, 그리고 세컨더리 조회(bookmark lookup)란?
> 클러스터드는 데이터 자체가 인덱스 순서로 저장돼 리프가 곧 실제 행이다(테이블당 하나, 보통 기본 키). 논클러스터드는 인덱스가 별도이고 리프가 행 위치(행 ID/기본 키)만 가리켜, 인덱스를 찾은 뒤 실제 행을 다시 읽는다. 이 추가 조회가 세컨더리 조회(bookmark lookup)로 비용이 든다.

> [!question]- 커버링 인덱스는 왜 빠른가?
> 쿼리에 필요한 모든 열이 인덱스 안에 있으면 실제 행을 조회할 필요 없이 인덱스만으로 답한다. 세컨더리 조회를 회피하기 때문에 빠르다. EXPLAIN에서 "COVERING INDEX"로 나타난다.

> [!question]- 인덱스를 무작정 많이 거는 것이 왜 역효과인가?
> 인덱스는 공짜가 아니다. INSERT/UPDATE/DELETE마다 인덱스도 B+tree 재조정으로 갱신돼 쓰기가 느려지고, 디스크 공간을 차지하며, 많으면 최적화기를 혼란시킨다. 읽기가 많으면 이득, 쓰기가 많으면 부담이라 조회 패턴에 맞는 인덱스만 걸어야 한다.

## 연습문제

> [!example]- 문제: 다음 두 쿼리에 대해 인덱스가 커버링인지 판정하라. 인덱스는 `idx(val)` 하나뿐이다. (a) `SELECT val FROM t WHERE val=50000` (b) `SELECT val, name FROM t WHERE val=50000`
> **풀이**
> (a) 필요한 열이 `val` 하나이고 인덱스가 `val`을 담으므로 인덱스만으로 답한다 → 커버링. 테이블(heap) 접근 없이 COVERING INDEX 스캔.
> (b) `name`이 인덱스에 없으므로 인덱스로 행을 찾은 뒤 실제 행을 다시 읽어 `name`을 가져와야 한다 → 커버링 아님(세컨더리 조회 발생).
> 개선: `name`도 자주 함께 조회한다면 `idx(val, name)`으로 만들어 (b)도 커버링으로 유도한다.

> [!example]- 문제: `WHERE age BETWEEN 20 AND 40 ORDER BY age` 쿼리가 있다. 해시 인덱스 대신 B+tree 인덱스를 써야 하는 이유를 실행 관점에서 설명하라
> **풀이**
> 해시 인덱스는 등호(`=`) 조회만 지원한다. 범위 조건(BETWEEN)은 해시로 특정 버킷을 좁힐 수 없어 인덱스를 못 타고 풀스캔이 된다.
> B+tree는 age=20에 해당하는 리프를 찾은 뒤 리프의 연결 리스트를 40까지 순회하면 되므로 범위가 O(범위 크기)로 처리된다. 게다가 리프가 이미 정렬돼 있어 ORDER BY age가 추가 정렬 없이 공짜로 해결된다. 따라서 범위+정렬 쿼리는 B+tree가 정답이다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 인덱스가 풀스캔(O(n))을 트리 탐색(O(log n))으로 바꾸는 원리와, 그 자료구조가 왜 B+tree인지(fanout·범위 스캔·정렬).
> 2. 클러스터드 vs 논클러스터드의 물리 배치 차이와 세컨더리 조회 비용, 커버링 인덱스가 그 비용을 없애는 방식.
> 3. 인덱스의 대가(쓰기 지연·공간)와 읽기/쓰기 트레이드오프. 쓰기가 많으면 LSM-tree 대안.
