# DB 파티셔닝 (Partitioning / Sharding)

## 한 줄 요약

데이터를 여러 노드에 나눠 저장해 한 서버 용량·처리량 한계를 넘는다. 파티션 키 선택이 핵심 - 잘못하면 핫스팟(부하 쏠림)이 생긴다. 복제와 함께 분산 DB의 두 축.

## 왜 필요한가

- 한 서버에 안 들어가는 데이터를 어떻게
- 파티션 키를 어떻게 정하나
- 핫스팟이 왜 생기고 어떻게 피하나

## 복제 vs 파티셔닝

분산 DB의 두 축:

- **복제**([[replication-db]]): **같은 데이터**를 여러 노드에 (가용성, 읽기 확장)
- **파티셔닝(샤딩)**: **다른 데이터**를 여러 노드에 (쓰기·용량 확장)

```
복제: 노드1[전체] 노드2[전체] 노드3[전체]  (사본)
파티션: 노드1[A~H] 노드2[I~P] 노드3[Q~Z]    (분할)
```

보통 **둘을 함께**: 각 파티션을 또 복제 (용량 + 가용성).

## 파티션 전략

데이터를 어떻게 나누나:

### 범위 파티셔닝 (range)

키 범위로 (A~H, I~P, ...):
- **범위 쿼리 효율**: 연속 범위가 한 파티션 (날짜별 등)
- **핫스팟 위험**: 최근 데이터에 쏠림 (시계열에서 오늘 파티션만 바쁨)

### 해시 파티셔닝 (hash)

키를 해시해 분배 (data-structures/[[hash-tables]]):
- **고른 분배**: 해시가 균등 → 부하 분산
- **범위 쿼리 불리**: 연속 키가 흩어짐 → 여러 파티션 조회

### consistent hashing

노드 추가/삭제 시 **최소한만 재배치**:
- 일반 해시(`% N`)는 노드 수 바뀌면 거의 다 재배치 → 재앙
- consistent hashing은 링에 노드·키 배치 → 노드 변경 시 인접만 이동
- Cassandra, DynamoDB → distributed-systems/[[partitioning]]

## 핫스팟 (hotspot)

파티셔닝의 핵심 함정 - **부하가 한 파티션에 쏠림**:

- **범위 + 순차 키**: 최신 데이터만 쓰임 (타임스탬프 키 → 오늘 파티션만)
- **인기 키**: 유명인 계정, 인기 상품 (한 키에 트래픽 집중)
- 결과: 한 노드만 바쁘고 나머지 논다 → 분산 이득 상실

해결:
- **해시 파티셔닝**: 균등 분배
- **키 조합**: 순차 키 앞에 랜덤/해시 접두사 (핫 파티션 분산)
- **인기 키 특별 처리**: 더 잘게 쪼개거나 캐싱 ([[caching-strategies]])

## 파티션 키 선택

가장 중요한 결정 - 잘못하면 되돌리기 어려움:

- **고른 분배**: 핫스팟 없게 (높은 카디널리티, 균등)
- **쿼리 패턴 일치**: 자주 함께 조회하는 걸 같은 파티션에 (교차 파티션 쿼리 회피)
- **교차 파티션 쿼리 최소화**: 여러 파티션에 걸친 JOIN·트랜잭션은 비쌈 (분산 → distributed-systems/[[distributed-transactions]])

트레이드오프: 분배 균등(해시) vs 범위 쿼리(범위) vs 관련 데이터 집중.

## 리밸런싱 (rebalancing)

노드 추가/제거 시 데이터 재분배:

- **고정 파티션 수**: 파티션을 미리 많이 만들고 노드에 할당 → 노드 변경 시 파티션 이동만 (재해시 없음). Elasticsearch, Riak
- **consistent hashing**: 인접 이동만
- 목표: 이동 최소화 + 부하 균등 유지
- 리밸런싱 중에도 서비스 계속 (온라인)

## 교차 파티션 연산

파티션을 넘는 연산이 어려움:

- **JOIN**: 다른 파티션의 데이터 결합 → 네트워크 비용, 종종 회피(역정규화 [[normalization]])
- **트랜잭션**: 여러 파티션에 걸치면 분산 트랜잭션 (2PC 등) → distributed-systems/[[distributed-transactions]]
- **집계**: 각 파티션 부분 집계 후 합침 (map-reduce 발상)
- 그래서 파티션 키를 "관련 데이터가 같은 파티션에" 오게 설계

## 셀프 체크

> [!question]- 복제와 파티셔닝의 차이는? 보통 왜 둘을 함께 쓰나?
> 복제는 **같은 데이터**를 여러 노드에 두어 가용성·읽기 확장을 얻고, 파티셔닝(샤딩)은 **다른 데이터**를 여러 노드에 나눠 쓰기·용량을 확장한다. 실무에선 각 파티션을 다시 복제해 용량과 가용성을 동시에 얻는다.

> [!question]- 범위 파티셔닝과 해시 파티셔닝은 각각 무엇에 유리하고 무엇에 불리한가?
> 범위는 연속 범위가 한 파티션에 모여 범위 쿼리(날짜별 등)에 효율적이지만 최근 데이터에 쏠리는 핫스팟 위험이 있다. 해시는 고르게 분배해 부하가 균등하지만 연속 키가 흩어져 범위 쿼리가 여러 파티션에 걸친다.

> [!question]- 핫스팟은 왜 생기며 어떻게 완화하나?
> 범위+순차 키(타임스탬프)로 최신 파티션만 쓰이거나 인기 키(유명인·인기 상품)에 트래픽이 집중되면 한 노드만 바쁘고 나머지는 논다. 완화책은 해시 파티셔닝, 순차 키 앞에 랜덤/해시 접두사 붙이기, 인기 키를 더 잘게 쪼개거나 캐싱하기.

> [!question]- 일반 해시(% N) 대신 consistent hashing을 쓰는 이유는?
> `% N`은 노드 수 N이 바뀌면 거의 모든 키를 재배치해야 해 재앙이다. consistent hashing은 링에 노드·키를 배치해 노드 추가/삭제 시 인접 구간만 이동하므로 재배치를 최소화한다.

## 연습문제

> [!example]- 문제: 주문 테이블을 `order_id = 타임스탬프 기반 순차 ID`로 범위 파티셔닝했더니 오늘 파티션 노드만 CPU가 100%다. 원인과 개선안을 제시하라.
> **풀이**
> 원인: 순차 키 + 범위 파티셔닝의 전형적 핫스팟. 최신 주문이 모두 마지막 파티션(오늘 구간)으로 몰려 그 노드만 쓰기 부하를 받는다.
> 개선안 1: **해시 파티셔닝**으로 전환 - `hash(order_id)`로 분배해 쓰기를 균등화. 단 order_id 범위 조회(기간별 조회)는 여러 파티션 스캔이 되므로 트레이드오프 검토.
> 개선안 2: **복합 키/접두사 샤딩** - 파티션 키를 `(shard_prefix, order_id)`로 두고 `shard_prefix = hash(user_id) % K`처럼 랜덤/해시 접두사를 앞에 붙여 최신 쓰기를 K개 파티션으로 분산. 기간 조회는 K개 파티션 병렬 조회로 감당.

> [!example]- 문제: 파티션 키를 고를 때 고려해야 할 세 기준을 들고, `user_id`로 샤딩한 소셜 앱에서 "특정 지역 사용자 전체 조회"가 느린 이유를 설명하라.
> **풀이**
> 세 기준: (1) 고른 분배 - 높은 카디널리티·균등 분포로 핫스팟 회피, (2) 쿼리 패턴 일치 - 자주 함께 조회하는 데이터를 같은 파티션에, (3) 교차 파티션 쿼리 최소화 - 여러 파티션에 걸친 JOIN·트랜잭션은 비쌈.
> 느린 이유: `user_id` 해시로 사용자가 전 파티션에 흩어져 있어 "지역별 조회"는 파티션 키와 무관한 조건이다. 따라서 모든 파티션을 스캔(scatter-gather)해야 하고, 각 파티션 부분 결과를 모아 합쳐야 하므로 느리다. 지역 조회가 잦다면 `region`을 파티션 키에 포함하거나 지역 기준 보조 인덱스를 검토한다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1. 복제 vs 파티셔닝의 축 차이와, 왜 실무에서 둘을 결합하는가.
> 2. 범위/해시/consistent hashing 각 전략의 트레이드오프(범위 쿼리 vs 균등 분배 vs 재배치 최소화).
> 3. 핫스팟의 원인(순차 키·인기 키)과 완화책, 그리고 파티션 키 선택이 왜 되돌리기 어려운 결정인가.
