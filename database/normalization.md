# 정규화 (Normalization)

## 한 줄 요약

데이터 중복을 제거해 이상 현상(갱신/삽입/삭제 이상)을 막는 스키마 설계 원칙. 함수 종속을 분석해 테이블을 쪼갠다. 하지만 실무에선 성능을 위해 의도적으로 역정규화하기도 한다.

## 왜 필요한가

- 왜 데이터를 여러 테이블로 쪼개나
- 중복이 왜 위험한가 (이상 현상)
- 언제 정규화를 깨야 하나

## 문제: 중복이 낳는 이상 현상

한 테이블에 다 넣으면 (비정규):

```
Orders
┌──────┬──────────┬───────────┬──────────┐
│ 주문 │ 고객이름  │ 고객주소   │ 상품     │
├──────┼──────────┼───────────┼──────────┤
│ 1    │ Alice    │ 서울       │ 노트북   │
│ 2    │ Alice    │ 서울       │ 마우스   │  ← Alice 정보 중복
└──────┴──────────┴───────────┴──────────┘
```

중복이 세 가지 **이상(anomaly)**:

- **갱신 이상**: Alice 주소 바꾸려면 **모든 Alice 행** 수정. 하나 빠뜨리면 불일치
- **삽입 이상**: 주문 없는 고객을 못 넣음 (주문 행이 있어야)
- **삭제 이상**: 마지막 주문 삭제 시 고객 정보도 사라짐

## 함수 종속 (functional dependency)

정규화의 분석 도구. **A → B**: A가 정해지면 B가 유일하게 결정:

```
고객id → 고객이름, 고객주소   (고객id가 이름·주소 결정)
주문id → 상품, 수량
```

- 이상 현상의 원인 = **기본 키가 아닌 것에 종속되는 속성** (고객이름이 주문id 아닌 고객id에 종속)
- 정규화 = 함수 종속을 분석해 "각 사실을 한 곳에만" 저장하도록 분해

## 정규형 단계

### 1NF (제1정규형)

- **원자값**: 각 셀이 하나의 값 (배열/중첩 불가)
- 반복 그룹 제거
- 예: `취미: "축구,농구"` → 별도 행/테이블로

### 2NF (제2정규형)

- 1NF + **부분 종속 제거**: 복합 키의 일부에만 종속되는 속성 분리
- (주문id, 상품id) 키에서 상품명이 상품id에만 종속 → 상품 테이블 분리

### 3NF (제3정규형)

- 2NF + **이행 종속 제거**: 키 아닌 것에 종속되는 것 분리
- 주문id → 고객id → 고객주소 (이행) → 고객 테이블 분리
- **실무 목표**: 대부분 3NF까지가 실용적 균형

### BCNF (Boyce-Codd)

- 3NF보다 엄격: 모든 결정자가 후보 키
- 드문 예외 케이스 처리. 대부분 3NF면 BCNF도 만족

정규화 결과: 위 예가 Customers, Orders, Products로 분리 → 각 사실이 한 곳에만.

## 정규화의 대가: JOIN

정규화하면 데이터가 여러 테이블로 흩어짐 → 조회 시 **JOIN 필요** ([[sql-deep-dive]]):

- 정보를 모으려면 여러 테이블 결합 → JOIN 비용
- 정규화가 심할수록 JOIN 많아짐 → 읽기 성능 부담
- 여기서 트레이드오프 발생

## 역정규화 (denormalization)

실무에선 **의도적으로 중복을 허용**해 성능 개선:

- 자주 함께 읽는 데이터를 한 테이블에 → JOIN 회피
- 집계값을 미리 저장 (주문 수 캐싱)
- 읽기 많은 시스템(분석, 대시보드)에서 흔함
- 대가: 중복 관리 (갱신 시 동기화), 이상 현상 위험 다시

**언제 역정규화**: 읽기가 압도적으로 많고, JOIN이 병목이며, 쓰기 시 일관성을 앱/트리거로 관리할 수 있을 때. 정규화를 먼저 하고 필요할 때 선택적으로 깸.

## OLTP vs OLAP

용도에 따라 정규화 정도가 다름:

- **OLTP**(트랜잭션, 주문 처리): 정규화 (쓰기 많음, 일관성 중요) → [[transactions-acid]]
- **OLAP**(분석, 데이터 웨어하우스): 역정규화 (읽기·집계 중심, star schema) → column store [[db-storage]]
- NoSQL([[nosql-landscape]])은 종종 역정규화를 기본으로 (문서에 중첩)

## 셀프 체크

> [!question]- 중복이 낳는 세 가지 이상 현상(anomaly)을 각각 예로 설명하라.
> 갱신 이상: 같은 사실이 여러 행에 있어 하나 수정 시 나머지를 빠뜨리면 불일치(예: Alice 주소를 한 행만 바꿈). 삽입 이상: 다른 정보 없이는 데이터를 못 넣음(주문 없는 고객을 못 등록). 삭제 이상: 한 행을 지우면 함께 저장된 다른 사실도 사라짐(마지막 주문 삭제 시 고객 정보 소멸).

> [!question]- 함수 종속 A → B는 무엇을 뜻하며 이상 현상의 원인과 어떻게 연결되나?
> A가 정해지면 B가 유일하게 결정된다는 뜻이다. 이상 현상의 원인은 기본 키가 아닌 속성에 종속되는 속성이다. 예로 고객이름이 주문id가 아니라 고객id에 종속되는데 같은 테이블에 있으면 고객id가 같은 여러 행에 이름이 중복 저장되어 이상이 생긴다.

> [!question]- 2NF, 3NF, BCNF가 각각 제거하는 종속은 무엇인가?
> 2NF는 복합 키의 일부에만 종속되는 부분 종속을 제거한다. 3NF는 키가 아닌 속성에 종속되는 이행 종속을 제거한다. BCNF는 모든 결정자가 후보 키가 되도록 하는 더 엄격한 형태로 3NF의 드문 예외를 처리한다.

> [!question]- 언제 역정규화를 택하며 그 대가는 무엇인가?
> 읽기가 압도적으로 많고 JOIN이 병목이며 쓰기 시 일관성을 앱/트리거로 관리할 수 있을 때 택한다. 자주 함께 읽는 데이터를 한 테이블에 두거나 집계값을 미리 저장해 JOIN을 회피한다. 대가는 중복 관리(갱신 시 동기화)와 이상 현상 위험의 재등장이다.

## 연습문제

> [!example]- 문제: 테이블 `수강(학번, 과목코드, 과목명, 학점, 성적)`에서 함수 종속은 `(학번, 과목코드) → 성적`, `과목코드 → 과목명, 학점`이다. 어떤 정규형을 위반하는지 판정하고 분해하라.
> **풀이**
> 후보 키는 (학번, 과목코드)이다. 과목명·학점이 키의 일부인 과목코드에만 종속(부분 종속) → 2NF 위반이다.
> 분해:
> - `수강(학번, 과목코드, 성적)` - 키 (학번, 과목코드)
> - `과목(과목코드, 과목명, 학점)` - 키 과목코드
> 이제 각 사실이 한 곳에만 저장되어 부분 종속이 제거되고 2NF(및 이 경우 3NF)를 만족한다.

> [!example]- 문제: 테이블 `직원(사번, 부서번호, 부서위치)`에서 `사번 → 부서번호`, `부서번호 → 부서위치`가 성립한다. 위반 정규형을 판정하고 분해하라.
> **풀이**
> 후보 키는 사번. 부서위치가 키가 아닌 부서번호에 종속되고 부서번호는 사번에 종속 → 사번 → 부서번호 → 부서위치의 이행 종속. 2NF는 만족하지만 이행 종속이 있어 3NF 위반이다.
> 분해:
> - `직원(사번, 부서번호)` - 키 사번
> - `부서(부서번호, 부서위치)` - 키 부서번호
> 이행 종속이 제거되어 부서위치 변경이 한 행에서만 일어난다(갱신 이상 해소).

> [!example]- 문제: OLTP 주문 처리 시스템과 OLAP 분석 대시보드에서 정규화 정도를 어떻게 다르게 가져갈지, 이유와 함께 설명하라.
> **풀이**
> OLTP는 쓰기가 많고 일관성이 중요하므로 3NF 수준으로 정규화해 이상 현상을 막고 갱신을 한 곳에서 처리한다.
> OLAP는 읽기·집계 중심이라 JOIN 비용을 줄이려 역정규화(star schema)하고, 컬럼 스토어로 대량 스캔·집계를 최적화한다. 쓰기가 적고 일괄 적재라 중복 동기화 부담이 작다.
> 요컨대 쓰기·일관성 우선이면 정규화, 읽기·집계 성능 우선이면 역정규화다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
> 1) 중복 → 세 가지 이상 현상 → 함수 종속 분석 → 테이블 분해로 이어지는 정규화의 논리를 예제 없이 재구성할 수 있는가.
> 2) 임의의 함수 종속 집합이 주어졌을 때 1NF~BCNF 중 어느 정규형을 위반하는지 판정하고 분해할 수 있는가.
> 3) 정규화의 대가(JOIN)와 역정규화의 판단 기준·대가를 OLTP/OLAP 맥락에서 설명할 수 있는가.
