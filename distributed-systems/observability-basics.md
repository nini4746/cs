# 관측성 기초 (Observability Basics)

## 한 줄 요약

분산 시스템에서 "무슨 일이 일어나는지" 파악하는 능력. 로그·메트릭·트레이스 세 기둥이 있고, 분산 추적이 여러 서비스에 걸친 요청을 따라간다. 로그만으로는 부족한 이유가 여기 있다.

## 왜 필요한가

- 분산 시스템은 왜 디버깅이 어려운가
- 로그·메트릭·트레이스의 차이
- 분산 추적이 뭘 해결하나

## 문제: 분산은 안 보인다

한 프로세스는 디버거로 보지만([[why-distributed]]), 분산은:

- 요청이 **여러 서비스를 거침** → 어디서 느린지/실패했는지 한눈에 안 보임
- **부분 실패**([[why-distributed]]): 일부만 죽음 → 증상이 간접적
- **비동기**([[message-queues]]): 흐름 추적 어려움
- 로그가 여러 서버에 흩어짐

**관측성(observability)** = 외부 출력(로그·메트릭·트레이스)으로 내부 상태를 추론하는 능력.

## 세 기둥 (three pillars)

### 1. 로그 (logs)

**이산 이벤트 기록** ("무슨 일이 일어났나"):

```
2026-07-08 10:00:01 ERROR order-service: payment failed order=123
```

- 상세한 맥락 (무엇이, 언제, 왜)
- **구조화 로그**(JSON): 검색·집계 가능
- **중앙 수집**: 여러 서버 로그를 한 곳에 (ELK, Loki)
- 문제: 대량, 여러 서비스에 흩어짐 → 요청 하나를 추적하기 어려움 (트레이스가 보완)

### 2. 메트릭 (metrics)

**수치 시계열** ("얼마나"):

```
request_count, latency_p99, error_rate, cpu_usage, queue_depth
```

- 집계된 수치 (카운터, 게이지, 히스토그램)
- 대시보드·알림 (Prometheus, Grafana)
- **백분위수**(p50, p99): 평균은 거짓말 - p99가 사용자 경험 (algorithms/[[selection]]의 백분위수)
- 추세·이상 감지에 강함, 하지만 "왜"는 로그·트레이스로

### 3. 트레이스 (traces)

**요청의 서비스 간 여정** ("어디서"):

```
요청 → [게이트웨이 5ms] → [주문 20ms] → [결제 100ms ← 느림!] → [재고 10ms]
```

- 한 요청이 여러 서비스를 거치는 전체 경로 + 각 단계 시간
- **분산 추적의 핵심** (아래)

## 분산 추적 (distributed tracing)

여러 서비스에 걸친 요청을 따라가는 것 - 관측성의 결정적 도구:

```
1. 요청 진입 시 고유 trace ID 생성
2. 각 서비스 호출에 trace ID 전파 (헤더로)
3. 각 단계가 span(구간)을 trace ID로 기록
4. 모아서 전체 요청의 waterfall 재구성
```

- **trace ID**: 한 요청의 전 여정을 묶는 ID (network/[[icmp-and-tools]]의 추적과 유사 발상)
- **span**: 각 작업 단위 (서비스 호출, DB 쿼리)
- 어느 서비스가 느린지/실패했는지 즉시 파악 (위 예의 결제 100ms)
- OpenTelemetry(표준), Jaeger, Zipkin

## 왜 로그만으론 부족한가

- **로그**: "이 서비스에서 이런 일" - 하지만 요청 하나가 여러 서비스면 흩어진 로그를 손으로 이어야
- **트레이스**: 요청 하나의 **전체 경로**를 자동으로 → "어디서" 문제인지
- **메트릭**: 추세·이상은 알지만 개별 요청은 모름
- 셋이 **상호 보완**: 메트릭으로 이상 감지 → 트레이스로 어느 서비스 → 로그로 왜

## SLO / SLI

관측성으로 **신뢰성 목표** 관리 (devops/[[observability]]):

- **SLI**(Service Level Indicator): 측정 지표 (가용성, 지연 p99)
- **SLO**(Service Level Objective): 목표 (99.9% 가용, p99 < 200ms)
- **에러 예산(error budget)**: 허용 실패량 → 다 쓰면 기능보다 안정성 우선
- 메트릭이 SLI의 기반

## 실무 원칙

1. **세 기둥 다**: 로그(상세) + 메트릭(추세) + 트레이스(경로)
2. **trace ID 전파**: 모든 서비스가 (로그에도 trace ID 넣어 연결)
3. **구조화**: 검색·집계 가능하게 (JSON 로그)
4. **백분위수**: 평균 아닌 p99
5. **알림**: SLO 위반 시 (하지만 알림 피로 주의)
6. **OpenTelemetry**: 표준으로 벤더 종속 회피

## 셀프 체크

> [!question]- 관측성(observability)의 정의와 세 기둥은?
> 외부 출력(로그·메트릭·트레이스)만으로 시스템 내부 상태를 추론하는 능력. 세 기둥은 로그(이산 이벤트 기록 - "무슨 일이"), 메트릭(수치 시계열 - "얼마나"), 트레이스(요청의 서비스 간 여정 - "어디서")다.

> [!question]- 로그·메트릭·트레이스가 각각 강한 것과 약한 것은?
> 로그는 상세한 맥락에 강하지만 여러 서비스에 흩어져 요청 하나 추적이 어렵다. 메트릭은 추세·이상 감지에 강하지만 개별 요청은 모르고 "왜"를 답하지 못한다. 트레이스는 요청 하나의 전체 경로("어디서")를 자동으로 보여주지만 그 자체로 상세 원인은 로그가 보완한다. 셋이 상호 보완한다.

> [!question]- 분산 추적(distributed tracing)은 어떻게 동작하나?
> 요청 진입 시 고유 trace ID를 생성하고, 각 서비스 호출에 헤더로 trace ID를 전파한다. 각 단계가 span(작업 단위)을 그 trace ID로 기록하고, 이를 모아 전체 요청의 waterfall을 재구성한다. 그러면 어느 서비스가 느린지/실패했는지 즉시 파악된다.

> [!question]- 왜 메트릭에서 평균이 아니라 p99를 봐야 하나?
> 평균은 소수의 느린 요청을 다수의 빠른 요청이 희석해 "거짓말"을 한다. p99 같은 백분위수는 꼬리 지연을 드러내며, 실제 사용자 경험(가장 느린 쪽)을 반영한다. 그래서 SLI로 p99 지연을 쓴다.

> [!question]- SLI, SLO, 에러 예산은 각각 무엇이며 어떻게 연결되나?
> SLI는 측정 지표(가용성, p99 지연), SLO는 그 목표치(99.9% 가용, p99 < 200ms). 에러 예산은 SLO가 허용하는 실패량으로, 다 쓰면 새 기능보다 안정성을 우선한다. 메트릭이 SLI의 기반이 된다.

## 연습문제

> [!example]- 문제: p99 지연이 갑자기 치솟았다. 세 기둥을 어떤 순서로 써서 원인을 좁혀갈지 절차를 설계하라.
> **풀이**
> 1. 메트릭으로 이상을 감지·확인한다(어느 엔드포인트의 p99가, 언제부터 올랐나).
> 2. 해당 시간대의 느린 요청 트레이스를 열어 waterfall에서 어느 span(서비스)이 지연의 대부분인지 특정한다(예: 결제 100ms).
> 3. 그 서비스의 로그를 trace ID로 필터링해 "왜"(에러, 재시도, 락 대기 등)를 확인한다.
> 4. 즉 메트릭(이상) → 트레이스(어디서) → 로그(왜) 순으로 좁힌다. 이 연결이 가능하려면 로그에도 trace ID가 들어 있어야 한다.

> [!example]- 문제: "로그만 다 모으면 충분하지 않나?"라는 주장에 대해, 로그만으로 부족한 경우를 판정하고 근거를 대라.
> **풀이**
> 1. 요청 하나가 여러 서비스를 거치면 로그가 여러 서버에 흩어져, 그 요청의 전체 경로를 손으로 이어 붙여야 한다.
> 2. 부분 실패·비동기 흐름에선 증상이 간접적이라 어느 단계가 원인인지 로그만으론 즉시 안 보인다.
> 3. 트레이스는 trace ID로 전체 경로를 자동 재구성해 "어디서"를 준다.
> 4. 메트릭은 추세·이상을 주지만 개별 요청은 모른다.
> 5. 판정: 로그만으론 부족하며, 트레이스(경로) + 메트릭(추세)과 상호 보완해야 한다.

> [!example]- 문제: 새 마이크로서비스를 관측 가능하게 만들라는 요구를 받았다. 무엇을 도입할지 실무 원칙에 근거해 설계하라.
> **풀이**
> 1. 세 기둥을 모두 심는다: 구조화(JSON) 로그, 메트릭, 트레이스.
> 2. OpenTelemetry로 계측해 벤더 종속을 피한다.
> 3. 모든 서비스가 trace ID를 헤더로 전파하고, 로그에도 trace ID를 넣어 트레이스-로그를 연결한다.
> 4. 메트릭은 평균이 아니라 p99 같은 백분위수를 대시보드/알림에 둔다(Prometheus/Grafana).
> 5. SLO를 정의하고 위반 시 알림하되, 알림 피로를 피하도록 임계치를 조절한다.
