# 조율 서비스 (Coordination Services)

## 한 줄 요약

ZooKeeper·etcd 같은 조율 서비스는 합의(Raft/Paxos)를 내장해 분산 락·리더 선출·설정 저장·서비스 디스커버리를 제공한다. 합의를 직접 구현하지 말고 이들에 위임하는 게 실무.

## 왜 필요한가

- 합의([[raft]])를 앱마다 구현 안 하고 재사용
- 분산 락·리더 선출을 어떻게 하나
- ZooKeeper/etcd의 용도

## 왜 조율 서비스인가

합의([[consensus-problem]], [[raft]])는 어렵고 미묘함 → **직접 구현하지 말고 검증된 서비스에 위임**:

- 조율 서비스가 합의를 내장 (Raft/Paxos 기반)
- 앱은 간단한 API로 조율 기능 사용
- security/[[crypto-basics]]의 "직접 만들지 마라"와 같은 정신

## 제공 기능

### 분산 락 (distributed lock)

여러 노드가 공유 자원에 접근 시 상호 배제 (os/[[locks]]의 분산 버전):

```
노드가 락 획득 시도 → 조율 서비스가 하나만 허용 → 그 노드만 진행
```

- 합의로 "누가 락을 쥐나" 동의
- **함정**: 락 쥔 노드가 죽으면? → TTL/세션으로 자동 해제. 하지만 GC 멈춤 등으로 락 만료 후에도 작업 중일 수 있음 → **fencing token**(단조 증가 번호)으로 방어

### 리더 선출 (leader election)

앱의 리더를 조율 서비스가 선출 ([[replication]]의 failover):
- 여러 노드가 후보 → 조율 서비스가 하나 선택
- 리더 죽으면 재선출
- 합의를 앱이 아니라 서비스가 처리

### 설정 저장 (configuration)

- 클러스터 설정을 중앙 저장 (일관된 뷰)
- 변경 시 모든 노드에 알림 (watch)

### 서비스 디스커버리 (service discovery)

- 어떤 서비스가 어디(IP:포트)에 있나 등록·조회
- 노드 추가/삭제 시 자동 갱신 → 마이크로서비스에 필수
- network/[[dns]]의 동적 버전

### 멤버십·헬스

- 클러스터에 누가 살아있나 (membership)
- 세션·heartbeat로 죽은 노드 감지

## ZooKeeper vs etcd

| | ZooKeeper | etcd |
|---|---|---|
| 합의 | ZAB | Raft ([[raft]]) |
| 모델 | 계층적 znode (파일시스템 유사) | 키-값 |
| 생태계 | Hadoop, Kafka | 쿠버네티스 (devops/[[kubernetes-basics]]) |
| API | 복잡 | 단순 (gRPC) |

- **etcd**: 쿠버네티스의 상태 저장소 → k8s의 모든 상태가 여기 (devops/[[kubernetes-basics]])
- **ZooKeeper**: Kafka([[message-queues]]), Hadoop 등 빅데이터 생태계
- **Consul**: 서비스 디스커버리 특화

## 데이터 모델 (etcd 예)

- **키-값** + watch (변경 알림)
- **강한 일관성**(linearizable [[consistency-models]]): 합의 기반이라 CP ([[cap-theorem]])
- **작은 데이터**: 설정·메타데이터용 (대량 데이터 저장소 아님)
- **TTL/lease**: 세션 만료로 자동 정리 (락·멤버십)

## 사용 원칙

- **직접 합의 구현 금지**: etcd/ZooKeeper 사용
- **작은 메타데이터만**: 설정·락·선출 (앱 데이터는 DB에)
- **일관성 우선(CP)**: 분할 시 소수파는 응답 안 함 ([[cap-theorem]]) → 조율은 정확성이 중요
- **fencing token**: 분산 락의 안전한 사용

## 셀프 체크

> [!question]- 합의를 앱마다 구현하지 않고 조율 서비스에 위임하는 이유는?
> 합의(Raft/Paxos)는 어렵고 미묘해서 직접 구현하면 버그가 나기 쉽다. ZooKeeper·etcd 같은 검증된 서비스에 위임하고 앱은 간단한 API로 락·선출·설정을 쓴다. crypto의 "직접 만들지 마라"와 같은 정신이다.

> [!question]- 분산 락에서 fencing token이 왜 필요한가?
> TTL/세션으로 죽은 노드의 락을 자동 해제하지만, GC 멈춤 등으로 락이 만료된 뒤에도 그 노드가 여전히 작업 중일 수 있다. 이때 **단조 증가하는 fencing token**을 자원 쪽에서 검사해 뒤늦은(오래된 토큰) 쓰기를 거부함으로써 방어한다.

> [!question]- ZooKeeper와 etcd의 주요 차이는?
> 합의는 ZooKeeper가 ZAB, etcd가 Raft를 쓴다. 데이터 모델은 ZooKeeper가 계층적 znode(파일시스템 유사), etcd가 키-값이다. 생태계는 ZooKeeper가 Hadoop·Kafka, etcd가 쿠버네티스의 상태 저장소다.

> [!question]- 조율 서비스가 CP인 이유와 네트워크 분할 시 동작은?
> 합의 기반이라 강한 일관성(linearizable)을 제공하며, 조율은 정확성이 중요하기 때문에 CP를 택한다. 분할 시 과반을 못 이룬 **소수파는 응답하지 않는다**(가용성보다 정확성 우선).

> [!question]- TTL/lease는 무엇을 해결하나?
> 세션 만료로 죽은 노드가 쥔 락이나 멤버십을 자동 정리한다. heartbeat가 끊기면 세션이 만료되어 락이 풀리고, ephemeral 노드가 사라져 멤버십에서 빠진다.

## 연습문제

> [!example]- 문제: 리더가 GC로 멈춰 락이 만료된 사이 다른 노드가 리더가 됐는데, 원래 리더가 깨어나 자원에 쓰기를 시도한다. 안전하게 막아라
> **풀이**
> 1. 문제: 락 TTL 만료 → 조율 서비스는 락을 회수하고 새 리더에게 부여. 하지만 옛 리더는 자기가 아직 락을 쥐었다고 믿는다.
> 2. TTL만으로는 옛 리더의 뒤늦은 쓰기를 막지 못한다(락 상태 확인과 실제 쓰기 사이에 시간차).
> 3. 해결: 락 획득 시 **단조 증가 fencing token**을 받고, 자원(스토리지)이 매 쓰기에서 토큰을 검사해 더 작은 토큰의 쓰기를 거부한다.
> 4. 결과: 옛 리더의 낡은 토큰 쓰기는 자원 단에서 거부되어 새 리더만 성공한다.

> [!example]- 문제: 수십 개 마이크로서비스가 뜨고 죽는 환경에서 서로를 찾게 하라. 어떤 조율 기능을 쓰나
> **풀이**
> 1. **서비스 디스커버리**: 각 서비스가 시작 시 IP:포트를 등록, 조회자는 이름으로 위치를 조회(동적 DNS).
> 2. **멤버십·헬스**: 세션/heartbeat로 죽은 인스턴스를 감지, ephemeral 등록이 자동으로 사라져 목록이 최신 유지.
> 3. **watch**: 인스턴스 추가/삭제 시 구독자에게 변경 알림 → 클라이언트가 즉시 갱신.
> 4. 저장은 작은 메타데이터만(주소·상태), 실제 앱 데이터는 별도 DB에 둔다.

> [!example]- 문제: 팀이 조율 서비스(etcd)에 대량의 애플리케이션 데이터를 저장하려 한다. 옳은지 판정하라
> **풀이**
> 1. 판정: **부적절**하다.
> 2. 근거: 조율 서비스는 합의 기반 CP라 모든 쓰기가 과반 동의를 거쳐 느리고 비싸다. 대량 데이터에 맞지 않는다.
> 3. 용도: 설정·락·리더 선출 같은 **작은 메타데이터**만 저장하도록 설계됐다.
> 4. 대안: 앱 데이터는 일반 DB에 두고, 조율 서비스에는 그 DB를 가리키는 메타데이터/설정만 저장한다.

## 파인만

> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
>
> 1. 조율 서비스가 제공하는 기능(분산 락·리더 선출·설정·디스커버리·멤버십)을 나열하고 각각이 왜 합의를 필요로 하는지 말할 수 있나.
> 2. fencing token이 TTL만으로 못 막는 무엇을 막는지 시나리오로 설명할 수 있나.
> 3. 왜 CP이고 작은 메타데이터만 저장하는지, ZooKeeper vs etcd 차이와 함께 설명할 수 있나.
