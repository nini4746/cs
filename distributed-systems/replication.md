# 복제 (Replication)

## 한 줄 요약

같은 데이터를 여러 노드에 복사하는 세 방식 - 단일 리더, 멀티 리더, 리더리스. 각각 쓰기 충돌 처리와 일관성 트레이드오프가 다르다. database/[[replication-db]]의 분산 시스템 일반화.

## 왜 필요한가

- 복제의 세 가지 방식과 트레이드오프
- 멀티 리더의 쓰기 충돌을 어떻게 해결하나
- 리더리스(Dynamo) 방식의 quorum

## 복제 = 데이터 사본

database/[[replication-db]]의 일반화. 왜 복제하나 ([[why-distributed]]): 가용성, 읽기 확장, 지리적 근접. 세 방식:

## 1. 단일 리더 (single-leader)

database/[[replication-db]]의 리더-팔로워:

```
쓰기 → 리더 → 로그 복제 → 팔로워들
읽기 ← 아무 노드
```

- **단순·일관**: 쓰기가 한 곳 → 순서 명확, 충돌 없음
- **복제 지연**: 팔로워가 뒤처짐 → read-your-writes 등 이상 (database/[[replication-db]])
- **리더 장애**: failover 필요 → 합의 ([[consensus-problem]])
- 대부분 관계형 DB, 가장 흔함

## 2. 멀티 리더 (multi-leader)

**여러 노드가 쓰기를 받음** (다중 데이터센터):

```
쓰기 → 리더A ⇄ 리더B ⇄ 리더C (서로 복제)
```

- **장점**: 각 지역에 리더 → 로컬 쓰기 빠름, 한 지역 장애에 강함
- **핵심 문제: 쓰기 충돌** - 두 리더가 같은 데이터를 동시에 다르게 수정
  ```
  리더A: x = 1
  리더B: x = 2   (동시에)
  → 어느 게 맞나? 충돌!
  ```

### 충돌 해결

- **LWW(Last Write Wins)**: 타임스탬프 큰 것 승리 → 하지만 시계 문제([[clocks]])로 데이터 유실 위험
- **버전 벡터**([[clocks]]): 동시 쓰기 감지 → 앱이 병합
- **CRDT**(충돌 없는 복제 자료형): 수학적으로 항상 병합 가능하게 설계 (카운터, 집합)
- **사용자에게 물어봄**: 충돌을 보존하고 사용자가 해결 (Git merge처럼)
- 충돌 해결이 멀티 리더의 근본 복잡도

## 3. 리더리스 (leaderless)

**리더 없이 여러 노드에 직접 쓰고 읽음** (Dynamo, Cassandra):

```
쓰기 → 여러 노드에 동시에 (W개 응답 기다림)
읽기 → 여러 노드에서 (R개 응답, 최신 선택)
```

### Quorum (정족수)

일관성을 **읽기/쓰기 노드 수**로 조절:

```
N = 전체 복제본 수
W = 쓰기 성공 필요 노드 수
R = 읽기 조회 노드 수

W + R > N  이면 → 읽기가 최신 쓰기를 반드시 포함 (겹침 보장)
```

- 예: N=3, W=2, R=2 → W+R=4>3 → 읽기 2개 중 하나는 최신 쓰기 봄
- **W 크게**: 쓰기 일관성↑, 가용성↓ (더 많은 노드 필요)
- **R 크게**: 읽기 일관성↑
- 조절 가능 → 일관성-가용성 트레이드오프를 앱이 선택 ([[cap-theorem]])

### 최신 값 판정

여러 노드가 다른 값 → 버전으로 최신 선택 ([[clocks]]의 벡터 시계). **read repair**(읽을 때 오래된 노드 갱신), **anti-entropy**(백그라운드 동기화).

## 세 방식 비교

| | 단일 리더 | 멀티 리더 | 리더리스 |
|---|---|---|---|
| 쓰기 | 한 곳 | 여러 곳 | 여러 노드 |
| 충돌 | 없음 | 있음 (해결 필요) | quorum + 버전 |
| 일관성 | 강함 | 약함 | 조절 (W/R) |
| 가용성 | 리더 의존 | 높음 | 높음 |
| 복잡도 | 낮음 | 높음 | 중간 |
| 예 | PostgreSQL | 다중 DC | Cassandra, Dynamo |
