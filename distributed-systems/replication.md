# 복제 (Replication)

## 한 줄 요약

같은 데이터를 여러 노드에 복사하는 세 방식 - 단일 리더, 멀티 리더, 리더리스. 각각 쓰기 충돌 처리와 일관성 트레이드오프가 다르다. database/[[replication-db]]의 분산 시스템 일반화.

## 왜 필요한가

- 복제의 세 가지 방식과 트레이드오프
- 멀티 리더의 쓰기 충돌을 어떻게 해결하나
- 리더리스(Dynamo) 방식의 quorum

## 복제 = 데이터 사본

database/[[replication-db]]의 일반화. 왜 복제하나 ([[why-distributed]]): 가용성, 읽기 확장, 지리적 근접. 세 방식:

## 1. 단일 리더 (single-leader)

database/[[replication-db]]의 리더-팔로워:

```
쓰기 → 리더 → 로그 복제 → 팔로워들
읽기 ← 아무 노드
```

- **단순·일관**: 쓰기가 한 곳 → 순서 명확, 충돌 없음
- **복제 지연**: 팔로워가 뒤처짐 → read-your-writes 등 이상 (database/[[replication-db]])
- **리더 장애**: failover 필요 → 합의 ([[consensus-problem]])
- 대부분 관계형 DB, 가장 흔함

## 2. 멀티 리더 (multi-leader)

**여러 노드가 쓰기를 받음** (다중 데이터센터):

```
쓰기 → 리더A ⇄ 리더B ⇄ 리더C (서로 복제)
```

- **장점**: 각 지역에 리더 → 로컬 쓰기 빠름, 한 지역 장애에 강함
- **핵심 문제: 쓰기 충돌** - 두 리더가 같은 데이터를 동시에 다르게 수정
  ```
  리더A: x = 1
  리더B: x = 2   (동시에)
  → 어느 게 맞나? 충돌!
  ```

### 충돌 해결

- **LWW(Last Write Wins)**: 타임스탬프 큰 것 승리 → 하지만 시계 문제([[clocks]])로 데이터 유실 위험
- **버전 벡터**([[clocks]]): 동시 쓰기 감지 → 앱이 병합
- **CRDT**(충돌 없는 복제 자료형): 수학적으로 항상 병합 가능하게 설계 (카운터, 집합)
- **사용자에게 물어봄**: 충돌을 보존하고 사용자가 해결 (Git merge처럼)
- 충돌 해결이 멀티 리더의 근본 복잡도

## 3. 리더리스 (leaderless)

**리더 없이 여러 노드에 직접 쓰고 읽음** (Dynamo, Cassandra):

```
쓰기 → 여러 노드에 동시에 (W개 응답 기다림)
읽기 → 여러 노드에서 (R개 응답, 최신 선택)
```

### Quorum (정족수)

일관성을 **읽기/쓰기 노드 수**로 조절:

```
N = 전체 복제본 수
W = 쓰기 성공 필요 노드 수
R = 읽기 조회 노드 수

W + R > N  이면 → 읽기가 최신 쓰기를 반드시 포함 (겹침 보장)
```

- 예: N=3, W=2, R=2 → W+R=4>3 → 읽기 2개 중 하나는 최신 쓰기 봄
- **W 크게**: 쓰기 일관성↑, 가용성↓ (더 많은 노드 필요)
- **R 크게**: 읽기 일관성↑
- 조절 가능 → 일관성-가용성 트레이드오프를 앱이 선택 ([[cap-theorem]])

### 최신 값 판정

여러 노드가 다른 값 → 버전으로 최신 선택 ([[clocks]]의 벡터 시계). **read repair**(읽을 때 오래된 노드 갱신), **anti-entropy**(백그라운드 동기화).

## 세 방식 비교

| | 단일 리더 | 멀티 리더 | 리더리스 |
|---|---|---|---|
| 쓰기 | 한 곳 | 여러 곳 | 여러 노드 |
| 충돌 | 없음 | 있음 (해결 필요) | quorum + 버전 |
| 일관성 | 강함 | 약함 | 조절 (W/R) |
| 가용성 | 리더 의존 | 높음 | 높음 |
| 복잡도 | 낮음 | 높음 | 중간 |
| 예 | PostgreSQL | 다중 DC | Cassandra, Dynamo |

## 셀프 체크

> [!question]- 질문: 복제의 세 방식은 무엇이고, 쓰기를 받는 노드 수로 어떻게 구분되나?
> 단일 리더(한 노드만 쓰기 수신), 멀티 리더(여러 노드가 쓰기 수신), 리더리스(리더 없이 여러 노드에 직접 쓰기)다. 쓰기 지점이 늘어날수록 로컬 쓰기·가용성은 좋아지지만 충돌 처리와 일관성 관리가 어려워진다.

> [!question]- 질문: 단일 리더 복제는 왜 충돌이 없지만, 어떤 이상 현상과 장애 문제를 갖나?
> 쓰기가 한 곳(리더)을 거쳐 순서가 명확하므로 쓰기 충돌이 없다. 하지만 팔로워가 뒤처지는 복제 지연 때문에 read-your-writes 같은 이상이 생기고, 리더가 죽으면 failover가 필요해 합의([[consensus-problem]]) 문제가 따라온다.

> [!question]- 질문: 멀티 리더의 근본 복잡도는 무엇이고, 충돌 해결 방법에는 어떤 것들이 있나?
> 근본 문제는 두 리더가 같은 데이터를 동시에 다르게 수정하는 쓰기 충돌이다. 해결책으로 LWW(타임스탬프 큰 것 승리, 시계 문제로 유실 위험), 버전 벡터(동시 쓰기 감지 후 앱 병합), CRDT(항상 병합 가능하게 설계), 사용자에게 물어 해결(Git merge식)이 있다.

> [!question]- 질문: 리더리스 quorum에서 W+R>N이 보장하는 것은 무엇인가?
> 읽는 노드 집합(R)과 마지막 쓰기가 성공한 노드 집합(W)이 반드시 겹치도록 만든다. 따라서 읽기가 최신 쓰기를 가진 노드를 최소 하나 포함해, 버전 비교로 최신 값을 골라낼 수 있다. 예: N=3, W=2, R=2면 W+R=4>3이라 겹침이 보장된다.

> [!question]- 질문: 리더리스에서 오래된 노드를 최신으로 되돌리는 두 메커니즘은?
> read repair(읽을 때 오래된 값을 발견하면 그 노드를 갱신)와 anti-entropy(백그라운드에서 노드 간 차이를 지속 동기화)다.

## 연습문제

> [!example]- 문제: N=5 리더리스 시스템에서 "쓰기 손실 없이 항상 최신 읽기"를 원하되 노드 2개 장애를 견디게 W, R을 정하라.
> **풀이**
> 1. 최신 읽기 보장 조건: W+R>N=5 → W+R≥6.
> 2. 2개 장애 내성: 쓰기가 성공하려면 W개가 필요하니 W≤N-2=3, 읽기도 R≤3이어야 2노드가 죽어도 남은 3노드로 정족수를 채운다.
> 3. W=3, R=3 선택 → W+R=6>5 만족, 각각 5-3=2개 장애를 견딤.
> 4. 트레이드오프: W를 더 키우면(예: W=4) 쓰기 일관성은 오르지만 쓰기 가용성이 떨어진다(4노드 필요). W/R로 일관성-가용성을 앱이 직접 조절한다([[cap-theorem]]).

> [!example]- 문제: 세 데이터센터에 각 지역 사용자가 쓰는 글로벌 앱을 설계한다. 단일 리더 대신 멀티 리더를 고른 이유와 반드시 함께 설계해야 할 것을 판정하라.
> **풀이**
> 1. 단일 리더면 모든 쓰기가 한 DC의 리더로 가야 해 타 지역 사용자의 쓰기 지연이 크고, 그 DC 장애 시 전체 쓰기가 멈춘다.
> 2. 멀티 리더는 각 지역에 리더를 둬 로컬 쓰기가 빠르고 한 지역 장애에 강하다.
> 3. 대가로 쓰기 충돌이 필연 → 충돌 해결 전략을 반드시 함께 설계해야 한다: LWW는 유실 위험이 크므로 카운터·집합류는 CRDT, 그 외는 버전 벡터로 동시 쓰기를 감지해 병합하거나 사용자에게 해결을 위임한다.
> 4. 결론: 멀티 리더의 이득은 가용성·지연이고 비용은 충돌 해결 복잡도다.

> [!example]- 문제: 같은 키에 대해 두 노드가 각각 다른 값을 반환했다. LWW로 하나를 고르는 것이 왜 위험한지, 대안은 무엇인지 설명하라.
> **풀이**
> 1. LWW는 타임스탬프가 큰 쪽을 선택하는데, 노드 간 시계가 어긋나면([[clocks]]) 실제로 나중에 일어난 쓰기가 더 작은 타임스탬프를 가질 수 있어 그 쓰기가 조용히 유실된다.
> 2. 또한 두 쓰기가 진짜 동시(인과적으로 무관)였는지 LWW는 구분하지 못하고 무조건 하나를 버린다.
> 3. 대안: 버전 벡터로 두 값이 동시 쓰기인지 판정하고, 동시라면 유실 대신 앱이 병합(또는 CRDT로 자동 병합, 사용자에게 위임)한다.
