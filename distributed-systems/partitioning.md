# 파티셔닝 (Partitioning)

## 한 줄 요약

데이터를 여러 노드에 나누는 것 (database/[[partitioning-db]]의 분산 시스템 관점). consistent hashing으로 노드 변경 시 최소 재배치를 하고, 라우팅으로 "어느 노드에 있나"를 찾는다.

## 왜 필요한가

- database/[[partitioning-db]]의 분산 일반화
- consistent hashing이 왜 중요한가
- 요청을 어느 노드로 보내나 (라우팅)

## 복습: 왜 파티셔닝

database/[[partitioning-db]]: 데이터를 노드에 나눠 용량·쓰기 확장. 복제([[replication]])와 두 축. 이 노트는 **분산 시스템 메커니즘**에 집중 (consistent hashing, 라우팅, 리밸런싱).

## consistent hashing 상세

database/[[partitioning-db]]에서 소개한 것을 심화. 일반 해시(`% N`)의 문제:

```
node = hash(key) % N
N이 바뀌면 (노드 추가/삭제) → 거의 모든 키의 node가 바뀜 → 대량 재배치!
```

**consistent hashing**: 노드 변경 시 최소 재배치:

```
1. 해시 공간을 링(원)으로 (0 ~ 2^k)
2. 노드들을 링에 배치 (hash(노드id))
3. 키도 링에 배치 (hash(key))
4. 키는 링에서 시계방향 다음 노드에 저장
```

- 노드 추가 → 그 노드와 이전 노드 사이 키만 이동 (나머지 무관)
- 노드 삭제 → 그 노드의 키만 다음 노드로
- **재배치가 1/N로 제한** (전체 아님) → 확장 유연

### 가상 노드 (virtual nodes)

문제: 노드가 링에 균등 분포 안 되면 → 부하 불균형 (핫스팟 [[partitioning-db]])

해결: **각 물리 노드를 링에 여러 지점(가상 노드)으로**:
- 물리 노드 하나 = 링의 여러 위치
- 더 고른 분포, 노드 제거 시 부하가 여러 노드로 분산
- Cassandra, DynamoDB가 사용

## 라우팅: 어느 노드?

클라이언트가 "이 키가 어느 노드에 있나"를 찾는 법:

1. **클라이언트가 앎**: 클라이언트가 파티션 맵 보유 → 직접 라우팅 (Cassandra)
2. **라우팅 계층**: 프록시/코디네이터가 라우팅 (중간 계층)
3. **아무 노드에 요청 → 전달**: 받은 노드가 올바른 노드로 (gossip으로 맵 공유)

파티션 맵(어느 키 범위가 어느 노드)을 어떻게 최신으로 유지 → 조율 서비스([[coordination-services]])나 gossip.

## 리밸런싱

노드 추가/제거 시 재분배 (database/[[partitioning-db]]):

- **고정 파티션 수**: 파티션을 미리 많이(예: 1000개) 만들고 노드에 할당 → 노드 변경 시 파티션 통째 이동 (재해시 없음). Elasticsearch
- **consistent hashing**: 위 링 방식
- **온라인**: 서비스 중단 없이 (읽기/쓰기 계속하며 이동)
- 목표: 이동 최소 + 균등 유지

## 파티셔닝 + 복제

실무는 **둘을 조합** ([[replication]], database/[[partitioning-db]]):

```
각 파티션을 여러 노드에 복제:
  파티션1 → 노드A(리더), B, C(복제본)
  파티션2 → 노드B(리더), C, D
  ...
```

- 파티셔닝: 확장 (다른 데이터를 다른 노드)
- 복제: 가용성 (같은 파티션을 여러 노드)
- 각 파티션이 자기 리더·복제본 → 리더 분산 (부하 분산)

## 보조 인덱스 파티셔닝

파티션 키 외 다른 열로 검색 시 (database/[[index-usage]]):

- **로컬 인덱스**: 각 파티션이 자기 데이터의 인덱스 → 검색 시 모든 파티션 조회 (scatter-gather)
- **글로벌 인덱스**: 인덱스 자체를 파티셔닝 → 검색은 빠르지만 쓰기가 여러 파티션 갱신
- 트레이드오프 (읽기 vs 쓰기)

## 셀프 체크

> [!question]- 일반 해시(`hash(key) % N`)의 문제는 무엇이고 consistent hashing이 이를 어떻게 푸는가?
> `% N`은 노드 수 N이 바뀌면 거의 모든 키의 대상 노드가 바뀌어 대량 재배치가 일어난다. consistent hashing은 해시 공간을 링으로 만들고 노드·키를 링에 배치해 키를 시계방향 다음 노드에 저장한다. 노드 추가/삭제 시 인접 구간의 키만 이동하므로 재배치가 1/N로 제한된다.

> [!question]- 가상 노드(virtual node)는 어떤 문제를 해결하나?
> 물리 노드가 링에 균등 분포하지 않으면 부하 불균형(핫스팟)이 생긴다. 각 물리 노드를 링의 여러 지점(가상 노드)으로 배치하면 분포가 고르게 되고, 노드 제거 시 그 부하가 여러 노드로 분산된다. Cassandra, DynamoDB가 사용한다.

> [!question]- 클라이언트가 "이 키가 어느 노드에 있나"를 찾는 세 가지 라우팅 방식은?
> (1) 클라이언트가 파티션 맵을 직접 보유해 라우팅(Cassandra), (2) 프록시/코디네이터 같은 라우팅 계층이 중개, (3) 아무 노드에 요청하면 받은 노드가 올바른 노드로 전달(gossip으로 맵 공유). 파티션 맵의 최신 유지는 조율 서비스나 gossip으로 한다.

> [!question]- 파티셔닝과 복제는 각각 무엇을 위한 것이고 실무에선 어떻게 조합되나?
> 파티셔닝은 확장(다른 데이터를 다른 노드에), 복제는 가용성(같은 파티션을 여러 노드에)을 위한 것이다. 실무는 각 파티션을 여러 노드에 복제하고 파티션마다 리더를 분산 배치해, 확장과 가용성을 함께 얻고 리더 부하도 분산한다.

> [!question]- 보조 인덱스의 로컬 인덱스와 글로벌 인덱스의 트레이드오프는?
> 로컬 인덱스는 각 파티션이 자기 데이터의 인덱스만 가져 쓰기는 싸지만 검색 시 모든 파티션을 조회해야 한다(scatter-gather). 글로벌 인덱스는 인덱스 자체를 파티셔닝해 검색은 빠르지만 쓰기가 여러 파티션의 인덱스를 갱신해야 한다. 즉 읽기 vs 쓰기의 트레이드오프다.

## 연습문제

> [!example]- 문제: 노드 3대 클러스터에 1대를 추가한다. `% N` 방식과 consistent hashing 각각에서 재배치되는 키의 양을 판정하라.
> **풀이**
> 1. `% N` 방식: N이 3에서 4로 바뀌면 대부분 키의 `hash(key) % N` 결과가 달라져 거의 전체 키가 재배치된다.
> 2. consistent hashing: 새 노드를 링에 배치하면, 그 노드와 링 상 이전 노드 사이 구간의 키만 새 노드로 이동한다.
> 3. 나머지 노드의 키는 그대로다. 이동량은 대략 1/N(여기선 신규 포함 약 1/4) 수준으로 제한된다.
> 4. 판정: consistent hashing이 재배치를 최소화하므로 확장에 유리하다. 다만 균등 분포를 위해 가상 노드를 함께 쓴다.

> [!example]- 문제: 특정 파티션에만 트래픽이 몰려 핫스팟이 생겼다. consistent hashing 환경에서 원인과 완화책을 설계하라.
> **풀이**
> 1. 원인 후보 1: 물리 노드가 링에 불균등 배치되어 특정 노드가 넓은 구간을 담당 → 가상 노드 수를 늘려 분포를 고르게 한다.
> 2. 원인 후보 2: 특정 키(예: 인기 항목)에 접근이 집중 → 키 분포 자체의 편향이라 해싱만으론 해결 안 됨.
> 3. 후자엔 해당 키만 별도 처리(캐싱, 키에 접미사를 붙여 여러 파티션으로 분산 등)를 검토한다.
> 4. 리밸런싱은 온라인으로(읽기/쓰기 계속하며) 수행해 서비스 중단을 피한다.
> 5. 목표는 이동 최소화와 균등 유지 둘 다.

> [!example]- 문제: 파티션당 여러 복제본을 두는 설계에서 노드 A가 죽었다. 데이터 가용성과 라우팅이 어떻게 유지되는지 시나리오를 추론하라.
> **풀이**
> 1. 각 파티션이 리더 + 복제본(예: A리더, B, C)으로 구성되어 있다고 가정.
> 2. A가 담당하던 파티션은 복제본 B/C가 데이터를 갖고 있으므로 유실 없이 그중 하나가 새 리더가 된다([[replication]]).
> 3. 파티션 맵이 갱신되어 "이 파티션의 리더는 이제 B"가 조율 서비스/gossip으로 전파된다.
> 4. 클라이언트/라우팅 계층은 갱신된 맵으로 새 리더에 요청을 보낸다.
> 5. 결과: 파티셔닝(확장)과 복제(가용성)의 조합 덕에 한 노드 장애에도 서비스가 계속된다.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
>
> - `% N`의 대량 재배치 문제와 consistent hashing 링이 이를 1/N로 줄이는 원리, 그리고 가상 노드의 역할.
> - 라우팅 세 방식과 파티션 맵을 최신으로 유지하는 방법(조율 서비스/gossip).
> - 파티셔닝(확장)과 복제(가용성)를 조합하는 이유와, 보조 인덱스 로컬 vs 글로벌의 읽기/쓰기 트레이드오프.
