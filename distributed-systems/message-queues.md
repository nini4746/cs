# 메시지 큐 (Message Queues)

## 한 줄 요약

서비스 간 비동기 통신 - 생산자가 메시지를 큐/로그에 넣고 소비자가 처리한다. Kafka는 로그 기반(파티션·오프셋)으로 높은 처리량과 재생을 제공한다. 느슨한 결합과 버퍼링이 핵심 이점.

## 왜 필요한가

- 동기 RPC([[rpc]]) 대신 비동기가 왜 필요한가
- Kafka가 어떻게 동작하나
- at-least-once 처리와 멱등성

## 왜 메시지 큐

동기 통신([[rpc]])의 한계:
- **강한 결합**: 상대가 살아있어야 함 (죽으면 실패)
- **부하 스파이크**: 갑작스런 트래픽에 하류가 못 버팀
- **느린 처리**: 하류가 느리면 상류가 대기

**메시지 큐 = 비동기 + 버퍼**:
- 생산자가 메시지를 큐에 넣음 → 즉시 반환 (하류 안 기다림)
- 소비자가 자기 속도로 처리
- **느슨한 결합**: 생산자·소비자가 서로 몰라도 됨 (큐가 중개)
- **버퍼링**: 스파이크를 큐가 흡수 (하류 보호)

## 두 모델

### 큐 (queue) - 작업 분배

- 각 메시지를 **소비자 하나가** 처리 (작업 분배)
- 여러 소비자가 나눠 처리 → 수평 확장
- 예: 작업 큐 (이메일 발송, 이미지 처리). RabbitMQ, SQS

### 발행-구독 (pub/sub) - 브로드캐스트

- 각 메시지를 **모든 구독자가** 받음 (브로드캐스트)
- 여러 서비스가 같은 이벤트에 반응 (이벤트 기반 → software-design/[[event-driven-architecture]])
- 예: 주문 이벤트 → 결제·재고·알림 서비스 각각

## Kafka: 로그 기반

가장 중요한 메시지 시스템 - **로그(log) 구조**:

```
토픽 = 로그 (추가만 되는 순서 있는 메시지 열)
파티션 = 토픽을 나눔 (병렬성 [[partitioning]])
오프셋 = 각 메시지의 위치 번호
```

- **파티션**([[partitioning]]): 토픽을 여러 파티션으로 → 병렬 처리, 확장
- **오프셋**: 소비자가 "어디까지 읽었나"를 오프셋으로 추적 → 각자 독립적 진행
- **로그 보존**: 메시지를 읽어도 삭제 안 함 (보존 기간) → **재생(replay)** 가능 (새 소비자가 처음부터, 재처리)
- database/[[lsm-tree]], [[recovery]]의 로그 구조와 같은 발상 - "로그가 진실"

### 소비자 그룹

- **컨슈머 그룹**: 여러 소비자가 파티션을 나눠 처리 (그룹 내 각 파티션은 한 소비자)
- 그룹 간엔 브로드캐스트 (각 그룹이 전체 받음) → 큐 + pub/sub 통합
- 소비자 추가 → 파티션 재분배 (확장)

## 전달 보장

[[rpc]]의 의미론이 여기도:

- **at-most-once**: 처리 전 오프셋 커밋 → 실패 시 유실
- **at-least-once**(흔함): 처리 후 오프셋 커밋 → 실패 시 재처리 (중복 가능)
- **exactly-once**: at-least-once + **멱등 처리**([[idempotency]]) 또는 트랜잭션 (Kafka 트랜잭션)

**at-least-once + 멱등성이 실무 표준**: 재처리해도 안전하게 소비자를 멱등하게.

## 이점 정리

- **비동기**: 생산자가 안 기다림 → 응답성
- **버퍼링**: 스파이크 흡수, 하류 보호 (backpressure)
- **느슨한 결합**: 서비스 독립 배포·확장
- **재생**: 로그 재처리 (버그 수정 후 재처리, 새 소비자)
- **순서**: 파티션 내 순서 보장

## 대가

- **복잡도**: 인프라 추가 (Kafka 클러스터 운영)
- **eventual consistency**: 비동기라 즉시 반영 안 됨 ([[consistency-models]])
- **중복·순서**: at-least-once 중복, 파티션 간 순서 없음 → 멱등·설계로 대응
- **디버깅 어려움**: 비동기 흐름 추적 ([[observability-basics]])

## 셀프 체크

> [!question]- 동기 RPC 대비 메시지 큐의 핵심 이점 세 가지는?
> 비동기(생산자가 하류를 안 기다림 → 응답성), 버퍼링(스파이크를 큐가 흡수해 하류 보호), 느슨한 결합(생산자·소비자가 서로 몰라도 큐가 중개 → 독립 배포·확장). 여기에 로그 기반이면 재생(replay)도 추가된다.

> [!question]- 큐(queue) 모델과 pub/sub 모델의 차이는?
> 큐는 각 메시지를 소비자 하나가 처리(작업 분배 → 수평 확장). pub/sub는 각 메시지를 모든 구독자가 받음(브로드캐스트 → 여러 서비스가 같은 이벤트에 반응). Kafka의 컨슈머 그룹은 그룹 내에선 큐(파티션 분배), 그룹 간에는 pub/sub(각 그룹이 전체 수신)로 둘을 통합한다.

> [!question]- Kafka에서 파티션·오프셋·컨슈머 그룹은 각각 무슨 역할인가?
> 파티션은 토픽을 나눠 병렬 처리·확장을 제공한다. 오프셋은 각 메시지의 위치 번호로, 소비자가 "어디까지 읽었나"를 추적해 각자 독립적으로 진행한다. 컨슈머 그룹은 그룹 내 소비자들이 파티션을 나눠 처리(각 파티션은 그룹 내 한 소비자)하고, 소비자 추가 시 파티션이 재분배되어 확장된다.

> [!question]- at-least-once + 멱등성이 실무 표준인 이유는?
> at-most-once는 처리 전 오프셋 커밋이라 실패 시 유실되고, exactly-once는 트랜잭션 비용이 크다. at-least-once(처리 후 커밋)는 유실이 없지만 실패 시 재처리로 중복이 생긴다. 그래서 소비자를 멱등하게 만들어 재처리해도 안전하게 하면, 유실 없이 중복도 무해해져 실용적 절충이 된다.

> [!question]- 메시지를 읽어도 삭제하지 않는 로그 보존이 무엇을 가능하게 하나?
> 재생(replay). 메시지가 보존 기간 동안 로그에 남아 있으므로, 새 소비자가 처음부터 다시 읽거나 버그 수정 후 과거 메시지를 재처리할 수 있다. "로그가 진실"이라는 발상으로, LSM-tree나 recovery 로그와 같은 구조다.

## 연습문제

> [!example]- 문제: 결제 이벤트를 at-least-once로 소비하는데 장애 후 재처리로 중복 청구가 발생한다. 유실 없이 중복을 막도록 설계하라.
> **풀이**
> 1. exactly-once 트랜잭션 대신 at-least-once + 멱등 처리를 택한다(유실 방지 우선).
> 2. 각 결제 이벤트에 고유 ID(예: order_id + payment_id)를 부여한다.
> 3. 소비자가 처리 전에 "이 ID를 이미 처리했는가"를 저장소(DB 유니크 제약/처리 이력 테이블)로 확인한다.
> 4. 미처리면 결제 + 처리 이력 기록을 같은 DB 트랜잭션으로 커밋, 이미 처리됐으면 무시한다.
> 5. 그 후 오프셋을 커밋한다. 이러면 재처리가 와도 유니크 제약에서 걸려 중복 청구가 없다(멱등성).

> [!example]- 문제: 사용자별 주문 이벤트의 순서를 반드시 지켜야 한다. Kafka 파티션 키를 어떻게 잡아야 하는가를 판정하라.
> **풀이**
> 1. Kafka는 파티션 내 순서만 보장하고 파티션 간 순서는 없음을 전제한다.
> 2. 따라서 "순서를 지켜야 하는 단위"를 파티션 키로 삼아야 한다. 여기선 user_id.
> 3. user_id를 파티션 키로 하면 같은 사용자의 모든 이벤트가 같은 파티션에 들어가 순서가 보장된다.
> 4. 서로 다른 사용자는 다른 파티션에 분산되어 병렬성·확장은 유지된다.
> 5. 주의: 특정 사용자에 트래픽이 몰리면 그 파티션이 핫스팟이 되므로, 순서 요구 범위가 좁을수록(예: 주문 단위) 분산이 좋아진다.

> [!example]- 문제: 하류 서비스가 순간적으로 느려져 처리가 밀린다. 상류를 지키면서 메시지를 잃지 않는 방식을 설계하라.
> **풀이**
> 1. 동기 RPC라면 상류가 하류를 기다리다 함께 느려지고 실패가 전파된다.
> 2. 메시지 큐를 두면 생산자는 큐에 넣고 즉시 반환하므로 상류 응답성이 유지된다.
> 3. 스파이크는 큐가 버퍼로 흡수하고, 소비자는 자기 속도로 밀린 메시지를 처리(backpressure).
> 4. 로그 기반(Kafka)이면 보존 기간 동안 메시지가 남으므로, 하류가 회복될 때까지 유실이 없다.
> 5. 대가로 eventual consistency(즉시 반영 안 됨)를 받아들이고, 큐 적체(queue depth)를 메트릭으로 모니터링한다.
