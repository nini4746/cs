# 메시지 큐 (Message Queues)

## 한 줄 요약

서비스 간 비동기 통신 - 생산자가 메시지를 큐/로그에 넣고 소비자가 처리한다. Kafka는 로그 기반(파티션·오프셋)으로 높은 처리량과 재생을 제공한다. 느슨한 결합과 버퍼링이 핵심 이점.

## 왜 필요한가

- 동기 RPC([[rpc]]) 대신 비동기가 왜 필요한가
- Kafka가 어떻게 동작하나
- at-least-once 처리와 멱등성

## 왜 메시지 큐

동기 통신([[rpc]])의 한계:
- **강한 결합**: 상대가 살아있어야 함 (죽으면 실패)
- **부하 스파이크**: 갑작스런 트래픽에 하류가 못 버팀
- **느린 처리**: 하류가 느리면 상류가 대기

**메시지 큐 = 비동기 + 버퍼**:
- 생산자가 메시지를 큐에 넣음 → 즉시 반환 (하류 안 기다림)
- 소비자가 자기 속도로 처리
- **느슨한 결합**: 생산자·소비자가 서로 몰라도 됨 (큐가 중개)
- **버퍼링**: 스파이크를 큐가 흡수 (하류 보호)

## 두 모델

### 큐 (queue) - 작업 분배

- 각 메시지를 **소비자 하나가** 처리 (작업 분배)
- 여러 소비자가 나눠 처리 → 수평 확장
- 예: 작업 큐 (이메일 발송, 이미지 처리). RabbitMQ, SQS

### 발행-구독 (pub/sub) - 브로드캐스트

- 각 메시지를 **모든 구독자가** 받음 (브로드캐스트)
- 여러 서비스가 같은 이벤트에 반응 (이벤트 기반 → software-design/[[event-driven-architecture]])
- 예: 주문 이벤트 → 결제·재고·알림 서비스 각각

## Kafka: 로그 기반

가장 중요한 메시지 시스템 - **로그(log) 구조**:

```
토픽 = 로그 (추가만 되는 순서 있는 메시지 열)
파티션 = 토픽을 나눔 (병렬성 [[partitioning]])
오프셋 = 각 메시지의 위치 번호
```

- **파티션**([[partitioning]]): 토픽을 여러 파티션으로 → 병렬 처리, 확장
- **오프셋**: 소비자가 "어디까지 읽었나"를 오프셋으로 추적 → 각자 독립적 진행
- **로그 보존**: 메시지를 읽어도 삭제 안 함 (보존 기간) → **재생(replay)** 가능 (새 소비자가 처음부터, 재처리)
- database/[[lsm-tree]], [[recovery]]의 로그 구조와 같은 발상 - "로그가 진실"

### 소비자 그룹

- **컨슈머 그룹**: 여러 소비자가 파티션을 나눠 처리 (그룹 내 각 파티션은 한 소비자)
- 그룹 간엔 브로드캐스트 (각 그룹이 전체 받음) → 큐 + pub/sub 통합
- 소비자 추가 → 파티션 재분배 (확장)

## 전달 보장

[[rpc]]의 의미론이 여기도:

- **at-most-once**: 처리 전 오프셋 커밋 → 실패 시 유실
- **at-least-once**(흔함): 처리 후 오프셋 커밋 → 실패 시 재처리 (중복 가능)
- **exactly-once**: at-least-once + **멱등 처리**([[idempotency]]) 또는 트랜잭션 (Kafka 트랜잭션)

**at-least-once + 멱등성이 실무 표준**: 재처리해도 안전하게 소비자를 멱등하게.

## 이점 정리

- **비동기**: 생산자가 안 기다림 → 응답성
- **버퍼링**: 스파이크 흡수, 하류 보호 (backpressure)
- **느슨한 결합**: 서비스 독립 배포·확장
- **재생**: 로그 재처리 (버그 수정 후 재처리, 새 소비자)
- **순서**: 파티션 내 순서 보장

## 대가

- **복잡도**: 인프라 추가 (Kafka 클러스터 운영)
- **eventual consistency**: 비동기라 즉시 반영 안 됨 ([[consistency-models]])
- **중복·순서**: at-least-once 중복, 파티션 간 순서 없음 → 멱등·설계로 대응
- **디버깅 어려움**: 비동기 흐름 추적 ([[observability-basics]])
