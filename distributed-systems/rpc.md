# RPC (원격 프로시저 호출)

## 한 줄 요약

원격 서버의 함수를 로컬 함수처럼 호출하는 추상. 편리하지만 네트워크의 부분 실패를 감추지 못한다 - "응답이 없다"가 성공인지 실패인지 모호해 at-least/at-most/exactly-once 의미론이 갈린다.

## 왜 필요한가

- 노드끼리 어떻게 통신하나
- 왜 "정확히 한 번"이 어려운가
- 재시도가 왜 위험한가

## RPC = 원격 함수 호출

**원격 서버의 함수를 로컬처럼**:

```
result = server.getUser(123)   # 로컬 함수 같지만 실제론 네트워크 요청
```

- 내부적으로: 인자 직렬화 → 네트워크 전송 → 서버 실행 → 결과 직렬화 → 반환
- **추상화**: 네트워크 세부를 감춤 → 개발 편의
- 예: gRPC(web/[[graphql-and-alternatives]]), 옛 CORBA, JSON-RPC

## 문제: 네트워크는 로컬이 아니다

RPC의 위험한 환상 - **로컬 호출처럼 보이지만 아니다**:

로컬 함수 호출: 항상 성공하거나 예외 (명확)
원격 호출: **부분 실패**([[why-distributed]]) - "응답 없음"이 여러 의미:

```
요청 보냄 → ??? (응답 없음)
  - 요청이 서버에 안 갔나? (네트워크 손실)
  - 서버가 실행했는데 응답이 손실됐나?
  - 서버가 죽었나?
  - 그냥 느린가?
```

**구별 불가**. 이게 exactly-once를 어렵게 만듦.

## 전달 의미론 (delivery semantics)

응답 없을 때 재시도하나에 따라:

### at-most-once (최대 한 번)

- 재시도 안 함 → 중복 실행 없음
- 하지만 **손실 가능** (응답 못 받으면 실패로 간주, 실제론 됐을 수도)
- 안전하지만 유실 감수

### at-least-once (최소 한 번)

- 응답 없으면 **재시도** → 최소 한 번은 실행 보장
- 하지만 **중복 실행 가능** (첫 요청이 실제론 됐는데 응답만 손실 → 재시도로 두 번)
- 예: "이체"를 재시도하면 두 번 이체될 수 있음!

### exactly-once (정확히 한 번)

- 이상적이지만 **네트워크만으론 불가능**
- 방법: at-least-once + **멱등성**([[idempotency]]) → 여러 번 실행해도 한 번 효과
- 또는 중복 제거(dedup) - 요청 ID로 이미 처리했나 확인

## 멱등성이 핵심

**at-least-once + 멱등 연산 = 사실상 exactly-once**:

- 멱등: 여러 번 해도 결과 같음 (web/[[rest-design]]의 PUT)
- 재시도해도 안전 → at-least-once로 유실 없이 + 중복 걱정 없이
- 멱등 키(Idempotency-Key)로 비멱등 연산도 안전하게 → [[idempotency]]
- 분산 시스템 설계의 핵심 도구

## RPC의 다른 함정

- **직렬화 비용**: 인자·결과를 바이트로 (web/[[graphql-and-alternatives]]의 Protocol Buffers)
- **타입/버전 불일치**: 클라-서버 스키마 다르면 (계약 필요)
- **지연 은폐**: 로컬처럼 보여 지연을 간과 (분산 오류 2번 [[why-distributed]])
- **타임아웃 설정**: 얼마나 기다리나 (너무 짧으면 오탐, 길면 늦게 감지)

## REST vs RPC vs 메시지

통신 스타일:
- **RPC**(gRPC): 함수 호출 스타일, 강타입 계약 → web/[[graphql-and-alternatives]]
- **REST**: 자원 중심 → web/[[rest-design]]
- **메시지 큐**: 비동기, 느슨한 결합 → [[message-queues]]

동기 RPC는 강한 결합(상대가 살아있어야), 메시지는 느슨(큐가 버퍼).

## 셀프 체크

> [!question]- 질문: RPC의 "위험한 환상"은 무엇이고, 로컬 호출과 근본적으로 다른 점은?
> 원격 함수를 로컬 함수처럼 보이게 하는 추상이 위험한 환상이다. 로컬 호출은 항상 성공하거나 예외로 명확히 끝나지만, 원격 호출은 부분 실패가 있어 "응답 없음"이 요청 손실·서버 실행 후 응답 손실·서버 죽음·단순 지연 중 무엇인지 구별할 수 없다.

> [!question]- 질문: "응답 없음"을 서로 구별할 수 없다는 사실이 왜 exactly-once를 어렵게 만드나?
> 재시도할지 말지를 결정하려면 첫 요청이 실행됐는지 알아야 하는데, 응답이 없을 때 실행 여부를 알 수 없기 때문이다. 재시도하면 중복 실행 위험, 재시도 안 하면 유실 위험이라 네트워크만으로는 정확히 한 번을 보장할 수 없다.

> [!question]- 질문: at-most-once와 at-least-once의 차이와 각각의 위험은?
> at-most-once는 재시도하지 않아 중복 실행은 없지만 응답 손실 시 실제로 성공한 연산을 실패로 간주해 유실할 수 있다. at-least-once는 응답이 없으면 재시도해 최소 한 번 실행을 보장하지만, 첫 요청이 성공하고 응답만 손실된 경우 중복 실행된다(예: 이체가 두 번).

> [!question]- 질문: 네트워크만으로 불가능한 exactly-once를 실무에서 어떻게 근사하나?
> at-least-once + 멱등성([[idempotency]]) 조합이다. 멱등 연산은 여러 번 실행해도 효과가 한 번과 같으므로, 재시도로 유실을 없애면서 중복을 걱정하지 않아도 된다. 비멱등 연산은 요청 ID/멱등 키로 중복 제거(dedup)해 이미 처리했는지 확인한다.

## 연습문제

> [!example]- 문제: 결제 API 클라이언트가 요청을 보냈으나 타임아웃 내 응답을 못 받았다. 재시도해도 안전하게 만들려면 어떻게 설계하는지 판정하라.
> **풀이**
> 1. 상황 진단: 응답 없음은 요청 미도달, 처리 후 응답 손실, 서버 다운, 지연을 구별할 수 없다. 그냥 재시도하면 at-least-once라 이미 처리된 결제가 이중 청구될 수 있다.
> 2. 멱등 키 도입: 클라이언트가 요청마다 Idempotency-Key를 붙인다.
> 3. 서버 dedup: 서버는 그 키로 이미 처리한 요청인지 확인해, 처리했다면 재실행 없이 저장해 둔 결과를 반환한다.
> 4. 결과: at-least-once + 멱등성으로 사실상 exactly-once를 얻어, 재시도해도 결제는 한 번만 일어난다([[idempotency]]).

> [!example]- 문제: 동기 RPC(gRPC)와 메시지 큐 중 무엇을 쓸지 판정하는 기준을 시나리오로 설명하라.
> **풀이**
> 1. 동기 RPC: 함수 호출 스타일에 강타입 계약, 즉시 응답이 필요할 때 적합. 단, 상대가 살아있어야 하는 강한 결합이라 상대 장애가 곧바로 전파된다.
> 2. 메시지 큐: 큐가 버퍼 역할을 해 상대가 잠시 죽어도 메시지가 남는 느슨한 결합. 비동기 처리·부하 평탄화에 적합하지만 즉시 응답이 없다.
> 3. 판정 예: "사용자 프로필 조회처럼 즉답이 필요한 읽기"는 RPC, "주문 후 이메일 발송·정산처럼 지연 허용되고 상대 가용성에 묶이면 안 되는 작업"은 메시지 큐.
> 4. 보강: RPC 경로에는 타임아웃과 circuit breaker로 실패 전파를 막고, 재시도는 멱등성을 전제로 한다.
