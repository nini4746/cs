# 멱등성 (Idempotency)

## 한 줄 요약

같은 연산을 여러 번 실행해도 결과가 한 번과 같은 성질. 분산 시스템의 재시도 안전성을 보장하는 핵심 도구 - 네트워크 불확실성 속에서 exactly-once를 근사한다.

## 왜 필요한가

- 분산 시스템 전반에서 계속 나온 개념의 종합
- 재시도가 왜 위험하고 멱등성이 어떻게 구하나
- 멱등 키 구현

## 멱등성 정의

**f(f(x)) = f(x)** - 여러 번 적용해도 한 번과 같음:

```
멱등: "잔액을 100으로 설정" → 여러 번 해도 100 (같음)
비멱등: "잔액에서 100 차감" → 여러 번 하면 계속 줄어듦 (다름)
```

web/[[rest-design]]의 PUT(멱등) vs POST(비멱등)의 그 개념.

## 왜 분산에서 필수

부분 실패([[why-distributed]])와 재시도의 만남:

```
요청 보냄 → 응답 없음 ([[rpc]]의 모호성)
  → 재시도? 
    - 실제론 성공했는데 응답만 손실 → 재시도로 중복 실행!
    - 비멱등이면 (이체) → 두 번 이체 (재앙)
    - 멱등이면 → 두 번 해도 한 번 효과 (안전)
```

**멱등성 = 재시도 안전성**. 네트워크가 불확실하니 재시도는 필수인데, 재시도가 안전하려면 멱등해야.

## 지금까지 나온 곳 (종합)

멱등성이 분산 시스템 전반의 핵심:

- **RPC**([[rpc]]): at-least-once + 멱등 = exactly-once 근사
- **분산 트랜잭션**([[distributed-transactions]]): saga 단계·보상이 멱등해야
- **메시지 큐**([[message-queues]]): at-least-once 소비를 멱등 처리로
- **복구**(database/[[recovery]]): redo/undo가 멱등 (여러 번 해도 안전)
- **REST**(web/[[rest-design]]): PUT/DELETE 멱등
- 이 노트가 그 종합.

## 자연히 멱등인 연산

- **설정(set)**: `x = 5` (몇 번 해도 5)
- **삭제(delete)**: 이미 없으면 그대로
- **삽입 with 고유 키**: 중복 키면 무시/에러
- **max/min 갱신**: `x = max(x, 5)`

## 비멱등을 멱등으로: 멱등 키

**증분·생성 같은 비멱등 연산**을 안전하게 (web/[[rest-design]]):

```
1. 클라이언트가 요청마다 고유 ID(Idempotency-Key) 부여
2. 서버가 그 ID를 기록 (처리했나)
3. 재시도(같은 ID) → 이미 처리했으면 → 저장된 결과 반환 (재실행 안 함)
```

- 결제 API의 표준 (Stripe 등): 같은 멱등 키면 중복 결제 안 됨
- 서버가 ID → 결과 매핑 저장 (data-structures/[[hash-tables]], 일정 기간)
- **중복 제거(dedup)**: 이미 본 요청 ID면 스킵

## 멱등성 설계 패턴

- **자연 멱등 선호**: 가능하면 set/upsert로 설계 (increment 대신)
- **멱등 키**: 비멱등 연산엔 클라이언트 생성 ID
- **버전/조건부**: "버전 5일 때만 갱신" (낙관적, 재시도 시 이미 갱신됐으면 무시) → database/[[mvcc]]
- **고유 제약**: DB 유니크 키로 중복 삽입 방지 (database/[[relational-model]])

## 정확히 한 번 (exactly-once) 재고

"정확히 한 번"은 불가능([[rpc]])하지만 **효과적 exactly-once**:

```
at-least-once 전달 (재시도로 유실 방지)
  + 멱등 처리 (중복 효과 제거)
= 효과적으로 정확히 한 번
```

- 전달은 여러 번일 수 있지만 **효과는 한 번** → 실용적 exactly-once
- 분산 시스템의 현실적 해법 ([[message-queues]], [[distributed-transactions]])

## 셀프 체크

> [!question]- 멱등성의 정의와 멱등/비멱등 연산 예는?
> f(f(x)) = f(x), 즉 여러 번 적용해도 한 번과 결과가 같은 성질이다. "잔액을 100으로 설정"은 멱등(여러 번 해도 100), "잔액에서 100 차감"은 비멱등(할 때마다 줄어듦). REST의 PUT(멱등) vs POST(비멱등)와 같은 개념.

> [!question]- 왜 분산 시스템에서 멱등성이 필수인가?
> RPC 응답이 없으면 요청이 성공했는지 알 수 없어 재시도가 필요한데(응답의 모호성), 실제로는 성공했던 요청을 재시도하면 중복 실행된다. 비멱등이면(이체) 두 번 실행되는 재앙이 나고, 멱등이면 두 번 해도 한 번 효과라 안전하다. 즉 **멱등성 = 재시도 안전성**.

> [!question]- 멱등 키(Idempotency-Key)는 어떻게 동작하나?
> (1) 클라이언트가 요청마다 고유 ID를 부여, (2) 서버가 그 ID의 처리 여부를 기록, (3) 같은 ID로 재시도가 오면 재실행하지 않고 저장된 결과를 반환한다. Stripe 등 결제 API의 표준으로 중복 결제를 막는다.

> [!question]- 효과적 exactly-once는 무엇의 조합인가?
> **at-least-once 전달(재시도로 유실 방지) + 멱등 처리(중복 효과 제거)**다. 전달은 여러 번일 수 있지만 효과는 한 번이 되어 실용적으로 정확히 한 번을 달성한다.

> [!question]- 자연히 멱등인 연산의 예는?
> set(`x = 5`), delete(이미 없으면 그대로), 고유 키를 쓴 insert(중복이면 무시/에러), max/min 갱신(`x = max(x, 5)`) 등이다. 설계 시 가능하면 increment 대신 이런 자연 멱등 연산으로 만든다.

## 연습문제

> [!example]- 문제: "잔액에서 100 차감" 같은 비멱등 연산을 재시도해도 안전하게 만들어라
> **풀이**
> 1. 문제: 차감은 재실행하면 계속 줄어드는 비멱등 연산이라 재시도가 위험하다.
> 2. 방법 A(멱등 키): 클라이언트가 요청마다 고유 ID 부여 → 서버가 ID→결과를 저장 → 같은 ID 재시도는 저장된 결과 반환, 재실행 안 함.
> 3. 방법 B(조건부/버전): "버전 5일 때만 차감"처럼 낙관적 갱신 → 재시도 시 이미 갱신됐으면(버전 불일치) 무시.
> 4. 둘 다 "전달 여러 번, 효과 한 번"을 만들어 재시도 안전성을 확보.

> [!example]- 문제: 결제 API에서 네트워크 재시도로 인한 이중 결제를 막아라
> **풀이**
> 1. 클라이언트가 결제 요청에 고유 Idempotency-Key를 붙여 전송.
> 2. 서버는 처리 전 그 키를 dedup 저장소(예: hash table)에서 조회.
> 3. 처음 보는 키면 결제 실행 후 (키 → 결과)를 기록, 이미 본 키면 실행을 스킵하고 저장된 결과를 그대로 반환.
> 4. 결과: 같은 키의 재시도가 몇 번 와도 실제 결제는 한 번. 키는 일정 기간 보관 후 정리.

> [!example]- 문제: at-least-once로 메시지가 전달되는 큐를 소비하는데 같은 메시지가 중복 도착한다. 안전하게 처리하라
> **풀이**
> 1. at-least-once는 유실을 막지만 중복을 허용하므로 소비 측이 중복을 감당해야 한다.
> 2. 각 메시지에 고유 ID를 부여(프로듀서 또는 메시지 자체)하고 소비자가 처리한 ID를 기록.
> 3. 이미 처리한 ID면 스킵(dedup), 처음이면 처리 후 ID 기록 → 멱등 소비.
> 4. 결과: at-least-once 전달 + 멱등 처리 = 효과적 exactly-once.

## 파인만

> [!note]- 백지에 이 노트 핵심을 남에게 설명하듯 써보라. 막히면 그 부분만 다시.
> **점검 포인트**: 이해했다면 답할 수 있어야 하는 핵심 3가지.
>
> 1. f(f(x))=f(x)의 정의와, 재시도 안전성이 왜 부분 실패·응답 모호성에서 필수인지 연결해 설명할 수 있나.
> 2. 비멱등 연산을 멱등 키/조건부 갱신/고유 제약으로 멱등화하는 방법을 예로 들 수 있나.
> 3. at-least-once + 멱등 = 효과적 exactly-once 공식과, 이것이 RPC·큐·saga·복구 전반에 어떻게 재등장하는지 말할 수 있나.
