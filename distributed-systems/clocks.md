# 시계와 순서 (Clocks and Ordering)

## 한 줄 요약

분산 노드의 물리 시계는 못 믿는다 (동기화 불완전). 그래서 논리 시계(Lamport, 벡터)로 "무엇이 무엇보다 먼저"라는 인과 순서를 추적한다. happened-before 관계가 핵심.

## 왜 필요한가

- 왜 노드 시계를 못 믿나
- "누가 먼저 했나"를 어떻게 정하나
- 인과 관계 추적의 기반

## 문제: 물리 시계는 못 믿는다

각 노드가 자기 시계를 가짐 → **동기화 안 됨**:

- **시계 드리프트(drift)**: 시계마다 미세하게 다른 속도 → 시간이 지나며 벌어짐
- **NTP 동기화**: 네트워크로 시계 맞추지만 오차 (수 ms~수십 ms)
- **비단조(non-monotonic)**: NTP 보정으로 시계가 **뒤로 갈 수도** → 타임스탬프로 순서 정하면 위험
- 결론: **타임스탬프로 이벤트 순서를 정하면 안 됨** (노드 간 비교 불가)

예: 노드 A가 "10:00:00.005"에 쓰고, 노드 B가 "10:00:00.003"에 썼다면 B가 먼저? → 시계가 안 맞으니 알 수 없음.

## happened-before 관계

Lamport의 핵심 개념 - **인과 순서**:

`a → b` (a가 b보다 먼저 일어남) 조건:
1. 같은 노드에서 a가 b보다 먼저
2. a가 메시지 전송, b가 그 수신
3. 전이성: a→b, b→c면 a→c

**동시(concurrent)**: a→b도 b→a도 아니면 → 인과 관계 없음 (순서 못 정함, 정할 필요 없음)

물리 시간이 아니라 **인과성**으로 순서를 정의. os/[[threads-and-races]]의 happens-before와 같은 개념.

## Lamport 시계

논리 시계 - 각 노드가 카운터:

```
1. 각 이벤트마다 카운터++
2. 메시지 보낼 때 카운터 첨부
3. 받을 때: max(내 카운터, 받은 값) + 1
```

성질:
- **a → b이면 L(a) < L(b)** (인과 순서면 시계 순서)
- 하지만 **역은 성립 안 함**: L(a) < L(b)여도 인과 관계 없을 수 있음 (동시일 수도)
- 전체 순서(total order)를 매길 순 있지만 (동점은 노드 ID로) 인과와 동시를 구별 못 함

## 벡터 시계 (vector clock)

Lamport의 한계 극복 - **인과성을 정확히 포착**:

```
각 노드가 [모든 노드의 카운터] 벡터 유지
자기 이벤트: 자기 위치++
메시지 받을 때: 각 위치를 max
```

- **인과 관계를 정확히 판정**: V(a) < V(b) ⟺ a → b
- **동시성 감지**: 두 벡터가 서로 크지도 작지도 않으면 → 동시 (충돌!)
- 대가: 벡터 크기 = 노드 수 (많으면 부담)

용도: 충돌 감지 (멀티 리더 복제 [[replication]]에서 동시 쓰기 감지 → 해결 필요).

## 순서가 왜 중요한가

- **일관성**([[consistency-models]]): 이벤트 순서가 일관성 정의의 기반
- **복제 충돌**([[replication]]): 동시 쓰기 감지 (벡터 시계)
- **인과 일관성**: 인과 관계 있는 것만 순서 보장 (동시는 자유)
- **분산 스냅샷**, 디버깅

## 실무: 하이브리드

- **TrueTime**(Google Spanner): GPS+원자시계로 시계 불확실성을 **범위로** 표현 → 그 범위만큼 기다려 순서 보장. 물리 시계를 정직하게 (오차 인정)
- **HLC**(Hybrid Logical Clock): 물리 시계 + 논리 시계 결합 → 대략 물리 시간 + 인과성
- 순수 논리 시계는 물리 시간 의미 없음, 순수 물리는 못 믿음 → 하이브리드

## 셀프 체크

> [!question]- 왜 물리 시계 타임스탬프로 이벤트 순서를 정하면 안 되나?
> 노드마다 시계가 있고 동기화가 불완전하기 때문이다. 시계 드리프트로 속도가 미세하게 다르고, NTP 동기화도 수~수십 ms 오차가 있으며, NTP 보정으로 시계가 뒤로 갈 수도(비단조) 있다. 그래서 노드 간 타임스탬프 비교로 순서를 정하면 위험하다.

> [!question]- happened-before(a → b) 관계의 성립 조건 세 가지는? concurrent는 무엇인가?
> (1) 같은 노드에서 a가 b보다 먼저, (2) a가 메시지 전송이고 b가 그 수신, (3) 전이성(a→b, b→c면 a→c). a→b도 b→a도 아니면 두 이벤트는 concurrent(동시) = 인과 관계가 없어 순서를 정할 필요도 없다.

> [!question]- Lamport 시계의 규칙과 한계는?
> 규칙: 각 이벤트마다 카운터++, 메시지 보낼 때 카운터 첨부, 받을 때 max(내 값, 받은 값)+1. 성질은 a→b이면 L(a)<L(b). 한계는 역이 성립 안 함 - L(a)<L(b)여도 인과 관계가 없을(동시) 수 있어, 인과와 동시를 구별하지 못한다.

> [!question]- 벡터 시계가 Lamport 시계보다 나은 점과 그 대가는?
> 각 노드가 모든 노드의 카운터 벡터를 유지해, V(a)<V(b) ⟺ a→b로 인과 관계를 정확히 판정하고, 두 벡터가 서로 크지도 작지도 않으면 동시(충돌)임을 감지한다. 대가는 벡터 크기가 노드 수에 비례해 노드가 많으면 부담이 크다는 점이다.

## 연습문제

> [!example]- 문제: 멀티 리더 복제에서 두 리더가 같은 키를 동시에 썼는지 판정하라
> **풀이**
> 1. 각 쓰기에 벡터 시계를 붙인다(각 노드 위치 카운터, 수신 시 각 위치 max).
> 2. 두 쓰기 A, B의 벡터를 비교: V(A)<V(B)이면 A→B(A가 먼저, B가 덮어씀), V(B)<V(A)이면 반대.
> 3. 서로 크지도 작지도 않으면 concurrent → 동시 쓰기 충돌로 판정, 해결(병합/최신 선택 등)이 필요하다.
> 4. Lamport 시계로는 전체 순서만 매겨져 이 동시성을 감지하지 못한다.

> [!example]- 문제: 노드 A가 "10:00:00.005", 노드 B가 "10:00:00.003"에 썼다. B가 먼저인가? 어떻게 순서를 정해야 하나
> **풀이**
> 1. 판정: 알 수 없음. 두 노드의 물리 시계가 안 맞으므로 타임스탬프 대소로 순서를 결론지으면 안 된다.
> 2. 올바른 접근: 두 쓰기 사이에 메시지 전달 같은 인과 연결이 있는지 본다(happened-before). 인과가 있으면 논리/벡터 시계로 순서가 정해진다.
> 3. 인과가 없으면 concurrent → 순서를 강제하지 말고 충돌로 다룬다.
> 4. 물리 시간이 꼭 필요하면 TrueTime(불확실성을 범위로)이나 HLC(물리+논리 결합)를 쓴다.
