# 정규식 엔진: DFA vs 백트래킹 (Regex Engines)

## 한 줄 요약

정규식 엔진은 두 방식으로 갈린다. DFA/NFA 시뮬레이션(RE2, grep)은 선형 시간 보장, 백트래킹(PCRE, Python, JS)은 역참조 등 강력한 기능을 주지만 최악 지수 시간(ReDoS)이 된다. 이론이 성능을 좌우하는 실전 사례.

## 왜 필요한가

- 왜 어떤 정규식이 서버를 멈추게 하나 (ReDoS)
- grep은 빠른데 Python re는 왜 느릴 수 있나
- 이론([[regex-and-finite-automata]])이 실전 성능에 직결

## 두 계열

### 1. 오토마타 기반 (DFA/NFA 시뮬레이션)

정규식을 오토마타로 컴파일해 실행 ([[regex-and-finite-automata]]):

- 입력을 한 번 훑으며 상태 전이 → **입력 길이에 선형 O(n)** 보장
- 백트래킹 없음 (여러 가능성을 상태 집합으로 동시 추적)
- **RE2**(Google), grep, awk, Go `regexp`, Rust `regex`
- 대가: 역참조 같은 정규 언어 밖 기능 **불가**

### 2. 백트래킹 기반

재귀적으로 매칭 시도, 실패하면 되돌아가 다른 경로:

- 역참조(`\1`), lookahead/lookbehind 등 **강력한 기능** 지원
- **PCRE, Python `re`, JavaScript, Java, Perl**
- 대가: 최악 **지수 시간** → ReDoS

## ReDoS: 지수 폭발 - 실측

백트래킹 엔진에서 특정 패턴 + 입력이 지수 시간. `(a+)+$`에 `aaa...X`:

```
n=15: 0.0011s
n=20: 0.0332s
n=25: 1.0813s
n=28: 9.2724s     ← 3글자 늘 때마다 ~9배
```

n이 3 늘 때마다 시간이 ~9배 → **지수적** (글자당 ~2배). 28글자에 9초, n=40이면 ~10시간, n=60이면 수천 년.

### 왜 폭발하나

`(a+)+`는 `a`들을 그룹으로 나누는 방법이 지수적으로 많음:
- `aaaa`를 `(aaaa)`, `(a)(aaa)`, `(aa)(aa)`... 여러 방식으로 나눔
- 끝의 `X`가 매칭 실패 → 엔진이 **모든 분할 방식을 되돌아가며 시도** → 2ⁿ
- 최종적으로 전부 실패하지만 그걸 확인하는 데 지수 시간

이론적으로 이 정규식 자체는 정규 언어(오토마타로 O(n) 매칭 가능)인데, **백트래킹 구현**이 지수로 만듦. RE2로 같은 패턴은 즉시.

## ReDoS 공격

- 사용자 입력을 정규식으로 검증하는 서버에서, 공격자가 폭발 유발 입력을 보냄
- 작은 입력(수십 바이트)으로 CPU 100% → 서비스 마비 (DoS)
- 취약 패턴: 중첩 수량자 `(a+)+`, `(a*)*`, 겹치는 대안 `(a|a)*`
- 실제 사고: Cloudflare 2019 장애(정규식), Stack Overflow 2016 다운

## 방어

1. **오토마타 엔진 사용**: RE2 등 선형 보장 엔진. 사용자 입력 검증엔 특히
2. **타임아웃**: 정규식 실행에 시간 제한
3. **패턴 검토**: 중첩 수량자 회피, 정규식 정적 분석 도구
4. **입력 제한**: 길이 제한으로 폭발 규모 제어
5. **역참조 안 쓰면** 오토마타 엔진으로 충분 (대부분 검증은 역참조 불필요)

## 트레이드오프 정리

| | 오토마타 (RE2) | 백트래킹 (PCRE) |
|---|---|---|
| 시간 보장 | O(n) 선형 | 최악 지수 |
| 역참조 | ✗ | ✓ |
| lookaround | 제한적 | ✓ |
| ReDoS | 불가능 | 취약 |
| 용도 | 대량/신뢰 못 할 입력 | 기능 필요 시 |

**교훈**: "정규식"이라 다 같지 않음. 역참조가 필요 없으면 오토마타 엔진이 안전하고 빠름. 이론(정규 언어 vs 그 너머)이 실전 성능·보안을 가름.
