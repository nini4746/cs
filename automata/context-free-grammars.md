# 문맥 자유 문법 (Context-Free Grammars)

## 한 줄 요약

정규 언어보다 강력한 문법 클래스. 생성 규칙으로 중첩 구조(괄호, 산술식)를 표현한다. 프로그래밍 언어 구문의 표준이며, 파스 트리로 구조를 드러낸다.

## 왜 필요한가

- 정규식이 못 하는 것(중첩, 균형) → CFG
- 프로그래밍 언어 문법의 기반 → compilers/[[parsing]]
- 파서가 실제로 인식하는 것

## CFG 정의

4요소 (V, Σ, R, S):
- **V**: 변수(비단말) 집합 - 문법 구조를 나타냄
- **Σ**: 단말(terminal) - 실제 토큰/문자
- **R**: 생성 규칙 A → (변수와 단말의 열)
- **S**: 시작 변수

규칙을 반복 적용해 변수를 확장 → 최종적으로 단말만 남으면 그게 생성된 문자열.

```
산술식 문법:
  E → E + E | E * E | ( E ) | num
E에서 시작해 규칙으로 확장 → "num + num * num" 등 생성
```

## 정규를 넘어서: 중첩

CFG가 정규 언어([[dfa-nfa]])보다 강한 핵심 = **재귀 규칙으로 중첩 표현**:

```
균형 괄호: S → ( S ) | S S | ε
```

- `S → (S)`가 재귀 → 중첩 깊이 무한 → aⁿbⁿ, 균형 괄호 표현 가능
- [[pumping-lemma]]에서 정규가 **아니라고** 증명한 것들을 CFG는 표현
- 이유: 스택 같은 무한 메모리(중첩 추적) → 유한 오토마타를 넘음 → [[pushdown-automata]]

## 파스 트리 (parse tree)

문자열이 문법으로 **어떻게 생성됐나**를 트리로:

```
"num + num * num":
        E
      / | \
     E  +  E
     |    /|\
    num  E * E
         |   |
        num num
```

- 잎 = 단말(실제 토큰), 내부 = 변수
- **구조를 드러냄**: 이 트리가 "곱셈이 먼저"라는 결합 구조를 보여줌
- 컴파일러가 이 트리(→ AST)로 의미를 해석 → compilers/[[ast-and-interpretation]]

## 모호성 (ambiguity)

한 문자열이 **여러 파스 트리**를 가지면 **모호(ambiguous)**:

```
"num + num * num"을 E → E+E | E*E로:
  (num+num)*num  vs  num+(num*num)  ← 두 트리!
```

- 모호하면 의미가 갈림 (연산 순서) → 컴파일러엔 치명적
- 해결: 문법 재작성으로 우선순위/결합 강제:

```
E → E + T | T       (덧셈)
T → T * F | F       (곱셈 - 더 깊음 = 먼저)
F → ( E ) | num
```

이러면 `num+num*num`이 유일한 트리 (곱셈 우선). 실용 문법은 이렇게 모호성 제거 → [[parsing]].

## 촘스키 표준형 (CNF)

모든 규칙을 A → BC 또는 A → a 꼴로 변환한 표준형:

- 파싱 알고리즘(CYK)이 요구
- 어떤 CFG든 CNF로 변환 가능
- 이론 증명·알고리즘에 편리

## CFG의 힘과 한계

**할 수 있음** (정규 넘어):
- 중첩 괄호, 산술식, 프로그래밍 언어 구문 대부분
- aⁿbⁿ, 회문

**못 함** (문맥 민감 필요):
- **aⁿbⁿcⁿ** (세 개 개수 맞추기) → [[cfl-pumping]]
- **변수 선언 후 사용 검사** (같은 이름 매칭) → 그래서 컴파일러가 파싱(CFG) 후 별도 의미 분석 → [[semantic-analysis]]
- 문맥 의존적 제약

이 한계 때문에 프로그래밍 언어는 CFG로 **구문**만 잡고, 타입/스코프 같은 **의미**는 파싱 후 따로 검사.

## 촘스키 위계에서의 위치

```
정규 (DFA) ⊂ 문맥자유 (CFG/PDA) ⊂ 문맥민감 ⊂ 재귀열거 (튜링)
```

CFG는 정규보다 강하고 튜링 머신보다 약함. → [[cfl-pumping]], [[turing-machines]]

## 셀프 체크

> [!question]- CFG의 4요소는 무엇인가?
> (V, Σ, R, S): V는 변수(비단말) 집합, Σ는 단말(실제 토큰/문자), R은 생성 규칙 A→(변수·단말의 열), S는 시작 변수다.

> [!question]- CFG가 정규 언어보다 강한 핵심 이유는?
> 재귀 규칙(예: S→(S))으로 중첩을 무한 깊이까지 표현할 수 있기 때문이다. 이는 스택 같은 무한 메모리로 중첩을 추적하는 것에 해당해, 유한 오토마타가 못 하던 aⁿbⁿ이나 균형 괄호를 표현한다.

> [!question]- 문법이 "모호(ambiguous)"하다는 것은? 왜 문제인가?
> 한 문자열이 두 개 이상의 파스 트리를 가지는 것이다. 트리가 여러 개면 의미(예: 연산 순서)가 갈려 컴파일러에 치명적이므로, 우선순위·결합을 강제하는 문법 재작성으로 제거한다.

> [!question]- 촘스키 표준형(CNF)의 규칙 꼴과 용도는?
> 모든 규칙이 A→BC 또는 A→a 꼴이다. CYK 같은 파싱 알고리즘이 이 형태를 요구하며, 어떤 CFG든 CNF로 변환 가능해 이론 증명·알고리즘에 편리하다.

## 연습문제

> [!example]- 문제: 알파벳 {a, b} 위에서 a와 b의 개수가 같은 문자열 전체의 CFG를 작성하라.
> **풀이**
> S → a S b S | b S a S | ε
> 직관: 어떤 균형 문자열이든 첫 글자가 a면 어딘가 짝이 되는 b가 있고, 그 사이(a...b 안쪽)와 뒤쪽 각각도 균형이다 → a S b S. 첫 글자가 b인 경우가 대칭으로 b S a S. 빈 문자열은 ε.
> 검증: "abba"는 S→aSbS→a(ε)b S→ab·(bSaS)... 등으로 유도 가능하며 a 2개, b 2개.

> [!example]- 문제: 모호한 산술 문법 `E → E + E | E * E | ( E ) | num`을 곱셈이 덧셈보다 먼저 묶이도록 재작성하라.
> **풀이**
> E → E + T | T       (덧셈 계층)
> T → T * F | F       (곱셈 계층 - 더 깊어서 먼저 묶임)
> F → ( E ) | num     (원자)
> 이제 `num + num * num`은 유일한 파스 트리를 가지며 `num + (num*num)`으로 해석된다. 좌재귀(E→E+T)로 좌결합도 강제된다.

## 파인만

> [!note]- 백지에 균형 괄호 문법 S→(S)|SS|ε를 적고, 왜 재귀 규칙이 정규 언어를 넘어서는지, 그리고 이것이 어떤 메모리 모델(스택)에 대응하는지 설명하라. 막히면 그 부분만 다시.
> **점검 포인트**: (1) 재귀 규칙과 중첩 깊이의 관계를 말할 수 있는가, (2) 모호성과 파스 트리 개수를 연결할 수 있는가, (3) CFG로도 못 하는 것(aⁿbⁿcⁿ, 변수 선언-사용)이 왜 CFG 밖인지 짚을 수 있는가.
